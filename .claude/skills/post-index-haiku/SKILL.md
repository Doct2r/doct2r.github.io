# POST_INDEX 엔트리 하이쿠 위임

## 언제 쓰는가

포스트 본문의 리서치·팩트체크·논지 구성이 **전부 끝난 뒤**에만 쓴다. 이 시점부터 남은 일은 "이미 확정된 사실을 정해진 포맷으로 다시 눌러 담기"뿐이라 창작적 판단이 필요 없다.

다음 판단은 계속 메인 모델(본인)이 한다 — 위임하지 않는다:
- 어떤 사실·날짜·인용을 포스트에 넣을지 고르는 리서치 자체
- 기존 POST_INDEX.md 엔트리와 핵심 논점이 겹치는지 판단(중복 글 여부 결정)
- About 페이지 재작성(20편 단위 갱신) — 여러 포스트를 종합해야 하는 판단이라 대상 아님

## 왜 하이쿠가 맞는가

POST_INDEX.md 엔트리는 창작이 아니라 압축이다 — 본문에 이미 쓴 날짜·인물·숫자·인용을 공백 없는 한국어 키워드 나열 포맷으로 다시 담는 기계적 작업이라, 오퍼스·소넷 수준의 판단력이 필요 없다. 이 저장소는 게시물마다 이 작업을 반복하므로 누적 절감 효과가 크다.

## 절차

1. 포스트 파일이 완성되고 본문 팩트체크가 끝난 뒤 이 스킬을 호출한다.
2. Agent 도구로 `model: "haiku"`를 지정한 서브에이전트를 백그라운드로 띄운다. 프롬프트에 반드시 포함할 것:
   - 완성된 포스트 전체 본문 (또는 파일 경로)
   - 삽입될 카테고리 섹션의 기존 엔트리 2~3개 (포맷·문체 참고용)
   - 지시: "본문에 실제로 쓰인 사실만 쓰고 새로 추론하지 말 것", "기존 엔트리와 동일한 공백 없는 키워드 나열 스타일 유지", "기존 유사 포스트와 각도가 어떻게 다른지 한 줄 언급"
   - 출력 형식: `| \`파일명.md\` | 제목 | 핵심 논점... |` 마크다운 표 행 한 줄만
3. 서브에이전트가 도는 동안 메인 모델은 다른 작업(빌드 확인 준비, 다음 포스트 착수 등)을 이어간다.
4. 반환된 행을 검증한 뒤 POST_INDEX.md의 해당 카테고리 섹션에 삽입한다 — 최근 관례상 섹션 최상단에 추가.
5. `hugo --quiet`로 빌드 확인 후 커밋·푸시한다.

## 하지 않는 것

- 포스트 본문 자체를 쓰거나 고치는 일 (본문은 항상 메인 모델이 작성)
- About 페이지 재작성
- 사실 검증·리서치
