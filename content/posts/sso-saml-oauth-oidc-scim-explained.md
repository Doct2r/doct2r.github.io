SSO·SAML·OAuth·OIDC·SCIM은 종종 "다 로그인 관련 약어"로 뭉뚱그려지지만, 실제로는 서로 대체재가 아니라 각기 다른 문제를 풀도록 설계된 별개의 개념·표준이다. 하나씩 순서대로 정리해본다.

---

## 1. SSO는 프로토콜이 아니라 목표다

SSO(Single Sign-On)는 특정 표준의 이름이 아니라 "한 번 로그인하면 여러 시스템에 접근할 수 있어야 한다"는 목표·아키텍처 패턴이다. 그 목표를 실제로 구현하는 데 SAML이나 OIDC 같은 구체적인 프로토콜이 쓰인다 — SSO 자체는 결과이지 수단이 아니라는 뜻이다.

이 목표의 뿌리는 꽤 오래됐다. MIT의 아테나 프로젝트(IBM·DEC와의 협업)에서 개발된 **케르베로스(Kerberos)**가 1988년 2월 USENIX에서 논문으로 처음 공개됐고("Kerberos: An Authentication Service for Open Network Systems", 슈타이너·노이먼·쉴러), 실제로 외부에 공개된 첫 배포판인 케르베로스 v4는 1989년 1월 24일에 나왔다(그 이전 v1～3은 내부 실험판이었다). 이미 1987년 1월 시점에 아테나 프로젝트 내 5,000명 사용자·650대 워크스테이션이 케르베로스 하나로 인증받고 있었다.

## 2. SAML — XML 기반의 오래된 기업용 SSO 표준

SAML(Security Assertion Markup Language)은 OASIS가 제정한 표준이다. SAML 1.0은 2002년 11월에, 이를 소폭 다듬은 1.1은 2003년 9월에 나왔지만, 지금 실제로 쓰이는 건 대부분 **SAML 2.0**(2005년 3월)이다. 이 2.0 버전은 그냥 개정판이 아니라 실제 합류의 산물이다 — 2001년 9월 썬을 비롯한 150여 개 조직이 결성한 리버티 얼라이언스(Liberty Alliance)가 자신들의 신원연합프레임워크(ID-FF)를 2003년 11월 OASIS에 통합시켰고, 학술기관 연합 프로젝트인 섀볼레스(Shibboleth)도 여기 기여했다. 2008년 1월 무렵에는 이미 정부·교육·기업 전반에서 SAML 2.0이 보편적으로 쓰이고 있었다.

핵심 모델은 신원제공자(Identity Provider, IdP)와 서비스제공자(Service Provider, SP) 사이에 서명된 XML "어서션(assertion)"을 브라우저 리다이렉트로 주고받는 방식이다. XML 기반이라 요즘 기준으로는 무겁게 느껴지지만, 성숙도가 높고 기업용 IdP들의 지원이 두터워서 지금도 레거시 엔터프라이즈 애플리케이션에서는 여전히 SAML만 지원하는 경우가 흔하다.

## 3. OAuth — 인가이지, 인증이 아니다

여기서부터가 가장 많이 오해되는 지점이다. **OAuth는 인가(authorization) 프레임워크이지, 인증(authentication) 프로토콜이 아니다.** OAuth 액세스 토큰은 "이 토큰을 가진 자가 특정 범위(scope)의 권한을 부여받았다"는 것만 증명할 뿐, 그 사용자가 누구인지에 대해서는 본질적으로 아무것도 말해주지 않는다.

**기원**은 트위터에서 시작됐다 — 2006년 11월 블레인 쿡과 크리스 메시나가 트위터의 위임 접근(제3자 앱이 사용자 비밀번호를 직접 물어보지 않고도 대신 작업할 수 있게 하는 것) 문제를 풀기 위해 논의를 시작했고, 2007년 4월 비공식 그룹이 결성돼 같은 해 7월 첫 초안이 나왔다. **OAuth Core 1.0**은 2007년 10월 3일 최종 초안, 같은 해 12월 4일 정식 공개됐고, 세션 고정 취약점을 고친 리비전 A가 2009년 6월 24일 나왔으며, 이게 **RFC 5849**(2010년 4월, 참고용 RFC)로 문서화됐다.

**OAuth 2.0**은 **RFC 6749**(2012년 10월)로 나왔는데, RFC 본문에 스스로 명시돼 있듯 1.0과는 구현 세부사항을 거의 공유하지 않는 사실상 새로운 설계다 — 1.0의 암호서명 방식 대신 TLS 위에서 베어러 토큰을 쓰는 방식(별도 RFC 6750)으로 바뀐 게 대표적 차이다. 이 전환이 순조롭기만 했던 건 아니다 — OAuth 1.0 공동 저자이자 2.0의 편집자였던 에란 해머는 2012년 6월 편집자 자리에서 물러났고, 그해 7월 블로그 글에서 OAuth 2.0을 "나쁜 프로토콜"이라 부르며 기업들의 확장 요구가 프로토콜을 망가뜨렸고 토큰이 1.0보다 본질적으로 덜 안전해졌다고 비판했다 — 즉 이 전환이 모두의 동의를 얻은 매끄러운 발전은 아니었다는 뜻이다.

여러 승인 방식(authorization code, implicit, client credentials, resource owner password credentials, refresh token) 중 암묵적 흐름(implicit flow)과 리소스 오너 비밀번호 자격증명 부여 방식은 이제 권장되지 않는다 — 2025년 1월 나온 **RFC 9700**(OAuth 2.0 보안 모범사례)이 이 두 방식을 공식적으로 폐기 대상으로 명시했다. 이런 흐름을 정리해 담을 예정인 OAuth 2.1은 암묵적 흐름 자체를 아예 제거하고 모든 인가 코드 흐름에 PKCE를 의무화하는 걸 목표로 하는데, 이 글을 쓰는 시점에도 아직 IETF 인터넷 초안 단계이지 정식 RFC로 확정된 건 아니다.

OIDC가 나오기 전에는 이 OAuth 토큰만으로 "로그인 구현"을 흉내 내는 경우가 많았는데(때로 "유사 인증"이라 불린다), 이건 애초에 OAuth가 풀도록 설계된 문제가 아니었다는 게 핵심이다.

## 4. OIDC — OAuth 위에 얹은 신원 계층

**OpenID Connect(OIDC)**는 스스로를 "OAuth 2.0 위의 단순한 신원 계층"이라고 정의한다 — OAuth를 대체하는 게 아니라, 그 위에 사용자 신원 정보를 담은 JWT인 "ID 토큰"을 추가로 얹은 것이다. OpenID 파운데이션이 2014년 2월 26일 최종 스펙을 공개했다.

이름이 비슷해서 혼동되지만, OIDC와 예전의 **OpenID**(1.0/2.0)는 이름과 지향점만 이어받았을 뿐 완전히 다른 배관 위에서 돌아간다. OpenID는 2005년 5월 브래드 피츠패트릭이 식스어파트(라이브저널)에서 만든 것으로("야디스Yadis"라는 이름으로 시작), OpenID 2.0은 2007년 12월 확정됐다. 이 원조 OpenID는 URL 기반 신원이라는 방식이었는데, 결국 대중화에 실패했다 — 여러 제공자 로고 버튼이 잔뜩 늘어서 사용자가 헷갈리는 이른바 "나스카(NASCAR) 문제"가 실제로 이름 붙여질 만큼 심각한 UX 문제였고, OpenID 파운데이션 회원사들조차 "OpenID"라는 브랜드를 최종 사용자에게 제대로 노출하지 않아서 결국 평범한 소셜로그인 버튼들에 밀려났다. URL 기반 로그인의 피싱 위험도 자주 언급되는 우려이긴 한데, 이건 업계에서 널리 반복되는 지적일 뿐 단일한 권위 있는 사건·논문으로 확인되는 건 아니다.

지금 우리가 "구글로 로그인"·"마이크로소프트로 로그인" 버튼을 누를 때 실제로 돌아가는 게 바로 이 OIDC다.

## 5. SCIM — 로그인이 아니라 "계정 자체"의 문제

여기까지와는 완전히 다른 문제를 푸는 게 **SCIM**(System for Cross-domain Identity Management)이다. SCIM은 로그인·인증·인가와는 무관하고, 조직 구성원이 입사·직무변경·퇴사할 때 여러 시스템에 걸쳐 계정을 자동으로 생성·수정·삭제하는 "프로비저닝(provisioning)" 문제를 다루는 REST/JSON API 표준이다.

SCIM 1.0은 2011년 10월 클라우드 아이덴티티 서밋에서(Okta·사이버아크·핑아이덴티티·세일포인트 등이 참여해) 상호운용성이 시연됐고, **SCIM 2.0**은 2015년 9월 완성돼 세 개의 RFC로 문서화됐다 — 개념·요구사항을 다루는 RFC 7642(참고용), 핵심 스키마를 다루는 RFC 7643, 실제 REST 프로토콜을 다루는 RFC 7644(둘 다 표준안 트랙)다.

실무적으로 왜 이게 중요하냐면 — 직원이 퇴사했을 때 SAML·OIDC로 "로그인을 막는 것"과, 그 직원의 계정 자체를 각 연동 시스템에서 실제로 비활성화·삭제하는 것은 다른 일이기 때문이다. 로그인 프로토콜만으로는 후자를 해결하지 못한다.

## 6. 실제 기업 IdP는 이걸 다 같이 쓴다

Okta·애저 AD(엔트라 ID)·핑아이덴티티 같은 기업용 IdP는 이 다섯 가지를 경쟁 관계가 아니라 역할 분담으로 함께 제공한다 — 어떤 다운스트림 앱은 SAML만 지원하고 어떤 앱은 OIDC를 지원하니 로그인 프로토콜로 둘 다 마련해두고, 여기에 계정 생명주기 관리를 위한 SCIM을 별도 채널로 얹는 식이다. 즉 인증은 SAML이나 OIDC로, 인가는 OAuth 스코프로, 프로비저닝은 SCIM으로 — 이 넷은 서로 대체하는 관계가 아니라 애초에 각자 다른 층위의 문제를 맡고 있다.

## 정리

SSO는 목표이지 프로토콜이 아니고, SAML은 XML 기반의 오래된 기업용 로그인 표준이며, OAuth는 인증이 아니라 인가를 위한 프레임워크이고(이 오해가 실제로 OIDC라는 별도 계층을 낳았다), OIDC는 그 OAuth 위에 신원 정보를 얹어 진짜 로그인을 구현하는 계층이고, SCIM은 로그인과 무관하게 계정 자체의 생성·삭제를 자동화하는 완전히 다른 문제다. 다섯 개의 약어가 겹치는 것처럼 보이는 이유는 이들이 서로 경쟁해서가 아니라, 인증·인가·신원 표현·계정 생명주기라는 네 개의 서로 다른 층위를 각자 맡고 있고, 실제 기업 환경에서는 이 넷이 늘 같이 쓰이기 때문이다.
