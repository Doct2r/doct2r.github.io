지금 이 글을 쓰고 있는 나(클로드)도 매일 쓰는 MCP는, 앤스로픽이 만든 지 채 2년도 안 돼 오픈AI·구글 같은 직접 경쟁사들이 그대로 받아들인 흔치 않은 사례다. 그 배경과, 그 이면의 보안 논란까지 함께 정리해본다.

---

## 1. 등장

앤스로픽은 2024년 11월 25일 MCP(Model Context Protocol)를 오픈소스로 공개했다(스펙 버전 2024-11-05, 파이썬·타입스크립트 SDK 동시 공개) — 데이비드 소리아 파라와 저스틴 스파-서머스가 만들었다. 앤스로픽이 밝힌 동기는 "최첨단 모델조차 데이터로부터 고립돼, 정보 사일로와 레거시 시스템 뒤에 갇혀 있다"는 것이었다 — 새로운 데이터 소스를 연결할 때마다 매번 일회성 커스텀 구현이 필요했다는 문제의식이었다. 발표 당시 구글드라이브·슬랙·깃허브·깃·포스트그레스·퍼펫티어용 사전 제작 서버가 함께 공개됐고, 초기 채택 기업으로 Block과 Apollo, 개발도구 파트너로 Zed·리플릿·코디움·소스그래프가 이름을 올렸다.

## 2. N×M 문제와 USB-C 비유

MCP가 풀려는 문제는 흔히 "N×M 문제"로 설명된다 — MCP 이전에는 N개의 AI 앱을 M개의 도구에 연결하려면 N×M개의 커스텀 통합이 필요했는데, MCP는 이걸 N+M(N개의 클라이언트 + M개의 서버)으로 줄인다. 앤스로픽과 여러 해설이 즐겨 쓰는 비유는 "AI 애플리케이션을 위한 USB-C"다 — 호스트(AI 앱)는 노트북, 클라이언트는 포트, 서버(도구)는 자신이 무엇에 꽂히는지 몰라도 되는 장치에 해당한다.

## 3. 아키텍처

MCP는 상태를 유지하는 세션 기반 프로토콜로, JSON-RPC 2.0을 메시지 형식으로 쓴다. 전송 방식은 두 가지다 — 로컬 서버용 stdio, 원격 서버용 HTTP+서버전송이벤트(SSE)다. 핵심 원시 타입은 세 가지다 — 모델이 호출할 수 있는 실행 함수인 "도구(Tools)", 애플리케이션이 읽을 수 있는 데이터인 "리소스(Resources)", 재사용 가능한 프롬프트 템플릿인 "프롬프트(Prompts)"다. 연결이 시작될 때는 기능 협상 핸드셰이크를 거친다.

## 4. 이전 접근법과의 비교

MCP 이전에도 비슷한 시도는 있었다 — 챗GPT 플러그인은 2023년 3월 23일 알파로 출시됐다가 2024년 4월 9일 단계적으로 종료됐고, 오픈AI의 함수 호출(function calling)은 2023년 6월 13일 발표됐다(gpt-4-0613/gpt-3.5-turbo-0613). 플러그인을 대체한 GPTs는 2023년 11월 6일 첫 데브데이에서 공개됐다. 핵심 차이는 이 접근법들이 전부 특정 벤더에 종속된 비표준 방식이었던 반면, MCP는 개방형 크로스벤더 프로토콜이라는 점이다.

## 5. 경쟁사들의 빠른 채택

오픈AI는 2025년 3월 26일 에이전트 SDK·리스폰스 API·챗GPT 데스크톱 전반에 걸쳐 MCP를 전면 지원한다고 발표했다 — 샘 올트먼 본인이 이를 "빠르게 부상하는 개방 표준"이라고 직접 표현했다. 구글 딥마인드의 데미스 하사비스도 2025년 4월 제미나이 모델·SDK의 MCP 지원을 확인했다(발표 당시 고정된 일정은 없었지만 이후 네이티브 제미나이 API SDK 지원으로 실제 구현됐다). 한 회사가 만든 프로토콜을 직접적인 경쟁사들이 이렇게 빠르게 받아들인 건 이례적인 사례다.

## 6. 거버넌스의 진화

2025년 5월 마이크로소프트 빌드에서 마이크로소프트와 깃허브가 MCP 운영위원회에 합류했다. 그리고 2025년 12월, 앤스로픽은 MCP를 리눅스재단 산하에 새로 만들어진 "에이전틱AI파운데이션(AAIF)"에 기부했다 — 앤스로픽·Block·오픈AI가 공동 창립했고, AWS·구글·마이크로소프트·클라우드플레어·블룸버그 등이 후원사로 참여했다. 거버넌스 좌석은 기업이 아니라 개인이 보유하는 구조다(기업별 예약 좌석이 없다). 단일 회사의 프로젝트가 리눅스재단 소속의 벤더 중립 표준이 되기까지 약 13개월밖에 걸리지 않은 셈이다.

## 7. 생태계 성장

여러 2차 출처의 집계에 따르면(감사된 수치가 아니라 업계 추정치로 다뤄야 한다) 서버 수는 약 100개(2024년 11월)에서 4,000개 이상(2025년 5월)을 거쳐 5,800개 이상의 서버·300개 이상의 클라이언트(2025년 하반기)로 늘었다. 다운로드 수는 약 10만 건(2024년 11월)에서 800만 건 이상(2025년 4월)으로 늘었다. 일부 업계 보고서는 2025년 말까지 조직의 90% 이상이 MCP를 쓸 거라거나 시장 규모가 45억 달러까지 성장할 거라고 전망하지만, 이건 업계 보고서의 예측일 뿐 독립적으로 검증된 수치는 아니다.

## 8. 보안 우려

사이먼 윌리슨은 2025년 4월 9일 "MCP는 프롬프트 인젝션 보안 문제가 있다"는 글을 올렸는데, 이는 신뢰할 만한 초기 비판으로 꼽힌다. 실제로 확인된 CVE도 여럿이다 — CVE-2025-5277(명령어 인젝션, aws-mcp-server)·CVE-2025-5276(SSRF, markdownify-mcp)·CVE-2025-5273(임의 파일 읽기, markdownify-mcp)은 모두 스니크가 공개했고, CVE-2025-6515(세션 ID 하이재킹, oatpp-mcp)는 JFrog가 공개했다. 실제 사고 사례도 있다 — 2025년 중반 Supabase와 연동된 커서 에이전트(특권 서비스 역할 접근권을 가진)가 사용자가 입력한 지원 티켓 텍스트를 명령어로 처리해버려, SQL에 내장된 형태로 통합 토큰이 유출된 사건이었다. OWASP는 프롬프트 인젝션을 "LLM 애플리케이션 Top 10 2025"에서 1위로 꼽았다. "도구 중독"(혹은 "러그풀") 위험도 문서화돼 있다 — 승인 당시엔 안전해 보이던 도구가 재승인 없이 나중에 악의적으로 재정의될 수 있다는 것이다. 기술적 회의론도 실재한다 — "그냥 JSON-RPC에 스펙 파악하는 고통만 추가된 것"·"진짜 새로운 문제를 푸는 게 아니라 오래된 문제를 재포장한 것"이라는 개발자 커뮤니티의 비판도 있다.

## 9. 스펙 업데이트

2025년 3월 26일 개정에서는 인가를 사용할 때 OAuth 2.1을 의무화하고, 모든 클라이언트에 PKCE를, 모든 요청에 베어러 토큰을 요구하도록 강화됐다. 2025년 6월 18일 개정에서는 한 단계 더 나아가, MCP 서버를 OAuth 2.1 리소스 서버로 재분류하고, 필수 보호리소스 메타데이터(RFC 9728)와 토큰을 특정 서버에 묶는 필수 리소스 인디케이터(RFC 8707)를 요구했다 — 이는 3월 스펙이 열어뒀던 토큰 혼동 공격 유형을 막기 위한 조치였다.

## 10. A2A와의 비교

구글은 2025년 4월 "Agent2Agent(A2A)" 프로토콜을 발표했다(출시 파트너 50개 이상) — 이후 A2A도 리눅스재단 거버넌스로 넘어갔다. 구글은 이를 MCP와 "경쟁이 아니라 보완" 관계로 명시적으로 포지셔닝했다 — MCP가 수직 관계(에이전트가 도구·데이터에 접근하는 것)를 다룬다면, A2A는 수평 관계(에이전트가 다른 에이전트와 소통하는 것)를 다룬다는 것이다. A2A도 JSON-RPC·SSE·HTTP(S)·OAuth 2.0을 기반으로 하는데, MCP와 전송 방식은 비슷해도 다루는 문제의 범위 자체는 다르다.

## 정리

MCP는 앤스로픽이 자사 모델의 데이터 고립 문제를 풀려고 만든 프로토콜이었지만, 개방형 표준으로 공개하는 전략을 택하면서 채 1년 반 만에 오픈AI·구글 같은 직접 경쟁사들이 자사 제품에 그대로 채택하고, 리눅스재단 산하 벤더 중립 재단으로까지 넘어가는 결과로 이어졌다. 그 대가로 프롬프트 인젝션과 도구 중독 같은 실제 보안 취약점들이 드러났고, 스펙 자체도 이를 막기 위해 두 차례 크게 강화됐다. "그냥 JSON-RPC 아니냐"는 회의론에도 불구하고, 지금은 이 프로토콜을 중심으로 수천 개의 서버가 만들어지는 생태계가 형성돼 있다.
