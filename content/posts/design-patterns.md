디자인 패턴은 "정답 코드"가 아니다. 특정 문제 상황에서 반복적으로 검증된 해결 구조의 이름표에 가깝다. 이름을 아는 것과 언제 써야 하는지 아는 것은 완전히 다른 문제이고, 실무에서 벌어지는 사고 대부분은 후자를 건너뛴 데서 나온다.

---

## 1. 무엇인가?

**디자인 패턴(Design Pattern)**은 소프트웨어 설계에서 반복적으로 등장하는 문제와, 그 문제를 푸는 재사용 가능한 해결 구조를 정형화한 것이다. 코드 조각이 아니라 **구조에 대한 템플릿**이다.

```
디자인 패턴이 아닌 것:
  - 복붙 가능한 코드 라이브러리
  - 특정 언어의 문법 기능
  - 알고리즘(정렬, 탐색 등)

디자인 패턴인 것:
  - "이런 상황에서는 이렇게 클래스/객체를 구성하면 문제가 풀린다"는 구조적 지식
```

1994년 Erich Gamma, Richard Helm, Ralph Johnson, John Vlissides 네 명이 **《Design Patterns: Elements of Reusable Object-Oriented Software》**를 출간하며 이 개념이 업계 표준 어휘가 됐다. 저자 넷을 합쳐 **GoF(Gang of Four)**라 부르고, 책에 실린 23개 패턴을 "GoF 패턴"이라 부른다.

원류는 소프트웨어가 아니다. 건축가 **Christopher Alexander**가 1977년 《A Pattern Language》에서 "좋은 건물에는 반복되는 공간 구성 패턴이 있다"고 주장한 것을 GoF가 소프트웨어 설계에 그대로 가져왔다. "패턴에 이름을 붙이면 설계자들이 같은 개념을 빠르게 주고받을 수 있다"는 것이 핵심 아이디어였다.

---

## 2. 근본적으로 왜 필요한가

디자인 패턴이 푸는 문제는 결국 하나다 — **변화에 대한 대응 비용을 낮추는 것**.

### 코드는 반드시 변한다

요구사항은 항상 바뀐다. 처음 짠 코드가 그대로 유지되는 경우는 거의 없다. 문제는 **어떻게 바뀌어도 코드 전체를 뜯어고치지 않게 만드느냐**다.

```
나쁜 설계: 요구사항 변경 1건 → 코드 10곳 수정
좋은 설계: 요구사항 변경 1건 → 코드 1곳 수정 (또는 확장만)
```

디자인 패턴 대부분은 이 목적을 위해 존재한다. 객체 간 **결합도(Coupling)를 낮추고 응집도(Cohesion)를 높이는** 구체적인 방법론들이다.

### 공통 어휘로서의 기능

"Observer 패턴을 쓰자"라는 한마디는 "이벤트가 발생하면 등록된 구독자들에게 자동으로 알림이 가도록, 발행자와 구독자를 느슨하게 연결하는 구조로 만들자"는 문장을 대체한다. 팀 내 커뮤니케이션 비용을 극적으로 줄인다. 이건 패턴의 코드적 가치보다 어휘적 가치에 가깝다.

### 검증된 해법의 재사용

디자인 패턴은 누군가 처음부터 발명한 게 아니라, 잘 짜인 여러 소프트웨어에서 **공통으로 발견된 구조를 사후에 추출**해 이름 붙인 것이다. 즉 "새로운 아이디어"가 아니라 "이미 잘 작동하는 걸 확인한 아이디어의 카탈로그"다. 바퀴를 재발명하지 않아도 되는 이유가 여기 있다.

## 3. 큰 분류 — GoF 3대 카테고리

GoF의 23개 패턴은 목적에 따라 세 갈래로 나뉜다.

| 분류 | 다루는 문제 | 핵심 질문 |
|---|---|---|
| **생성(Creational)** | 객체를 어떻게 만들 것인가 | "new를 직접 쓰지 않고 어떻게 유연하게 객체를 생성할까?" |
| **구조(Structural)** | 객체를 어떻게 조합할 것인가 | "서로 다른 인터페이스를 가진 것들을 어떻게 엮을까?" |
| **행동(Behavioral)** | 객체 사이의 책임과 상호작용을 어떻게 나눌 것인가 | "누가 어떤 일을 언제 처리하게 할까?" |

이 3분류가 존재하는 이유는 명확하다. 소프트웨어 설계에서 마주치는 문제는 결국 **"만들기 → 조립하기 → 동작시키기"** 세 단계로 수렴하기 때문이다.

---

## 4. 작은 분류 — 자주 쓰이는 패턴들

### 생성 패턴 (Creational)

| 패턴 | 해결하는 문제 |
|---|---|
| **Singleton** | 인스턴스가 애플리케이션 전체에서 딱 하나만 존재해야 할 때 (설정 객체, 로거) |
| **Factory Method** | 어떤 클래스의 인스턴스를 만들지를 서브클래스가 결정하게 하고 싶을 때 |
| **Abstract Factory** | 서로 연관된 객체 묶음(제품군)을 일관되게 생성하고 싶을 때 (OS별 UI 위젯 세트) |
| **Builder** | 생성자 인자가 너무 많거나, 객체를 단계별로 조립해야 할 때 |
| **Prototype** | 기존 객체를 복제해서 새 객체를 만드는 게 새로 생성하는 것보다 쌀 때 |

### 구조 패턴 (Structural)

| 패턴 | 해결하는 문제 |
|---|---|
| **Adapter** | 호환되지 않는 인터페이스 두 개를 연결해야 할 때 (레거시 API 감싸기) |
| **Decorator** | 상속 없이 객체에 기능을 동적으로 덧붙이고 싶을 때 |
| **Facade** | 복잡한 서브시스템을 단순한 인터페이스 하나로 감싸고 싶을 때 |
| **Proxy** | 실제 객체 접근을 제어하거나 지연시키고 싶을 때 (캐싱, 권한 체크, 지연 로딩) |
| **Composite** | 개별 객체와 객체 묶음을 같은 방식으로 다루고 싶을 때 (파일/폴더 트리) |

### 행동 패턴 (Behavioral)

| 패턴 | 해결하는 문제 |
|---|---|
| **Observer** | 상태 변화를 여러 곳에 자동으로 알리고 싶을 때 (이벤트, 구독 시스템) |
| **Strategy** | 알고리즘을 런타임에 교체 가능하게 만들고 싶을 때 (정렬 기준, 결제 방식) |
| **Command** | 요청 자체를 객체로 캡슐화해 큐에 쌓거나, 취소(undo)하고 싶을 때 |
| **State** | 객체의 상태에 따라 행동이 달라지는 걸 조건문 없이 구조화하고 싶을 때 |
| **Template Method** | 알고리즘의 뼈대는 고정하고, 일부 단계만 서브클래스가 바꾸게 하고 싶을 때 |
| **Iterator** | 컬렉션의 내부 구조를 노출하지 않고 순회하고 싶을 때 |

### GoF 이후 — 아키텍처 레벨 패턴

GoF는 클래스 단위 패턴에 집중했지만, 이후 더 큰 단위의 패턴들이 정착했다. 이들도 "디자인 패턴"의 확장선상에서 자주 함께 다뤄진다.

```
MVC / MVVM / MVP     → UI와 로직의 책임 분리 (프레젠테이션 계층 패턴)
Repository            → 데이터 접근 로직을 도메인 로직에서 분리
Dependency Injection  → 객체가 의존성을 직접 생성하지 않고 외부에서 주입받음
Pub/Sub               → Observer의 분산 시스템 버전 (메시지 큐, 이벤트 버스)
CQRS                  → 명령(쓰기)과 조회(읽기)의 책임을 분리
```

## 5. 주의사항 — 잊지 말아야 할 것

**패턴은 목적이 아니라 수단이다.** "이 프로젝트에 패턴을 몇 개 썼는가"는 좋은 설계의 지표가 아니다. 오히려 패턴 개수가 늘수록 간접 계층(Indirection)이 늘어나 코드를 따라가기 어려워지는 경우가 많다.

**이름이 같다고 구현이 같지 않다.** GoF는 1994년 C++/Smalltalk 기준으로 작성됐다. 언어에 따라 패턴의 필요성 자체가 사라지기도 한다.

```
예: Strategy 패턴
  Java/C++ (1994년 기준): 인터페이스 + 여러 구현 클래스 필요
  JavaScript/Python/Kotlin: 함수를 값으로 전달하면 끝 (별도 클래스 불필요)
```

Peter Norvig는 GoF 23개 패턴 중 상당수가 **동적 타입 언어나 일급 함수를 지원하는 언어에서는 문법 자체로 흡수되거나 불필요해진다**고 지적한 바 있다. 패턴은 언어의 한계를 우회하는 목적도 있었다는 뜻이다.

**과잉 설계(Over-engineering)를 경계하라.** 요구사항이 바뀔 가능성이 낮은 곳에 미리 Strategy, Factory, Observer를 겹겹이 깔아두면, "혹시 모를 유연성"을 위해 지금 당장의 가독성을 희생하는 셈이다. 실제로 바뀔 일이 없는 곳에 추상화 계층을 쌓는 것은 **YAGNI(You Aren't Gonna Need It)** 원칙과 정면으로 충돌한다.

**패턴 이름을 아는 것과 문제를 인식하는 능력은 다르다.** "이 상황에 뭘 써야 하지"를 패턴 목록에서 거꾸로 찾는 접근은 위험하다. 올바른 순서는 "지금 이 코드의 어떤 부분이 자주 바뀌는가, 어떤 결합이 문제인가"를 먼저 진단하고, 그 진단에 맞는 패턴이 있으면 가져다 쓰는 것이다.

**팀 전체가 같은 어휘를 공유해야 효과가 있다.** 한 명만 패턴 용어를 쓰고 나머지가 모르면, 오히려 코드 리뷰와 온보딩 비용이 늘어난다. 패턴은 "나 혼자 아는 지식"이 아니라 "팀의 공통 언어"일 때 가치가 생긴다.

## 6. 한계성

**패턴은 문제를 해결하지 않고 재배치할 뿐인 경우가 있다.** 예를 들어 Singleton은 전역 상태 문제를 "전역 변수"에서 "전역 접근 가능한 클래스"로 이름만 바꾼 것이라는 비판이 꾸준히 있다. 테스트하기 어렵고, 숨겨진 의존성을 만든다는 점은 그대로 남는다.

**클래스 기반 객체지향 언어에 편향돼 있다.** GoF의 원류가 C++/Smalltalk이다 보니, 함수형 프로그래밍이나 데이터 지향 설계(Data-Oriented Design)에는 그대로 적용하기 어렵거나 아예 무의미한 패턴이 많다. 게임 엔진 개발에서 캐시 효율을 위해 OOP 상속 구조 자체를 지양하는 흐름(ECS, Entity-Component-System)이 대표적 반례다.

**정적 카탈로그라 새로운 패러다임을 못 따라간다.** 1994년 GoF 책 이후 마이크로서비스, 반응형 프로그래밍(Reactive), 이벤트 소싱 같은 아키텍처가 등장했지만 GoF 23종에는 이런 것들이 없다. 커뮤니티가 새 패턴(위 5번 항목의 MVC/DI/CQRS 등)을 계속 추가해왔지만, "공식 카탈로그"라는 개념 자체가 계속 확장되는 것이지 완결된 목록이 아니다.

**패턴 선택이 트레이드오프라는 사실이 자주 생략된다.** 모든 패턴은 유연성을 얻는 대신 복잡도를 지불한다. 이 교환 비율이 상황마다 다른데, 패턴을 "배웠으니 써야 한다"는 태도로 접근하면 이 트레이드오프 감각이 무뎌진다. 숙련된 설계자와 초보자의 차이는 패턴을 아느냐가 아니라, **패턴을 안 쓰기로 결정할 줄 아느냐**에 가깝다.

## 정리

```
무엇인가        → 반복되는 설계 문제에 대한 이름 붙은 해결 구조 (GoF, 1994)
왜 필요한가      → 변화 대응 비용 절감 + 팀 공통 어휘 + 검증된 해법 재사용
큰 분류         → 생성(만들기) / 구조(조립하기) / 행동(동작시키기)
작은 분류       → Singleton·Factory·Builder / Adapter·Decorator·Facade / Observer·Strategy·Command 등
GoF 이후        → MVC·DI·Repository·CQRS 등 아키텍처 레벨로 확장
주의사항        → 목적이 아닌 수단, 언어에 따라 불필요해짐, 과잉 설계 경계, 진단이 먼저
한계성          → 문제 재배치에 그칠 수 있음, OOP 편향, 정적 카탈로그, 트레이드오프 생략 위험
```

디자인 패턴은 "이렇게 짜면 무조건 좋다"는 규칙집이 아니라, 특정 시대(1990년대 객체지향 전성기)의 특정 언어군에서 반복 검증된 해법 모음이다. 지금도 유효한 이유는 여전히 "변화에 강한 구조를 어떻게 짤 것인가"라는 근본 문제가 사라지지 않았기 때문이고, 동시에 맹신하면 안 되는 이유는 그 해법이 모든 언어, 모든 상황에 그대로 옮겨지지 않기 때문이다.
