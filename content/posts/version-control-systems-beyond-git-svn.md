[Git](/posts/git-collaboration-branch-fork-guide/)과 [SVN](/posts/svn-subversion-history-explained/)을 다뤘지만, 형상관리 시스템의 계보는 이 둘로 끝나지 않는다. Git 이전부터 지금까지 살아남은 것도 있고, Git이 표준이 된 지금도 특정 업계에서 여전히 왕좌를 지키는 것도 있고, 최근 몇 년 새 "Git보다 나은 인터페이스"를 내걸고 새로 뜨는 것도 있다. 각각 정확히 어떤 문제를 겨냥해 태어났는지 정리했다.

## 1. CVS — 모두가 결국 벗어나려 했던 공통 조상

CVS(Concurrent Versions System)는 딕 그루너(Dick Grune)가 1986년 7월 comp.sources.unix 뉴스그룹에 올린 셸 스크립트 뭉치에서 출발했다 — 지금 CVS 코드에 이 스크립트 자체는 남아 있지 않지만, 충돌 해결 알고리즘의 상당 부분은 여기서 이어졌다. 이후 1989년 4월 브라이언 벌리너(Brian Berliner)가 이를 정식으로 설계·재작성했고, 1990년 11월 19일 CVS 1.0이 자유소프트웨어재단(FSF)에 제출되며 정식 배포가 시작됐다. CVS는 그보다 앞선 단일 파일 버전관리 시스템 RCS(Revision Control System) 위에 저장소 단위 추적과 클라이언트-서버 모델을 얹은 것이었는데, [SVN 글](/posts/svn-subversion-history-explained/)에서 다뤘듯 바로 이 CVS의 결함(원자적이지 않은 커밋, 부실한 디렉터리 버전관리)을 정확히 겨냥해 2000년 SVN이 태어났다. CVS는 지금은 거의 쓰이지 않지만, 이후 태어난 거의 모든 형상관리 시스템이 "CVS가 뭘 못했는가"를 출발점으로 삼았다는 점에서 진짜 공통 조상이라 할 만하다.

## 2. Perforce — 게임·반도체 업계가 지금도 떠나지 않는 이유

퍼포스(Perforce, 지금은 브랜드명 P4로 재통합된 Helix Core)는 1995년 캘리포니아 앨러미다에서 크리스토퍼 세이월드(Christopher Seiwald)가 창업한 회사다. 처음부터 소스코드뿐 아니라 대용량 바이너리 파일까지 다루는 걸 염두에 둔 중앙집중형 시스템으로 설계됐고, 엔비디아·EA 같은 초기 고객이 대규모 바이너리 애셋을 소스코드와 함께 관리하려고 채택했다. 지금도 상위 20개 AAA 게임 스튜디오 중 19곳, 상위 10개 반도체 기업 중 9곳이 Perforce를 쓴다고 회사 측이 밝힐 정도로 이 두 업계에서 압도적 지위를 유지하고 있다 — [SVN 글](/posts/svn-subversion-history-explained/)에서 짚었듯 이런 업계는 대용량 바이너리 파일의 부분 체크아웃과 파일 잠금(locking)이 Git의 분산 복제 모델보다 여전히 실질적으로 유리하기 때문이다.

## 3. ClearCase — IBM까지 흘러들어간 무거운 엔터프라이즈 표준

클리어케이스(ClearCase)는 1990년 1월 11일 데이비드 르블랑(David Leblang)을 비롯한 창업자들이 세운 애트리아 소프트웨어(Atria Software, HP의 아폴로컴퓨터 인수 후 퇴사한 개발자들이 만든 회사)가 만들어 1992년 첫 출시했다. 이후 1996년 퓨어소프트웨어와 합병해 퓨어애트리아가 됐고, 1997년 래셔널 소프트웨어(Rational Software)에 인수됐으며, 그 래셔널이 2003년 2월 21일 IBM에 21억 달러에 팔리면서 지금은 "IBM 래셔널 클리어케이스"라는 이름으로 남아 있다. 멀티사이트 복제(전 세계 여러 지역에 저장소를 복제해 지연 없이 협업)와 "뷰(view)"라는 독자적인 파일시스템 통합 개념이 특징이었지만, 그만큼 설치·운영이 무겁고 느리다는 게 오래된 정설이다 — 지금도 일부 대형 금융·통신·항공 등 수십 년 된 엔터프라이즈 인프라에 남아 있는 경우가 있지만, 신규 프로젝트가 이를 선택하는 일은 사실상 없다.

## 4. Mercurial — Git과 같은 주에 태어난 "쌍둥이"

[Git 글](/posts/git-collaboration-branch-fork-guide/)에서 다룬 2005년 비트키퍼 라이선스 철회 사태는 사실 두 개의 대체 시스템을 동시에 낳았다. 리누스 토르발스가 4월 3일 Git 작업을 시작한 지 며칠 뒤인 4월 19일, 또 다른 리눅스 커널 개발자 맷 매콜(Matt Mackall)이 같은 위기에 대응해 머큐리얼(Mercurial)을 공개했다 — 목표도 동일했다(비트키퍼를 대신할 분산 버전관리 시스템). 결국 리눅스 커널 자체는 Git을 택했지만, 머큐리얼은 그 후로도 별도의 생태계를 꾸렸다 — 모질라(Mozilla)가 채택했고, 파이썬은 [SVN 글](/posts/svn-subversion-history-explained/)에서 다뤘듯 2008~2009년 SVN에서 머큐리얼로 옮겨갔다가 2017년에야 Git으로 다시 이전했으며, 페이스북(메타)도 초기부터 대규모로 채택해 자체 확장까지 개발했다. 명령어 체계가 Git보다 단순하고 일관적이라는 평가를 받았음에도, 깃허브라는 압도적 생태계 중력을 가진 Git에 결국 개발자 점유율에서 크게 밀렸다는 게 지금 머큐리얼의 위치다.

## 5. Darcs와 Pijul — "수학적으로 옳은 병합"이라는 다른 길

대부분의 형상관리 시스템이 스냅샷이나 파일 이력을 중심에 두는 것과 달리, 다스(Darcs, 2003년 데이비드 라운디가 하스켈로 개발)는 "패치 이론(patch theory)"이라는 수학적 틀 위에서 변경 자체를 병합의 기본 단위로 삼았다. 문제는 특정 상황(여러 충돌하는 패치가 얽힌 경우)에서 병합 계산 시간이 기하급수적으로 폭증하는 이른바 "지수 병합 문제"였다 — 실제로 병합 자체가 끝나지 않는 사례까지 보고됐다. 피에르-에티엔 뫼니에(Pierre-Étienne Meunier)가 만든 피줄(Pijul)은 사뮈엘 미맘(Samuel Mimram)과 친지아 디주스토(Cinzia Di Giusto)의 패치 이론을 이어받되 범주론(category theory)을 접목해 이 문제를 이론적으로 해결했다고 주장한다 — 충돌 없는 병합은 이력 크기에 로그 시간, 충돌이 있는 경우에도 선형 시간을 넘지 않도록 설계됐다는 것이다. Rust로 작성된 피줄은 아직 알파 단계를 벗어난 지 얼마 안 된 니치 프로젝트지만, "병합을 수학적으로 증명 가능하게 만든다"는 접근 자체는 Git·SVN류의 실용주의적 설계와는 완전히 다른 계보를 보여준다.

## 6. Fossil — SQLite 저자가 만든 "자기완결형" 형상관리

파슬(Fossil)은 SQLite의 저자인 D. 리처드 힙(D. Richard Hipp)이 2006년 내놓은 분산 버전관리 시스템으로, 접근 자체가 독특하다 — 소스 코드 이력뿐 아니라 위키, 버그 트래커, 포럼까지 하나의 단일 실행 파일과 단일 SQLite 데이터베이스 파일 안에 전부 담는다. 외부 `diff`나 `patch` 같은 프로그램에 의존하지 않고, 그 실행 파일 하나가 웹 요청 처리(소스 브라우징·이슈 관리·저장소 동기화)까지 직접 담당한다. 힙은 이 저장소 형식을 "아직 태어나지도 않은 사람들이 읽고 검색하고 확장할 수 있어야 한다"는 목표로 설계했다고 밝혔다. 실제로 SQLite 프로젝트 자신이 Fossil로 관리되고 있으며, Fossil 자기 자신의 개발도 Fossil로 이뤄진다 — "형상관리·이슈트래커·위키를 따로 붙이지 않고 하나로 묶는다"는 이 접근은 지금도 깃허브 같은 외부 플랫폼 의존을 최소화하고 싶은 소규모·장기 프로젝트에서 꾸준히 언급된다.

## 7. 지금 뜨는 것들 — Sapling과 Jujutsu, "Git 위에서 더 나은 경험"

최근 몇 년 새 주목받는 두 시스템은 공통적으로 Git 생태계 자체를 버리지 않으면서 그 위에 더 나은 사용자 경험을 얹는 전략을 택했다. 메타(옛 페이스북)는 2022년 11월 자사 초대형 모노레포를 위해 내부에서 써오던 세이플링(Sapling)을 오픈소스로 공개했다 — 원래 머큐리얼의 확장으로 시작했지만 저장 포맷·프로토콜·알고리즘이 완전히 새로 발전해 독자적인 시스템이 됐고, 한눈에 자기 작업 스택을 보여주는 "스마트로그(Smartlog)" 같은 기능으로 사용성을 강조하면서도 Git 저장소를 그대로 클론·연동할 수 있게 했다. 주지츠(Jujutsu, 명령어는 `jj`)는 구글 엔지니어 마틴 폰 츠바이베르크(Martin von Zweigbergk, 흥미롭게도 이전엔 구글 내부에서 머큐리얼 유지보수를 맡았던 인물)가 2019년 말 개인 프로젝트로 시작했고, 2022년 5월 Git을 저장소 백엔드로 쓸 수 있는 기능이 추가되면서 동료들에게 시스템 전환을 요구하지 않고도 `.git` 디렉터리 위에 그대로 얹어 쓸 수 있게 되자 확산이 빨라졌다. 스테이징 영역이 따로 없고 모든 작업이 곧바로 커밋으로 기록되며, 충돌이 나도 즉시 해결을 강제하지 않고 "충돌 상태의 커밋"으로 남겨둔 채 나중에 정리할 수 있다는 게 Jujutsu의 핵심 차별점으로 꼽힌다. 둘 다 "Git을 완전히 대체"하기보다 "Git 저장소는 그대로 두고 그 위의 경험만 바꾼다"는 점에서, 처음부터 새 프로토콜을 들고나왔던 이전 세대(SVN, Mercurial, Fossil, Pijul)와는 접근 자체가 다르다.

## 정리

CVS 이후 태어난 형상관리 시스템들은 저마다 "이전 시스템이 못 푼 정확히 하나의 문제"를 겨냥해 갈라져 나왔다 — Perforce는 대용량 바이너리와 파일 잠금(그래서 게임·반도체 업계에 지금도 남아 있고), ClearCase는 엔터프라이즈급 멀티사이트 복제(그래서 무거워졌고), Mercurial은 Git과 똑같이 비트키퍼 위기에서 나왔지만 생태계 중력에서 밀렸고, Darcs·Pijul은 병합을 수학적으로 증명 가능하게 만드는 이론적 순수성, Fossil은 외부 도구·플랫폼 의존 없는 자기완결성을 각각 목표로 삼았다. 그리고 가장 최근 세대인 Sapling·Jujutsu는 아예 프로토콜 전쟁을 포기하고 "Git 저장소는 그대로 둔 채 그 위의 경험만 새로 짠다"는, 앞선 어떤 세대와도 다른 전략을 택했다. 결국 형상관리 시스템의 역사 전체를 관통하는 질문은 하나다 — 지금 내가 겪는 협업의 고통이 바이너리 파일 크기 때문인지, 병합의 정확성 때문인지, 아니면 그냥 명령어가 손에 안 익어서인지, 그 답에 따라 골라야 할 도구가 완전히 달라진다는 것이다.
