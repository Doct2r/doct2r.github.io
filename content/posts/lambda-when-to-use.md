`list.map(x => x * 2)` 한 줄은 우아하다. 그런데 화살표 뒤에 조건문 세 개와 중첩 콜백이 붙기 시작하면 그 우아함은 순식간에 디버깅 지옥으로 바뀐다. 람다는 "짧게 쓸 수 있다"는 이유만으로 남용되기 쉬운 문법이다. 어디서 왔고, 뭐가 다르고, 언제 써야 실제로 이득인지 정리한다.

---

## 1. 람다란 무엇인가

"람다(lambda)"라는 이름은 1930년대 수학자 **알론조 처치(Alonzo Church)**가 만든 **람다 계산법(lambda calculus)**에서 왔다. 함수를 이름 없이 순수한 입력-출력 대응 관계로만 표현하는 수학적 표기법이다. 1958년 존 매카시(John McCarthy)가 **리스프(Lisp)**를 설계하며 이 개념을 그대로 언어 문법으로 가져와 `LAMBDA` 키워드를 만들었고, 이후 대부분의 언어가 이 이름을 물려받았다.

프로그래밍에서 람다는 결국 **이름이 없는 함수(익명 함수)**를 뜻한다. 핵심 특징은 두 가지다.

- **일급 객체(first-class citizen)**: 변수에 담고, 인자로 넘기고, 반환값으로 돌려줄 수 있다.
- **클로저(closure)**: 자신이 정의된 시점의 주변 스코프 변수를 캡처해 나중에 실행될 때도 그 값을 참조할 수 있다.

---

## 2. 언어별 문법 — 같은 개념, 다른 얼굴

| 언어 | 문법 | 도입 시점 |
|---|---|---|
| Lisp | `(lambda (x) (* x 2))` | 1958 |
| Python | `lambda x: x * 2` | 초기부터, 단일 표현식만 허용 |
| C++ | `[capture](int x) { return x * 2; }` | C++11 (2011) |
| Java | `x -> x * 2` | Java 8 (2014) |
| C# | `x => x * 2` | C# 3.0 (2007), LINQ와 함께 |
| JavaScript | `x => x * 2` | ES2015 |
| Kotlin/Swift | `{ x -> x * 2 }` / `{ x in x * 2 }` | 초기부터 |

문법은 제각각이지만 "이름 짓지 않고 즉석에서 함수를 만든다"는 본질은 동일하다. 차이가 있다면 캡처 방식이다. C++만 유일하게 `[&]`, `[=]`처럼 **캡처 목록을 명시**해야 한다 — 참조로 캡처할지 값으로 복사할지를 개발자가 직접 선택해야 하는데, 이 지점이 실제로 많은 버그의 근원이 된다(뒤에서 다룬다).

---

## 3. 왜 쓰는가 — 람다가 해결하는 문제

람다가 널리 퍼진 이유는 **고차 함수(higher-order function)**와 궁합이 좋기 때문이다. `map`, `filter`, `reduce`, 정렬 비교자(comparator), 이벤트 핸들러, 비동기 콜백처럼 "짧은 로직을 함수 인자로 즉시 전달"해야 하는 상황에서, 일일이 이름을 짓고 별도 함수로 선언하는 건 과한 격식이다.

```java
// 이름을 짓기엔 너무 사소한 로직
list.sort((a, b) -> a.getAge() - b.getAge());
```

이런 자리에 매번 `compareByAge`라는 이름의 별도 메서드를 만드는 건 오히려 코드를 읽는 사람의 시선을 엉뚱한 곳으로 흩어놓는다. 람다는 **로직이 쓰이는 자리와 정의되는 자리를 붙여놓아** 문맥을 한눈에 보이게 한다.

---

## 4. 남용했을 때 실제로 터지는 문제들

### 클로저 캡처 버그 — 가장 유명한 함정

C#에는 오랫동안 유명했던 버그 패턴이 있다. C# 5(2012) 이전에는 `foreach` 루프 안에서 만든 람다가 **반복 변수 자체를 참조**로 캡처했다. 그 결과 루프가 다 끝난 뒤 람다들이 실행되면 전부 마지막 값 하나만 참조하는 버그가 났다.

```javascript
// 흔한 JavaScript 초심자 버그
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100);
}
// 0, 1, 2가 아니라 3, 3, 3이 찍힌다 (var는 함수 스코프)
```

`var`를 `let`으로 바꾸면 해결되는데, 이는 `let`이 반복마다 새 바인딩을 만들기 때문이다. 파이썬도 같은 함정이 있다 — 루프 안 람다가 변수를 **늦은 바인딩(late binding)**으로 참조해 마지막 값만 캡처한다. 세 언어 모두 다른 방식으로 같은 문제를 겪었다는 건, 이게 문법의 우연한 결함이 아니라 **클로저라는 개념 자체가 가진 근본적인 함정**이라는 뜻이다.

### 중첩되면 콜백 지옥이 된다

Node.js 초기 비동기 코드는 콜백을 계속 중첩시키는 방식으로 짜였고, 그 결과가 악명 높은 **콜백 지옥(callback hell)**이다. 람다/익명 함수 하나하나는 짧아도, 중첩된 깊이가 늘어나면 코드 전체의 들여쓰기와 흐름을 추적하기가 급격히 어려워진다. 이 문제 때문에 JS 생태계는 Promise, 그다음 `async/await`로 옮겨갔다 — 결국 "람다를 계속 겹쳐 쓰는 방식"이 한계에 부딪혔다는 방증이다.

### 스택 트레이스와 디버깅

이름이 없다는 특성은 디버깅에서 그대로 대가로 돌아온다. 예외가 발생했을 때 스택 트레이스에 `lambda$processOrder$3` 같은 컴파일러 생성 이름이 찍히면, 정작 어떤 의도로 만든 람다였는지 코드를 열어 위아래를 뒤져야 알 수 있다. 이름 있는 메서드였다면 이름만으로 의도가 드러났을 정보다.

### 숨은 성능 비용

람다가 클로저로 외부 변수를 캡처하면, 대부분의 런타임은 이를 위해 **힙에 객체를 새로 할당**한다. Java의 스트림 API(`stream().map(...).filter(...)`)가 단순 for 루프보다 느릴 수 있다는 지적이 꾸준히 나오는 이유 중 하나가 이 캡처·오토박싱 비용이다. C++에서도 `std::function`으로 람다를 저장하면 타입 소거(type erasure) 과정에서 힙 할당이 발생할 수 있어, 핫패스(hot path)에서는 템플릿 기반 설계로 이를 피하는 경우가 많다.

### 재사용과 테스트의 어려움

이름이 없다는 건 **참조할 손잡이가 없다**는 뜻이기도 하다. 람다로 작성된 로직은 다른 곳에서 재사용하려면 결국 어딘가로 끌어올려 이름을 붙여야 하고, 단위 테스트도 람다를 감싼 상위 코드를 통째로 실행해야 검증할 수 있다. 로직이 복잡해질수록 이 비용은 누적된다.

---

## 5. 그래서 언제 쓰는 게 맞는가

정리하면 판단 기준은 **"이 로직에 이름을 지을 필요가 있는가"**로 좁혀진다.

**람다가 적합한 경우**
- 한두 줄 안에 끝나는 짧은 로직
- 정의되는 자리에서 바로, 단 한 번만 쓰이는 경우 (정렬 비교자, 즉시 실행 콜백)
- 캡처하는 외부 변수가 적고 수명이 짧은 경우 (버튼 클릭 핸들러, 단발성 비동기 콜백)

**이름 있는 함수로 승격해야 하는 경우**
- 조건 분기나 반복문이 들어갈 만큼 로직이 길어질 때
- 같은 로직이 두 곳 이상에서 쓰일 때
- 단위 테스트로 별도 검증이 필요한 로직일 때
- 콜백이 2단 이상 중첩되기 시작할 때 (즉시 별도 함수로 분리하거나 `async/await`류 문법으로 전환)
- 예외 발생 시 스택 트레이스에서 의도가 바로 읽혀야 하는 프로덕션 크리티컬 경로

---

## 6. 결론

람다는 "이름을 지을 필요가 없을 만큼 짧고, 그 자리에서만 쓰이는 지역적 로직"을 위해 만들어진 문법이다. 캡처·중첩·디버깅·성능 문제는 전부 이 전제를 벗어났을 때 — 즉 람다 안의 로직이 이름을 가질 자격을 갖췄는데도 익명으로 방치됐을 때 — 터진다.

그러니 람다를 쓸지 말지 고민될 때는 문법이 아니라 로직의 성격을 봐야 한다. 로직이 자라서 재사용되거나, 테스트가 필요해지거나, 남이 읽고 이해해야 하는 순간이 오면 망설이지 말고 이름을 붙여라. 람다는 편의 문법이지 만능 해법이 아니다.
