FastAPI는 파이썬 웹 프레임워크치고 이례적으로 빨리 퍼졌다. 2018년 12월 8일 세바스티안 라미레즈("tiangolo")가 0.1.0을 내놓은 뒤, JetBrains·PSF의 파이썬 개발자 설문에서 2021년 14%였던 채택률이 2023년에는 25%까지 올랐다(같은 기간 플라스크는 29%에서 26%로 내려왔다). 그런데 FastAPI 자체 문서를 들여다보면, 이 프레임워크가 정말로 새로 발명한 기술은 거의 없다는 게 드러난다. 대신 여러 프로젝트의 아이디어를 정직하게 이어붙였고, 그 이어붙이는 방식 자체가 무기였다.

---

## 1. 탄생 — APIStar가 버린 아이디어를 이어받다

FastAPI의 직계 조상은 톰 크리스티가 만든 APIStar다. APIStar는 파이썬 타입 힌트로 파라미터와 요청을 선언하면 검증·직렬화·OpenAPI 스키마 생성까지 자동으로 되는, 이 방식을 처음 시도한 프레임워크 중 하나였다. 문제는 크리스티가 이후 Starlette 개발에 집중하면서 APIStar의 방향이 바뀌었다는 점이다 — 지금 APIStar는 웹 프레임워크가 아니라 OpenAPI 스펙을 검증하는 도구 모음으로 남아 있다. FastAPI 공식 문서는 이 경위를 직접 설명하며 자신을 APIStar의 "정신적 후계자"라고 표현한다.

## 2. 영향받은 것들 — FastAPI가 스스로 밝힌 계보

FastAPI 문서의 "대안들(Alternatives)" 페이지는 어디서 뭘 가져왔는지를 유난히 솔직하게 밝힌다. 플라스크에서는 데코레이터 기반 라우팅이라는 마이크로프레임워크의 단순함을, Requests에서는 직관적인 API와 합리적인 기본값이라는 개발자 경험 철학을, 장고 REST 프레임워크(DRF)에서는 자동 API 문서 UI라는 핵심 기능 자체를(FastAPI가 DRF에서 정말로 원했던 건 이거 하나였다고 명시한다), 마시멜로우에서는 스키마 기반 검증을, 웹아그스에서는 요청 자동 검증을, APISpec·Flask-apispec에서는 검증 코드에서 그대로 OpenAPI 문서를 뽑아내는 방식을, Hug에서는 타입 힌트로 파라미터를 선언하는 패턴을, Sanic에서는 순수 성능 지향점을 가져왔다고 스스로 정리해뒀다.

## 3. 핵심 구조 — Starlette과 Pydantic이라는 두 기둥

`FastAPI` 클래스는 실제로 톰 크리스티가 만든 Starlette 클래스를 그대로 상속한다. Starlette이 라우팅·미들웨어·웹소켓·백그라운드 태스크 같은 웹 처리 부분을 담당하고, 여기엔 검증이나 문서화 기능이 없다. 그 빈자리를 새뮤얼 콜빈이 만든 Pydantic이 채운다 — 타입 힌트로부터 검증·직렬화·JSON 스키마 기반 문서를 만들어내는 역할이다. 즉 FastAPI 자체는 새로운 엔진이 아니라, 이 두 라이브러리를 정확한 지점에서 접합한 결과에 가깝다.

## 4. ASGI라는 배경 — WSGI가 못 하던 것

Starlette과 FastAPI가 둘 다 비동기 우선으로 설계될 수 있었던 건 ASGI(Asynchronous Server Gateway Interface)라는 표준 덕분이다. ASGI는 앤드류 갓윈이 장고 채널스 프로젝트를 만들던 2016년 무렵 시작됐다 — 기존 WSGI 표준이 요청-응답 한 사이클로만 설계돼 있어서 웹소켓처럼 오래 유지되는 연결을 다루지 못한다는 한계를 풀기 위해서였다. 이후 2017년 11월 ASGI 2.0 스펙이, 2019년 3월 3.0 스펙이 나왔고 톰 크리스티도 이 프로토콜을 다듬는 데 참여했다. FastAPI를 실행하는 표준 서버로 쓰이는 유비콘(Uvicorn) 역시 크리스티가 몸담은 Encode 소속 프로젝트다.

## 5. 자동 문서화 — 타입 힌트 하나로 세 가지를 동시에

FastAPI가 가장 많이 홍보하는 특징은 함수 시그니처와 Pydantic 모델만 작성하면 OpenAPI 스키마가 자동으로 만들어지고, `/docs`(스웨거 UI)와 `/redoc` 경로에서 대화형 문서가 바로 뜬다는 점이다. 별도의 스키마 정의 파일이나 직렬화 클래스를 따로 안 써도 된다는 뜻이다 — DRF가 시리얼라이저 클래스를 명시적으로 작성해야 하는 것과 대비되는 지점이다.

## 6. 성능 주장 — 실제로는 정직한 편이다

FastAPI가 "Node.js·Go급 성능"이라는 식으로 부풀려 알려진 경우가 많은데, 정작 FastAPI 공식 문서 자체의 표현은 훨씬 절제돼 있다 — "유비콘 위에서 실행되는 FastAPI 애플리케이션은 스탈렛과 유비콘 자체보다 아주 약간 아래에 있을 뿐, 사용 가능한 파이썬 프레임워크 중 가장 빠른 축에 속한다"는 게 공식 문서의 표현이다. 즉 "자신이 올라탄 두 기반기술 자체보다 빠를 수는 없다"는 정직한 한계를 스스로 못 박아둔 셈이고, "Node·Go를 이긴다"는 식의 문구는 FastAPI 공식 문서가 실제로 하는 주장보다 부풀려진 버전으로 봐야 한다.

## 7. Pydantic v2 전환 — 러스트로 다시 쓴 검증 엔진

Pydantic은 2023년 6월 30일 v2.0을 냈는데, 이건 점진적 개선이 아니라 핵심 검증 로직을 통째로 러스트로 새로 짠(pydantic-core, PyO3 바인딩 사용) 재작성이었다 — 콜빈이 2022년 초부터 약 1년 반을 풀타임으로 투입한 작업이다. 콜빈 본인이 공개한 수치로는 v1.9 대비 검증 속도가 약 17배 빨라졌고, 작업 종류에 따라 4배에서 최대 50배까지 차이가 났다. 흥미로운 지점은 FastAPI를 만든 라미레즈 본인이 Pydantic 프로젝트의 실제 기여자 명단에 이름을 올리고 있다는 것이다 — 단순히 의존성 관계가 아니라 두 프로젝트 사이에 실제 인적 연결이 있다는 뜻이다.

## 8. 채택 사례 — 정확히 인용된 것만

FastAPI 공식 사이트의 "실사용 사례" 페이지에는 몇 개의 구체적인 증언이 실려 있다. 마이크로소프트의 카비르 칸은 "우리 팀의 머신러닝 서비스 전체에 FastAPI를 쓸 계획이고, 일부는 이미 윈도우 핵심 제품과 오피스 제품 일부에 들어갔다"고 밝혔다. 우버는 피에로 몰리노 등이 자사 머신러닝 툴킷 Ludwig에서 "예측 결과를 조회할 수 있는 REST 서버를 띄우려고 FastAPI를 채택했다"고 밝혔고, 넷플릭스는 자사 위기관리 프레임워크 Dispatch를 FastAPI로 만들어 오픈소스로 공개했다. 시스코의 데온 필스버리도 "API 우선 개발 전략의 핵심 요소"라고 언급했다. 다만 이런 인용을 훨씬 구체적으로 부풀린 2차 SEO성 기사들(예: 우버가 이걸로 "전 세계 배차 운영의 고처리량 데이터 파이프라인"을 돌린다는 식의 서술)은 원문 인용에는 없는 내용이라 신뢰하기 어렵다 — 실제 공식 인용문 이상으로 각색된 부분은 걸러서 봐야 한다.

## 정리

FastAPI의 재료 하나하나는 다 어딘가에서 가져온 것들이다 — 문법은 플라스크, 자동 문서화라는 목표는 DRF, 타입 힌트로 파라미터를 선언하는 방식은 Hug, 애초에 이 모든 걸 타입 힌트 하나로 엮는다는 발상 자체는 APIStar에서 왔다. FastAPI가 실제로 한 일은 새 부품을 만드는 게 아니라, Starlette과 Pydantic이라는 이미 잘 만들어진 두 부품을 정확한 접합면에서 붙이고, 그 결과로 검증·직렬화·문서화가 타입 힌트 하나에서 동시에 나오게 만든 것이다. 빠르게 퍼진 이유도 여기 있다 — 완전히 새로운 걸 배워야 했던 게 아니라, 이미 흩어져 있던 익숙한 아이디어들이 한 프레임워크 안에서 처음으로 맞물렸기 때문이다.
