화면을 안 보고도 앱을 쓰는 순간이 있다 — 주머니 속 진동, 알림음, 잠금화면의 알림 문구. 1편이 화면 뒤의 뼈대였다면, 이번 편은 화면 밖에서 사용자와 OS를 연결하는 요소들이다. 사운드, 알림, 권한, 백그라운드 실행, 다국어까지 — GUI가 아니지만 사용자 경험을 결정하는 층위를 다룬다.

---

## 1. 사운드 디자인 — 눈이 아니라 귀와 피부로 전달하는 피드백

### 효과음(SFX)과 브랜드 사운드

버튼 클릭음, 성공/실패 알림음, 알림 도착음은 단순 장식이 아니라 **상태를 전달하는 피드백 채널**이다. 화면을 안 보고 있어도 "저장이 됐다", "오류가 났다"를 소리만으로 알 수 있게 설계하는 것이 목표다. 카카오톡 알림음, 넷플릭스 재생 사운드처럼 특정 사운드가 브랜드 그 자체로 인식되는 경우도 있다.

### 햅틱 피드백

터치에 반응하는 미세한 진동은 iOS의 Taptic Engine, Android의 Vibration API로 구현된다. 햅틱은 단순 재미 요소가 아니라 **시각·청각 장애가 있는 사용자에게 대체 피드백 채널**로 작동하기도 한다. 버튼을 눌렀는데 화면 반응이 0.1초 늦어도, 즉각적인 햅틱이 있으면 사용자는 "눌렸다"고 인지한다.

### 오디오 세션과 포커스 관리

```
문제 상황: 사용자가 음악 앱을 틀어둔 채 다른 앱에서 알림음이 울리면?
  → 두 소리가 겹쳐 들리거나, 음악이 갑자기 끊기거나, 볼륨이 이상하게 섞인다

해결 구조:
  Android: AudioFocus API — "지금 내가 소리를 낼 차례"를 OS에 요청하고 양보받음
  iOS    : AVAudioSession — 카테고리(재생/녹음/믹싱 허용 여부)를 명시적으로 선언
```

이 오디오 포커스 관리를 제대로 안 하면, 사용자가 음악을 듣다가 알림 하나로 재생이 끊기는 불쾌한 경험을 만들게 된다. 또한 사용자가 **무음 모드/방해금지 모드**를 켰을 때 이를 존중하는지 여부도 OS 정책 준수의 일부다.

## 2. 알림 — 앱이 닫혀 있어도 사용자에게 닿는 유일한 통로

| 종류 | 특징 |
|---|---|
| **로컬 알림** | 기기 내부에서 예약·발송(예: 타이머 종료 알림). 서버 불필요 |
| **푸시 알림** | 서버가 FCM(Android)/APNs(iOS)를 통해 원격으로 발송. 마케팅·실시간 이벤트 전달에 사용 |

알림에는 **채널/카테고리**와 **우선순위**가 있어 사용자가 종류별로 켜고 끌 수 있어야 하며(예: "주문 알림은 받되 마케팅 알림은 끄기"), 알림을 탭했을 때 관련 화면으로 바로 이동시키는 **딥링크** 처리가 따라와야 한다. 알림만 오고 탭해도 앱 홈 화면만 뜨는 경험은 알림의 가치를 절반 이상 깎아먹는다.

---

## 3. 권한 요청 흐름 — OS가 사용자 대신 지키는 최소한의 선

카메라, 위치, 마이크, 연락처 같은 민감한 기능은 OS가 런타임에 사용자 동의를 요구한다. 설계 시 지켜야 할 원칙은 명확하다.

```
원칙: 앱 실행 시점이 아니라, "그 권한이 실제로 필요한 행동을 하는 순간"에 요청한다
  나쁜 예: 앱 켜자마자 카메라·위치·알림 권한을 한꺼번에 요청 → 대부분 거부당함
  좋은 예: 사용자가 "사진 찍기" 버튼을 눌렀을 때 그제서야 카메라 권한 요청
```

권한을 거부당했을 때의 **폴백(fallback) 동작**도 설계에 포함돼야 한다 — 위치 권한이 없으면 수동 지역 검색으로 대체하는 식으로, 권한 거부가 곧 기능 정지로 이어지지 않게 만드는 것이 좋은 설계다.

## 4. 백그라운드 실행 정책 — OS는 배터리를 앱보다 우선한다

앱이 화면에서 사라진 뒤에도 계속 실행되길 원하는 건 개발자의 바람일 뿐, OS는 배터리 보호를 위해 이를 강하게 제한한다.

```
iOS  : 백그라운드 실행이 허용된 목적(오디오 재생, 위치 추적, VoIP 등)이 화이트리스트로 제한됨
       Background App Refresh는 시스템이 알아서 스케줄링 — 개발자가 실행 시점을 강제할 수 없음

Android: Doze 모드 — 화면이 꺼지고 일정 시간 정지 상태면 백그라운드 작업을 그룹으로 묶어 지연 실행
         배터리 최적화 예외 목록에 없으면 알람·동기화가 지연되거나 스킵될 수 있음
```

이 제약을 무시하고 "항상 떠 있는 서비스"를 억지로 구현하면 OS가 강제 종료시키거나, 최악의 경우 사용자가 배터리 소모 앱으로 인식해 직접 삭제한다. 안정적인 백그라운드 동작은 OS가 제공하는 예약 실행 API(1편에서 언급한 WorkManager, BackgroundTasks)를 통해 "OS의 스케줄에 맞춰 협조하는" 방식으로만 가능하다.

## 5. 로컬라이제이션 — 번역이 아니라 지역화

다국어 지원은 문자열을 번역하는 것보다 훨씬 넓은 작업이다.

| 항목 | 함정 |
|---|---|
| **복수형 규칙** | "1개 항목", "2개 항목" — 언어마다 복수형 규칙 개수가 다름(영어 2개, 아랍어는 6개 형태) |
| **RTL(우→좌) 레이아웃** | 아랍어·히브리어는 레이아웃 자체가 좌우 반전되어야 함. 아이콘 방향까지 미러링 필요 |
| **날짜·통화·숫자 포맷** | 같은 날짜도 지역마다 표기 순서·구분자가 다름(MM/DD/YYYY vs DD.MM.YYYY), 소수점 구분자(`,` vs `.`) |
| **텍스트 길이 변화** | 같은 문장도 언어에 따라 길이가 2～3배 차이 나 UI가 깨질 수 있음(독일어가 대표적으로 김) |

문자열을 하드코딩하지 않고 리소스 파일로 분리하는 것은 "번역 작업의 편의"를 위해서가 아니라, 위 항목들을 코드 수정 없이 지역별로 다르게 처리하기 위한 구조적 필요다.

## 정리

```
사운드      → 효과음은 상태 피드백 채널, 햅틱은 접근성 대체 채널, 오디오 포커스로 다른 앱과 충돌 방지
알림        → 로컬/푸시 구분, 채널별 온오프, 딥링크로 알림→화면 연결까지가 한 세트
권한        → "필요한 순간"에 요청, 거부 시 폴백 동작까지 설계에 포함
백그라운드   → OS가 배터리 보호를 위해 강하게 제한 — 전용 스케줄링 API로 "협조"해야 안정적
로컬라이제이션→ 번역이 아니라 복수형·RTL·날짜포맷·텍스트 길이까지 포함하는 지역화 작업
```

이 요소들의 공통점은 "앱 혼자만의 문제가 아니라 OS·다른 앱·사용자의 물리적 환경과 어떻게 공존하느냐"의 문제라는 것이다. 마지막 3편에서는 이 앱이 출시된 이후 안전하게 운영되게 만드는 보안·로깅·테스트·배포 요소를 다룬다.
