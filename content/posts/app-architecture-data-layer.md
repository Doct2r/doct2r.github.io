앱을 만들 때 가장 먼저 이야기되는 건 화면 디자인과 사용성이다. 하지만 그 화면 뒤에는 사용자 눈에 절대 보이지 않는 뼈대가 있고, 이 뼈대가 부실하면 UI가 아무리 예뻐도 앱은 버벅이거나 데이터를 잃어버린다. UX/GUI를 걷어내고 남는 것들 — 아키텍처, 데이터, 네트워킹, 동시성 — 을 정리한다. (사운드·알림·시스템 통합은 2편, 보안·운영은 3편에서 다룬다.)

---

## 1. 상태 관리 아키텍처 — UI와 로직을 어떻게 나눌 것인가

앱이 커지면 "어떤 데이터가 어디서 바뀌었는지"를 추적하기가 기하급수적으로 어려워진다. 상태 관리 아키텍처는 이 문제를 구조적으로 통제하기 위한 규칙이다.

```
공통 목표: 데이터 흐름을 예측 가능하게 만든다
  나쁜 상태: 아무 코드나 아무 화면의 데이터를 직접 건드릴 수 있음
  좋은 상태: 데이터 변경 경로가 한 방향으로 정해져 있어, 버그가 나면 그 경로만 추적하면 됨
```

| 패턴 | 핵심 아이디어 | 대표 구현체 |
|---|---|---|
| **MVVM** | View는 ViewModel의 상태를 구독만 하고, 로직은 ViewModel이 전담 | Android Jetpack(StateFlow), iOS Combine/Observation |
| **MVI** (Model-View-Intent) | 상태를 하나의 불변 객체로 통일하고, 모든 변경은 "의도(Intent)"를 거침 | Android Compose+Orbit, 일부 Flutter 아키텍처 |
| **Redux/Flux 계열** | 전역 단일 스토어 + 순수 함수(리듀서)로만 상태 변경 | React+Redux, Flutter BLoC/Riverpod |
| **Clean Architecture** | UI-도메인-데이터 계층을 완전히 분리해 프레임워크 교체에도 로직이 안 깨지게 함 | 대규모 안드로이드/iOS 앱에서 자주 채택 |

패턴 이름보다 중요한 원칙은 하나다 — **단방향 데이터 흐름(Unidirectional Data Flow)**. 상태가 어디서 왜 바뀌었는지 역추적이 가능해야, 앱이 커져도 디버깅이 선형적으로 가능하다. 패턴을 안 쓰는 것 자체가 문제가 아니라, "상태를 아무 데서나 바꿀 수 있는 구조"가 문제다.

## 2. 로컬 데이터 계층 — 네트워크가 없어도 앱은 죽지 않아야 한다

### 저장 방식의 선택

| 저장 방식 | 적합한 용도 |
|---|---|
| **관계형 DB** (SQLite, Room, Core Data) | 구조화된 데이터, 복잡한 조회·관계가 필요할 때 |
| **키-값 저장** (SharedPreferences, UserDefaults) | 설정값, 로그인 토큰처럼 단순한 값 |
| **파일 시스템** | 이미지·캐시 파일 등 이진 데이터 |
| **인메모리 캐시** | 세션 동안만 유지되면 되는 임시 데이터 |

### 캐싱 전략

```
캐시가 하는 일 = "다시 계산하거나 다시 받아오는 비용"을 없애는 것
  TTL(Time To Live)   : 일정 시간이 지나면 자동 무효화
  LRU(Least Recently Used) : 공간이 부족하면 가장 안 쓴 데이터부터 삭제
  무효화(Invalidation) : 서버 데이터가 바뀌었을 때 캐시를 강제로 갱신
```

### 오프라인 우선(Offline-first) 설계

네트워크가 불안정한 환경(지하철, 해외 로밍)에서도 앱이 정상 동작해야 한다면, "서버 응답을 받아야 화면이 뜬다"가 아니라 **"로컬 데이터로 먼저 그리고, 서버 동기화는 뒤에서 처리한다"**는 순서로 설계해야 한다. 이때 반드시 따라오는 문제가 **충돌 해결(Conflict Resolution)**이다 — 오프라인 상태에서 로컬과 서버 데이터가 각각 바뀌었다면, 어느 쪽을 우선할지(마지막 수정 시각 기준, 서버 우선, 병합 등) 규칙이 있어야 한다.

## 3. 네트워킹 계층 — 프로토콜은 취향이 아니라 요구사항의 결과다

| 방식 | 적합한 상황 |
|---|---|
| **REST** | 리소스 중심 CRUD, 캐싱이 중요한 일반적인 API |
| **GraphQL** | 화면마다 필요한 데이터 모양이 다양하고, 오버페칭/언더페칭을 줄이고 싶을 때 |
| **WebSocket** | 채팅, 실시간 협업처럼 서버가 먼저 데이터를 밀어줘야 할 때 |
| **gRPC** | 마이크로서비스 간 통신처럼 성능과 스키마 강제가 중요할 때 |

프로토콜 선택보다 앱 안정성에 더 직접적인 영향을 주는 건 아래 디테일들이다.

- **재시도와 백오프**: 네트워크 요청이 실패했을 때 무한 재시도로 서버를 두들기지 않도록 지수 백오프(exponential backoff) 적용
- **인증 토큰 갱신**: 액세스 토큰 만료 시 리프레시 토큰으로 자동 재발급하는 흐름이 없으면, 사용자는 이유 없이 로그아웃당하는 경험을 한다
- **오프라인 큐잉**: 네트워크가 끊긴 동안 발생한 요청(좋아요, 댓글 등)을 큐에 쌓아뒀다가 연결 복구 시 순서대로 재전송

## 4. 동시성과 백그라운드 처리

UI가 멈추지 않으려면 무거운 작업(파일 읽기, 네트워크, 이미지 처리)은 반드시 **메인/UI 스레드 밖**에서 처리해야 한다.

```
스레드 모델의 진화:
  콜백 지옥(Callback Hell) → Promise/Future → async/await → 구조화된 동시성(Structured Concurrency)
                                                            (코루틴, Swift Concurrency 등)
```

구조화된 동시성이 중요한 이유는 **작업의 생명주기를 화면의 생명주기에 묶을 수 있기 때문**이다. 사용자가 화면을 떠났는데 백그라운드에서 계속 돌아가는 작업이 있으면 메모리 누수와 크래시의 원인이 된다. 화면이 사라지면 관련 작업도 자동으로 취소되는 구조가 안전하다.

백그라운드 작업 예약(예: 주기적 동기화, 파일 업로드)에는 OS가 제공하는 전용 API(Android WorkManager, iOS BackgroundTasks, 웹의 Service Worker)를 쓰는 게 원칙이다. 직접 스레드를 띄워 무한정 돌리는 방식은 OS의 배터리 최적화 정책에 걸려 강제 종료당하기 쉽다 — 이 부분은 2편에서 더 자세히 다룬다.

---

## 정리

```
상태 관리   → MVVM/MVI/Redux 등 패턴 이름보다 "단방향 데이터 흐름" 원칙이 핵심
로컬 데이터  → 관계형DB/키값저장/파일/캐시를 용도별로 구분, 오프라인 우선 설계 시 충돌 해결 필수
네트워킹    → REST/GraphQL/WebSocket/gRPC는 요구사항이 결정, 재시도·토큰갱신·오프라인 큐잉이 체감 안정성 좌우
동시성      → 무거운 작업은 UI 스레드 밖으로, 구조화된 동시성으로 화면 생명주기에 작업을 묶어야 안전
```

이 네 가지는 사용자에게 절대 직접 보이지 않는다. 하지만 앱이 "빠르다", "안정적이다", "네트워크 나빠도 잘 된다"고 느껴지는 체감 품질의 8할은 여기서 결정된다. 다음 편에서는 사용자의 감각(청각·촉각)과 OS 시스템에 앱을 통합하는 요소들을 다룬다.
