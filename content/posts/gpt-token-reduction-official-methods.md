지난 글 [클로드에서 토큰을 줄이는 공식 기법들](/posts/claude-token-reduction-official-methods/)에서 정리한 원칙을 그대로 가져온다 — "다른 모델 쓰세요"나 "사람이 중간에서 결과물을 수동으로 옮기세요" 같은 답은 운영 부담을 떠넘기는 것일 뿐 API 자체가 제공하는 절감 기법이 아니다. 여기서는 오픈AI가 API·플랫폼에 실제로 구현해둔, 개발자가 매번 손으로 개입하지 않아도 되는 공식 토큰·비용 절감 기법만 정리했다. 다만 클로드와 겹치지 않는 지점도 여럿이라 그 차이를 위주로 짚었다.

## 1. 자동 프롬프트 캐싱 — 마커를 걸 필요조차 없다

클로드는 `cache_control` 블록을 직접 지정해야 캐싱이 걸리지만, 오픈AI의 프롬프트 캐싱(2024년 10월 데브데이 발표)은 완전 자동이다. 프롬프트가 **1,024토큰 이상**이면 이전 요청과 겹치는 가장 긴 프리픽스를 API가 알아서 찾아 캐시하고, 128토큰 단위로 캐시 범위를 늘려간다. 1,024토큰 미만이면 그냥 캐시가 안 걸릴 뿐이고, 캐시가 실제로 잡혔는지는 응답의 `usage.prompt_tokens_details.cached_tokens`(챗 컴플리션) 또는 `usage.input_tokens_details`(리스폰스 API)로 확인한다.

할인율은 시기별로 달랐다 — 2024년 10월 출시 당시 GPT-4o 계열은 캐시 히트 시 약 **50% 할인**이었는데, 지금(2026년 중반) GPT-5 계열 가격표에서는 캐시된 입력 토큰이 **90% 할인**(표준 입력가의 10%만 과금)으로 뛰었다. 예를 들어 가격표상 gpt-5.6 계열 상위 모델은 표준 입력가 100만 토큰당 5달러인데 캐시 히트 시 0.5달러로 떨어지는 식이다.

GPT-5.6 계열부터는 캐시 적중률을 높이려고 요청마다 동일하게 지정하는 `prompt_cache_key` 파라미터가 생겼고, `prompt_cache_options: {"mode": "explicit"}`로 캐시 경계를 직접 지정하는 것도 가능해졌다 — 완전 자동이던 초기 설계가 클로드식 명시적 제어 쪽으로 조금씩 수렴하는 셈이다. 기본 TTL은 GPT-5.6 계열 기준 `prompt_cache_options.ttl`로 **30분**이 문서화된 유일한 값이고, 이전 세대 모델들은 대략 5~10분 비활성 시 만료·최대 1시간 정도로 알려져 있었다(오픈AI 공식 문서보다는 커뮤니티 보고에 가까운 수치라 참고용).

한 가지 반전은 캐시 "쓰기" 비용이다 — GPT-5.6 계열부터는 캐시를 처음 채우는 쪽(cache write)에 **표준 입력가의 1.25배**가 붙는 `cache_write_tokens` 항목이 새로 생겼다. 이전 세대는 캐시 쓰기가 무료였다. 즉 "캐싱은 공짜로 켜두면 무조건 이득"이라는 2024~2025년식 설명이 최신 모델에는 더 이상 정확하지 않다 — 첫 호출에 프리미엄을 내는 대신 이후 히트마다 90% 할인을 받는 구조이므로, 같은 프리픽스를 충분히 재사용할 때만 순이익이 난다.

## 2. 배치 API — 할인율은 그대로, 처리 범위는 넓어졌다

`POST /v1/batches`는 지금도 **정확히 50% 할인**이다 — 2024년 출시 이후 이 수치는 안 바뀌었다는 점이 흥미롭다(캐싱 할인은 50%에서 90%로 올랐는데 배치는 그대로다). 지원 엔드포인트는 `/v1/responses`·`/v1/chat/completions`·`/v1/embeddings`·`/v1/completions`·`/v1/moderations`·`/v1/images/generations`·`/v1/images/edits`, 그리고 최근 추가된 `/v1/videos`까지 넓어졌다. 배치 하나에 요청 5만 건 또는 입력 파일 200MB 중 먼저 도달하는 한도까지 담을 수 있고, 배치 생성 자체는 시간당 2,000건으로 제한된다. 결과는 24시간 안에 보장되고(실제로는 대부분 그보다 훨씬 빠르다), 결과 파일은 완료 후 30일 뒤 자동 삭제된다. 배치 안에서도 프롬프트 캐싱 할인이 별도로 얹히므로, 반복 프리픽스가 있는 대량 작업이면 두 할인이 중첩된다.

## 3. 추론 강도(`reasoning_effort`)와 응답 길이(`verbosity`) — 두 축이 분리돼 있다

o시리즈·GPT-5 계열 추론 모델은 `reasoning_effort` 파라미터로 `minimal`/`low`/`medium`/`high` 네 단계를 고를 수 있다(기본값 `medium`). `minimal`은 숨겨진 추론 토큰을 거의 쓰지 않도록 설계돼 있어, 분류·포맷 변환처럼 정답이 명확한 작업에 적합하다.

여기에 짝을 이루는 게 클로드에는 없는 `verbosity` 파라미터다(`low`/`medium`/`high`, 기본값 `medium`) — 추론 깊이와 별개로 최종 응답의 길이·상세도만 조절한다. 즉 리스폰스 API에서 `reasoning: {"effort": "minimal"}`과 `text: {"verbosity": "low"}`를 따로 조합할 수 있어, "깊게 생각하되 짧게 답하기"나 "가볍게 생각하되 자세히 답하기" 같은 조합이 가능하다. `reasoning_effort`는 숨겨진 추론 토큰(과금되지만 화면엔 안 보이는)을 줄이고, `verbosity`는 눈에 보이는 출력 토큰 자체를 줄인다는 점에서 두 파라미터가 서로 다른 비용 축을 건드린다.

## 4. 예측 출력(Predicted Outputs) — 레거시 모델 전용으로 남았다

2024년 11월 나온 기능으로, `prediction: {"type": "content", "content": "<예상 텍스트>"}`를 요청에 함께 보내면 코드 편집처럼 결과 대부분이 입력과 겹치는 작업에서 추측 디코딩과 비슷한 방식으로 생성을 빠르고 싸게 만든다. 다만 **GPT-5나 추론 모델로는 확장되지 않았고, gpt-4o·gpt-4o-mini·gpt-4.1 계열에만 남아 있는 레거시 기능**이다. `n`이나 `logprobs`, 0이 아닌 페널티 파라미터, 도구 호출, 텍스트 외 모달리티, `max_completion_tokens`와는 함께 쓸 수 없다. 주의할 점은 예측이 틀리면 거절된 예측 토큰도 정상 출력 토큰 요금 그대로 과금된다는 것 — 응답의 `accepted_prediction_tokens`/`rejected_prediction_tokens`로 얼마나 맞았는지 확인할 수 있는데, 예측이 자주 빗나가면 오히려 비용이 늘어날 수 있다.

## 5. 서비스 티어(Flex/Priority) — 반대 방향 두 가지 트레이드오프

`service_tier` 파라미터로 `flex`·`priority`·`default` 중 고를 수 있다. **flex**는 배치 API와 같은 수준(약 50% 할인)으로 가격이 내려가는 대신 지연이 느려지고 최선노력(best-effort) 처리라 기본 10분 타임아웃 안에 `429 Resource Unavailable`로 실패할 수도 있다 — 이 경우 과금되지 않는다. 실시간성이 필요 없는 비동기 워크로드에 적합하다. 반대로 **priority**는 표준가의 약 2배(가격표 기준 상위 모델이 100만 토큰당 5달러 → 10달러)를 내는 대신 지연이 더 낮고 일정하게 유지된다 — 두 티어 모두 프롬프트 캐싱 할인은 별도로 적용된다. priority는 트래픽이 15분 안에 50% 이상 급증하면 자동으로 표준 티어로 강등될 수 있다는 안전장치도 있다.

## 6. `previous_response_id` — 이건 사실 별도 할인이 아니다

리스폰스 API의 `previous_response_id`(+`store: true`)를 쓰면 이전 대화 전체를 매번 배열로 재구성해서 보낼 필요 없이 서버가 대화 상태를 들고 있어준다. 그런데 여기서 중요한 건 **이 기능 자체가 과금되는 입력 토큰 수를 줄여주지는 않는다**는 점이다 — 오픈AI는 여전히 재구성된 전체 컨텍스트(시스템 프롬프트+이전 턴)를 입력 토큰으로 과금한다. 실제 절감 효과는 이 기능이 만들어내는 프리픽스가 턴마다 바이트 단위로 동일해서 위의 자동 프롬프트 캐싱이 걸리기 딱 좋은 조건을 만들어준다는 데서 나온다 — 즉 `previous_response_id`는 그 자체로 할인 메커니즘이 아니라, 캐싱 할인을 안정적으로 받게 해주는 개발 편의 기능으로 이해하는 게 정확하다.

## 7. GPT-5 "라우터" — API 개발자에게는 해당 사항 없음

챗GPT(소비자 앱)에는 대화 신호나 도구 필요 여부, "깊게 생각해줘" 같은 사용자 의도에 따라 가볍고 빠른 경로와 깊은 추론 경로를 자동으로 골라주는 실시간 라우터가 있다. 하지만 이건 **챗GPT 제품 UX 기능이지 API 파라미터가 아니다** — API를 쓰는 개발자는 이런 자동 라우팅 혜택을 받지 못하고, `gpt-5`/`gpt-5-mini`/`gpt-5-nano`(또는 현재 가격표의 gpt-5.6 계열) 중 모델을 직접 골라야 하며 비용 조절은 앞서 나온 `reasoning_effort`·`verbosity`로 명시적으로 해야 한다.

## 8. 토큰 카운팅 — 오픈AI도 결국 무료 호스팅 엔드포인트를 냈다

한동안 오픈AI는 클로드의 `count_tokens` 같은 무료 호스팅 측정 엔드포인트가 없어서, 로컬에서 도는 `tiktoken` 라이브러리로 텍스트만 대략 셀 수 있었다(이미지·파일·도구 스키마는 tiktoken으로 정확히 세기 어렵다). 그런데 2026년 현재 오픈AI도 `POST /v1/responses/input_tokens` 엔드포인트를 냈다 — 리스폰스 API와 동일한 입력 형식(텍스트·메시지 배열·이미지·파일·도구·대화 이력)을 받아 `{"input_tokens": <정수>}`를 반환하는 무료 측정용 엔드포인트로, 이미지·파일·도구 스키마까지 정확히 셀 수 있다는 점에서 tiktoken의 한계를 보완한다. tiktoken은 여전히 오프라인·무료 텍스트 카운팅용 표준 도구로 남아 있지만, "오픈AI엔 호스팅 카운팅 엔드포인트가 없다"는 설명은 더 이상 맞지 않는다.

## 9. (보너스) 구조화 출력과 이미지 디테일 — 간접적이지만 실질적인 절감

`response_format`/`text.format`에 `strict: true` JSON 스키마를 지정하는 구조화 출력(2024년 8월 출시)은 그 자체로 할인 기법은 아니지만, 느슨한 JSON 모드에서 흔했던 스키마 불일치 실패율을 크게 낮춰 재시도로 낭비되는 왕복 토큰을 줄여준다. 이미지 입력에서는 `detail` 파라미터(`low`/`high`/`auto`)로 세밀한 인식이 필요 없는 이미지를 `low`로 낮춰 토큰 소비 자체를 줄일 수 있고, 반복되는 이미지에 대해 `detail` 값을 동일하게 맞춰야 위 1번의 캐싱 대상이 된다는 점도 챙길 만하다. 참고로 o시리즈 이후 모델은 숨겨진 추론 토큰까지 포함해 총 생성량을 제한하는 `max_completion_tokens`만 쓸 수 있고, 레거시 `max_tokens`는 gpt-3.5-turbo 같은 구형 모델에서만 동작한다.

## 정리

| 기법 | 핵심 파라미터 | 효과 |
|---|---|---|
| 자동 프롬프트 캐싱 | `prompt_cache_key`, `prompt_cache_options` | 캐시 히트 시 입력가의 10%(90% 할인), GPT-5.6+는 쓰기에 1.25배 프리미엄 |
| 배치 API | `POST /v1/batches` | 전체 50% 할인, 24시간 내 처리 |
| 추론 강도 | `reasoning_effort` | 낮출수록 숨겨진 추론 토큰 감소 |
| 응답 길이 | `verbosity` | 추론 깊이와 별개로 출력 토큰 조절 |
| 예측 출력 | `prediction` | GPT-4o/4.1 전용, 예측 실패 시 오히려 손해 |
| Flex 티어 | `service_tier: "flex"` | 배치 수준 할인, 느린 최선노력 처리 |
| Priority 티어 | `service_tier: "priority"` | 약 2배 가격, 낮고 안정적인 지연 |
| 토큰 카운팅 | `POST /v1/responses/input_tokens` | 무료, 이미지·도구 스키마까지 정확 측정 |

클로드 쪽과 비교하면 비슷한 축(자동 캐싱, 배치 50%, 추론 강도 조절, 무료 토큰 카운팅)이 대부분 대응되지만, 오픈AI 쪽엔 클로드의 컨텍스트 압축(Compaction)이나 컨텍스트 편집(Context Editing)에 정확히 대응하는 공식 기능이 보이지 않는다는 게 눈에 띄는 차이다. 반대로 `verbosity`처럼 추론 깊이와 출력 길이를 별도 축으로 쪼갠 파라미터나, flex/priority처럼 지연-비용을 양방향으로 조절하는 티어는 오픈AI 쪽에만 있다. 실무에서는 자동 캐싱을 기본으로 깔고(단, GPT-5.6 이후 캐시 쓰기 프리미엄을 감안해 재사용 빈도가 낮은 프리픽스엔 굳이 걸 필요 없다), 실시간이 필요 없는 대량 작업은 배치나 flex로, 작업 난이도에 맞춰 reasoning_effort와 verbosity를 따로 조절하는 조합이 가장 자연스럽다.
