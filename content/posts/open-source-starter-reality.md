"나도 유명한 오픈소스 하나 만들고 싶다"는 생각, 개발자라면 한 번쯤 해봤을 것이다. 그런데 이 문장을 조금만 뜯어보면 이상한 게 보인다 — 대부분은 "만들고 싶다"고 말하면서 실제로는 "이미 완성돼서 사람들이 써주는 상태"를 원한다. 그건 닭을 키우지 않고 알만 얻으려는 것과 다르지 않다. 냉정하게 시작해서, 그래도 하고 싶은 사람을 위한 현실적인 경로까지 정리한다.

---

## 1. 냉정하게 시작하자

GitHub에는 매일 수많은 저장소가 새로 만들어진다. 그중 압도적 다수는 커밋 몇 번으로 멈추고, 스타 하나 못 받은 채 조용히 방치된다. "만들었다"와 "오픈소스 프로젝트가 됐다" 사이에는 대부분의 사람이 넘지 못하는 큰 간극이 있다.

```
알(결과물)만 원하는 태도  : 코드를 공개하면 사람들이 알아서 쓰고, 기여하고, 유명해진다
닭(과정)을 키우는 태도    : 코드 공개는 시작일 뿐, 그 뒤로 몇 달~몇 년의 무보수 노동이 이어진다
```

시작하기 전에 스스로에게 먼저 물어야 할 질문이 있다. **왜 오픈소스를 만들고 싶은가?**

- 이력서에 한 줄 넣고 싶어서인가
- 유명해지고 싶어서인가
- 아니면 정말로 풀고 싶은 문제가 있어서인가

이 답에 따라 이 글 뒤의 내용을 받아들이는 방식이 완전히 달라진다. 앞의 두 동기라면 뒤에 나올 현실이 꽤 실망스러울 것이고, 세 번째 동기라면 오히려 지금부터가 진짜 유용한 이야기다.

---

## 2. "만든다"는 것의 실체 — 코드는 시작의 일부일 뿐

오픈소스를 "운영"하는 데 필요한 능력은 코딩 실력 하나가 아니다.

| 필요 능력 | 구체적으로 무엇인가 |
|---|---|
| **기술력** | 코드 품질, 아키텍처 설계, 테스트 — 남이 봐도 이해할 수 있는 수준 |
| **문서화 능력** | README, 설치 가이드, 기여 가이드(CONTRIBUTING.md) — 문서가 없으면 아무도 못 씀 |
| **커뮤니케이션·거버넌스** | 낯선 사람이 던진 이슈·PR에 매번 대응, 의견 충돌 조정, 무례한 댓글도 감내 |
| **노출·마케팅** | 아무리 잘 만들어도 아무도 모르면 그냥 개인 저장소다 — 알리는 것도 일이다 |
| **라이선스·법적 이해** | 어떤 라이선스를 쓸지, 기여자 라이선스 계약(CLA)이 필요한지 판단 |
| **지구력** | 위 모든 걸 무보수로, 기약 없이 계속할 수 있는가 |

이 중 가장 과소평가되는 건 마지막, **지구력**이다. 오픈소스 유지보수는 대부분 무보수이고 끝이 정해져 있지 않다. 이 지점에서 실제로 터진 사건들이 있다.

- **event-stream 사건(2018)**: 주당 수백만 다운로드를 기록하던 npm 패키지의 원작자가 유지보수에 지쳐 낯선 사람에게 관리 권한을 넘겼는데, 그 사람이 악성 코드를 몰래 심어 암호화폐 지갑 라이브러리를 공격하는 사건이 발생했다
- **colors.js·faker.js 사건(2022)**: 두 인기 npm 패키지의 원작자가 "기업들이 무보수 오픈소스 노동으로 이익만 취한다"는 항의로, 자기 손으로 패키지를 망가뜨려 이를 의존하던 수많은 프로젝트를 한꺼번에 멈춰 세웠다

두 사건 모두 기술 문제가 아니라 **"무보수로 계속 유지보수해야 한다는 압박"이 한계에 다다른 결과**였다. 코드를 잘 짜는 능력과, 그 코드를 몇 년간 책임지는 능력은 완전히 다른 종류의 역량이다.

## 3. 그래도 하고 싶다면 — 전제 조건 체크리스트

```
체크 1: 이미 실무에서 반복적으로 부딪힌 "진짜 내 문제"가 있는가
  → 성공한 오픈소스 대부분은 저자가 자기 문제를 풀려다 나온 부산물이다
    리눅스(토르발스가 자기 PC용 유닉스 계열 OS가 필요해서)
    깃(토르발스가 커널 버전 관리 도구가 필요해서)
    레디스(산필리포가 자기 스타트업의 실시간 분석 문제를 풀려고)
    레일즈(DHH가 베이스캠프 개발 과정에서 뽑아낸 프레임워크)
  → "재밌어 보여서" 만든 프로젝트보다 생존율이 훨씬 높다

체크 2: 이미 존재하는 대안을 다 찾아봤는가
  → 비슷한 게 이미 있다면, 새로 만들기보다 그 프로젝트에 기여하는 게
    훨씬 적은 노력으로 더 큰 영향력을 낼 수 있다

체크 3: 최소 6개월~1년, 무보수로 유지보수할 각오가 있는가
  → 이 각오가 없다면 애초에 "운영"이 아니라 "공개"만 목표로 잡는 게 맞다
```

## 4. 실전 경로 — 능력별 분기

모두가 처음부터 창시자가 될 필요는 없다. 자신의 강점에 따라 시작점을 다르게 잡는 게 현실적이다.

```
코드 실력이 강점인 경우
  → 거창한 프레임워크 말고, 아주 작은 문제 하나만 확실히 푸는 라이브러리로 시작
  → 범위가 작을수록 유지보수 부담도 작고, 완성도도 높이기 쉽다

커뮤니티·기획력이 강점인 경우
  → 처음부터 창시자가 되려 하지 말고, 기존 프로젝트의 기여자로 시작
  → 이슈 트리아지, 문서 정리, 리뷰만 해도 충분한 기여이고,
    메인테이너의 실제 고충을 창시자가 되기 전에 미리 체감할 수 있다

지구력에 자신이 없는 경우
  → "완전한 커뮤니티 운영"까지 갈 필요는 없다
  → 내가 필요해서 만든 개인 도구를 그냥 공개해두는 것도 오픈소스는 오픈소스다
  → 기대치를 "많은 사람이 쓰는 프로젝트"가 아니라 "필요한 사람이 찾아 쓸 수 있는 상태"로 낮추는 것도 정답이다
```

## 정리

```
냉정한 진입    → "만들고 싶다"는 대부분 "이미 완성된 상태를 원한다"는 뜻 — 동기 점검이 먼저
만든다는 것의 실체→ 코드는 시작일 뿐, 문서화·커뮤니케이션·노출·법적 이해·지구력까지 필요
                  event-stream·colors.js/faker.js 사건 — 무보수 지구력 한계가 실제로 터진 사례
전제 조건      → 진짜 내 문제인가, 대안 조사는 끝났는가, 6개월~1년 무보수 각오가 있는가
실전 경로      → 코드형은 작게 시작, 기획형은 기여자로 시작, 지구력 약하면 기대치를 낮춰 공개만
```

"알을 얻으려면 닭부터 키워야 한다"는 말의 진짜 의미는, 오픈소스가 **만드는 순간이 아니라 유지하는 기간으로 정의된다**는 것이다. 그리고 다행인 건, 그 닭을 처음부터 혼자 키울 필요가 없다는 점이다 — 이미 누군가 키우고 있는 닭장에 들어가 함께 돌보는 것도, 충분히 "오픈소스를 하는 것"이다.
