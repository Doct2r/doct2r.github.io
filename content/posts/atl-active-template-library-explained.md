ATL(Active Template Library)을 "MFC 안에 있는 라이브러리"로 기억하는 사람이 많다 — 비주얼 스튜디오 프로젝트 마법사에서 늘 "MFC와 ATL"이 한 세트로 묶여 나오고, 실제로 두 라이브러리를 섞어 쓰는 "MFC/ATL 혼합 프로젝트"도 흔하기 때문이다. 하지만 정확히 말하면 ATL은 MFC의 하위 라이브러리가 아니라, MFC가 COM·ActiveX 컴포넌트를 만들기엔 너무 무겁다는 문제의식에서 별도로 태어난 독립 라이브러리다. 왜 따로 만들어야 했는지, 어떻게 그 가벼움을 실제로 구현했는지 정리했다.

## 1. 오해가 생긴 이유 — 같은 팀, 같은 마법사, 그러나 다른 목적

ATL과 [MFC](/posts/framework-pattern-limits/)는 둘 다 마이크로소프트 비주얼 C++ 팀이 만든 C++ 클래스 라이브러리이고, 프로젝트 마법사 안에서 나란히 제공되며, 2003년 비주얼 스튜디오 .NET(ATL 7.0)부터는 ATL 프로젝트에서 MFC 클래스를 가져다 쓸 수 있게 해주는 "ATLMFC"라는 공유 레이어까지 추가됐다. 이런 표면적 결합 때문에 ATL을 MFC의 부속물로 오해하기 쉽지만, 둘은 애초에 겨냥한 문제 자체가 다르다 — MFC는 1992년부터 윈도우 GUI 애플리케이션 전체(창·메시지 루프·문서/뷰)를 프레임워크로 감싸는 게 목적이었고, ATL은 GUI가 아니라 COM 컴포넌트 하나를 최대한 작고 빠르게 만드는 게 목적이었다. 마법사가 둘을 나란히 보여주는 건 "관련이 있어서"이지 "포함 관계라서"가 아니다.

## 2. 태어난 이유 — 웹에 띄우기엔 MFC 기반 ActiveX 컨트롤이 너무 무거웠다

ATL은 1997년 2월 비주얼 C++ 5.0 프로페셔널 에디션과 함께 정식 출시됐다. COM 객체는 이미 MFC로도 만들 수 있었지만, MFC로 COM·ActiveX 컨트롤을 만들면 실행 파일이 MFC 런타임 지원 DLL(예: MFC42.DLL)에 의존하는 훨씬 큰 바이너리가 됐다. 당시는 인터넷 익스플로러 웹페이지에 ActiveX 컨트롤을 심어 다이얼업 모뎀으로 다운로드하게 하던 시절이었다는 걸 감안하면, 이 바이너리 크기 차이는 추상적 효율 문제가 아니라 사용자가 실제로 느끼는 로딩 지연이었다. ATL은 이 문제를 정면으로 겨냥해, MFC나 순수 COM을 직접 손으로 짜는 것보다 실행 속도는 빠르고 바이너리는 작은 COM 컴포넌트를 만들 수 있게 설계됐다.

## 3. 설계 — 다중상속과 CRTP로 vtable 오버헤드를 피하다

ATL이 가벼움을 실제로 구현한 핵심 기법은 특이 재귀 템플릿 패턴(CRTP, Curiously Recurring Template Pattern)과 다중 상속의 조합이다. [MFC의 메시지맵](/posts/framework-pattern-limits/)이 매크로로 런타임 디스패치 테이블을 만들어 메시지를 처리하듯, ATL도 `BEGIN_COM_MAP`/`COM_INTERFACE_ENTRY` 같은 매크로로 한 클래스가 어떤 COM 인터페이스들을 구현하는지 선언한다. 다만 결정적 차이는, ATL 클래스가 `CComObjectRootEx<T>`처럼 자기 자신을 템플릿 인자로 넘기는 CRTP 구조를 쓴다는 점이다 — 이 덕분에 어떤 기능이 필요한지(스레딩 모델, 참조 카운팅 방식 등)를 컴파일 타임에 조합해 정하고, 가상 함수 오버헤드 없이 딱 필요한 코드만 인라인·정적으로 붙여넣을 수 있다. MFC가 "모든 걸 포함한 무거운 기반 클래스 하나"를 상속받는 방식이라면, ATL은 "필요한 조각만 템플릿으로 조립"하는 방식으로 같은 문제(코드 재사용)를 정반대 전략으로 풀었다.

## 4. 맹점 — ATL엔 원래 창(window)이 없다

ATL은 철저히 COM 컴포넌트 자체에 집중한 라이브러리라, 정작 윈도우·메시지 루프 같은 GUI 요소는 처음부터 범위 밖이었다. 이 공백을 메운 게 ATL 개발에 참여했던 마이크로소프트 엔지니어 네나드 스테파노비치(Nenad Stefanovic)가 1990년대 후반 사내용으로 만든 WTL(Windows Template Library)이다 — MFC 없이도 ATL 위에서 대화상자·컨트롤 같은 UI를 가볍게 그릴 수 있게 해주는, 사실상 "ATL의 GUI 확장판"이었다. 마이크로소프트는 이를 비주얼 스튜디오·Win32 프레임워크 SDK의 비공식·무지원 애드온으로만 배포하다가, 2004년 전체 소스코드를 커먼퍼블릭라이선스(Common Public License)로 소스포지(SourceForge)에 공개했다. 마이크로소프트는 2007년 WTL 8.0 릴리스 이후 직접 관여를 접었지만, 스테파노비치 본인이 이후로도 계속 유지보수를 이어왔다.

## 5. 버전의 궤적 — 무거워졌다가 다시 가벼워지다

ATL 3.0(1998년 비주얼 C++ 6.0)은 초기 형태가 자리 잡은 버전이었고, 이후 2003년 비주얼 스튜디오 .NET에서 나온 ATL 7.0은 앞서 언급한 ATLMFC 공유 레이어와 함께 C++ 속성(attribute) 기반 코드 생성을 도입했다 — 어트리뷰트로 COM 관련 상용구 코드를 자동 생성하려는 시도였지만 디버깅이 어렵다는 불만이 컸고, 2005년 비주얼 스튜디오 2005의 ATL 8.0에서는 마법사가 기본적으로 어트리뷰트 코드를 더는 생성하지 않게 되며 사실상 폐기 수순을 밟았다. 이후 2013년 비주얼 스튜디오 2013(ATL 12 전후)부터는 ATL 코드 전체가 정적 링크되도록 바뀌어, 애초에 ATL이 없애려 했던 것과 같은 종류의 문제(별도 지원 DLL 의존성)를 이번엔 ATL 자신에게서 완전히 없앴다.

## 6. 지금 — 클래식 COM이 남아 있는 한 함께 남는다

지금도 비주얼 스튜디오는 ATL을 계속 포함해 배포하며, [.NET Native AoT·WinUI3로의 마이그레이션](/posts/mfc-to-winui3/)을 다루는 최신 윈도우 개발 논의에서도 ATL은 여전히 등장한다 — 상당수 레거시 MFC 애플리케이션이 내부적으로 ATL 기반 COM 컴포넌트에 의존하고 있기 때문이다. 마이크로소프트가 최신 COM(WinRT) 상호운용을 위해 내놓은 C++/WinRT가 새 프로젝트에서는 ATL의 역할 일부를 대체하고 있지만, 수십 년 묵은 클래식 COM·ActiveX·OLE 인터롭 코드베이스가 존재하는 한 ATL은 그 코드를 유지보수할 유일한 실질적 선택지로 남아 있다.

## 정리

ATL은 MFC의 하위 기능이 아니라, "MFC로 COM 컴포넌트를 만들면 웹에 띄우기엔 너무 무겁다"는 1990년대 중반의 구체적 문제(ActiveX 컨트롤의 다운로드 크기)에서 출발한 별도 라이브러리였다. CRTP와 다중 상속으로 가상 함수 오버헤드 없이 필요한 기능만 조립하는 설계가 그 가벼움의 실체였고, GUI가 빠진 자리는 훗날 WTL이라는 별도 프로젝트로 메워졌다. 이후 어트리뷰트 기반 코드 생성처럼 오히려 무거워지는 방향으로 갔다가 되돌아온 우여곡절을 거쳐, 2013년 이후에는 정적 링크로 DLL 의존성 자체를 없애며 원래의 문제의식으로 돌아왔다. ATL과 MFC가 프로젝트 마법사에서 나란히 묶여 있는 건 두 라이브러리가 서로를 감싸고 있어서가 아니라, 같은 팀이 GUI(MFC)와 COM 컴포넌트(ATL)라는 서로 다른 문제를 각자 다른 전략으로 풀어놓고 이후 필요할 때 섞어 쓸 수 있게 다리를 놓아둔 결과다.
