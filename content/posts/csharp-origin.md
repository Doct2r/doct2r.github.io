C#은 처음부터 "더 나은 언어를 만들자"는 순수한 동기로 시작하지 않았다. 시작은 소송이었다. 마이크로소프트가 자바를 자기 방식대로 바꿔 쓰려다 선(Sun Microsystems)에게 고소당했고, 그 결과 자바를 계속 쓸 수 없게 된 자리에서 C#이 태어났다. 이 탄생 배경을 알아야 C#이 왜 지금 같은 모양을 하고 있는지도 이해할 수 있다.

---

## 1. 발단 — 자바와의 소송

1990년대 중반, 자바가 "한 번 작성하면 어디서든 실행된다(Write Once, Run Anywhere)"는 구호로 업계를 흔들자 마이크로소프트도 자바 라이선스를 받아 자체 자바 가상머신과 개발 도구인 **Visual J++**를 내놨다.

문제는 마이크로소프트가 여기에 **J/Direct, WFC(Windows Foundation Classes)** 같은 윈도우 전용 확장 기능을 몰래 얹었다는 것이다. 이 확장으로 짠 코드는 윈도우에서만 돌아가고 다른 플랫폼에서는 실행되지 않았다 — 자바의 존재 이유 자체를 무력화하는 방향이었다. 선은 이를 라이선스 계약 위반으로 보고 **1997년 마이크로소프트를 고소**했다.

법정 다툼은 2001년 합의로 끝났다. 마이크로소프트는 배상금을 지불했고, 결국 자체 자바 가상머신 배포를 단계적으로 중단해야 했다(2004년 완전 종료). 마이크로소프트 입장에서는 이제 **윈도우 생태계에서 쓸 수 있는 "자기 소유의" 현대적 언어**가 필요해졌다.

---

## 2. 그래서 만든 대안 — Cool에서 C#으로

마이크로소프트는 1996년, **덴마크 출신 개발자 앤더스 헤일스버그(Anders Hejlsberg)**를 영입했다. 그는 이전에 볼랜드(Borland)에서 **터보 파스칼(Turbo Pascal)**과 **델파이(Delphi)**를 설계한 인물로, 생산성 높은 언어와 개발 도구를 만드는 데 이미 정평이 나 있었다.

헤일스버그 팀이 개발한 새 언어는 내부적으로 **"Cool"(C-like Object Oriented Language)**이라는 코드명으로 불렸다. 이 프로젝트는 2000년 **.NET 플랫폼**과 함께 **C#**이라는 이름으로 공개됐고, 2002년 정식 출시됐다.

```
1997: 마이크로소프트, Java 라이선스 위반으로 Sun에게 피소
2001: 소송 합의, MS 자체 JVM(Visual J++) 단계적 폐기 확정
2000: 내부 코드명 "Cool" → C#으로 공개 발표 (.NET과 함께)
2002: C# 1.0 및 .NET Framework 1.0 정식 출시
```

---

## 3. 설계 목표 — C++의 힘과 자바의 안전함 사이

C#의 문법이 C++과 자바 둘 다와 닮아 보이는 건 우연이 아니다. 설계 목표 자체가 **두 언어의 단점을 서로의 장점으로 메우는 것**이었다.

| 문제였던 것 | C#의 해법 |
|---|---|
| C++의 수동 메모리 관리(직접 delete 필요, 메모리 누수 위험) | 가비지 컬렉션 도입 — 자바처럼 자동 메모리 관리 |
| C++의 헤더 파일 분리, 취약한 타입 시스템 | 단일 파일 클래스 정의, 강타입·타입 안전성 강화 |
| 자바에 없던 편의 기능 | **프로퍼티**(getter/setter를 문법으로 흡수), **이벤트·델리게이트**(함수를 값으로 다루는 콜백 체계), **연산자 오버로딩**, **구조체(값 타입)** |

특히 프로퍼티·이벤트·델리게이트는 당시 자바에는 없던 기능으로, 헤일스버그가 델파이에서 이미 검증했던 개념을 C#에 그대로 옮겨온 것이다. "자바를 베꼈다"는 초기 비판과 별개로, C#은 처음부터 자바보다 한발 더 개발자 편의를 챙기는 방향으로 설계됐다.

---

## 4. 언어보다 플랫폼이 목적이었다 — CLR과 .NET

C#만 따로 떼어 보면 잘 만든 언어 정도지만, 마이크로소프트의 진짜 목표는 언어가 아니라 **.NET이라는 플랫폼 자체**였다.

```
기존 COM(Component Object Model)의 문제
  → 바이너리 호환성은 있지만 다루기 매우 복잡함(IDL, 레퍼런스 카운팅 수동 관리 등)
  → C++ 개발자에게도 진입장벽이 높았음

.NET의 CLR(Common Language Runtime)
  → 어셈블리·메타데이터 기반으로 컴포넌트 모델을 단순화
  → 처음부터 "여러 언어가 하나의 런타임 위에서 상호 운용"되도록 설계
    (C#, VB.NET, C++/CLI, 이후 F# 등)
```

자바의 JVM이 애초에 "자바 전용"으로 출발했던 것과 달리, CLR은 설계 단계부터 **다국어 런타임**을 목표로 했다. C#은 이 플랫폼을 가장 잘 보여주기 위해 새로 설계된 "쇼케이스 언어"에 가까웠다 — 이는 이후 시리즈에서 다룬 픽셀과 안드로이드의 관계와 비슷한 구도다: 플랫폼을 파는 회사가 그 플랫폼의 이상적인 모습을 보여주기 위해 직접 만든 결과물.

---

## 5. 표준화 — 소송에서 배운 교훈

선이 자바를 독점적으로 통제하다 마이크로소프트와 충돌했던 전례가 있었기 때문에, 마이크로소프트는 C#을 처음부터 다르게 다뤘다. **C#의 언어 명세와 CLI(Common Language Infrastructure)를 ECMA와 ISO에 정식 표준으로 제출**한 것이다(ECMA-334, ISO/IEC 23270). 이는 당시 반독점 감시 속에 있던 마이크로소프트가 "이번엔 독점적으로 가두지 않는다"는 신호를 시장에 보내야 했던 사정과도 맞물린다.

이 결정 덕분에 이후 모노(Mono) 프로젝트 같은 제3자 구현체가 리눅스·macOS에서도 C#을 돌릴 수 있는 길이 일찍부터 열렸다 — 훗날 C#이 크로스플랫폼 언어로 전환하는 데 중요한 밑거름이 됐다.

---

## 6. 그 후 — 소송의 자식이 오픈소스가 되기까지

C#은 태생부터 "윈도우 전용 자바 대체재"였지만, 지금은 완전히 다른 정체성을 갖고 있다.

```
2014  : 컴파일러(Roslyn) 오픈소스 공개
2016  : .NET Core로 리눅스·macOS 네이티브 지원, .NET Foundation 출범
지금  : 유니티(Unity) 게임 엔진의 주력 스크립팅 언어,
        Xamarin/MAUI로 모바일까지 확장, 완전한 크로스플랫폼·오픈소스 생태계
```

소송을 피하려고 시작된 "마이크로소프트만의 자바"가 20여 년 뒤에는 마이크로소프트의 울타리를 벗어나 리눅스 서버부터 게임 엔진까지 쓰이는 범용 언어가 됐다는 점은, 이 언어의 출발점을 알고 나면 더 흥미롭게 다가온다.

---

## 정리

```
발단        → 마이크로소프트의 Visual J++ 자바 라이선스 위반, 1997년 Sun의 소송, 2001년 합의로 자체 JVM 폐기
대안 개발    → 델파이 설계자 앤더스 헤일스버그 영입, 내부명 "Cool" → 2000년 C#·.NET 공개, 2002년 정식 출시
설계 목표    → C++의 힘(성능)과 자바의 안전함(가비지컬렉션) 절충, 프로퍼티·이벤트·델리게이트로 자바보다 앞서 나감
진짜 목적    → 언어가 아니라 CLR/.NET 플랫폼 — 여러 언어를 한 런타임에서 돌리는 것이 핵심 목표
표준화       → ECMA/ISO 표준 제출 — 자바처럼 독점 통제해 소송당한 전례를 반복하지 않기 위한 전략
현재         → Roslyn 오픈소스화, .NET Core로 크로스플랫폼 전환, 유니티 등에서 광범위 사용
```

C#의 역사는 "좋은 기술은 순수한 이상에서만 나온다"는 통념을 깨는 사례다. 소송에서 밀려 자바를 못 쓰게 된 위기가, 오히려 자바의 단점을 반면교사 삼아 더 다듬어진 언어와, 처음부터 다국어·표준 기반으로 설계된 플랫폼을 낳았다. 지금 C#을 쓰는 대다수는 이 언어가 소송의 부산물이라는 사실조차 모르지만, 그 배경이 지금 C#의 설계 곳곳에 여전히 남아 있다.
