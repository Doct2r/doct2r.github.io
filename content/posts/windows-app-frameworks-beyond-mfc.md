윈도우 네이티브 앱을 새로 시작할 때 실질적인 첫 질문은 "무엇으로 만들 것인가"가 아니라 "MFC로 만들지 않으려면 무엇을 감수해야 하는가"다. [MFC](/posts/framework-pattern-limits/)는 여전히 안정적이고 유지보수는 되지만, [GDI 기반 렌더링](/posts/gdi-gdiplus-explained/)과 1990년대식 리소스 에디터에 묶여 있어 실제 디자인 작업은 고통스럽다. 그 자리를 대신할 후보들을 각각의 트레이드오프와 함께 쭉 훑었다.

## 1. WinForms(2002) — 가장 순한 첫걸음, 그러나 결국 GDI+ 그대로

닷넷 프레임워크 1.0과 함께 2002년 2월 나온 WinForms는 MFC에서 옮겨오기 가장 쉬운 선택지다. C#이라는 현대적 언어, 가비지 컬렉션, 훨씬 나은 디자이너(드래그 앤 드롭 폼 편집)를 준다는 점에서 확실히 MFC보다 낫다. 하지만 결국 내부적으로는 [GDI+를 그대로 감싼 것](/posts/gdi-gdiplus-explained/)이라, 데이터 바인딩은 있지만 얕고, 스타일 시스템(CSS 같은 일관된 테마 언어) 자체가 없어 룩앤필을 근본적으로 바꾸려면 결국 컨트롤을 손으로 다시 그려야 한다 — "MFC보다 편해졌다"이지 "디자인의 자유"까지 온 건 아니다.

## 2. WPF(2006) — 디자인을 프로그래밍의 일급 시민로 만든 전환점

WPF는 2006년 11월 코드네임 "에이번(Avalon)"으로 닷넷 프레임워크 3.0과 함께 나왔다. 여기서 처음으로 UI가 XAML이라는 선언형 마크업으로 분리되고, 데이터 바인딩·스타일·템플릿·애니메이션이 언어 차원에서 지원되기 시작했다 — 디자이너와 개발자가 각자 XAML과 코드를 따로 작업할 수 있게 된 첫 사례다. 렌더링도 GDI를 벗어나 DirectX 기반으로 옮겨가 부드러운 벡터·애니메이션 표현이 가능해졌다. [MFC의 한계](/posts/framework-pattern-limits/)를 다룰 때 짚었듯, 이런 유연성의 대가로 WPF는 학습 곡선이 가파르고(XAML+MVVM 패턴을 새로 익혀야 함), 단일 UI 스레드 구조라 무거운 작업을 잘못 배치하면 여전히 멈춘다.

## 3. UWP(2015) — 실험이었지만 샌드박스가 발목을 잡았다

윈도우 10과 함께 나온 UWP(Universal Windows Platform)는 "PC·태블릿·Xbox·홀로렌즈에 하나의 앱으로"라는 야심을 내걸었다. 문제는 UWP 앱이 AppContainer라는 엄격한 보안 샌드박스 안에서만 돌아가도록 설계됐다는 점이다 — 파일시스템 접근이 사용자 폴더로 제한되고, 웹캠·마이크 같은 하드웨어도 매번 권한을 물어야 하며, 임의의 Win32 API를 자유롭게 호출할 수 없었다. [윈도우 스마트폰 실패사](/posts/windows-phone-failure-history/)에서 다룬 것과 비슷한 패턴으로, 이 제약은 데스크톱 생산성 앱 개발자들을 붙잡기엔 너무 컸고 UWP는 결국 주류가 되지 못했다.

## 4. WinUI3(2021) — UWP의 후계, 그러나 여전한 마찰

UWP의 UI 계층(WinUI)을 데스크톱 Win32 앱에서도 쓸 수 있게 분리해낸 게 WinUI3다. 2021년 11월 16일 Windows App SDK 1.0 정식 버전과 함께 나왔고, MSIX 패키징 없이도 배포할 수 있게 되면서 UWP의 샌드박스 제약 상당수를 벗어났다. 하지만 [WinUI3 마이그레이션 글](/posts/mfc-to-winui3/)과 [Native AoT 글](/posts/winui3-native-aot/)에서 다뤘듯, MFC 코드와 섞어 쓰려 하면 곧바로 Native AoT 미지원 같은 구체적 벽에 부딪힌다 — 겉모습은 현대적이지만 아직 MFC와의 상호운용이라는 실무 지점에서는 손이 많이 간다.

## 5. Qt — 네이티브 크로스플랫폼, 그러나 라이선스가 진짜 리스크

[Qt 글](/posts/qt-framework-explained/)에서 다뤘듯, Qt는 1995년부터 이어진 성숙한 크로스플랫폼 프레임워크로 Qt Designer라는 검증된 비주얼 디자이너와 QML 선언형 UI를 함께 제공한다. 윈도우뿐 아니라 리눅스·macOS까지 같은 코드베이스로 갈 수 있다는 게 WPF·WinUI3에는 없는 장점이다. 다만 Qt의 진짜 트레이드오프는 기술이 아니라 라이선스에 있다 — 2020년 오프라인 인스톨러 유료 제한, 오픈소스 바이너리도 계정 의무화 같은 조치가 이어졌고, 상용 배포를 하려면 라이선스 비용과 조건을 계속 챙겨야 한다.

## 6. Avalonia — WPF의 DNA를 그대로 크로스플랫폼으로

WPF에 가장 가까운 개발 경험을 크로스플랫폼으로 옮기고 싶다면 애벌로니아(Avalonia)가 있다. 2013년 12월 스티븐 커크가 "퍼스펙스(Perspex)"라는 이름으로 시작한 프로젝트로, WPF 아키텍처를 그대로 개념적 토대로 삼았다. 상표권 문제로 2016년 5월 "Avalonia"로 개명했고 같은 해 8월 첫 정식 릴리스가 나왔다. MIT 라이선스라 Qt 같은 상용 라이선스 부담이 없고, XAML·데이터 바인딩·스타일 시스템까지 WPF 경험을 거의 그대로 가져가면서 윈도우·리눅스·macOS·iOS·안드로이드·웹어셈블리까지 뻗어나간다. 대가는 생태계 크기다 — WPF나 Qt만큼 축적된 서드파티 컨트롤·경험자 풀이 아직 두텁지 않다.

## 7. .NET MAUI — 모바일 우선 유산이 데스크톱에 남긴 흔적

2022년 5월 자마린(Xamarin.Forms)의 후계로 정식 출시된 .NET MAUI는 원래 모바일 앱(iOS·안드로이드)을 위해 설계된 크로스플랫폼 UI 틀을 윈도우·macOS 데스크톱까지 확장한 것이다. 하나의 코드베이스로 모바일과 데스크톱을 동시에 노려야 하는 조직에는 매력적이지만, 애초에 모바일 우선으로 설계된 컨트롤·레이아웃 관성이 데스크톱에서는 어색하게 느껴지는 경우가 많다는 게 실무에서 자주 나오는 불만이다 — 데스크톱 전용 생산성 앱이 목표라면 최우선 후보는 아니다.

## 8. 웹 기술 진영 — Electron·Tauri·Wails

디자인 자유도를 극단까지 밀어붙이고 싶다면 웹 기술로 UI를 그리는 진영이 있다. [Electron](/posts/electron-explained/)은 크로미움을 통째로 번들해 가장 무겁지만 생태계가 가장 크고, [Tauri](/posts/tauri-explained/)는 OS 내장 웹뷰를 써서 훨씬 가볍지만 웹뷰 엔진 파편화(macOS WKWebView, 윈도우 WebView2, 리눅스 WebKitGTK)라는 대가를 치르며, [Wails](/posts/wails-go-framework-explained/)는 같은 OS 웹뷰 전략을 Go로 구현해 자동 TypeScript 생성이라는 강점과 상대적으로 약한 권한 모델을 함께 갖는다. 셋 다 공통적으로 "네이티브 룩이 아니라 웹 UI를 그대로 데스크톱에 심는다"는 접근이라, MFC를 대체한다기보다는 아예 다른 종류의 앱(디자인 자유도가 최우선인 도구)을 만들 때 고르는 선택지에 가깝다.

## 9. 결론 — 무엇을 포기할지가 선택을 결정한다

| 선택지 | 얻는 것 | 포기하는 것 |
|---|---|---|
| WinForms | 가장 쉬운 이전, 익숙한 이벤트 모델 | 결국 GDI+, 얕은 스타일링 |
| WPF | 완전한 데이터 바인딩·스타일·애니메이션 | 가파른 학습 곡선, 단일 UI 스레드 |
| UWP | 통합 플랫폼 야심 | 샌드박스 제약으로 사실상 도태 |
| WinUI3 | 현대적 룩, MSIX 없는 배포 | MFC 상호운용·Native AoT 마찰 |
| Qt | 성숙한 크로스플랫폼, 네이티브 디자이너 | 상용 라이선스 리스크 |
| Avalonia | WPF 경험 그대로 크로스플랫폼, MIT | 아직 얇은 생태계 |
| .NET MAUI | 모바일+데스크톱 통합 코드베이스 | 모바일 우선 설계의 어색함 |
| Electron/Tauri/Wails | 최대의 디자인 자유도 | 네이티브 룩 포기, 각자 다른 무게·파편화 대가 |

## 정리

MFC를 떠나는 결정은 "더 나은 프레임워크를 찾는" 문제가 아니라 "어떤 트레이드오프를 감수할지 먼저 정하는" 문제다. 순수하게 이전 비용을 최소화하고 싶다면 WinForms가 가장 순하지만 결국 GDI+라는 천장을 다시 만나고, 디자인의 자유도 자체가 목적이라면 WPF나 Avalonia가 정답에 가깝지만 학습 곡선이나 생태계 크기라는 대가가 따른다. 크로스플랫폼이 진짜 요구사항이라면 Qt의 성숙함과 라이선스 리스크, Avalonia의 가벼움과 얇은 생태계, 웹 기술 진영의 자유도와 무게 중 하나를 골라야 한다. 결국 "MFC를 이긴다"는 건 하나의 우월한 후계자를 찾는 게 아니라, 자신의 프로젝트가 정확히 무엇을 포기해도 괜찮은지를 먼저 정의하는 일이다.
