"토큰을 줄이는 법"을 검색하면 "다른 모델 쓰세요"나 "소넷으로 초안 만들고 하이쿠에 수동으로 던지세요" 같은 답이 자주 나온다. 하지만 이런 답은 사실 운영 부담을 사람에게 떠넘기는 것일 뿐, 클로드 자체가 제공하는 기능이 아니다. 여기서는 앤스로픽이 API·클로드 코드에 실제로 구현해둔, 사용자가 매번 손으로 개입하지 않아도 자동으로 동작하는 공식 토큰·비용 절감 기법만 정리했다.

## 1. 프롬프트 캐싱 — 반복되는 컨텍스트를 매번 다시 읽지 않는다

가장 기본이면서 효과가 큰 기법이다. 요청 안의 특정 위치까지를 "캐시 경계"로 지정(`cache_control: {"type": "ephemeral"}`)해두면, 그 경계까지의 내용이 프리픽스 일치(prefix match) 방식으로 캐시된다 — 경계 이전 바이트 하나만 바뀌어도 그 뒤 전체가 무효화된다는 점이 핵심 제약이다. 렌더링 순서는 `tools → system → messages`이므로, 시스템 프롬프트의 마지막 블록에 캐시 경계를 걸면 도구 목록과 시스템 프롬프트를 한꺼번에 캐시할 수 있다.

TTL은 두 가지다 — 기본값 5분짜리 `{"type": "ephemeral"}`과, `{"type": "ephemeral", "ttl": "1h"}`로 지정하는 1시간짜리. 가격 구조가 흥미롭다 — 캐시에 쓸 때는 5분 TTL이면 기본 입력가의 1.25배, 1시간 TTL이면 2배를 내지만, 캐시에서 읽을 때는 기본 입력가의 약 0.1배만 낸다. 즉 5분 TTL은 딱 2번째 요청부터 손익분기(1.25+0.1=1.35 vs 캐시 없이 2회×1=2)를 넘기고, 1시간 TTL은 최소 3번은 읽어야(2+0.2=2.2 vs 3) 본전을 뽑는다 — 요청 빈도가 낮고 간격이 길수록 1시간 TTL을 고르는 게 유리하다.

실무 제약도 몇 가지 짚어둘 만하다. 요청당 캐시 경계는 최대 4개까지만 걸 수 있고, 캐시 가능한 최소 프리픽스 길이는 모델별로 다르다(구형 소넷 계열 1024토큰, 파이블5·소넷4.6·구형 하이쿠 2048토큰, 오퍼스 계열·하이쿠4.5는 4096토큰) — 이보다 짧은 프리픽스는 에러 없이 그냥 조용히 캐시되지 않는다. 실제로 캐시가 동작하는지는 응답의 `usage.cache_read_input_tokens`로 확인할 수 있는데, 이 값이 반복 요청에서도 계속 0이라면 시스템 프롬프트에 타임스탬프나 UUID가 섞여 매번 프리픽스가 달라지는 "조용한 무효화" 버그를 의심해야 한다.

## 2. 컨텍스트 압축(Compaction) — 대화가 길어지면 서버가 알아서 요약한다

베타 기능으로, `compact-2026-01-12` 헤더를 붙이면 대화가 일정 길이(기본 트리거 15만 토큰)에 근접할 때 서버가 이전 컨텍스트를 자동으로 요약해준다. 클로드 코드를 써본 사람이라면 이미 익숙한 동작이다 — 대화가 컨텍스트 한도에 가까워지면 시스템이 이전 메시지를 자동으로 압축한다는 안내를 본 적이 있을 텐데, 그게 바로 이 기능이다. 주의할 점은, 압축 블록이 포함된 응답을 다음 요청에 그대로 다시 넣어줘야 한다는 것이다 — 텍스트만 추출해서 넣으면 압축 상태 자체가 사라져 버린다.

## 3. 컨텍스트 편집(Context Editing) — 압축과는 다르다, "요약"이 아니라 "삭제"

역시 베타(`context-management-2025-06-27` 헤더)인 이 기능은 압축과 헷갈리기 쉽지만 동작 방식이 완전히 다르다. 압축은 오래된 내용을 요약해서 남겨두는 반면, 컨텍스트 편집은 오래된 도구 호출 결과나 사고(thinking) 블록을 아예 지워버린다. `clear_tool_uses_20250919`(오래된 도구 결과 삭제, `clear_tool_inputs: true`로 입력 파라미터까지 삭제 가능)와 `clear_thinking_20251015`(사고 블록 삭제) 두 전략이 있다 — 이미 다 쓴 도구 출력이 컨텍스트에 계속 쌓이는 에이전틱 루프에서 특히 효과적이다.

## 4. 추론 강도(Effort)와 적응형 사고 — 얼마나 생각할지를 조절한다

`output_config.effort` 파라미터는 `low`/`medium`/`high`/`xhigh`/`max` 다섯 단계로 사고 깊이와 전체 토큰 소비를 조절한다. 별도 베타 헤더 없이 정식 기능이며, 기본값은 `high`다. 이 값을 낮추면 도구 호출 수가 줄고 더 통합적으로 이뤄지며, 서두 설명이나 확인 문구도 짧아진다 — 단순 분류·짧은 작업이라면 `low`로 내려도 충분한 경우가 많다.

여기에 짝을 이루는 게 `thinking: {"type": "adaptive"}`다. 예전에는 `budget_tokens`로 사고에 쓸 토큰량을 고정값으로 지정해야 했는데, 적응형 사고는 이 값을 클로드가 작업 난이도에 맞춰 스스로 조절하게 한다 — 쉬운 작업엔 적게, 복잡한 작업엔 필요한 만큼만 쓰는 식이라 고정 예산을 과도하게 잡아 낭비하는 일이 줄어든다.

## 5. 지연 로딩 도구(Tool Search) — 도구 설명서를 미리 다 읽지 않아도 된다

도구가 많은 애플리케이션에서는 매 요청마다 모든 도구의 스키마(이름·설명·입력 파라미터)를 전부 컨텍스트에 넣어야 했다. 툴 서치 서버 도구(`tool_search_tool_regex_20251119` 또는 `tool_search_tool_bm25_20251119`)를 선언하고 나머지 도구들을 `defer_loading: true`로 표시해두면, 클로드가 필요할 때 도구 목록을 검색해서 관련 있는 스키마만 그때 불러온다. 이 방식이 특히 좋은 이유는, 새로 불러온 도구 스키마가 기존 프리픽스에 "덧붙는" 것이라 위에서 다룬 프롬프트 캐시를 깨뜨리지 않는다는 점이다 — 도구를 교체하면 캐시가 무효화되지만, 검색으로 추가하는 건 무효화되지 않는다.

## 6. 프로그래매틱 도구 호출(PTC) — 중간 결과를 컨텍스트로 돌려보내지 않는다

일반적인 도구 호출은 매번 왕복이 발생한다 — 클로드가 도구를 부르고, 결과가 클로드의 컨텍스트로 들어오고, 그걸 보고 다음 도구를 부른다. 프로필 조회 → 주문 내역 조회 → 재고 확인처럼 순차적으로 도구를 세 번 불러야 하는 작업이면 왕복이 세 번 발생하고, 그 중간 결과 대부분은 다시 쓰이지도 않는다. 프로그래매틱 도구 호출은 클로드가 이런 호출들을 코드 실행 컨테이너 안에서 하나의 스크립트로 짜서 실행하게 한다 — 스크립트가 도구를 부르면 컨테이너 안에서 실행되고 결과는 스크립트 쪽으로 돌아갈 뿐 클로드의 컨텍스트에는 들어오지 않는다. 오직 스크립트의 최종 출력만 클로드에게 전달되므로, 토큰 비용이 중간 결과의 양이 아니라 최종 출력량에 비례하게 된다.

## 7. 배치 API — 실시간이 필요 없다면 토큰 가격이 반값이다

`POST /v1/messages/batches`로 요청을 한 번에 최대 10만 건(또는 256MB)까지 모아 보내면, 입력·출력 토큰 전체에 걸쳐 일반 요금의 50% 할인이 적용된다. 대부분 1시간 안에, 늦어도 24시간 안에 처리되고 결과는 생성 후 29일간 보관된다. 실시간 응답이 필요 없는 대량 분류·요약·추출 작업이라면, 모델을 바꾸거나 사람이 개입할 필요 없이 같은 모델·같은 품질을 절반 가격에 쓸 수 있는 가장 확실한 방법이다.

## 8. (보너스) 토큰 카운팅 — 보내기 전에 미리 재본다

`POST /v1/messages/count_tokens`는 실제로 과금되지 않는 무료 측정 엔드포인트로, 요청을 보내기 전에 정확한 토큰 수를 미리 확인할 수 있다. 클로드가 아닌 다른 토크나이저(예: OpenAI의 tiktoken)로 추정하면 텍스트 기준 15~20%, 코드나 비영어권 텍스트에서는 그보다 훨씬 크게 틀리기 때문에, 정확한 비교·최적화가 필요하다면 이 엔드포인트로 직접 재는 게 유일하게 신뢰할 수 있는 방법이다.

## 정리

| 기법 | 핵심 파라미터 | 효과 |
|---|---|---|
| 프롬프트 캐싱 | `cache_control: {type: "ephemeral", ttl: "5m"\|"1h"}` | 캐시 읽기 시 입력가의 ~0.1배 |
| 컨텍스트 압축 | 베타 `compact-2026-01-12` | 긴 대화를 서버가 자동 요약 |
| 컨텍스트 편집 | 베타 `context-management-2025-06-27` | 오래된 도구결과·사고블록 삭제 |
| 추론 강도 | `output_config.effort` | 낮출수록 도구호출·서두 감소 |
| 적응형 사고 | `thinking: {type: "adaptive"}` | 난이도에 맞춰 사고 토큰 자동 조절 |
| 지연 로딩 도구 | `defer_loading: true` + tool search | 필요한 도구 스키마만 검색해서 로드 |
| 프로그래매틱 도구 호출 | 코드 실행 + `allowed_callers` | 중간 결과가 컨텍스트를 거치지 않음 |
| 배치 API | `POST /v1/messages/batches` | 토큰 전체 50% 할인 |

이 여덟 가지는 전부 앤스로픽이 API·클로드 코드 안에 이미 구현해둔 공식 메커니즘이고, 어느 것도 다른 모델로 바꾸거나 사람이 중간에서 결과물을 수동으로 옮겨줘야 하는 방식이 아니다. 실무에서는 대개 여러 개를 동시에 켜는 게 정답이다 — 프롬프트 캐싱으로 반복 컨텍스트를 지키고, 긴 세션은 압축·컨텍스트 편집으로 다이어트시키고, 작업 난이도에 맞춰 effort를 조절하고, 도구가 많으면 지연 로딩을 켜고, 실시간이 필요 없는 대량 작업은 배치로 돌리는 조합이 가장 자연스럽다.
