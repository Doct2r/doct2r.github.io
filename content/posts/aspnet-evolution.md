[고전 ASP](/posts/classic-asp-explained/)를 뒤엎고 나온 ASP.NET은 그 자체로 안정된 결과물이 아니었다. 2002년 웹폼으로 시작한 이 기술은 이후 20년 동안 최소 세 번 스스로의 아키텍처를 통째로 갈아엎었다 — MVC로, 그다음 크로스플랫폼 코어로, 그다음 브라우저에서 자바스크립트 없이 돌아가는 블레이저로. 매번 이전 모델이 감추려 했던 것을 다시 드러내는 방향이었다는 공통점이 있다.

---

## 1. 웹폼 — 데스크톱 개발자를 위해 설계된 웹

ASP.NET 1.0의 웹폼은 처음부터 노림수가 분명했다. 마이크로소프트가 2001년 5월 MSDN 매거진에 실은 기사 제목부터가 "ASP.NET: 웹폼으로 드래그앤드롭만으로 강력한 웹 앱을 만든다"였다 — VB6·윈폼에 익숙한 데스크톱 개발자가 웹을 캔버스처럼 다루게 만드는 게 설계 의도였다는 뜻이다.

문제는 HTTP 자체가 상태를 기억하지 않는다는 사실이었다. 웹폼은 이걸 뷰스테이트(ViewState)로 가렸다 — 서버 컨트롤의 상태를 직렬화해 Base64로 인코딩한 뒤 숨겨진 `__VIEWSTATE` 필드에 통째로 밀어넣고, 페이지를 다시 제출(postback)할 때마다 이걸 복원하는 방식이다. 초기 버전에서는 이 값이 암호화도 안 돼 있어서 그냥 소스 보기 후 Base64만 풀면 내용이 그대로 보였다. 실제 페이지에서 뷰스테이트 크기가 800자에서 7,800자까지 늘어난 사례가 보고됐고, 심한 경우 페이지 전체 용량을 두 배로 불려 로딩 속도를 떨어뜨린다는 지적이 이어졌다. 여기에 PreInit→Init→InitComplete→PreLoad→Load→(컨트롤 이벤트)→LoadComplete→PreRender→PreRenderComplete→SaveStateComplete→Unload로 이어지는 페이지 생명주기까지 겹치면서, 지금 이 이벤트에서 값을 바꾸면 실제로 어떻게 반영될지 예측하기 어렵다는 불만이 쌓였다.

## 2. 레일즈의 그림자 — ASP.NET MVC의 등장

2007년 10월 5～7일 ALT.NET 컨퍼런스에서 스콧 거스리가 ASP.NET MVC 프레임워크 베타를 발표했다. 당시 기사들은 이 프레임워크가 "루비 온 레일즈와 상당히 비슷한 방식으로 ASP.NET 위에 MVC를 구현한 것"이라고 대놓고 언급했다 — 레일즈가 되살린 MVC 패턴의 인기를 마이크로소프트도 그대로 따라간 셈이다. 핵심 목표는 뷰스테이트가 뒤섞인 마크업 대신 렌더링되는 HTML을 온전히 통제하는 것, 그리고 정적인 `HttpContext`에 묶여 있던 웹폼과 달리 컨트롤러를 평범한 C# 클래스로 만들어 테스트하기 쉽게 만드는 것이었다.

소스 공개도 단계적으로 이뤄졌다. 2008년 3월에는 소스 코드를 보고 직접 빌드할 수 있게만 풀었고, 아직 OSI 승인 라이선스는 아니었다. 정식으로 OSI 승인 라이선스인 Ms-PL 아래 공개된 건 2009년 4월이었다 — 이건 나델라 시대의 마이크로소프트 오픈소스 전환보다 5년 가까이 앞선 일이다. ASP.NET MVC 1.0 정식판은 2009년 3월 18일 MIX09 첫날 키노트에서 발표됐다.

## 3. Web API — 또 하나의 갈래

2012년, .NET 프레임워크 4.5와 함께 ASP.NET Web API가 별도로 나왔다. RESTful HTTP 서비스를 만드는 데 특화된 프레임워크였는데, MVC의 `Controller`가 아니라 `ApiController`라는 별도 기반 클래스를 썼다 — 공식 문서조차 "Web API를 쓰는 데 ASP.NET MVC를 알 필요는 없다"고 명시할 만큼 의도적으로 분리된 스택이었다. 화면을 그리는 프레임워크와 API를 만드는 프레임워크가 따로 존재하는 이 어색한 이원 구조는 몇 년 뒤 통째로 정리된다.

## 4. ASP.NET Core — 이름부터 다시 지은 전면 재작성

2014년 무렵 "ASP.NET vNext"라는 코드네임으로 시작된 다음 세대는 처음엔 "ASP.NET 5"라는 이름을 달고 있었다. 그런데 2016년 1월 20일 마이크로소프트는 이 이름을 "ASP.NET Core 1.0"으로 바꾼다. 이유는 명확했다 — "5"라는 숫자가 기존 4.6을 대체하는 점진적 업그레이드처럼 보이게 만드는데, 실제로는 핵심 플랫폼 라이브러리까지 통째로 다시 짠 완전히 다른 물건이었기 때문이다. "Core"라는 이름은 이게 새 플랫폼의 첫 출시라는 신호였고, 동시에 새로운 .NET Core와 기존 .NET 프레임워크 4.6 양쪽에서 돌아간다는 사실도 반영한 것이었다.

2016년 6월 27일 .NET Core 1.0·ASP.NET Core 1.0·EF Core 1.0이 동시에 정식 출시됐다. 윈도우·macOS·리눅스를 처음부터 동등하게 지원하는 크로스플랫폼이었고, 개발 초기부터 깃허브에서 오픈소스로 진행됐다. 내장 웹서버 케스트럴(Kestrel)은 원래 libuv 기반으로 만들어졌다가, ASP.NET Core 2.1(2018)부터는 매니지드 .NET 소켓 기반이 기본값으로 바뀌었다 — 유닉스 도메인 소켓이나 systemd 소켓 활성화처럼 TCP가 아닌 특수한 바인딩을 쓰는 경우에만 libuv가 남았다. 그리고 처음 재작성될 때부터 MVC와 Web API가 하나로 합쳐져서, 이제 컨트롤러 하나가 HTML 뷰든 JSON이든 같은 파이프라인으로 반환할 수 있게 됐다. `HttpModule`·`HttpHandler`로 이뤄지던 낡은 파이프라인은 `IApplicationBuilder`의 미들웨어 체인으로 바뀌었고, 서드파티 DI 컨테이너 없이도 의존성 주입이 프레임워크 자체 기능으로 들어갔다.

## 5. .NET 통합이라는 배경

2020년 11월 10일 나온 .NET 5는 이름에서 아예 "Core"를 뗐다 — 마이크로소프트의 설명은 "앞으로 .NET이 하나뿐이라면 굳이 구분하는 수식어가 필요 없다"는 것이었다. 이게 윈도우 전용이던 옛 .NET 프레임워크에서 오픈소스 크로스플랫폼 .NET으로 완전히 넘어가는 전환점이었고, ASP.NET Core도 이 시점부터는 그냥 ".NET" 위에서 돌아가는 셈이 됐다.

## 6. 블레이저 — 자바스크립트 없이 웹을 만든다는 시도

블레이저의 시작은 실험이었다. 마이크로소프트의 스티브 샌더슨이 2017년 NDC 오슬로에서 브라우저 안에서 C#·레이저 문법이 웹어셈블리로 돌아가는 실험적 프레임워크를 시연했다 — 플러그인 없이, 크리스 베이컨이 2010년부터 만들어온 이식형 .NET 런타임 DotNetAnywhere를 확장해 웹어셈블리로 컴파일한 결과였다.

이 실험은 두 갈래로 정식 출시됐다. 블레이저 서버는 2019년 9월 .NET Core 3.0과 함께 나왔는데, UI 로직 자체는 서버에서 돌고 화면 변경 사항만 시그널알(SignalR) 연결을 통해 브라우저로 밀어주는 방식이다 — 초기 다운로드는 가볍지만 지연시간에 민감한 상시 연결이 필요하다. 블레이저 웹어셈블리는 2020년 5월 3.2.0 버전으로 정식 출시됐는데, .NET 런타임 자체를 웹어셈블리로 통째로 내려받아 브라우저 안에서 완전히 클라이언트 쪽으로 돌아간다 — 처음 받는 용량은 크지만 이후엔 서버 왕복 없이 오프라인에서도 동작한다. 둘 다 목표는 같다 — 자바스크립트를 한 줄도 안 쓰고 C# 개발자가 SPA를 만들 수 있게 하는 것이다.

## 정리

세 번의 재발명은 매번 같은 방향을 향했다. 웹폼은 HTTP의 무상태성을 뷰스테이트로 가리려다 실패했고, MVC는 그 가림막을 걷어내 HTML 제어권을 개발자에게 돌려줬다. ASP.NET Core는 그다음 단계로 IIS·윈도우라는 플랫폼 자체의 경계를 걷어냈고, 블레이저는 마지막 남은 경계 — 브라우저에서는 자바스크립트를 써야 한다는 전제 — 마저 지우려는 시도다. ASP.NET의 역사는 결국 "무엇을 숨길 것인가"에서 "무엇을 더는 숨기지 않을 것인가"로 계속 옮겨간 역사였다.
