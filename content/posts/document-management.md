문서 관리가 무너지는 패턴은 거의 같다. 어느 순간 문서가 너무 많아서 찾을 수가 없거나, 문서가 하나뿐인데 너무 길어서 읽을 수가 없거나, 문서가 있는데 아무도 업데이트를 안 해서 믿을 수가 없는 상태가 된다. 이 세 가지 실패를 피하려면 **무엇을 어디에, 어떻게 나누어 관리할지**를 처음부터 설계해야 한다.

---

## 1. 문서의 종류를 먼저 분류한다

문서를 관리하기 전에, 문서를 용도로 나눠야 한다. 용도가 다른 문서는 관리 방법도 다르다.

```
문서의 용도별 분류:

① 개념 문서 (Conceptual)
   → 이것이 무엇인지, 왜 이렇게 동작하는지
   → 예: "우리 인증 시스템의 설계 원칙"
   → 독자: 처음 접하는 사람, 배경 이해가 필요한 사람
   → 변경 빈도: 낮음 (방향이 바뀔 때만)

② 가이드 문서 (How-to / Tutorial)
   → 특정 작업을 어떻게 하는지
   → 예: "로컬 개발 환경 세팅하는 법", "배포 절차"
   → 독자: 특정 작업을 수행하려는 사람
   → 변경 빈도: 중간 (절차가 바뀔 때)

③ 레퍼런스 문서 (Reference)
   → 정확한 스펙, API, 파라미터, 에러코드
   → 예: "API 엔드포인트 목록", "설정 옵션 전체"
   → 독자: 이미 알고 있고 세부 사항을 확인하는 사람
   → 변경 빈도: 높음 (코드 바뀔 때마다)

④ 결정 문서 (Decision Record)
   → 왜 이 선택을 했는지의 이유
   → 예: ADR(Architecture Decision Record)
   → 독자: 나중에 "왜 이렇게 했지?"를 묻는 사람
   → 변경 빈도: 추가만 함, 수정 안 함

⑤ 상태 문서 (Status / Living Document)
   → 지금 어떤 상태인지
   → 예: "현재 진행 중인 프로젝트 현황", "온콜 핸드오프 노트"
   → 독자: 현재 상황을 파악해야 하는 사람
   → 변경 빈도: 매우 높음 (수시로 갱신)
```

이 분류가 중요한 이유는, 용도가 다른 문서를 한 곳에 섞으면 독자가 "내가 찾는 게 개념 설명인지, 따라하는 절차인지, API 스펙인지"를 모르는 상태로 읽게 되기 때문이다.

## 2. 문서를 쪼개야 할 때

### 독자가 다를 때

가장 명확한 분리 기준이다.

```
잘못된 예: 하나의 README에 전부
  # 프로젝트 소개
  ## 사용법 (일반 사용자용)
  ## 개발 환경 세팅 (개발자용)
  ## API 레퍼런스 (통합 개발자용)
  ## 운영 가이드 (DevOps용)

문제:
  일반 사용자: 개발 환경 세팅을 읽어야 함
  개발자: API 레퍼런스를 찾으러 스크롤해야 함
  DevOps: 사용법을 건너뛰어야 함

올바른 분리:
  README.md         → 프로젝트 소개 + 빠른 시작
  docs/user.md      → 일반 사용자 가이드
  docs/dev-setup.md → 개발 환경 세팅
  docs/api.md       → API 레퍼런스
  docs/ops.md       → 운영 가이드
```

### 업데이트 주기가 다를 때

```
함께 두면 안 되는 조합:

API 레퍼런스 + 시스템 개념 설명
  API: 버전마다 바뀜
  개념: 거의 안 바뀜
  → 함께 두면 개념 문서가 오래된 API 정보와 섞여 신뢰성 하락

배포 절차 + 아키텍처 설명
  배포 절차: 인프라 변경 때마다 업데이트
  아키텍처: 큰 그림은 오래 유지됨
  → 함께 두면 "이 부분 최신인가?" 혼란 발생
```

### 독립적으로 링크해야 할 때

```
"이 문서의 3번 섹션을 참고하세요"
→ 받은 사람이 문서 열고 3번 찾아야 함
→ 섹션이 바뀌면 링크 의미 변함

"배포 절차 문서를 참고하세요" + 직접 링크
→ 문서 자체로 완결됨
→ 링크 하나가 항상 올바른 내용으로 연결됨

독립 링크가 필요한 순간:
  PR 설명에 관련 문서 첨부할 때
  슬랙에서 "이거 어떻게 해요?" 질문에 답할 때
  온보딩 체크리스트에 항목으로 넣을 때
```

### 문서가 일정 길이를 넘을 때

```
기준 (절대적이지 않음):
  스크롤 없이 한 화면에 안 보이면 → 구조화 필요
  목차가 5개 이상 → 분리 고려
  "이 문서는 A에 대한 것인데, B도 설명해야 해서..." → 분리 신호

단, 길이보다 중요한 것은 "한 주제인가":
  API 레퍼런스는 길어도 하나의 문서
  "개요 + 튜토리얼 + API + FAQ + 트러블슈팅"은 분리 대상
```

## 3. 문서를 같이 가야 할 때

### 내러티브가 이어질 때

```
쪼개면 안 되는 예: 튜토리얼

Step 1에서 만든 것을 Step 2에서 쓴다
Step 2에서 설정한 것을 Step 3에서 확인한다
→ 단계 사이가 분리되면 흐름이 끊어짐
→ 독자가 맥락을 잃음

올바른 판단:
  "이 섹션이 없으면 다음 섹션을 이해할 수 없는가?"
  → Yes: 같이 두어야 함
  → No: 분리 가능
```

### 항상 함께 참조되는 것들

```
예: 에러 코드 목록 + 각 에러의 해결책
  에러 코드만 따로 → 해결책 찾으러 다른 곳 가야 함
  해결책만 따로 → 에러 코드 보고 찾아와야 함
  → 함께 두는 것이 맞음

예: 설정 옵션 목록 + 예시 설정 파일
  옵션만 나열 → 실제 어떻게 쓰는지 모름
  예시만 → 각 옵션의 의미를 모름
  → 함께 두거나 옵션 설명 안에 예시 포함
```

### 문서가 짧고 단일 주제일 때

```
"이것만 읽으면 되는" 짧은 문서를 더 쪼개는 것은 불필요한 클릭을 추가한다.

나쁜 분리:
  "환경변수 목록.md" (3줄)
  "환경변수 설명.md" (5줄)
  "환경변수 예시.md" (4줄)
  → 3개의 파일을 열어야 하나의 정보를 얻음

올바른 처리:
  "환경변수 가이드.md"에 목록 + 설명 + 예시 통합
  → 12줄 짜리 완결된 문서 하나
```

## 4. 각 도구/엔진을 어떻게 활용할 것인가

도구마다 강점이 다르다. 같은 정보를 다른 도구에 넣으면 관리가 힘들어지거나 아무도 안 보게 된다.

### 코드 저장소 (Git + Markdown)

```
적합한 문서:
  코드와 함께 버전 관리되어야 하는 것
  - API 레퍼런스 (코드가 바뀌면 함께 수정)
  - 개발 환경 세팅
  - Architecture Decision Record (ADR)
  - CHANGELOG

강점:
  코드와 같은 PR에서 리뷰 가능
  "이 커밋 때 문서도 이렇게 바뀌었다" 추적 가능
  브랜치 전략 그대로 적용 가능

약점:
  비개발자 접근 어려움
  렌더링 없이 raw Markdown은 불편
  실시간 협업 불가
  검색 능력 제한적
```

### Wiki 도구 (Confluence, Notion, Obsidian)

```
적합한 문서:
  여러 사람이 함께 편집하고 탐색하는 것
  - 프로젝트 계획, 회의록
  - 팀 온보딩 자료
  - 운영 절차, 가이드
  - 조직 지식

강점:
  비개발자도 편집 가능
  링크, 태그, 검색 강력
  구조화된 탐색 (계층, 사이드바)
  실시간 협업

약점:
  코드와 버전 동기화 어려움
  장기적으로 문서가 쌓이면 정리 필요
  "누가 최신 버전을 가지고 있나" 신뢰 문제
  사용 도구에 종속 (이전 어려움)
```

### 스프레드시트 (Excel, Google Sheets)

```
적합한 문서:
  구조화된 데이터, 비교, 계산이 필요한 것
  - 기능 목록 + 지원 여부 (O/X 행렬)
  - 에러 코드 + 설명 + 조치 방법
  - 일정 + 담당자 + 상태
  - 설정값 + 기본값 + 범위

강점:
  필터, 정렬, 수식
  많은 항목을 한 화면에 비교
  비개발자도 편집 쉬움

약점:
  서술형 설명에 부적합
  버전 관리 어려움
  스프레드시트 URL이 없어지면 찾기 어려움
  "최신 파일이 어디 있는가" 혼란
```

### 다이어그램 도구 (draw.io, Miro, Lucidchart)

```
적합한 문서:
  관계, 흐름, 구조를 시각으로 표현
  - 시스템 아키텍처
  - 데이터 플로우
  - 사용자 여정 (User Journey)
  - 시퀀스 다이어그램

강점:
  텍스트로 설명하기 어려운 것을 직관적으로
  아키텍처 논의에서 공통 언어 역할

약점:
  업데이트 잘 안 됨 ("그건 작년 다이어그램이에요")
  텍스트 검색 불가
  버전 관리 어려움 → 다이어그램 as Code로 해결 가능
    (Mermaid, PlantUML: 텍스트로 다이어그램 작성)
```

### 이슈 트래커 (Jira, Linear, GitHub Issues)

```
적합한 문서:
  작업 단위, 버그, 결정 이력
  - 기능 요청의 배경 설명
  - 버그 재현 절차 + 해결 방법
  - 스프린트 계획

부적합한 것:
  장기 참조 문서 (이슈는 닫히면 찾기 어려움)
  → "이슈 #1234에 설명 있어요" → 나중에 못 찾음
  → 결론만 별도 문서로 이전 필요

이슈 트래커 문서의 수명:
  작업 중: 이슈에 메모
  완료 후: 중요한 결정이나 방법은 Wiki로 이전
  이슈 자체: 히스토리 용도로 닫아둠
```

### 채팅 (Slack, Teams)

```
적합한 문서:
  일시적, 맥락 있는 커뮤니케이션
  - 지금 이 순간의 질문과 답
  - 장애 대응 실시간 소통
  - 빠른 결정

절대 채팅에만 두면 안 되는 것:
  나중에 다시 찾아야 할 정보
  → 슬랙 검색은 시간이 지나면 실용적이지 않음
  → 중요한 내용은 반드시 Wiki나 문서로 이전

"채팅은 문서의 입구다":
  슬랙에서 나온 결정 → Wiki에 기록
  슬랙에서 해결된 문제 → FAQ 또는 트러블슈팅 문서에 추가
```

## 5. 구글의 3레이어 기법

구글은 대규모 개발자 문서(Google Cloud, Android, Firebase 등)를 관리하면서 독자가 "지금 무엇을 원하는가"에 따라 문서를 세 레이어로 나눈다. 이 체계는 Daniele Procida가 정리한 **Diátaxis** 프레임워크와 구글의 실무가 결합된 형태로, 구글 개발자 문서 전반에 실제로 적용돼 있다.

### 레이어 1 — 개념 (Conceptual)

```
목적: 이것이 무엇인지, 왜 이렇게 동작하는지 이해
독자 상태: "처음 들어봄", "배경이 필요함"
독자의 질문: "이게 뭐야?", "왜 이렇게 설계됐어?"

내용:
  - 시스템/기능의 존재 이유
  - 핵심 개념과 용어 정의
  - 다른 것과 어떻게 다른가
  - 어떤 문제를 해결하는가

형식:
  설명 위주, 코드 최소화
  다이어그램과 비유 활용
  "왜"에 집중

예: Google Cloud의 "Cloud Run이란?"
  → 컨테이너 기반 서버리스의 개념 설명
  → 언제 쓰는 게 맞는지
  → Cloud Functions와의 차이
  → 코드 없이 개념으로 설명
```

### 레이어 2 — 가이드 (How-to / Tutorial)

```
목적: 특정 작업을 완료하거나 목표를 달성
독자 상태: "개념은 알겠고, 이제 해보고 싶어"
독자의 질문: "어떻게 해?", "이걸 따라하면 되는 거야?"

두 가지 하위 유형:
  ① 튜토리얼 (Tutorial)
     → 처음부터 끝까지 따라가며 배움
     → 학습 자체가 목적
     → "Hello World 앱 만들기"

  ② How-to 가이드
     → 특정 작업 완료가 목적
     → 이미 기본은 앎, 이걸 어떻게 하는지만 필요
     → "Cloud Run에 커스텀 도메인 연결하는 법"

형식:
  번호 있는 단계 (Step 1, Step 2...)
  복사 가능한 코드 스니펫
  "이 단계가 끝나면 이렇게 됩니다" 확인 포인트
  완료 상태 명확히 제시

구분 기준:
  "이걸 모르면 처음부터 읽어야 해" → 튜토리얼
  "이것만 하면 돼" → How-to 가이드
```

### 레이어 3 — 레퍼런스 (Reference)

```
목적: 정확한 정보 조회
독자 상태: "이미 알고 있고, 세부 사항 확인만 필요"
독자의 질문: "이 파라미터 타입이 뭐지?", "이 에러 코드 의미가?"

내용:
  - API 엔드포인트 + 파라미터 + 반환값
  - 설정 옵션 전체 목록
  - 에러 코드 + 의미 + 해결 방법
  - 명령어 + 플래그 + 예시

형식:
  서술 최소화, 표와 목록 중심
  기계적으로 완전해야 함 (빠진 항목 없음)
  알파벳 순 또는 일관된 정렬
  코드 저장소와 동기화 (자동 생성이 이상적)

레퍼런스의 핵심 특성:
  독자가 위에서 아래로 읽지 않음
  → 찾고자 하는 것을 검색하거나 Ctrl+F
  → 한 항목만 읽고 닫음
  → 그래서 각 항목이 독립적으로 완결돼야 함
```

### 3레이어가 실제로 작동하는 방법

```
독자 진입 방식:

① 새로 접한 사람:
   개념 문서 읽음 → 이해 → 튜토리얼 → 직접 해봄
   → How-to 가이드 → 레퍼런스 (필요할 때만)

② 경험자가 새 기능 접할 때:
   개념 문서 훑음 → How-to 가이드 → 레퍼런스 (확인용)

③ 이미 쓰고 있는 사람:
   레퍼런스만 → "이 파라미터 타입이 뭐지?"

문서 간 연결:
  개념 문서 → "더 알고 싶다면 How-to 가이드"
  How-to 가이드 → "파라미터 전체 목록은 레퍼런스에"
  레퍼런스 → "처음 접하신다면 개념 문서부터"

구글 Firebase 문서에서 실제 적용 확인:
  Get started (튜토리얼)
  Guides (How-to)
  Reference (API 레퍼런스)
  Samples (예제)
  → 왼쪽 네비게이션 구조 자체가 3레이어
```

## 6. 문서 관리의 실용 원칙

### 문서는 누가 쓰는지 명시한다

```
나쁜 예:
  # API 가이드
  (누구를 위한 것인지 모름)

좋은 예:
  # API 가이드
  > 이 문서는 서버 통합을 개발하는 백엔드 개발자를 위한 것입니다.
  > 프론트엔드 연동은 [클라이언트 SDK 가이드]를 참조하세요.

결과:
  독자가 자신이 맞는 문서를 읽고 있는지 즉시 판단
  틀린 문서라면 올바른 문서로 바로 이동
```

### 문서에 날짜와 소유자를 명시한다

```
---
최종 수정: 2025-03-15
작성자: 서버팀 김OO
리뷰: 분기마다 (다음 리뷰: 2025-06)
---

이유:
  "이 문서 최신이에요?" → 날짜 보면 판단 가능
  "이 내용 틀렸는데요" → 소유자에게 바로 연락
  리뷰 일정 → 문서 부패 방지
```

### "문서 부패(Documentation Rot)" 막기

```
문서는 코드보다 빠르게 틀려진다:
  코드: 틀리면 컴파일 오류 or 테스트 실패 → 즉시 발견
  문서: 틀려도 에러 없음 → 누군가가 따라하다가 실패하기 전까지 모름

방법:
  ① 코드와 함께 리뷰: PR에 관련 문서 업데이트 포함
  ② 자동 생성: API 레퍼런스는 코드에서 자동 추출 (JSDoc, Swagger/OpenAPI)
  ③ 정기 리뷰: 분기별로 문서 오너가 검토
  ④ 독자 피드백: "이 내용이 틀렸나요?" 피드백 버튼

가장 효과적인 방법:
  레퍼런스 문서 → 코드에서 자동 생성
  → 코드가 바뀌면 문서도 자동으로 바뀜
  → 사람이 업데이트 기억할 필요 없음
```

### 문서 중복 제거

```
같은 내용이 두 군데:
  어느 것이 최신인지 모름
  하나 업데이트하면 다른 하나는 틀린 채로 남음

원칙: "진실의 단일 출처(Single Source of Truth)"
  정보는 한 곳에만 존재
  다른 곳에서 필요하면 링크로 참조

나쁜 예:
  Confluence에 배포 절차
  README.md에 배포 절차 (다름)
  팀 노션에 배포 절차 (또 다름)

좋은 예:
  배포 절차: Confluence에만 존재
  README.md → "배포 절차는 [Confluence 링크] 참조"
  팀 노션 → 동일 링크
```

## 정리

```
쪼개야 하는 경우:
  독자가 다를 때 (역할, 경험 수준)
  업데이트 주기가 다를 때
  독립적으로 링크될 때
  한 문서에 여러 주제가 섞일 때

같이 가야 하는 경우:
  단계가 이어지는 튜토리얼
  항상 함께 참조되는 내용
  문서 자체가 짧고 단일 주제

도구 배치 원칙:
  코드 저장소(Git+MD) → API 레퍼런스, ADR, 개발 가이드
  Wiki(Confluence/Notion) → 팀 지식, 운영 절차, 온보딩
  스프레드시트 → 구조화된 데이터, 비교표
  다이어그램 → 아키텍처, 흐름 (가능하면 코드로 관리)
  이슈 트래커 → 작업 단위 (완료 후 중요 내용은 Wiki로)
  채팅 → 일시적 커뮤니케이션 (결론은 문서로)

구글 3레이어:
  레이어 1 (개념): 이게 뭔지, 왜 존재하는지
  레이어 2 (가이드): 어떻게 하는지, 따라하기
  레이어 3 (레퍼런스): 정확한 스펙 조회

공통 원칙:
  독자가 누군지 명시
  소유자 + 날짜 + 리뷰 일정
  자동 생성 가능한 것은 자동화
  중복 없이 단일 출처
```

문서가 없는 것보다 틀린 문서가 더 나쁘다. 잘못된 문서는 사람을 잘못된 방향으로 이끌고, 없는 문서는 찾으러 다니게 만든다. 좋은 문서 관리의 목표는 **"찾을 수 있고, 믿을 수 있고, 최신인"** 상태를 유지하는 것이다.
