C++은 자바처럼 오라클이 쥐고 있는 것도 아니고, C#처럼 마이크로소프트가 주도권을 갖는 것도 아니다. 그리고 흔히 알려진 것과 달리 "C에 몇 가지 기능을 얹은 확장판"이라는 말도 지금 기준으로는 절반만 맞다. C++이 정확히 누구의 것이고, C와 지금 어떤 관계인지 정리한다.

---

## 1. 탄생 — "C with Classes"에서 시작

C++은 1979년 벨 연구소의 **비야네 스트롭스트룹(Bjarne Stroustrup)**이 시작한 프로젝트였다. 그는 시뮬레이션 언어 **Simula 67**의 객체지향 개념(클래스)이 마음에 들었지만, 실행 성능은 C만큼 빠르길 원했다. 그래서 C에 클래스 개념을 얹은 **"C with Classes"**를 만들었고, 1983년 **C++**로 이름을 바꿨다("C보다 한 단계 더 나아갔다"는 뜻의 증가 연산자 `++`를 딴 언어유희다).

최초의 C++ 컴파일러 **Cfront**는 C++ 코드를 직접 기계어로 컴파일하지 않고, **C 코드로 변환한 뒤 그 C 코드를 다시 컴파일**하는 방식이었다. 이 구조 자체가 "C++은 결국 C의 확장일 뿐"이라는 인식이 굳어진 근원이다.

---

## 2. 그래서 C의 확장판인가 — 예전엔 맞고, 지금은 아니다

### 초기에는 사실상 맞는 말이었다

Cfront의 존재 자체가 증거다. C++ 코드가 C 코드로 변환 가능했다는 건, 초기 C++이 **C의 상위 집합(Superset)**이 되도록 의도적으로 설계됐다는 뜻이다. 기존 C 코드베이스를 거의 그대로 가져다 쓸 수 있어야 한다는 게 초기 설계의 핵심 목표였다.

### 지금은 갈라진 친척에 가깝다

문제는 이후 C와 C++이 **완전히 별개의 표준화 기구**를 통해 각자 진화했다는 점이다. C는 ISO의 **WG14** 위원회가, C++은 **WG21** 위원회가 따로 표준을 발전시켜왔다. 40년 넘게 따로 진화한 결과, 지금은 서로 호환되지 않는 지점이 여럿 생겼다.

| 항목 | C | C++ |
|---|---|---|
| 가변 길이 배열(VLA) | 지원(C99～) | 표준에서 지원 안 함 |
| `_Generic` (제네릭 매크로) | 지원(C11～) | 없음(대신 템플릿·오버로딩 사용) |
| 문자열 리터럴 타입 | `char[]` | `const char[]` |
| `void*` → 다른 포인터 암시적 변환 | 허용 | 금지(명시적 캐스팅 필요) |
| 구조체 태그 이름 | `struct`/`union`/`enum` 키워드 없이 못 씀 | 태그 이름 자체가 타입명으로 바로 사용 가능 |

이 표에서 보듯, **C99 이후에 나온 문법으로 짠 C 코드는 C++에서 그대로 컴파일되지 않는 경우가 실제로 있다.** "C++은 C의 확장판"이라는 말은 1980년대 기준으로는 맞았지만, 지금은 **문법이 크게 겹치는 사촌 언어** 정도로 이해하는 게 정확하다.

---

## 3. 누구의 것인가 — 회사가 아니라 위원회

C++의 소유 구조는 처음부터 지금까지 특정 회사 소유였던 적이 없다는 점에서 특이하다.

```
자바   → 썬(이후 오라클)이 사실상 소유, 라이선스 통제
C#    → 마이크로소프트가 주도 개발 후 표준 제출(ECMA/ISO는 사후 절차)
C++   → 처음부터 표준화 기구가 주도. 특정 기업이 언어의 방향을 단독 결정할 수 없음
```

C++은 1989년 **ANSI X3J16** 위원회에서 표준화 작업이 시작됐고, 1990년대에 ISO의 **WG21**(ISO/IEC JTC1/SC22/WG21)로 이어져 1998년 최초의 정식 표준 **ISO/IEC 14882:1998(C++98)**로 결실을 맺었다. 이 위원회에는 구글, 마이크로소프트, 메타, 블룸버그, 엔비디아, 인텔 같은 여러 기업의 대표와 학계·개인 전문가들이 함께 참여해 논문(Paper) 제출과 투표를 거쳐 표준을 결정한다.

또한 처음부터 **경쟁하는 여러 독립 구현체**가 존재했다 — Cfront 이후 GCC, Clang, MSVC, 인텔 컴파일러가 각자 따로 C++ 표준을 구현해왔다. 어느 한 회사의 컴파일러가 "사실상 표준"이 되는 구조가 아니라는 뜻이다. 스트롭스트룹 본인도 지금까지 존경받는 원로 설계자로 위원회에 참여하지만, 파이썬의 귀도 반 로섬 같은 "종신 자애로운 독재자(BDFL)" 위치는 아니다 — 그의 의견도 위원회 투표를 거쳐야 반영된다.

---

## 4. 느리지만 민주적인 표준화 과정

다자 합의 구조는 안정성을 주는 대신 속도를 희생시킨다. 실제로 C++98 이후 다음 대규모 개정판인 **C++11이 나오기까지 13년**이 걸렸다 — 그사이 시도됐던 "C++0x" 계획이 여러 차례 지연된 끝에야 나온 결과다. 이후 위원회는 교훈을 얻어 **3년 주기 정기 릴리즈**(C++14 → C++17 → C++20 → C++23)로 체제를 바꿨다.

```
C++98 -----------------(13년)----------------- C++11
                                        C++14 → C++17 → C++20 → C++23 (3년 주기 정착)
```

---

## 5. 소유 구조가 진화 속도에 미치는 영향

단일 기업이 주도하는 언어(C#, 자바 초기)는 의사결정이 빠르고 방향이 명확한 대신, 그 기업의 이해관계에서 자유롭기 어렵다. C++처럼 다자 위원회가 주도하는 언어는 반대다 — 특정 기업의 입김만으로 방향이 뒤집히지 않는 안정성을 얻는 대신, 논쟁이 길어지고 기능 하나가 표준에 들어가기까지 몇 년씩 걸리는 게 당연한 일이 된다. C++가 지금도 하위 호환성을 극단적으로 중시하며 천천히 움직이는 이유는, 이 소유 구조 자체가 "빠르게 바꾸는 것"보다 "누구도 일방적으로 못 바꾸는 것"을 우선하도록 설계돼 있기 때문이다.

---

## 정리

```
탄생        → 1979년 스트롭스트룹이 벨 연구소에서 "C with Classes"로 시작, 1983년 C++로 개명
C의 확장인가 → 초기(Cfront 시절)엔 사실상 맞음. 지금은 WG14/WG21이 따로 진화시켜
              VLA·_Generic·문자열 리터럴 타입 등에서 서로 호환 안 되는 갈라진 친척 관계
누구의 것인가 → 특정 기업 소유 아님. ISO WG21 위원회가 다자 합의로 표준 결정(1998년 첫 표준 C++98)
              GCC·Clang·MSVC 등 경쟁 구현체가 처음부터 공존, 스트롭스트룹도 단독 결정권 없음
표준화 속도  → C++98→C++11 13년 소요 후 3년 주기 정착. 다자 합의 구조라 신중하지만 느림
소유 구조 영향→ 단일기업 주도(빠름·종속적) vs 위원회 주도(느림·안정적) — C++은 후자를 택해 하위호환성 극단적으로 중시
```

C++을 "C의 확장판"이라고만 알고 있다면 1980년대에서 멈춘 지식이고, 어느 한 회사가 이 언어를 좌우한다고 생각한다면 그것도 틀렸다. C++은 처음부터 지금까지 **누구의 소유도 아니면서, 그래서 누구도 마음대로 못 바꾸는** 언어로 남기 위해 설계된 표준이다.
