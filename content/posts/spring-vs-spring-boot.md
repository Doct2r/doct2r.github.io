"스프링 배워야지" 하고 검색하면 튜토리얼은 죄다 Spring Boot로 시작한다. `@SpringBootApplication` 하나 붙이고 `main` 메서드를 실행하면 웹서버가 뜬다. 그런데 자바 채용 공고에는 "Spring 경험자"라고 적혀 있고, 어떤 사내 문서는 "우리는 Spring Boot를 안 쓰고 순수 Spring으로 간다"고 적혀 있다. 이 둘은 같은 것인가, 다른 것인가. 정확히는 **같은 층위가 아니다**.

---

## 1. Spring은 왜 나왔나 — EJB에 대한 반발

2000년대 초 자바 엔터프라이즈 개발의 표준은 **J2EE**, 그중에서도 **EJB(Enterprise JavaBeans)**였다. EJB 2.x로 컴포넌트 하나를 만들려면 홈 인터페이스, 리모트 인터페이스, 빈 구현 클래스, XML 배포 서술자까지 최소 네 벌의 코드를 맞춰 써야 했다. 게다가 EJB 컴포넌트는 원격 호출을 전제로 설계돼 있어서, 같은 서버 안에서 호출하는 로컬 객체조차 무거운 프록시를 거쳐야 했다. 테스트를 하려면 실제 애플리케이션 서버를 띄워야 하는 경우가 많아, 단위 테스트 자체가 고역이었다.

2002년 로드 존슨(Rod Johnson)은 저서 《Expert One-on-One J2EE Design and Development》에서 이 무거움을 정면 비판하며, 책에 실은 약 3만 줄의 프레임워크 코드를 공개했다. 이게 **Spring**의 뿌리다. 2004년 정식 1.0이 나왔고, 핵심 철학은 단순했다.

- 비즈니스 객체는 EJB 없이 **평범한 자바 객체(POJO)**로 만든다.
- 객체 간 의존 관계는 프레임워크가 대신 조립해준다 — **의존성 주입(Dependency Injection)**, 넓게는 **제어의 역전(IoC)**.
- 트랜잭션·로깅 같은 부가 기능은 **관점지향 프로그래밍(AOP)**으로 필요한 지점에만 끼워 넣는다.

즉 "컨테이너 없이도 엔터프라이즈 자바를 테스트 가능하게 짤 수 있다"는 게 Spring의 원래 약속이었다.

---

## 2. Spring이 스스로 무거워진 과정

역설적이게도 Spring은 시간이 지나며 자기만의 무게를 쌓기 시작했다. 2000년대 중후반 전형적인 Spring 프로젝트는 빈(bean) 정의를 담은 대형 XML 파일 여러 개, 스프링 코어·MVC·시큐리티·하이버네이트 사이의 버전 호환 조합을 직접 맞춰야 하는 의존성 관리, 그리고 완성된 앱을 WAR로 패키징해 별도의 톰캣 같은 애플리케이션 서버에 배포하는 절차까지 요구했다.

"EJB의 설정 지옥에서 벗어나려고 만든 프레임워크가, 결국 자기만의 XML 설정 지옥을 만들었다"는 비아냥이 이 시기에 나왔다. Spring 2.5(2007)와 3.0(2009)에서 애노테이션 기반 설정(`@Component`, `@Autowired`, `@Configuration`)이 도입되며 XML은 크게 줄었지만, **의존성 버전 조합을 직접 맞춰야 하는 문제와 배포 복잡성**은 그대로 남았다.

---

## 3. Spring Boot의 등장 — 설정을 없애는 게 아니라 감추는 방식

2014년 4월, 필 웹(Phil Webb)과 데이브 사이어(Dave Syer)를 중심으로 한 팀이 **Spring Boot 1.0**을 내놨다. 당시 Spring을 이끌던 회사는 VMware에서 분리된 **피봇탈(Pivotal)**이었다. Spring Boot가 실제로 해결한 문제는 세 가지다.

- **스타터 의존성(starter dependency)**: `spring-boot-starter-web` 하나만 추가하면, 서로 호환되는 버전의 라이브러리 묶음이 한 번에 딸려 온다. 버전 조합을 직접 맞추던 부담이 사라졌다.
- **자동 설정(auto-configuration)**: 클래스패스에 무엇이 올라와 있는지 보고(`@ConditionalOnClass` 등), 필요한 빈을 프레임워크가 알아서 등록한다. 데이터베이스 드라이버가 클래스패스에 있으면 데이터소스 빈을 자동으로 만들어주는 식이다.
- **내장 서블릿 컨테이너**: 톰캣·제티·언더토우를 애플리케이션 안에 내장시켜, `java -jar app.jar` 한 줄로 실행되는 독립 실행형 앱을 만든다. 별도 애플리케이션 서버에 WAR를 배포하는 절차 자체가 필요 없어졌다.

여기에 **액추에이터(Actuator)**로 헬스 체크·메트릭 같은 운영 기능까지 기본 제공하면서, Spring Boot는 "설정을 없앤 것"이 아니라 **"관용적인(opinionated) 기본값으로 설정을 대신 해주는 것"**에 가깝다.

---

## 4. 그래서 관계를 어떻게 정리해야 하나

**Spring Boot는 Spring을 대체하지 않는다.** IoC 컨테이너, AOP, Spring MVC, Spring Data, Spring Security 같은 실질적인 기능은 여전히 Spring Framework의 몫이다. Spring Boot는 그 위에 얹혀서 **"이 조합을 빠르게, 이견 없이 시작할 수 있게 해주는 배포·설정 계층"**일 뿐이다.

비유하자면 Spring이 엔진과 부품이라면, Spring Boot는 그 부품들을 미리 조립해 시동만 걸면 되게 만든 완성차에 가깝다. "우리는 Spring Boot 없이 순수 Spring으로 간다"는 팀은 부품을 사서 직접 조립하겠다는 뜻이지, 다른 엔진을 쓰겠다는 뜻이 아니다.

---

## 5. Spring Boot의 트레이드오프

자동 설정의 편리함은 공짜가 아니다. `@ConditionalOnClass`, `@ConditionalOnMissingBean` 같은 조건부 설정이 클래스패스 스캔 결과에 따라 암묵적으로 빈을 등록하다 보니, 초심자는 "이 빈이 왜, 어떻게 만들어졌는지" 코드만 봐서는 알기 어려운 경우가 많다. 원인을 알 수 없는 설정 충돌이 발생하면 Spring Boot의 자동 설정 리포트(`--debug` 옵션)를 켜서 어떤 조건이 어떻게 평가됐는지 역추적해야 하는 상황도 흔하다.

관용적 기본값이 프로젝트 상황과 안 맞을 때도 있다. 이미 운영 중인 톰캣 클러스터에 배포해야 하는 조직은 내장 서버 대신 다시 WAR 패키징으로 돌아가야 하고, 이 경우 Spring Boot가 자동으로 처리해주던 부분을 되짚어 이해해야 하는 역설이 생긴다.

---

## 6. 오라클의 그림자 — javax에서 jakarta로

Spring 생태계도 오라클의 자바 상표권 통제에서 자유롭지 않다. 오라클이 자바 상표를 계속 쥐고 있던 탓에([[oracle-litigation-machine]] 참고), Java EE는 2017년 이클립스 재단으로 이관되며 **Jakarta EE**로 이름이 바뀌었고, 2020년 Jakarta EE 9부터는 패키지 네임스페이스 자체가 `javax.*`에서 `jakarta.*`로 강제 변경됐다. 이 여파로 2022년 나온 **Spring Framework 6 / Spring Boot 3**는 `jakarta.*` 네임스페이스로 완전히 갈아타며 구버전 `javax` 기반 라이브러리와의 하위 호환을 끊었다. 자바 생태계 어디에 있든 오라클의 상표권 결정 하나가 몇 년 뒤 마이그레이션 비용으로 돌아온다는 걸 보여주는 사례다.

---

## 7. 기업 계보 — 이름은 바뀌어도 이어진 프로젝트

Spring을 만든 로드 존슨의 회사 Interface21은 2007년 **SpringSource**로 개명했고, 2009년 VMware가 인수했다. 2013년에는 EMC·VMware·GE가 합작해 세운 **Pivotal**로 Spring 사업부가 넘어갔고, 2019~2020년 VMware가 Pivotal을 다시 흡수했다. 2023년 브로드컴이 VMware를 인수하면서, Spring은 현재 브로드컴 산하 VMware Tanzu 조직에서 관리되고 있다. 회사 이름은 네 번 바뀌었지만 프로젝트 자체는 20년 넘게 끊기지 않고 이어지고 있다.

---

## 8. 결론

Spring은 EJB의 무거움에 반발해 태어난 **철학**(POJO 기반 DI/AOP)이고, Spring Boot는 그 철학을 실제로 빠르게 시작할 수 있게 만든 **실행·배포 계층**이다. 초심자용 튜토리얼이 Spring Boot로 시작하는 건 당연하다 — 어차피 밑바닥의 Spring Framework를 그대로 쓰면서 설정 부담만 덜어주기 때문이다. 다만 "Spring을 안다"는 게 진짜로 뭘 안다는 뜻인지 물으면, 답은 결국 자동 설정 뒤에 숨어 있는 IoC 컨테이너와 AOP를 이해하고 있느냐로 돌아온다.
