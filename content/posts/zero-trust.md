"내부망에 있으면 안전하다"는 전제가 깨진 지 오래다. 그 전제 위에 세워진 보안 모델이 반복적으로 실패하면서 나온 대안이 제로 트러스트(Zero Trust)다. 개념은 단순하지만 구현은 조직 전체를 바꾸는 작업이다.

---

## 1. 왜 나왔나

### 경계 기반 보안의 붕괴

전통적인 보안 모델은 **성벽(Perimeter)** 모델이었다. 내부망은 신뢰, 외부망은 불신. 방화벽으로 경계를 긋고, 경계 안에 들어온 것은 믿는다.

```
[인터넷 - 불신]  ──방화벽──  [내부망 - 완전 신뢰]
                              ├ 서버
                              ├ DB
                              └ 직원 PC
```

이 모델은 세 가지 가정 위에 서 있었다.

1. 공격자는 항상 외부에 있다.
2. 내부에 들어온 사용자는 신뢰할 수 있다.
3. 내부망은 하나의 통제된 공간이다.

이 가정들이 현실과 맞지 않게 된 계기들이 있다.

**클라우드와 SaaS의 확산**
내부망이라는 개념 자체가 흐려졌다. 직원들은 Google Workspace, Salesforce, AWS 콘솔에 브라우저로 접근한다. 데이터가 사내 서버가 아니라 클라우드에 있다. 방화벽으로 경계를 그을 수 있는 "내부"가 사라졌다.

**재택근무의 일반화**
직원이 집, 카페, 호텔에서 업무를 본다. VPN으로 내부망에 연결하면 "내부"에 있다고 간주했는데, VPN 자격증명 하나만 탈취하면 내부 전체가 열린다.

**내부자 위협과 횡적 이동**
2013년 Edward Snowden 사건, 2020년 SolarWinds 해킹은 공통점이 있다. 이미 내부에 있거나 내부 접근 권한을 가진 존재가 피해를 일으켰다. SolarWinds의 경우 공급망을 통해 내부망에 진입한 공격자가 수개월간 탐지 없이 횡적으로 이동(Lateral Movement)했다.

**공격자는 이미 안에 있다**
Mandiant(FireEye) 보고서에 따르면 침해 사고의 평균 탐지 기간은 수십 일에서 수백 일이다. 공격자가 내부에 들어온 뒤 오랫동안 발각되지 않는다는 뜻이다. "내부에 있으면 안전"이라는 전제가 실제로는 "공격자에게 수백 일의 자유를 주는 것"이었다.

### 제로 트러스트의 등장

2010년 Forrester Research의 존 킨더박(John Kindervag)이 처음 "Zero Trust"라는 용어를 공식화했다. 2014년 Google이 자사 내부에 **BeyondCorp**라는 이름으로 구현해 발표하면서 업계 전반으로 확산됐다.

NIST는 2020년 SP 800-207 문서로 제로 트러스트 아키텍처를 표준화했다.

## 2. 원리

제로 트러스트를 한 문장으로 정의하면 이렇다.

> **"절대 신뢰하지 말고, 항상 검증하라. (Never trust, always verify.)"**

위치(내부망/외부망)나 장치 유형이 아니라, **매 요청마다** 신원과 맥락을 검증하고 최소한의 권한만 부여한다.

### 핵심 원칙

**① 모든 요청을 명시적으로 검증**

사용자가 누구인지, 어떤 기기를 쓰는지, 어디서 접속하는지, 무엇에 접근하려 하는지를 매번 확인한다. 한 번 인증했다고 영구적으로 신뢰하지 않는다.

```
요청 발생
  ↓
ID 확인    → 누가? (사용자 계정, MFA)
기기 확인  → 어떤 기기? (관리된 기기인가, 패치됐는가)
위치 확인  → 어디서? (평소와 다른 국가/IP인가)
행동 확인  → 무엇을? (이 사람이 이 리소스에 접근한 전례가 있는가)
  ↓
접근 허용 / 거부 / 조건부 허용(MFA 추가 요구)
```

**② 최소 권한 원칙 (Least Privilege)**

필요한 것에만, 필요한 시간만 접근 권한을 부여한다. DBA에게 서비스 서버 접근 권한이 없어야 하고, 개발자에게 프로덕션 DB 직접 접근 권한이 없어야 한다.

**③ 침해를 전제로 설계 (Assume Breach)**

"언젠가는 뚫린다"를 전제한다. 공격자가 내부에 진입해도 횡적 이동이 불가능하도록, 리소스를 마이크로 세그먼트로 격리한다.

```
[기존 성벽 모델]          [제로 트러스트]
외부 ──벽──  내부          리소스마다 개별 검증
           자유 이동           A ─검증─ B
                               A ─검증─ C
                               (A가 B에 접근해도 C는 별개)
```

**④ 지속적 모니터링**

접근을 허용한 이후에도 행동을 계속 관찰한다. 정상 범위를 벗어나는 행동(평소에 없던 대량 파일 다운로드, 새벽 접속 등)이 감지되면 세션을 끊거나 재인증을 요구한다.

### 구성 요소

제로 트러스트 아키텍처는 단일 제품이 아니라 여러 요소의 조합이다.

| 구성 요소 | 역할 | 예시 |
|---|---|---|
| **IdP (Identity Provider)** | 신원 관리, SSO, MFA | Okta, Azure AD, Google Workspace |
| **PEP (Policy Enforcement Point)** | 정책에 따라 접근 허용/차단 | Cloudflare Access, Zscaler |
| **PDP (Policy Decision Point)** | 실시간으로 접근 정책 판단 | 각 솔루션의 컨트롤 플레인 |
| **MDM (Mobile Device Management)** | 기기 상태 관리 및 신뢰도 평가 | Jamf, Intune, CrowdStrike |
| **SIEM / UEBA** | 행동 분석, 이상 탐지 | Splunk, Microsoft Sentinel |
| **PAM (Privileged Access Management)** | 관리자 계정 접근 제어, 세션 기록 | CyberArk, HashiCorp Vault |

## 3. 실제 사용 사례

### 사례 1 — Google BeyondCorp

Google은 2011년 중국발 Operation Aurora 해킹 이후 내부 보안 모델을 전면 재설계했다. VPN 없이 모든 내부 서비스에 접근하는 구조를 만들었다.

핵심은 **"기기와 사용자를 각각 검증"**이다.

```
직원이 내부 서비스에 접속
  ↓
Access Proxy가 가로챔
  ↓
사용자 인증 (Google 계정 + 2FA)
  +
기기 신뢰도 확인 (Google 발급 인증서 있는가, 최신 패치인가)
  ↓
둘 다 통과하면 해당 서비스로 포워딩
```

결과: 직원이 카페에서 노트북으로 사내 서비스에 접속해도 VPN 없이, 내부망과 같은 보안 수준으로 접근 가능해졌다.

---

### 사례 2 — 금융권 내부망 마이크로 세그멘테이션

은행 내부 시스템은 전통적으로 하나의 거대한 내부망이었다. 핵심 뱅킹 서버와 직원 PC가 같은 VLAN에 있는 경우도 있었다. 공격자가 직원 PC를 감염시키면 뱅킹 서버로 자유롭게 이동할 수 있었다.

마이크로 세그멘테이션 적용 후:

```
[직원 PC 영역]
  ↓ (인증 필요)
[업무 시스템 영역]
  ↓ (별도 인증 + 권한 확인)
[핵심 뱅킹 서버 영역]
  ↓ (PAM을 통한 세션 기록)
[DB 영역]
```

각 경계를 넘을 때마다 별도 검증이 필요하다. 공격자가 직원 PC를 장악해도 다음 단계로 이동하려면 또 다른 인증을 뚫어야 한다.

---

### 사례 3 — 원격 근무 환경의 ZTNA

재택근무자가 사내 ERP에 접근하는 기존 방식:
```
직원 → VPN 연결 → 내부망 전체 접근 가능
```

ZTNA(Zero Trust Network Access) 적용 후:
```
직원 → Cloudflare Access / Zscaler
  ↓
신원 확인 (IdP 인증 + MFA)
기기 확인 (회사 관리 기기인가)
  ↓
ERP만 접근 가능 (내부망 전체가 아님)
다른 서비스는 별도 인증 필요
```

VPN 자격증명 하나를 탈취해도 내부망 전체가 아니라 그 계정이 허가받은 서비스만 볼 수 있다.

---

### 사례 4 — 개발팀 프로덕션 접근 제어

개발자가 프로덕션 서버에 SSH로 접근할 때 PAM(Privileged Access Management)을 적용한다.

```
개발자 → Vault / CyberArk에 접근 요청
  ↓
Just-in-Time 임시 계정 발급 (1시간짜리)
  +
세션 전체 기록 (화면/명령어)
  ↓
1시간 후 계정 자동 삭제
```

영구적인 SSH 키나 계정이 없다. 필요할 때만, 필요한 시간만 권한이 생긴다. 퇴사자 계정이 남아있어도 자동 만료되므로 탈취 위협이 줄어든다.

## 4. 주의점

### "제로 트러스트 제품 하나 사면 된다"는 착각

제로 트러스트는 제품이 아니라 전략이다. Cloudflare Access나 Zscaler를 도입한다고 제로 트러스트가 완성되지 않는다. IdP, MDM, SIEM, PAM이 유기적으로 연동되어야 한다. 도입 전에 "우리 조직의 어떤 위협 모델을 해결하려 하는가"를 먼저 정의해야 한다.

### 사용자 경험 저하

매 요청마다 검증하면 사용자 입장에서 불편하다. MFA를 너무 자주 요구하면 직원들이 인증을 대충 처리하는 **MFA 피로 공격(MFA Fatigue)** 이 생긴다. 맥락 기반 조건부 접근(저위험 상황에서는 추가 인증 생략)으로 균형을 잡아야 한다.

### 레거시 시스템과의 충돌

제로 트러스트는 모든 요청에 현대적인 인증 프로토콜(OAuth 2.0, SAML, OIDC)을 요구한다. 10년 전에 만들어진 사내 시스템이 이를 지원하지 않으면 연동이 안 된다. 마이그레이션 비용이 생각보다 크다.

### 단계적으로 가야 한다

제로 트러스트를 한 번에 전환하려 하면 실패한다. Google BeyondCorp도 완전 전환에 수년이 걸렸다. 현실적인 접근은 다음과 같다.

```
1단계: 신원 강화 — MFA 전면 도입, SSO 통합
2단계: 기기 관리 — MDM 도입, 기기 상태 가시성 확보
3단계: 최소 권한 — 역할 기반 접근 제어(RBAC) 재정비
4단계: 네트워크 분리 — 마이크로 세그멘테이션, ZTNA 도입
5단계: 모니터링 — SIEM/UEBA로 이상 행동 탐지
```

### 내부자 위협은 완전히 막을 수 없다

제로 트러스트는 내부자 위협을 **줄이고 탐지**하는 것이지, 완전히 제거하지는 못한다. 정당한 권한을 가진 직원이 악의적으로 데이터를 유출하면 탐지는 할 수 있어도 사전 차단은 한계가 있다.

### 비용

제로 트러스트 도입은 단순한 소프트웨어 구매가 아니다. 솔루션 라이선스, 컨설팅, 기존 시스템 개조, 직원 교육, 운영 인력까지 포함하면 대규모 조직에서 수억～수십억 원 규모의 프로젝트가 된다. 중소기업은 Tailscale + Cloudflare Access + Google Workspace 조합처럼 비용 효율적인 경로를 먼저 탐색하는 게 현실적이다.

---

## 5. 결론

제로 트러스트는 유행어가 아니다. 클라우드, 원격 근무, 공급망 공격이라는 현실의 변화에 기술적으로 대응하는 불가피한 방향이다.

핵심을 세 줄로 요약하면:

- **위치를 믿지 않는다** — 내부망이든 외부망이든 동일하게 검증한다.
- **최소한만 허용한다** — 필요한 것에, 필요한 시간만, 필요한 사람에게.
- **침해를 전제한다** — 이미 뚫렸다고 가정하고, 피해를 격리하고 탐지하는 데 집중한다.

조직의 규모와 예산에 따라 구현 수준은 다를 수 있다. 하지만 MFA 전면 도입과 최소 권한 원칙 적용은 규모에 상관없이 지금 당장 시작할 수 있는 제로 트러스트의 출발점이다.

VPN으로 내부망에 연결한다고 안심하는 시대는 이미 지났다.
