Claude Code를 쓰다 보면 반복하는 작업이 생긴다. "이 프로젝트에서 PR 만들 때는 항상 이 형식으로", "코드 리뷰할 때는 이 관점에서" 같은 패턴이다. 이것을 매번 긴 문장으로 설명하는 대신, `/pr-create` 한 단어로 끝낼 수 있다. 그것이 스킬이다.

---

## 스킬이란

Claude Code의 **스킬(Skill)**은 사용자가 직접 정의하는 슬래시 커맨드(`/command-name`)다. Markdown 파일 하나가 하나의 커맨드가 된다. 파일 이름이 커맨드 이름이 되고, 파일 내용이 Claude에게 전달되는 프롬프트가 된다.

```
/review        → .claude/commands/review.md 의 내용이 프롬프트로 전달됨
/deploy-check  → .claude/commands/deploy-check.md 의 내용이 프롬프트로 전달됨
```

Claude의 내장 슬래시 커맨드(`/help`, `/clear`, `/bug`)와 별개로, 프로젝트나 개인 용도에 맞는 커맨드를 추가할 수 있다.

---

## 스킬 파일의 위치

스킬은 두 가지 범위로 나뉜다.

### 프로젝트 스킬

```
프로젝트 루트/
└── .claude/
    └── commands/
        ├── review.md
        ├── pr-create.md
        └── deploy-check.md
```

해당 프로젝트 디렉터리에서 Claude Code를 열었을 때만 나타난다. 팀원과 git으로 공유할 수 있다. `.claude/commands/` 폴더를 리포지터리에 커밋하면 모든 팀원이 동일한 스킬을 쓴다.

### 사용자 스킬

```
~/.claude/
└── commands/
    ├── daily-standup.md
    └── explain-simple.md
```

홈 디렉터리의 `~/.claude/commands/`에 두면 모든 프로젝트에서 사용 가능하다. 개인 습관·스타일을 담는 용도에 맞다.

**우선순위**: 같은 이름이 있으면 프로젝트 스킬이 우선된다.

---

## 파일 형식

`.md` (Markdown) 파일이다. 특별한 문법은 없다. 파일 내용이 그대로 Claude에게 전달될 프롬프트다.

### 가장 단순한 예

```
.claude/commands/greet.md
```

```markdown
현재 파일을 읽고, 이 코드에서 흥미로운 점 세 가지를 찾아 친근한 말투로 설명해줘.
```

`/greet`를 입력하면 Claude가 이 내용을 받아 실행한다.

### `$ARGUMENTS` — 입력값 받기

커맨드에 추가 인자를 넘기려면 `$ARGUMENTS` 변수를 사용한다.

```markdown
<!-- .claude/commands/explain.md -->

$ARGUMENTS 에 대해 설명해줘.
- 초등학생도 이해할 수 있는 수준으로
- 현실 세계의 비유를 하나 포함해서
- 코드 예시는 Python으로
```

사용:
```
/explain 클로저
/explain 의존성 역전 원칙
```

`$ARGUMENTS`는 `/커맨드명` 뒤에 오는 텍스트 전체로 치환된다.

## 실용적인 스킬 예시

### PR 생성 템플릿

```markdown
<!-- .claude/commands/pr.md -->

현재 브랜치와 main 브랜치의 차이를 분석해서 PR을 만들어줘.

PR 제목: 50자 이내, 동사로 시작
PR 본문 구성:
- ## 변경 내용 (bullet point 3개 이내)
- ## 테스트 방법
- ## 주의사항 (있을 경우만)

언어는 한국어로. gh pr create 명령어를 실제로 실행해줘.
```

### 코드 리뷰

```markdown
<!-- .claude/commands/review.md -->

$ARGUMENTS 파일을 코드 리뷰해줘.

다음 관점에서만 검토해:
1. 버그 가능성 (확실한 것만)
2. 성능 문제 (명백한 것만)
3. 보안 취약점

스타일, 네이밍, 문서화는 지적하지 마. 
각 항목은 파일명:라인번호 형식으로 위치를 알려줘.
```

사용: `/review src/auth/login.ts`

### 커밋 메시지 작성

```markdown
<!-- .claude/commands/commit.md -->

staged된 변경사항을 분석해서 커밋 메시지를 제안해줘.

규칙:
- 첫 줄: 영어, 50자 이내, 현재형 동사로 시작 (Add / Fix / Update / Remove)
- 빈 줄
- 본문: 한국어, 왜 이 변경이 필요했는지 설명 (무엇을 했는지 말고)
- 마지막 줄: Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>

제안만 해줘. 실제로 커밋하지 마.
```

### 일일 작업 정리

```markdown
<!-- ~/.claude/commands/daily.md -->

오늘 작업을 정리해줘.

1. git log --oneline --since="yesterday" --author="$(git config user.name)" 실행
2. 변경된 파일 목록 확인
3. 다음 형식으로 출력:

## 오늘 한 일
- (작업 내용)

## 내일 이어서 할 것
- (미완료 작업 추정)
```

### 의존성 감사

```markdown
<!-- .claude/commands/audit.md -->

이 프로젝트의 의존성을 감사해줘.

확인할 것:
- package.json (또는 requirements.txt, go.mod, Cargo.toml) 읽기
- 메이저 버전이 오래된 패키지 (2년 이상)
- 보안 이슈가 알려진 패키지 (알고 있는 것만)
- 실제로 사용되지 않을 가능성이 있는 패키지

추측하지 말고, 확실한 것만 알려줘.
```

## CLAUDE.md와의 관계

프로젝트 루트의 `CLAUDE.md`는 Claude가 프로젝트를 열 때마다 자동으로 읽는 컨텍스트 파일이다. 스킬과 역할이 다르다.

```
CLAUDE.md     → "이 프로젝트는 이런 곳이야" (상시 컨텍스트)
스킬(.md)     → "이 작업을 이 방식으로 해줘" (특정 작업 트리거)
```

두 가지를 함께 쓰면 효과적이다.

```markdown
<!-- CLAUDE.md -->
이 프로젝트는 Go로 작성된 REST API 서버입니다.
테스트는 반드시 실제 DB를 사용합니다 (mock 금지).
커밋 메시지는 영어, PR 본문은 한국어입니다.

스킬:
- /pr  : PR 생성 (이 프로젝트 컨벤션에 맞게)
- /review : 코드 리뷰
```

CLAUDE.md에 사용 가능한 스킬 목록을 적어두면, 팀원이 어떤 커맨드가 있는지 쉽게 파악한다.

## 스킬 작성 팁

### 구체적일수록 좋다

모호한 지시는 모호한 결과를 낳는다.

```markdown
<!-- 나쁜 예 -->
코드 리뷰해줘.

<!-- 좋은 예 -->
이 파일에서 NullPointerException이 발생할 수 있는 지점을 찾아줘.
각 지점을 파일명:라인번호 형식으로 알려주고, 왜 위험한지 한 줄로 설명해줘.
발견 못하면 "없음"이라고만 해줘.
```

### 범위를 제한한다

Claude가 너무 많은 것을 하려 하면 결과가 산만해진다.

```markdown
<!-- 스코프 제한 예 -->
$ARGUMENTS 함수만 분석해줘. 
다른 함수나 파일은 건드리지 마.
리팩터링 제안도 하지 마.
오직 이 함수가 올바르게 동작하는지만 판단해줘.
```

### 출력 형식을 지정한다

```markdown
결과는 다음 형식으로만 출력해줘:

**발견된 문제**: X개
1. [심각도: 높음/중간/낮음] 설명 (파일:줄번호)
2. ...

문제가 없으면 "문제 없음" 한 줄만 출력해줘.
```

### 실행 여부를 명시한다

스킬이 명령을 제안만 할지, 실제로 실행할지를 명시한다.

```markdown
<!-- 제안만 -->
커밋 메시지를 세 가지 제안해줘. 실제로 커밋하지 마.

<!-- 실행까지 -->
분석 후 git commit까지 실행해줘. push는 하지 마.
```

## 현재 한계

스킬은 편리하지만 몇 가지 제약이 있다.

**실행 환경 의존성**: 스킬 파일은 텍스트 프롬프트다. Claude가 실제로 어떤 도구를 쓸 수 있는지는 실행 환경의 권한 설정에 달렸다. `Bash(git *)` 권한이 없으면 git 관련 스킬이 실행 도중 막힐 수 있다.

**버전 관리 없음**: 스킬 자체는 단순한 Markdown 파일이라 버전 히스토리를 추적하려면 git에 커밋해야 한다. 자동 버전 관리 기능은 없다.

**조건 분기 없음**: 스킬 파일 안에서 "if 조건 then A else B" 같은 로직을 넣을 수 없다. 모든 분기 처리는 Claude가 프롬프트를 해석해서 결정한다. 복잡한 분기가 필요하면 여러 스킬로 나누는 것이 낫다.

**인자는 단순 텍스트**: `$ARGUMENTS`는 텍스트 치환이다. 플래그(`--verbose`), 다중 인자(`file1 file2`) 처리도 가능하지만, 파싱은 Claude가 프롬프트를 읽고 해석하는 방식이다.

## 정리

```
스킬 = .claude/commands/<이름>.md

파일 내용 = Claude에게 전달할 프롬프트
$ARGUMENTS = 커맨드 뒤에 붙인 텍스트로 치환

프로젝트 스킬  → .claude/commands/  (팀 공유 가능)
사용자 스킬   → ~/.claude/commands/ (모든 프로젝트에서 사용)
```

반복하는 작업이 있다면 스킬로 만들어라. 작성하는 데 5분, 이후 매번 아끼는 시간은 그 몇 배다.
