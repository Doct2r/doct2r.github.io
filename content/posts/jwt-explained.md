[지난 글](/posts/sso-saml-oauth-oidc-scim-explained/)에서 OIDC의 ID 토큰이 JWT라는 걸 한 줄로만 짚고 넘어갔는데, 이번엔 그 JWT 자체를 제대로 파본다. 가장 먼저 짚어야 할 오해가 있다 — JWT는 "서명"되는 것이지 기본적으로 "암호화"되는 게 아니다.

---

## 1. JWT는 혼자가 아니다 — JOSE 워킹그룹의 6형제

JWT(JSON Web Token)는 **RFC 7519**(2015년 5월)로 정의됐는데, 이건 단독으로 나온 스펙이 아니다. IETF JOSE(JSON Object Signing and Encryption) 워킹그룹이 같은 시기에 발표한 여섯 개 RFC 중 하나다 — RFC 7515(JWS, JSON Web Signature)·RFC 7516(JWE, JSON Web Encryption)·RFC 7517(JWK, JSON Web Key)·RFC 7518(JWA, JSON Web Algorithms)·RFC 7519(JWT)·RFC 7520(예제 모음)이 전부 2015년 5월 한 묶음으로 확정됐다(마이크 존스가 자신의 블로그에 "JWT and JOSE are now RFCs!"라는 제목으로 이 소식을 직접 알렸다). 저자는 마이크 존스(마이크로소프트)·존 브래들리(핑아이덴티티)·나츠 사키무라(NRI)다.

왜 여섯 개로 쪼개졌을까 — JWT 자체는 "클레임셋(claims-set)을 어떻게 담을지"에 대한 규약일 뿐, 실제 서명(JWS)이나 암호화(JWE) 메커니즘은 각각 별도 스펙에 있고, 알고리즘 식별자는 JWA가, 키 표현 형식은 JWK가 담당한다. 즉 JWT는 그 자체로 독립된 암호 메커니즘이 아니라, JWS나 JWE 위에 JSON 클레임셋을 얹는 특정한 사용법이다.

## 2. 구조 — header.payload.signature

JWT는 점(.)으로 구분된 세 부분, `header.payload.signature`로 이뤄진다. 이때 일반 base64가 아니라 **base64url**(URL-세이프 변형)을 쓰는데, 이는 URL·헤더·쿠키에 토큰을 실을 때 일반 base64에 포함되는 `+`·`/`·`=` 문자를 퍼센트 인코딩할 필요가 없게 하려는 목적이다.

헤더에는 서명 알고리즘(`alg`)과 타입(`typ`, 보통 "JWT")이 들어가고, 페이로드에는 "클레임(claim)"이 들어간다. RFC 7519 4.1절이 정의한 등록된 클레임은 `iss`(발급자)·`sub`(주체)·`aud`(대상)·`exp`(만료시각)·`nbf`(이 시각 이전엔 무효)·`iat`(발급시각)·`jti`(토큰 고유 ID)다. 이 외에 공개 클레임과 서비스별 사설 클레임을 자유롭게 추가할 수 있다.

## 3. 가장 흔한 오해 — 서명됐다고 암호화된 게 아니다

여기가 핵심이다. 일반적으로 쓰이는 서명된 JWT(JWS)는 **무결성·진위만 보장**할 뿐 기밀성은 전혀 보장하지 않는다. 페이로드는 그냥 base64url로 인코딩돼 있을 뿐이라, jwt.io 같은 사이트에 붙여넣으면 키 없이도 누구나 그대로 디코딩해서 읽을 수 있다. 서명은 "이 내용이 위조되지 않았고 주장된 발급자가 맞다"는 것만 증명하지, "이 내용을 남이 못 읽는다"는 것과는 별개 문제다. 실제로 개발자들이 이걸 혼동해 페이로드에 비밀번호나 민감한 개인정보를 그대로 넣었다가 유출되는 사고가 반복되는 이유이기도 하다. 진짜 기밀성이 필요하면 JWE를 쓰거나 JWS를 JWE 안에 중첩해야 하는데, 실무에서는 이 방식이 평범한 서명 JWT보다 훨씬 덜 쓰인다.

## 4. 보안 취약점 역사 — 2015년 한 번의 공개로 드러난 두 가지 공격

2015년 3월, Auth0의 팀 매클린은 "JSON 웹 토큰 라이브러리의 치명적 취약점"이라는 글에서 두 가지 공격 유형을 한 번에 공개했다.

첫째는 **"alg: none" 취약점**이다 — 일부 라이브러리가 헤더에 `{"alg":"none"}`을 넣고 서명 부분을 비워둔 토큰을, 서명 검증을 아예 건너뛴 채 "유효하다"고 받아들였다. 공격자는 아무 서명 없이도 원하는 클레임을 위조할 수 있었다.

둘째는 **알고리즘 혼동 공격(RS256→HS256 다운그레이드)**이다 — 서버가 RS256(비대칭키, 공개키/개인키 쌍)으로 서명된 토큰을 기대하는데, 검증 코드가 알고리즘을 엄격히 고정하지 않고 "토큰이 주장하는 알고리즘 + 애플리케이션이 넘긴 키"로 그냥 검증해버리면 문제가 생긴다. 공격자가 서버의 (원래 공개돼 있는) RSA 공개키를 HMAC 비밀키로 그대로 사용해 HS256으로 서명한 위조 토큰을 만들면, 순진한 검증 코드는 이걸 그대로 "유효한 서명"으로 받아들인다.

이 외에도 헤더의 `kid`(키 ID) 필드를 검증자가 그대로 신뢰해 SQL 쿼리나 파일 경로에 삽입하는 **kid 헤더 인젝션 공격**도 잘 알려진 공격 유형이다 — SQL 인젝션으로 쿼리 결과를 공격자가 아는 값(예: 빈 문자열)으로 만들어 그 값을 비밀키로 서명하거나, `/dev/null` 같은 빈 파일을 가리키게 해 빈 문자열 키로 서명하는 식이다.

지금의 모범사례는 2020년 2월 나온 **RFC 8725**("JSON 웹 토큰 모범사례")에 명시돼 있다 — 검증 라이브러리는 호출자가 허용할 알고리즘을 명시적으로 지정하도록 해야 하고, 그 외의 알고리즘은 절대 받아들이면 안 된다는 것이다. 즉 토큰 자신이 `alg`나 `kid` 헤더에 뭐라고 주장하든, 검증자는 자기가 미리 정해둔 알고리즘·키만 신뢰해야 한다는 것 — 이 원칙 하나가 위 두 공격 유형을 모두 막는다.

## 5. 무상태라는 장점, 그리고 폐기 불가라는 대가

JWT가 인기를 끈 이유는 "무상태(stateless)"이기 때문이다 — 서명만 확인하면 되니 데이터베이스 조회 없이도 검증이 끝난다. 그런데 이게 곧바로 진짜 문제로 이어진다 — 자연 만료 전에 특정 토큰 하나만 콕 집어 폐기하려면, 결국 서버 쪽에 다시 상태를 둬야 한다(보통 만료시각과 같은 TTL을 가진 jti 기반 차단목록, 실무에서는 Redis 같은 걸 많이 쓴다). 한 문장으로 요약한 표현이 정곡을 찌른다 — "차단목록이 커질수록, 당신의 '무상태' 서버는 그만큼 더 많은 상태를 갖게 된다."

흔한 완화책은 액세스 토큰의 유효기간을 5～15분처럼 짧게 잡고, 서버에서 별도로 추적·폐기 가능한 리프레시 토큰을 함께 쓰는 것이다.

## 6. OAuth 액세스 토큰과 JWT는 사실 별개다

또 하나 자주 혼동되는 지점 — OAuth 2.0의 기본 스펙(RFC 6749)은 액세스 토큰이 반드시 JWT여야 한다고 요구하지 않는다. 아무 의미 없는 문자열인 "불투명(opaque) 참조 토큰"으로 만들고, 이걸 검증할 때는 인가서버의 인트로스펙션 엔드포인트(RFC 7662, 2015년 10월)에 물어보는 방식도 똑같이 유효한 방법이다. 반면 RFC 9068("OAuth 2.0 액세스 토큰을 위한 JWT 프로파일", 2021년 10월)은 액세스 토큰을 JWT 구조로 표준화한 별도 스펙이다.

이 둘의 트레이드오프는 명확하다 — 불투명 토큰은 매 요청마다 인가서버에 네트워크 호출을 해야 하지만 즉시 폐기가 가능하고, JWT 액세스 토큰은 리소스서버가 네트워크 호출 없이 로컬에서 바로 검증할 수 있어 빠르지만, JWT 전반이 갖는 그 "폐기 지연" 문제를 그대로 물려받는다.

## 7. 비판도 있다 — "세션에 JWT 쓰지 마라"

JWT를 향한 유명한 비판도 있다. 2016년 joepie91이 쓴 "Stop using JWT for sessions"라는 글은, JWT의 원래 설계 의도가 5분 이하의 아주 짧은 수명을 가진 토큰이었는데 이걸 일반적인 세션 메커니즘으로 확장해 쓰는 건 기존 서버측 세션 쿠키보다 더 안전하지도, 더 효율적이지도 않다고 주장한다 — "안전한 방식의 무상태 인증은 애초에 불가능하다. 어딘가엔 반드시 상태가 있어야 한다"는 게 이 글의 핵심 문장이다.

## 정리

JWT는 그 자체로 독립된 암호 메커니즘이 아니라 JOSE 워킹그룹의 여섯 스펙 중 하나이고, 흔히 쓰이는 서명된 형태(JWS)는 무결성만 보장할 뿐 암호화가 아니라서 페이로드는 누구나 읽을 수 있다. 2015년 한 번의 공개로 alg:none과 알고리즘 혼동 공격이라는 두 가지 취약점이 드러났고, 지금은 검증자가 알고리즘을 스스로 고정해야 한다는 게 표준 모범사례다. 그리고 "무상태"라는 장점은 곧 "만료 전 개별 폐기가 어렵다"는 대가와 맞물려 있어서, 짧은 만료시간과 별도의 폐기 가능한 리프레시 토큰으로 절충하는 게 일반적이다. OAuth 액세스 토큰조차 JWT일 필요가 없다는 것, 그리고 애초에 JWT를 장기 세션에 쓰는 것 자체에 대한 비판이 있다는 것까지 감안하면, JWT는 만능 토큰 포맷이 아니라 특정 트레이드오프(빠른 로컬 검증 vs 즉시 폐기 불가)를 감수하겠다는 명시적 선택으로 봐야 한다.
