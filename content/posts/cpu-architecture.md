"M4가 더 좋아요, x86이 더 좋아요?" 라는 질문은 아키텍처를 모르면 답하기 어렵다. 단순히 클럭 수나 코어 수가 아니라, 설계 철학 자체가 다르기 때문이다. 아키텍처를 이해하면 왜 같은 GHz라도 성능이 다른지, 왜 같은 앱이라도 플랫폼마다 다시 빌드해야 하는지가 보인다.

---

## 아키텍처란

CPU 아키텍처는 두 개의 층으로 나뉜다. 이 둘을 혼동하면 대화가 어긋난다.

### ISA (Instruction Set Architecture, 명령어 집합 구조)

소프트웨어와 하드웨어 사이의 **계약**이다. "이 명령어를 주면 이런 결과를 낸다"는 인터페이스 규격이다.

```
소프트웨어 (컴파일된 바이너리)
        ↓  ISA에 정의된 명령어
CPU 하드웨어 (ISA를 구현한 실리콘)
```

같은 ISA라면 제조사가 달라도 같은 바이너리가 돌아간다. x86-64 ISA로 컴파일한 프로그램은 Intel CPU에서도, AMD CPU에서도 실행된다. 이것이 ISA의 역할이다.

### 마이크로아키텍처 (Microarchitecture)

ISA를 **어떻게 구현하는가**에 대한 설계다. 같은 ISA를 쓰더라도 파이프라인 길이, 캐시 계층, 분기 예측기, 비순서 실행(Out-of-Order Execution) 방식이 제조사마다, 세대마다 다르다.

Intel의 Alder Lake와 Raptor Lake는 둘 다 x86-64 ISA지만 마이크로아키텍처가 다르다. 같은 코드가 다른 성능으로 돌아간다.

**흔한 혼동:**
- "ARM 아키텍처" → ISA를 가리킴
- "Cortex-A78 아키텍처" → 마이크로아키텍처를 가리킴
- 둘 다 "아키텍처"라고 부르므로 문맥으로 구분해야 한다.

## 어떤 종류가 있는가

### CISC vs RISC — 근본적 갈림길

모든 현대 아키텍처는 이 두 철학에서 출발한다.

**CISC (Complex Instruction Set Computer)**
- 복잡한 명령어를 하드웨어가 직접 처리
- 명령어 하나로 여러 연산을 수행 가능
- 코드 밀도가 높음 (같은 일을 더 적은 명령어로)
- 명령어 길이가 가변적
- 대표: x86, x86-64

```asm
; x86: 메모리에서 바로 더하기 (CISC)
ADD EAX, [mem_address]   ; 메모리 읽기 + 덧셈을 한 명령어로
```

**RISC (Reduced Instruction Set Computer)**
- 단순한 명령어, 하드웨어는 단순하게
- 명령어 하나는 하나의 연산만
- 명령어 길이가 고정 (ARM은 32비트)
- 파이프라이닝에 유리
- 대표: ARM, RISC-V, MIPS, PowerPC

```asm
; ARM: 메모리 읽기 따로, 덧셈 따로 (RISC)
LDR R1, [R0]    ; 메모리 읽기
ADD R2, R2, R1  ; 덧셈
```

RISC가 단순하다고 느리지 않다. 오히려 단순한 명령어를 고속 파이프라인으로 병렬 처리하면 CISC보다 빠를 수 있다. 현대 Intel/AMD CPU는 내부에서 CISC 명령어를 RISC 스타일의 마이크로 연산(μop)으로 분해해서 처리한다. 사실상 RISC 코어 위에 CISC 디코더를 얹은 구조다.

### 주요 아키텍처별 특성

#### x86 / x86-64

- **소유**: Intel (원작), AMD (호환 구현 후 공동 사실상 표준화)
- **비트**: x86은 32비트, x86-64(AMD64)는 64비트 확장
- **특징**: 수십 년의 소프트웨어 생태계. Windows, 기업 서버, 데스크탑의 지배자
- **약점**: 명령어 복잡성으로 인한 높은 전력 소비, 발열

Intel과 AMD는 같은 x86-64 ISA를 구현하지만 마이크로아키텍처가 다르다. Intel은 Ring Bus → Mesh 구조로, AMD는 Chiplet(MCM) 구조로 설계 방향이 갈렸다.

```
x86-64 레지스터 (일부):
RAX, RBX, RCX, RDX  (범용 64비트)
RSP (스택 포인터), RBP (베이스 포인터)
R8 ~ R15 (추가 범용)
```

---

#### ARM (Advanced RISC Machine)

- **소유**: ARM Holdings (영국). 설계만 하고 제조는 라이선스 받은 회사가 함
- **버전**: ARMv7(32비트), ARMv8/AArch64(64비트), ARMv9(최신)
- **특징**: 모바일의 표준. 낮은 전력, 높은 성능/전력 비율
- **구현체**: Apple A/M 시리즈, Qualcomm Snapdragon, Samsung Exynos, MediaTek, AWS Graviton, Ampere Altra

ARM의 핵심은 **라이선스 모델**이다. ARM은 칩을 만들지 않는다. 설계를 판다. 두 종류의 라이선스가 있다.

- **IP 라이선스**: 완성된 코어 설계(Cortex 시리즈)를 그대로 사용. 삼성 Exynos 일부, MediaTek
- **아키텍처 라이선스**: ISA만 라이선스 받고 마이크로아키텍처는 직접 설계. **Apple, Qualcomm(Oryon), Amazon(Graviton)**

Apple의 M 시리즈가 ARM ISA를 쓰면서도 Intel/Qualcomm보다 월등한 성능을 내는 이유가 여기 있다. ARM이 설계한 Cortex 코어를 쓰는 게 아니라, Apple이 ARMv8/v9 ISA를 기반으로 자체 마이크로아키텍처를 설계했기 때문이다.

---

#### RISC-V

- **소유**: RISC-V International (비영리). 누구나 무료로 사용 가능
- **특징**: 완전한 오픈소스 ISA. 로열티 없음
- **현황**: 임베디드, IoT, 학술 분야에서 빠르게 확산 중. 서버급 구현도 등장 중
- **의의**: ARM에 대한 종속성을 탈피하려는 움직임의 중심

중국이 미국의 ARM 라이선스 제재에 대응하기 위해 RISC-V 투자를 급격히 늘리고 있다. SiFive, Alibaba T-Head, StarFive 등이 RISC-V 기반 칩을 출시했다. 장기적으로 ARM의 유력한 대안이 될 가능성이 있다.

---

#### PowerPC / POWER

- **소유**: IBM (POWER 아키텍처)
- **역사**: 1990년대 Apple Mac이 PowerPC를 사용했다가 2006년 Intel로 전환
- **현재**: IBM의 고성능 서버(POWER10), PlayStation 3, Xbox 360, Wii에 사용됐음
- **특징**: 빅엔디안(Big-Endian) 기본 (x86은 리틀엔디안)

---

#### MIPS

- **역사**: 1985년. 초기 RISC 아키텍처의 교과서
- **현재**: 임베디드, 네트워크 장비, 일부 게임 콘솔(PlayStation 1/2)
- **현황**: 사실상 새 설계에서는 RISC-V에 자리를 내주는 추세

---

#### LoongArch

- **소유**: 중국 Loongson
- **배경**: x86, ARM 없이 자체 아키텍처 구축을 목표로 한 중국의 전략적 산물
- **현황**: 중국 공공기관, 군사 용도로 채택 중. 민간 시장 점유는 아직 미미

---

## 뭐가 다른가

같은 ISA라도, 다른 ISA라도 성능 차이를 만드는 요소들이 있다.

### 파이프라인과 비순서 실행

명령어를 순서대로 처리하면 앞 명령어가 끝날 때까지 다음이 기다린다. 현대 CPU는 명령어 간 의존성이 없으면 순서를 바꿔 동시에 처리한다(Out-of-Order Execution). 이 능력이 높을수록 IPC(Instructions Per Cycle)가 높아진다.

### 캐시 계층

메인 메모리(RAM)는 CPU보다 수십~수백 배 느리다. 자주 쓰는 데이터를 CPU 가까이 두는 캐시(L1/L2/L3)가 실질적 성능을 결정한다.

```
접근 속도 (대략):
L1 캐시:  ~1ns    (수십 KB)
L2 캐시:  ~5ns    (수백 KB ~ 수 MB)
L3 캐시:  ~20ns   (수십 MB)
RAM:      ~70ns   (수 GB ~ 수백 GB)
SSD:      ~100μs
```

Apple M 시리즈의 체감 속도가 높은 이유 중 하나는 유난히 큰 L2/L3 캐시와 낮은 메모리 레이턴시(SoC 내부 통합 메모리)다.

### 메모리 아키텍처

- **분리 메모리**: CPU와 RAM이 별도. 전통적 PC 구조
- **통합 메모리(UMA)**: CPU, GPU, NPU가 같은 메모리 풀을 공유. Apple Silicon, 최근 모바일 AP

UMA에서는 CPU가 처리한 데이터를 GPU로 넘길 때 복사가 필요 없다. 이것이 Apple M 시리즈의 GPU/ML 성능이 상대적으로 우수한 구조적 이유다.

### 엔디안 (Endianness)

같은 데이터를 메모리에 저장하는 바이트 순서.

```
숫자 0x12345678을 저장할 때:

리틀엔디안 (x86, ARM 기본):  78 56 34 12  (낮은 바이트 먼저)
빅엔디안 (PowerPC, 네트워크): 12 34 56 78  (높은 바이트 먼저)
```

네트워크 프로토콜은 빅엔디안(네트워크 바이트 오더)을 사용한다. x86에서 네트워크 코드를 짤 때 `htonl()`, `ntohl()` 같은 변환 함수가 필요한 이유다.

## 세대(Gen)로 구별할 수 있는가

아키텍처마다 세대 구분이 있지만, 의미 있는 단절과 점진적 개선을 구분해야 한다.

### x86 / x86-64 세대

| 시기 | 마이크로아키텍처 | 주요 변화 |
|---|---|---|
| 1978 | 8086 | x86 ISA 시작. 16비트 |
| 1985 | 386 (IA-32) | 32비트 확장 |
| 2003 | AMD64 (x86-64) | 64비트 확장. AMD가 먼저 제안 |
| 2006 | Intel Core (Yonah/Merom) | 효율성 중심으로 방향 전환 |
| 2011 | Sandy Bridge | CPU+GPU 통합(내장 그래픽) 본격화 |
| 2021 | Alder Lake (12세대) | **P코어+E코어 이종 설계** 도입. x86 역사적 전환 |
| 2023 | Raptor Lake (13/14세대) | Alder Lake 개선 |
| 2024 | Arrow Lake (15세대) | 아키텍처 분리 심화 |

Alder Lake는 단순한 업그레이드가 아니다. ARM의 big.LITTLE처럼 성능 코어(P-core)와 효율 코어(E-core)를 섞은 이종 설계(Hybrid Architecture)를 x86에 도입한 패러다임 전환이다. OS의 스케줄러가 이를 인식해야 제대로 작동한다.

### ARM ISA 세대

| 버전 | 특징 | 대표 기기 |
|---|---|---|
| ARMv6 | 32비트. Cortex-A8 | 초기 iPhone (1세대~3GS) |
| ARMv7 | 32비트 개선. NEON SIMD | iPhone 4~5, 초기 Android |
| ARMv8 / AArch64 | **64비트 도입** | iPhone 5s(A7), 현대 Android |
| ARMv9 | 보안 강화(CCA), SVE2 | Cortex-X3, Apple A16~ |

ARMv8(AArch64) 전환이 가장 큰 단절이다. 32비트에서 64비트로 넘어가면서 레지스터가 16개에서 31개로 늘고, 부동소수점·SIMD가 통합됐다. 64비트 전용 앱이 32비트 기기에서 돌지 않는 이유다.

### Apple Silicon 세대

| 칩 | 연도 | 주요 변화 |
|---|---|---|
| M1 | 2020 | Apple Silicon 첫 세대. x86 Mac 대체 시작 |
| M1 Pro/Max/Ultra | 2021 | 코어 수·메모리 대역폭 확장 |
| M2 시리즈 | 2022~2023 | ARMv9, 미디어 엔진 강화 |
| M3 시리즈 | 2023~2024 | 3nm 공정, 하드웨어 레이 트레이싱 |
| M4 시리즈 | 2024~ | NPU 강화 (Apple Intelligence), 2nm 공정 |

Apple Silicon은 ISA 세대(ARMv8→v9)와 마이크로아키텍처 개선이 동시에 일어나므로 세대 구분이 명확하다.

### 모바일 AP — Cortex 코어 네이밍

ARM이 설계한 코어는 성능 등급에 따라 세 계열로 나뉜다.

| 계열 | 역할 | 예시 |
|---|---|---|
| Cortex-X | 최고 성능, 고전력 | X1, X2, X3, X4 |
| Cortex-A7xx | 성능/효율 균형 | A710, A720 |
| Cortex-A5xx | 효율 우선, 저전력 | A510, A520 |

Snapdragon 8 Gen 3의 구성 예:
```
1× Cortex-X4   (최고 성능 코어)
5× Cortex-A720 (성능 코어)
2× Cortex-A520 (효율 코어)
```
이 구성이 big.LITTLE의 현대 버전이다.

## 주의사항 및 한계

### ISA가 같아도 바이너리가 안 돌 수 있다

ARM ISA가 같아도 ARMv7용 바이너리는 ARMv8에서 돌지만, ARMv8(64비트) 전용 바이너리는 ARMv7(32비트) 기기에서 돌지 않는다. 상위 호환이지 하위 호환이 아니다.

Android가 32비트 앱 지원을 끊는 이유, iOS가 32비트 앱 지원을 iOS 11에서 종료한 이유가 여기 있다.

### 클럭(GHz)으로 성능을 비교하면 안 된다

다른 ISA, 다른 마이크로아키텍처 간의 클럭 비교는 의미가 없다. Apple M4 3.4GHz와 Intel Core i9 5.8GHz를 클럭으로 비교하면 틀린다. IPC(클럭당 처리 명령어 수)와 아키텍처 효율이 다르기 때문이다.

같은 ISA, 같은 마이크로아키텍처 내에서는 클럭이 의미 있다. 다른 순간 클럭 비교는 항상 의심해야 한다.

### 세대 번호 = 성능 순서가 아니다

Intel 12세대(Alder Lake)가 13세대(Raptor Lake)보다 나은 경우도 있다. 세대 번호는 출시 순서지, 절대 성능 순서가 아니다. 벤치마크 없이 세대 번호만 보고 판단하면 틀린다.

### 열 설계 한계 (TDP)

같은 칩이라도 노트북(저전력 모드)과 데스크탑(고전력)에서 성능이 크게 다르다. 클럭이 같아도 서멀 스로틀링(과열로 인한 클럭 감속)이 얼마나 일어나는지가 실 성능을 결정한다.

### RISC-V의 파편화 가능성

RISC-V는 오픈소스라 누구든 확장 명령어를 추가할 수 있다. 이것이 강점이지만, 표준 확장 외에 벤더별 커스텀 확장이 많아지면 ARM처럼 단일 바이너리가 모든 RISC-V 기기에서 돌지 않는 문제가 생긴다. 현재 진행 중인 과제다.

### 크로스컴파일의 필요

다른 아키텍처용 바이너리를 만들려면 크로스컴파일러가 필요하다. x86 개발 머신에서 ARM용 임베디드 펌웨어를 빌드하거나, M1 Mac에서 x86 Linux 바이너리를 빌드할 때 발생하는 복잡성이다. Docker의 `--platform` 옵션이 이 문제를 다루는 이유다.

```bash
# x86 머신에서 ARM용 Docker 이미지 빌드
docker buildx build --platform linux/arm64 -t myapp:arm64 .
```

## 정리

| 구분 | x86-64 | ARM (AArch64) | RISC-V |
|---|---|---|---|
| 소유 | Intel/AMD | ARM Holdings | 오픈소스 |
| 주 용도 | 데스크탑, 서버 | 모바일, 임베디드, 서버 증가 | 임베디드, 미래 범용 |
| 전력 효율 | 낮음 | 높음 | 높음 (설계에 따라) |
