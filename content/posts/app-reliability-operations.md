앱은 출시하는 순간 끝나는 게 아니라 그때부터 진짜 시작이다. 화면 밖에서 사용자 데이터를 지키고, 문제가 생겼을 때 알아채고, 고치기 전에 미리 잡아내고, 새 버전을 안전하게 내보내는 일 — 이 마지막 편은 앱을 "믿고 계속 쓸 수 있게" 만드는 운영 요소들이다.

---

## 1. 보안 — 사용자가 절대 보지 못하는 방어선

### 저장 데이터 암호화

로그인 토큰이나 민감 정보를 일반 파일이나 평문 설정값에 저장하면 기기를 탈취당했을 때 그대로 노출된다. iOS의 **키체인(Keychain)**, Android의 **키스토어(Keystore)**는 OS 수준에서 암호화·격리된 저장소를 제공하며, 민감 정보는 반드시 여기 저장해야 한다.

### 통신 암호화

모든 네트워크 통신은 TLS로 암호화되어야 하며, 금융·인증처럼 민감한 앱은 **인증서 피닝(Certificate Pinning)**을 추가해 중간자 공격(MITM)을 막는다. 공개 와이파이 환경에서 앱을 쓰는 사용자가 실제로 존재하는 이상 이건 선택이 아니라 기본값이다.

### 인증과 인가

```
인증(Authentication) : 이 사용자가 누구인가 — 로그인, OAuth2/OIDC
인가(Authorization)  : 이 사용자가 이 작업을 할 권한이 있는가 — 역할·스코프 기반 접근 제어
```

토큰을 어디에 저장하는지(위 키체인/키스토어), 토큰 만료 시 갱신 흐름이 매끄러운지(1편에서 다룬 리프레시 토큰)까지가 보안 설계의 연장선이다.

## 2. 로깅 · 크래시 리포팅 · 텔레메트리 — 사용자가 말 안 해도 문제를 알아채는 법

대부분의 사용자는 앱이 이상해도 신고하지 않고 그냥 지운다. 그래서 개발자가 먼저 알아채는 체계가 없으면 이탈 원인을 영영 모른다.

| 종류 | 목적 | 대표 도구 |
|---|---|---|
| **크래시 리포팅** | 앱이 강제 종료된 순간의 스택 트레이스 자동 수집 | Firebase Crashlytics, Sentry |
| **구조화 로깅** | 크래시까진 아니지만 추적이 필요한 이벤트를 기록 | 자체 로그 서버, Datadog 등 |
| **텔레메트리/애널리틱스** | 기능 사용률, 이탈 지점 등 사용자 행동 데이터 | Firebase Analytics, Amplitude |

세 가지는 목적이 다르지만 공통 함정이 있다 — **개인정보(PII)를 로그에 그대로 남기면 안 된다.** 이메일, 전화번호, 위치 같은 정보는 마스킹하거나 아예 수집 대상에서 제외해야 하며, 이는 GDPR 같은 규제 준수 이전에 기본적인 설계 책임이다.

---

## 3. 테스트 전략 — 무엇을, 얼마나 자동으로 검증할 것인가

```
테스트 피라미드 (아래로 갈수록 많이, 위로 갈수록 적게):
  E2E 테스트    : 실제 사용자 흐름 전체를 자동화로 검증 (느리고, 깨지기 쉬움 — 최소한만)
  통합 테스트   : 여러 모듈이 맞물려 동작하는지 검증
  유닛 테스트   : 함수·클래스 단위 최소 단위 검증 (빠르고, 가장 많이 작성)
```

E2E 테스트가 많을수록 안심될 것 같지만 실제로는 환경 의존성 때문에 **플레이키 테스트(Flaky Test)** — 코드는 안 바뀌었는데 실행할 때마다 성공/실패가 뒤바뀌는 테스트 — 가 늘어나 오히려 신뢰도를 깎아먹는다. 피라미드 아래쪽(유닛 테스트)에 많이 투자하고, 위쪽(E2E)은 핵심 시나리오만 최소한으로 유지하는 게 정석이다.

## 4. 배포 안전장치 — "언제 낼지"가 아니라 "어떻게 사고 없이 낼지"

이 주제는 릴리즈 주기(얼마나 자주 배포할지, CI/CD 파이프라인 구조)와는 다른 질문이다. 여기서 다루는 건 **"새 버전을 실제로 사용자에게 내보낼 때 사고를 최소화하는 장치"**다.

- **단계적 롤아웃(Staged Rollout)**: 새 버전을 전체 사용자가 아니라 1% → 10% → 50% → 100% 순으로 점진 배포해, 문제가 있으면 초기에 적은 피해로 멈출 수 있게 함
- **피처 플래그(Feature Flag)**: 새 기능을 앱 업데이트 없이 서버 설정값으로 켜고 끌 수 있게 만드는 장치. 문제가 생기면 스토어 재배포 없이 즉시 기능을 끌 수 있다
- **킬 스위치(Kill Switch)**: 심각한 결함이 발견됐을 때 특정 기능(또는 앱 전체)을 원격으로 강제 비활성화하는 최후 수단

이 세 가지는 "완벽하게 테스트했으니 안전하다"는 가정 대신 **"문제는 반드시 생기니, 생겼을 때 빨리 되돌릴 수 있게 만든다"**는 사고방식에 기반한다.

---

## 5. 성능 모니터링(APM)

기능이 정상 작동해도 느리면 사용자는 이탈한다. APM(Application Performance Monitoring)은 아래 지표들을 실사용 환경에서 지속적으로 추적한다.

```
앱 시작 시간(Cold/Warm Start)  : 첫 화면이 뜨기까지 걸리는 시간
프레임 드랍(Jank)              : 애니메이션·스크롤이 끊기는 구간
메모리 사용량과 누수           : 오래 켜둘수록 느려지거나 강제 종료되는 원인
네트워크 응답 시간             : API별 실제 사용자 체감 지연
```

개발 환경에서는 문제없던 것이 실제 사용자의 저사양 기기·느린 네트워크에서는 완전히 다르게 나타나는 경우가 흔하기 때문에, 실사용 데이터 기반 모니터링 없이는 "내 폰에서는 잘 되는데요"의 함정에서 벗어날 수 없다.

## 정리

```
보안        → 저장 데이터 암호화(키체인/키스토어), 통신 암호화(TLS+인증서 피닝), 인증/인가 분리
로깅·리포팅  → 크래시/로그/애널리틱스는 목적이 다름, 공통적으로 PII 마스킹 필수
테스트      → 유닛(많이)-통합-E2E(최소한) 피라미드, 플레이키 테스트 관리가 핵심
배포 안전장치→ 단계적 롤아웃·피처 플래그·킬 스위치 — "안 터지게"가 아니라 "터져도 빨리 되돌리게"
성능 모니터링→ 시작시간·프레임드랍·메모리·네트워크를 실사용 환경 기준으로 추적
```

1편의 아키텍처·데이터, 2편의 사운드·시스템 통합, 그리고 이번 편의 신뢰성·운영까지가 UX/GUI 뒤에 숨어 있는 "앱을 앱답게 만드는" 나머지 절반이다. 화면은 사용자가 매일 보는 부분이지만, 이 나머지 절반이 없으면 그 화면은 오래 버티지 못한다.
