[부트스트랩](/posts/bootstrap-framework-explained/)이 미리 스타일이 입혀진 컴포넌트 단위로 접근한다면, 테일윈드CSS는 정반대다 — `flex`·`pt-4`·`text-center` 같은 원자 단위 클래스를 직접 조합해서 쓴다. 이 선택은 처음부터 논쟁적이었고, 그 논쟁을 감당하고 성장하는 과정에서 테일윈드는 자기 엔진을 세 번이나 갈아치웠다.

---

## 1. 유틸리티 퍼스트 논쟁 — "이게 관심사 분리를 어긴 거 아닌가"

아담 와단은 테일윈드가 나오기 석 달 전인 2017년 8월 7일 "CSS 유틸리티 클래스와 '관심사 분리'"라는 글을 썼다. 논리 구조는 리액트의 JSX 논쟁 때 피트 헌트가 폈던 주장과 정확히 같다 — "관심사 분리"가 "언어·파일별 분리"를 뜻할 필요는 없고, 스타일링과 마크업은 애초에 같은 관심사인 경우가 많다는 것이다.

이 주장이 맞서야 했던 비판은 명확했다. 버튼 하나에 `inline-flex items-center justify-center rounded-md px-4 py-2 text-sm font-medium...` 식으로 클래스가 15개 넘게 붙는 걸 두고 "클래스 수프"라는 조롱이 따라다녔고, 지금도 여러 해에 걸쳐 반복되는 비판이다. 이에 대한 표준적인 반론은 두 가지다 — 리액트·뷰 같은 컴포넌트 기반 프레임워크에서는 이 "수프"를 재사용 컴포넌트 안에 한 번만 써두면 호출하는 쪽에서는 다시 볼 일이 없고, VS Code용 공식 확장 프로그램 "Tailwind CSS IntelliSense"가 자동완성·호버 미리보기·린팅을 지원해 실제로 코드를 짜는 경험은 원본 코드 샘플만 봤을 때보다 훨씬 덜 고통스럽다는 것이다.

## 2. 핵심 기술 — 유틸리티가 쌓이며 생기는 문제를 어떻게 풀었나

모든 값의 조합에 대해 클래스를 미리 다 만들어두면 스타일시트가 감당 못 할 만큼 커진다는 게 유틸리티 퍼스트 접근의 뻔한 약점이다. 초기 버전(1.x·2.x)에서는 PurgeCSS라는 별도 도구를 후처리 단계로 붙여, 최종 HTML·JS 결과물을 스캔해서 실제로 안 쓰인 클래스를 스타일시트에서 걷어내는 방식으로 이 문제를 풀었다.

2021년 3월 `@tailwindcss/jit`라는 실험적 패키지로 JIT(Just-In-Time) 엔진이 나오면서 접근 자체가 바뀐다. 모든 유틸리티를 미리 다 만들어놓고 나중에 걷어내는 대신, 소스 파일을 스캔해서 실제로 쓰인 클래스에 대한 CSS만 그때그때 즉석에서 생성하는 방식이다. 이 엔진은 2.1 버전에서 코어에 병합됐고, 2021년 12월 9일 나온 3.0에서 마침내 기본이자 유일한 엔진이 됐다 — 이때부터 `w-[137px]`처럼 디자인 스케일에 없는 임의의 값을 대괄호 문법으로 바로 쓸 수 있게 됐다.

## 3. 4.0 — 러스트로 다시 짠 Oxide 엔진

2025년 1월 말 나온 4.0은 엔진 자체를 러스트로 전면 재작성한 "Oxide" 엔진을 채택했다 — 기존 자바스크립트 구현 대신 러스트 기반 파서·압축기인 Lightning CSS를 쓴다. 테일윈드 랩스 자체 발표 수치로는 전체 빌드가 378ms에서 100ms로, 새 CSS가 있는 증분 빌드가 44ms에서 5ms로, CSS 변경이 없는 재빌드는 35ms에서 192마이크로초로 줄었다고 하는데, 이건 회사가 직접 발표한 벤치마크지 독립적으로 검증된 수치는 아니라는 점을 감안해야 한다. 더 실질적인 변화는 설정 방식이다 — 자바스크립트 파일인 `tailwind.config.js` 대신 네이티브 CSS의 `@theme` 지시문으로 설정하는 CSS 우선 방식으로 바뀌었고, 네이티브 CSS 캐스케이드 레이어 지원과 스캔할 파일 목록을 일일이 지정하지 않아도 되는 자동 콘텐츠 감지도 함께 들어왔다.

## 4. 사업 모델 — Tailwind UI에서 Tailwind Plus로

테일윈드의 지속적인 개발 자금줄은 특이하다. 아담 와단은 2019년 1월 테일윈드에 풀타임으로 전념하기 시작했고, 2020년 2월 26일 유료 컴포넌트 라이브러리 Tailwind UI를 출시했다 — 첫날에만 50만 달러 매출을 올렸고, 이후 200만 달러를 넘겼다. 구독이 아니라 한 번 사면 평생 쓰는 일회성 구매 모델이었고, 이 방식으로 2년이 채 안 되는 기간에 400만 달러 넘는 매출을 냈다 — 벤처 투자 없이 순수하게 이 수익만으로 오픈소스 개발을 지탱한 셈이다. 2025년 3월 4일에는 이 제품이 Tailwind Plus로 리브랜딩됐는데, 기존 라이선스 보유자는 추가 비용 없이 그대로 이전됐다.

## 5. 생태계 중력 — shadcn/ui

테일윈드가 리액트·Next.js 생태계에서 얼마나 중심적인 위치를 차지하는지 보여주는 사례가 shadcn/ui다. 제작자는 "shadcn"이라는 가명만 공개돼 있고 실명을 밝히지 않는 게 이 프로젝트의 의도된 정체성 중 하나다. 2022년 말 깃허브에 처음 올라왔고 2023년 1월 정식 출시됐는데, 이 프로젝트의 특이한 점은 npm으로 설치하는 패키지가 아니라는 것이다 — CLI가 컴포넌트 소스 코드 자체를 프로젝트 안에 그대로 복사해 넣는 "카피 앤 페이스트" 방식이라, 블랙박스 의존성도 없고 버전 관리 문제도 없다. Radix UI(접근성·키보드·포커스 처리)와 테일윈드CSS(스타일링)를 조합해 만들어졌다.

## 6. 채택 통계의 함정

테일윈드의 실제 채택 규모를 이야기할 때는 조심할 게 있다. 2025년 State of CSS 설문에서는 테일윈드가 압도적 1위였지만, 웹사이트를 직접 크롤링해 집계하는 W3Techs 기준으로는 2026년 2월 현재 전체 웹사이트의 약 0.3%(CSS 프레임워크를 쓰는 걸로 확인된 사이트 중에서도 1.6%)에서만 테일윈드가 검출됐다 — 같은 비교에서 부트스트랩은 74.9%였다. 이 엄청난 격차의 이유는 기술적이다 — 테일윈드는 빌드 후 평범한 CSS로 완전히 컴파일돼버려서, 부트스트랩의 `.navbar-toggler`·`.container-fluid` 같은 식별 가능한 클래스 흔적이 전혀 안 남는다. 그래서 크롤러 기반 탐지 도구는 테일윈드를 체계적으로 과소 집계할 수밖에 없다. 개발자 설문(자기 응답, 아주 높게 나옴)과 크롤링 탐지(W3Techs, 아주 낮게 나옴) 중 어느 하나만 놓고 "이게 진짜 채택률"이라고 말하면 안 된다는 게 이 사례가 주는 정직한 교훈이다.

## 정리

부트스트랩이 jQuery라는 외부 의존성 하나 때문에 발목이 잡혔던 것과 대칭적으로, 테일윈드는 유틸리티가 쌓이며 생기는 성능 문제를 PurgeCSS에서 JIT로, 다시 Oxide로 엔진 자체를 세 번이나 갈아치우며 스스로 풀어왔다. 그리고 이 개발을 지탱한 돈은 벤처 투자가 아니라 창업자가 직접 판 유료 컴포넌트 라이브러리였다는 점에서, 테일윈드는 기술적 선택뿐 아니라 사업 모델에서도 이례적인 궤적을 그려온 프로젝트다.
