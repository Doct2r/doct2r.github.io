"에뮬레이터랑 시뮬레이터가 뭐가 달라요?" 라고 물으면 대부분 얼버무린다. 가상머신과 컨테이너의 차이를 정확히 설명할 수 있는 사람도 생각보다 적다. 이 개념들은 모두 "실제가 아닌 환경을 만든다"는 공통점이 있지만, 어떻게 만드는지가 근본적으로 다르다.

---

## 용어들이 왜 헷갈리는가

혼란의 근원은 일상 언어에서 이 단어들이 섞여 쓰인다는 것이다.

- 게임보이 에뮬레이터를 "게임보이 시뮬레이터"라고 부르는 사람이 있다.
- Xcode의 "iOS 시뮬레이터"는 사실 시뮬레이터가 맞지만, Android의 "에뮬레이터"는 에뮬레이터가 맞다.
- Docker를 "가상머신"이라고 부르는 사람이 있지만 정확하지 않다.
- Wine을 "Windows 에뮬레이터"라고 소개하지만, 공식 명칭은 "호환성 레이어"다.

각각이 정확히 무엇인지 정의부터 보자.

---

## 1. 시뮬레이터 (Simulator)

> **행동(결과)을 흉내 내지만, 내부 구조까지 재현하지는 않는다.**

시뮬레이터는 대상 시스템의 **외부 동작**을 모방한다. 내부 메커니즘이 실제와 같을 필요가 없다. 결과가 같으면 된다.

**Xcode iOS 시뮬레이터**가 대표적인 예다.

```
실제 iPhone:  ARM 아키텍처 CPU에서 ARM 코드 실행
iOS 시뮬레이터: Mac의 ARM/x86 CPU에서 네이티브 코드로 실행
               (CPU 명령어를 번역하지 않음)
```

iOS 시뮬레이터에서 앱은 실제 iPhone CPU 명령어를 번역하는 게 아니라, Mac에서 바로 돌아가도록 컴파일된 버전으로 실행된다. UI와 API 동작을 흉내 낼 뿐, iPhone 하드웨어를 재현하지 않는다. 그래서 빠르다. 그래서 카메라, Bluetooth, NFC 같은 하드웨어 의존 기능은 시뮬레이터에서 동작하지 않는다.

**비행 시뮬레이터**도 마찬가지다. 조종 물리를 수식으로 모델링할 뿐, 실제 항공기 엔진 내부 연소 과정을 재현하지 않는다.

**핵심:** 목적은 훈련, 테스트, 동작 검증. 정확성보다 속도와 편의성이 우선이다.

## 2. 에뮬레이터 (Emulator)

> **내부 구조(하드웨어, 명령어 셋)까지 소프트웨어로 재현한다.**

에뮬레이터는 대상 시스템의 **하드웨어를 소프트웨어로 구현**한다. 대상 시스템의 CPU 명령어를 하나씩 해석하고 번역해서 실행한다. 느리지만 정확하다.

**게임보이 에뮬레이터(mGBA 등)**가 대표적이다.

```
실제 게임보이:  Sharp LR35902 (Z80 계열) CPU가 직접 ROM 실행
에뮬레이터:    x86/ARM CPU가 Z80 명령어를 소프트웨어로 해석 및 번역
               Z80 레지스터, 메모리 맵, 타이밍까지 재현
```

에뮬레이터는 게임보이의 CPU, 메모리 컨트롤러, PPU(픽셀 처리 유닛), APU(오디오)를 소프트웨어로 구현한다. 실제 칩이 하는 일을 소프트웨어가 대신한다.

**Android 에뮬레이터**도 에뮬레이터다. x86 Mac에서 ARM 기반 Android 기기를 에뮬레이션한다. ARM CPU 명령어를 x86으로 변환하기 때문에 느리다. Apple Silicon Mac이 등장한 뒤로는 ARM-on-ARM이라 훨씬 빨라졌다.

**QEMU**는 에뮬레이터의 정점이다. x86 머신에서 MIPS, RISC-V, PowerPC 등 전혀 다른 아키텍처를 완전히 에뮬레이션할 수 있다.

**핵심:** 목적은 정확한 재현. 속도보다 정확성이 우선이다. 하드웨어 없이 소프트웨어로 그 하드웨어가 필요한 바이너리를 실행할 수 있다.

## 3. 가상머신 (Virtual Machine, VM)

> **물리 하드웨어 위에 가상의 하드웨어를 만들어 완전한 OS를 올린다.**

에뮬레이터처럼 하드웨어를 소프트웨어로 재현하지만, 핵심 차이가 있다. VM은 보통 **같은 아키텍처**를 가상화한다. x86 CPU 위에서 x86 가상 CPU를 만든다. 그래서 대부분의 명령어를 번역 없이 직접 실행(Native Execution)할 수 있다.

```
[물리 하드웨어: x86 CPU]
        ↓
[하이퍼바이저]
        ↓
[가상 하드웨어: 가상 CPU, 가상 RAM, 가상 디스크, 가상 NIC]
        ↓
[게스트 OS: Windows, Linux, ...]
```

게스트 OS는 자신이 가상 환경에 있는지 모른다(모를 수도 있다). 실제 하드웨어 위에 있다고 생각하며 동작한다.

**하이퍼바이저(Hypervisor)** 가 가상 하드웨어를 관리하는 소프트웨어다. 두 종류가 있다.

### Type 1 하이퍼바이저 (Bare-metal)

OS 없이 하드웨어 위에 직접 설치된다. 성능이 높다.

```
[하드웨어]
[Type 1 하이퍼바이저] ← OS 없음. 하이퍼바이저가 곧 OS
[VM1] [VM2] [VM3]
```

| 제품 | 용도 |
|---|---|
| VMware ESXi | 기업 서버 가상화 표준 |
| Microsoft Hyper-V (Server) | Windows Server 기반 인프라 |
| KVM (Linux 커널 내장) | 리눅스 기반 서버 가상화 |
| Xen | AWS EC2 초기 기반 (현재는 KVM + Nitro) |

### Type 2 하이퍼바이저 (Hosted)

일반 OS 위에서 애플리케이션처럼 실행된다. 편리하지만 호스트 OS를 거쳐야 해서 Type 1보다 느리다.

```
[하드웨어]
[호스트 OS: Windows / macOS / Linux]
[Type 2 하이퍼바이저 앱]
[VM1] [VM2]
```

| 제품 | 특징 |
|---|---|
| VirtualBox | 오픈소스, 무료, 크로스플랫폼 |
| VMware Workstation / Fusion | 성능 좋음, 유료 (개인은 무료) |
| Parallels Desktop | macOS 전용, Apple Silicon 최적화 |
| QEMU | 오픈소스, 에뮬레이션도 지원 |

### 하드웨어 가상화 지원 (VT-x / AMD-V)

현대 VM이 빠른 이유는 CPU가 직접 가상화를 돕기 때문이다. Intel의 VT-x, AMD의 AMD-V는 CPU 명령어 수준에서 가상화를 지원해, 게스트 OS의 명령어를 소프트웨어 번역 없이 직접 실행한다. 이 기능이 없으면 모든 명령어를 소프트웨어로 번역해야 해서 극히 느려진다.

## 4. 컨테이너 (Container)

> **OS를 가상화하는 게 아니라, OS 위의 실행 환경을 격리한다.**

컨테이너는 VM과 완전히 다른 접근이다. VM은 하드웨어를 가상화해서 OS를 통째로 올린다. 컨테이너는 **호스트 OS의 커널을 공유**하고, 프로세스와 파일시스템만 격리한다.

```
[VM 방식]                    [컨테이너 방식]
하드웨어                      하드웨어
하이퍼바이저                   호스트 OS (커널)
게스트 OS (수 GB)              ─────────────────
앱                            컨테이너A  컨테이너B
                              (앱+라이브러리)
                              커널 공유 ↑
```

Docker가 대표적이다. `docker run ubuntu`를 실행하면 Ubuntu 커널이 올라가는 게 아니라, Ubuntu의 파일시스템(라이브러리, 명령어 등)이 격리된 공간에서 호스트 커널 위에 실행된다.

**컨테이너가 VM보다 가벼운 이유:**
- 게스트 OS 커널이 없다 → 수 GB가 아닌 수십～수백 MB
- 부팅 시간이 없다 → 밀리초 단위로 시작
- 오버헤드가 적다 → 거의 네이티브 성능

**컨테이너의 한계:**
- Linux 컨테이너는 Linux 커널만 공유할 수 있다. Mac에서 Docker를 쓸 때 내부적으로 Linux VM이 먼저 뜨는 이유다(Docker Desktop = Linux VM + 컨테이너).
- VM만큼 강한 격리가 아니다. 커널 취약점은 같은 커널을 쓰는 모든 컨테이너에 영향을 줄 수 있다.

## 5. 호환성 레이어 (Compatibility Layer)

> **다른 OS의 API 호출을 현재 OS의 API 호출로 실시간 번역한다.**

에뮬레이터와 비슷해 보이지만 다르다. 에뮬레이터는 하드웨어(CPU)를 재현한다. 호환성 레이어는 **소프트웨어 인터페이스(API)**를 번역한다.

**Wine**이 대표적이다.

```
일반 Windows 에뮬레이터:
  Windows 앱 → 가상 x86 CPU → 가상 Windows 커널 → 호스트 OS

Wine (호환성 레이어):
  Windows 앱 → Wine이 Win32 API 호출을 POSIX 호출로 번역 → Linux/macOS 커널 직접 실행
```

Wine은 "Wine Is Not an Emulator"의 재귀 약자다. 의도적으로 "에뮬레이터가 아님"을 강조한다. CPU를 에뮬레이션하는 게 아니라, Windows API를 리눅스/macOS API로 번역하는 것이기 때문이다. 그래서 성능 손실이 에뮬레이터보다 훨씬 적다.

**macOS의 Rosetta 2**도 호환성 레이어와 에뮬레이션의 중간쯤이다. Intel 바이너리를 Apple Silicon에서 실행할 때 x86 명령어를 ARM으로 JIT 번역한다. 정확히는 **바이너리 트랜슬레이션(Binary Translation)**이라 부른다.

**WSL (Windows Subsystem for Linux)**도 호환성 레이어다. WSL1은 Linux 시스템 콜을 Windows NT 커널 콜로 번역했다. WSL2는 방식을 바꿔 경량 Linux VM을 올린다.

## 6. 파라가상화 (Paravirtualization)

> **게스트 OS가 자신이 가상 환경에 있음을 알고 협력한다.**

일반 VM(완전 가상화)에서 게스트 OS는 자신이 가상 환경에 있는지 모른다. 모든 하드웨어 접근을 하이퍼바이저가 가로채야 한다.

파라가상화에서는 게스트 OS 커널을 수정해서 하이퍼바이저와 직접 통신하게 만든다. 하드웨어를 흉내 낼 필요가 없어 성능이 좋다.

```
완전 가상화:    게스트 OS → 가상 하드웨어 접근 → 하이퍼바이저 가로채기
파라가상화:     게스트 OS → 하이퍼바이저 직접 API 호출 (hypercall)
```

Xen의 초기 방식이 파라가상화였다. AWS EC2 초기 인스턴스들이 이 방식이었다. 현재는 CPU 하드웨어 가상화(VT-x)가 보편화되면서 완전 가상화가 성능 면에서도 충분히 좋아졌고, 파라가상화의 필요성이 줄었다.

## 한눈에 비교

| 방식 | 재현 대상 | 커널 공유 | 성능 | 격리 수준 | 대표 사례 |
|---|---|---|---|---|---|
| 시뮬레이터 | 외부 동작 | 호스트 그대로 | 빠름 | 낮음 | iOS 시뮬레이터 |
| 에뮬레이터 | 하드웨어 전체 | 없음 | 느림 | 높음 | 게임보이 에뮬레이터, QEMU |
| VM (Type 1) | 하드웨어 | 없음 | 높음 | 매우 높음 | VMware ESXi, KVM |
| VM (Type 2) | 하드웨어 | 없음 | 중간 | 높음 | VirtualBox, Parallels |
| 컨테이너 | OS 프로세스 공간 | 공유 | 네이티브급 | 중간 | Docker, LXC |
| 호환성 레이어 | 소프트웨어 API | 공유 | 높음 | 낮음 | Wine, Rosetta 2, WSL1 |
| 파라가상화 | 하드웨어 (협력형) | 없음 | 높음 | 높음 | Xen (초기), Virtio |

## 어떤 것을 써야 하나

**다른 아키텍처의 바이너리를 실행해야 한다** → 에뮬레이터 (QEMU)

**Windows 앱을 Linux에서 돌리고 싶다** → 호환성 레이어 (Wine)

**격리된 완전한 OS 환경이 필요하다** → VM (VirtualBox, KVM)

**앱 배포 환경을 통일하고 싶다** → 컨테이너 (Docker)

**iOS 앱을 빠르게 테스트하고 싶다** → 시뮬레이터 (Xcode Simulator)

**게임보이 게임을 PC에서 하고 싶다** → 에뮬레이터 (mGBA, RetroArch)

이름이 비슷해서 혼용되지만, 선택이 달라지면 결과도 달라진다. 개념을 정확히 알면 엉뚱한 도구를 고르는 시행착오를 줄일 수 있다.
