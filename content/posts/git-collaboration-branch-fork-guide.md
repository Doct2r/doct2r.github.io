"형상관리(SCM)"라는 말은 딱딱하게 들리지만 실제로 팀이 매일 부딪히는 문제는 단순하다 — 여러 사람이 같은 코드를 동시에 건드릴 때 누구의 작업도 잃어버리지 않으면서 합치는 방법이다. 지금은 사실상 Git 하나로 답이 정해진 문제지만, Git이 왜 지금 형태로 만들어졌는지, "브랜치"와 "포크"라는 두 협업 단위가 개념적으로 왜 다른지, 그리고 실무에서 어떤 브랜치 전략을 골라야 하는지를 정리했다.

## 1. Git은 왜 지금 형태로 만들어졌나 — 2005년의 라이선스 분쟁

리눅스 커널 개발팀은 2002년부터 래리 맥보이(Larry McVoy)의 회사 비트무버(BitMover)가 만든 상용 분산 버전관리 시스템 비트키퍼(BitKeeper)를 무료 라이선스로 써왔는데, 여기엔 "경쟁 버전관리 도구 개발에 참여할 수 없고 역공학도 금지한다"는 조건이 붙어 있었다. 2005년 커널 개발자 앤드루 트리젤(Andrew Tridgell)이 비트키퍼 서버와 통신하는 프로토콜을 직접 관찰해 호환 클라이언트를 만들자, 비트무버는 이를 라이선스 위반으로 간주하고 리눅스 커뮤니티에 대한 무료 라이선스를 철회했다. 리누스 토르발스(Linus Torvalds)는 대체 도구를 처음부터 새로 만들기로 하고 2005년 4월 3일 작업을 시작해 4월 6일 공개, 4월 7일에는 벌써 Git 자신의 소스코드를 Git으로 관리하는 자기 호스팅(self-hosting) 단계에 도달했다. 그해 6월 16일 리눅스 커널 2.6.12가 Git으로 관리된 첫 릴리스로 기록됐다 — 필요에서 실전 투입까지 채 석 달이 걸리지 않은 셈이다. 이 사건 자체가 왜 Git이 처음부터 "중앙 서버 없이도 완전히 동작하는 분산 모델"로 설계됐는지를 설명한다 — 특정 회사가 통제하는 서버에 다시 발목 잡히지 않겠다는 문제의식이 설계 밑바탕에 깔려 있었다.

## 2. 브랜치는 파일 복사가 아니라 "포인터"다

Git 협업을 이해하는 데 가장 중요한 사실 하나는, 브랜치가 커밋을 통째로 복사하는 무거운 단위가 아니라 특정 커밋을 가리키는 41바이트짜리 참조(포인터) 하나에 불과하다는 점이다. 브랜치를 새로 만드는 연산이 사실상 즉시 끝나는 이유이자, 개발자가 기능 하나·버그 하나마다 부담 없이 브랜치를 새로 파고 버릴 수 있는 이유이기도 하다. 이 가벼움이 뒷받침되지 않았다면 지금부터 다룰 브랜치 전략들 자체가 성립하지 않았을 것이다.

## 3. 브랜치 전략 ① Git Flow — 릴리스 주기가 있는 프로젝트용

가장 널리 알려진 브랜치 전략은 네덜란드 개발자 빈센트 드리센(Vincent Driessen)이 2010년 1월 5일 자신의 블로그 nvie.com에 올린 "A successful Git branching model"(흔히 git-flow)이다. 이 모델은 브랜치를 다섯 종류로 나눈다 — 영구 보존되는 `master`(항상 배포 가능한 상태)와 `develop`(다음 릴리스를 위한 통합 브랜치), 그리고 임시로 파고 병합 후 삭제하는 `feature`(기능 개발)·`release`(릴리스 준비·QA)·`hotfix`(운영 중인 버전의 긴급 수정) 브랜치다. 이 구조는 "정해진 주기로 버전을 릴리스하는" 소프트웨어(데스크톱 앱, 패키지 배포판 등)에 잘 맞는다. 다만 드리센 본인이 이후 이 모델은 애초에 그런 릴리스 주기가 있는 프로젝트를 염두에 두고 만든 것이며, 지속적 배포(continuous delivery)를 하는 팀에는 뒤에 나올 더 단순한 워크플로우를 권한다고 스스로 밝혔다 — 만능 답이 아니라 특정 배포 방식에 최적화된 모델이라는 뜻이다.

## 4. 브랜치 전략 ② GitHub Flow — 매일 배포하는 팀용

깃허브의 스콧 채콘(Scott Chacon)은 2011년 8월 31일 자신의 블로그에서 훨씬 단순한 워크플로우를 제안했다. 그는 "깃허브에서는 git-flow를 쓰지 않는다 — 우리는 항상 배포하기 때문에 사실상 '릴리스'라는 개념 자체가 없고, 하루에도 여러 번 프로덕션에 배포한다"고 이유를 밝혔다. GitHub Flow의 구조는 단일 기본 브랜치(`main`)와, 그 위에서 파생돼 작업이 끝나면 풀 리퀘스트(Pull Request, PR)를 거쳐 곧바로 `main`에 합쳐지고 삭제되는 짧은 수명의 기능 브랜치뿐이다 — `develop`이나 `release` 같은 중간 단계가 아예 없다. release 브랜치가 빠진 대신 PR 단계에서의 코드 리뷰와 CI 통과가 품질 게이트 역할을 대신한다.

## 5. 브랜치 전략 ③ 트렁크 기반 개발 — 브랜치를 아예 최소화하는 극단

세 번째 축은 브랜치 자체를 최소화하는 트렁크 기반 개발(Trunk-Based Development)이다. 2002년부터 쏘트웍스(ThoughtWorks)에서 이 관행을 전도해온 폴 해먼트(Paul Hammant)는 2007년 구글에 파견됐을 때, 사실상 전 세계에서 가장 큰 규모의 트렁크 기반 개발 현장을 목격했다고 회고한다 — 구글 개발자의 약 95%(약 2만 5천 명)가 단일 트렁크(모노레포)를 공유하며 작업한다는 사실이 라헬 포트빈(Rachel Potvin)과 조시 레벤버그(Josh Levenberg)의 ACM 기고문으로 알려져 있다. 이 방식에서는 미완성 기능도 기능 플래그(feature flag)로 꺼둔 채 트렁크에 자주 병합하는 걸 원칙으로 삼는다 — 오래 사는 브랜치가 쌓일수록 통합 시점의 충돌이 기하급수적으로 커진다는 문제의식이 출발점이다. Git Flow가 "브랜치로 단계를 분리"하고 GitHub Flow가 "브랜치를 짧게 유지"한다면, 트렁크 기반 개발은 "브랜치 자체의 수명을 하루 이내로 극단적으로 줄인다"는 세 번째 지점에 있다.

## 6. 포크(Fork) — 원래는 브랜치가 아니라 "거버넌스의 분열"이었다

브랜치가 한 저장소 "안"에서 갈라지는 참조라면, 포크는 저장소 자체를 통째로 복제해 별도의 소유권 아래 두는 행위다. 이 단어는 원래 소프트웨어 프로젝트가 의견 차이로 완전히 갈라서는, 훨씬 무거운 사건을 가리켰다. 대표적 사례가 2003년 3월 X 윈도우 시스템 구현체인 XFree86에서 핵심 개발자 키스 패커드(Keith Packard)가 폐쇄적인 저장소 접근 권한과 불투명한 의사결정에 반발해 축출된 사건이다 — 이는 결국 2004년 X.Org 프로젝트의 포크로 이어졌고, 지금은 대부분의 유닉스 계열 OS가 XFree86 대신 X.Org를 쓴다. 2010년 오라클이 썬(Sun)을 인수해 MySQL을 손에 넣자 MySQL 원저자 몬티 비데니우스(Michael "Monty" Widenius)가 라이선스·거버넌스 우려로 MariaDB를 포크한 사례, 같은 해 오라클의 OpenOffice.org 인수에 반발해 커뮤니티가 LibreOffice를 포크한 사례도 같은 패턴이다 — 포크는 원래 "이 프로젝트의 미래를 더는 같이 결정할 수 없다"는 결별 선언에 가까웠다.

## 7. 깃허브가 포크를 다시 정의하다 — "결별"에서 "기여 수단"으로

2008년 4월 톰 프레스턴워너(Tom Preston-Werner)·크리스 완스트래스(Chris Wanstrath)·PJ 하이엇(PJ Hyett)·스콧 채콘이 정식 출시한 깃허브는 이 무거운 단어를 훨씬 가벼운 의미로 재사용했다 — 누구든 원 저장소에 대한 쓰기 권한 없이도 자기 계정 아래 사본(포크)을 만들고, 거기서 자유롭게 수정한 뒤 풀 리퀘스트로 원 저장소에 병합을 "제안"할 수 있게 한 것이다. 이 재정의 덕분에 포크는 더 이상 프로젝트의 분열이 아니라 오히려 협업 참여의 첫 단계가 됐다. 2009년 2월 24일 깃허브가 공개한 창립 1주년 통계를 보면 이미 공개 저장소 4만 6천여 개 중 약 6,200개가 최소 한 번 포크된 상태였고 4,600개는 병합까지 이뤄졌다 — 서비스 초기부터 "포크 후 PR"이 오픈소스 기여의 기본 동선으로 자리 잡았다는 뜻이다.

## 8. 두 협업 모델 — 팀 내부는 브랜치, 외부 기여자는 포크

실무에서 이 둘은 "누가 원 저장소에 대한 쓰기 권한을 갖고 있는가"에 따라 자연스럽게 갈린다. **공유 저장소 모델(Shared Repository Model)**은 팀원 전원이 같은 저장소에 쓰기 권한을 갖고, 각자 브랜치를 파서 작업한 뒤 PR로 리뷰·병합하는 방식이다 — 사내 팀, 신뢰된 협업자 그룹에 적합하고, `main` 브랜치에 보호 규칙(리뷰 승인·CI 통과 없이는 직접 푸시 금지)을 거는 게 일반적이다. **포크 & 풀 모델(Fork & Pull Model)**은 외부 기여자가 쓰기 권한이 전혀 없는 오픈소스 프로젝트에 기여할 때 쓰인다 — 자기 계정에 포크를 뜨고, 거기서 브랜치를 파 작업한 뒤, 원 저장소를 향해 PR을 보낸다. 리눅스 커널, 대부분의 깃허브 공개 오픈소스 프로젝트가 이 방식을 쓰고, 회사 내부 프로젝트라도 외주·인턴 등 신뢰 수준이 낮은 기여자에게는 같은 방식을 적용하는 경우가 많다. 결국 브랜치가 "같은 집 안의 방 나누기"라면 포크는 "옆집에 사본을 두고 작업한 뒤 원래 집 대문에 편지(PR)를 넣는" 구조에 가깝다.

## 9. 병합 방식 논쟁 — merge, rebase, squash

PR을 실제로 합치는 단계에서도 팀마다 정책이 갈린다. 일반 `merge`는 두 브랜치의 이력을 그대로 보존하며 병합 커밋 하나를 새로 만든다 — 실제로 무슨 일이 있었는지 그대로 남지만 이력이 지저분해지기 쉽다. `rebase`는 브랜치의 커밋들을 대상 브랜치 끝으로 재배치해 이력을 일직선으로 만든다 — 읽기는 깔끔해지지만 이미 공유된 브랜치에 rebase를 하면 다른 사람의 로컬 이력과 어긋나 버리므로, "아직 아무도 안 가져간 로컬 브랜치에서만 쓴다"는 게 정설이다. `squash merge`는 기능 브랜치의 커밋 여러 개를 PR 하나당 커밋 하나로 뭉개 병합한다 — GitHub Flow처럼 짧은 기능 브랜치를 자주 병합하는 팀에서 `main`의 이력을 "기능 단위로 한 줄씩" 깔끔하게 남기고 싶을 때 널리 쓰인다.

## 정리

Git 자체는 2005년 비트키퍼 라이선스 분쟁이라는 구체적 위기에서 "중앙 서버에 다시 발목 잡히지 않겠다"는 문제의식으로 태어났고, 그 결과 브랜치를 극도로 가벼운 포인터로 설계한 게 이후 모든 협업 전략의 토대가 됐다. 그 위에서 Git Flow(2010, 릴리스 주기형)·GitHub Flow(2011, 상시 배포형)·트렁크 기반 개발(브랜치 수명 최소화형)이라는 세 가지 브랜치 전략이 서로 다른 배포 리듬에 맞춰 갈라져 나왔다. 포크는 이와 별개의 축이다 — 원래 XFree86·MySQL·OpenOffice.org 사례처럼 프로젝트가 거버넌스 갈등으로 완전히 갈라서는 무거운 사건을 뜻했지만, 2008년 깃허브가 "쓰기 권한 없는 사람도 기여할 수 있게 하는 개인 사본"이라는 훨씬 가벼운 의미로 재사용하면서 오픈소스 협업의 기본 동선이 됐다. 결국 어떤 전략·어떤 병합 방식을 쓸지는 "우리 팀이 누구에게 쓰기 권한을 주는가"와 "우리가 얼마나 자주 배포하는가"라는 두 질문에 대한 답에서 거꾸로 도출된다.
