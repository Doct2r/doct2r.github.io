HTTP를 쓰면 될 것 같은데 굳이 소켓을 써야 하는 순간이 온다. 채팅, 실시간 게임, 센서 데이터 수집 — 이런 것들을 만들다 보면 결국 소켓으로 귀결된다. 이 글에서는 소켓이 무엇인지부터, 어떤 상황에서 쓰고 어떤 상황에서 피해야 하는지를 정리한다.

---

## 1. 소켓 통신이란

소켓(Socket)은 네트워크 통신의 끝점(endpoint)이다. 두 프로세스가 데이터를 주고받으려면 각자 소켓을 열고 연결한다. 파일 디스크립터처럼 다루며, 파일을 읽고 쓰듯 소켓에 데이터를 읽고 쓴다.

```
[클라이언트]              [서버]
  socket()                socket()
  connect() ─────────→   accept()
  send()    ─────────→   recv()
  recv()    ←─────────   send()
  close()                close()
```

소켓은 크게 두 종류다.

### TCP 소켓 (SOCK_STREAM)

연결 지향. 데이터가 순서대로, 빠짐없이 도달하는 것을 보장한다. 신뢰성이 핵심이다.

```python
import socket

# 서버
srv = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
srv.bind(("0.0.0.0", 9000))
srv.listen(5)
conn, addr = srv.accept()
data = conn.recv(1024)
conn.sendall(b"received: " + data)

# 클라이언트
cli = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
cli.connect(("127.0.0.1", 9000))
cli.sendall(b"hello")
print(cli.recv(1024))
```

### UDP 소켓 (SOCK_DGRAM)

비연결. 패킷 단위로 던지고 끝이다. 순서 보장도, 재전송도 없다. 대신 빠르고 가볍다.

```python
import socket

# 서버
srv = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
srv.bind(("0.0.0.0", 9001))
data, addr = srv.recvfrom(1024)
srv.sendto(b"pong", addr)

# 클라이언트
cli = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
cli.sendto(b"ping", ("127.0.0.1", 9001))
print(cli.recvfrom(1024))
```

### 프로토콜별 특성 요약

| 항목 | TCP | UDP |
|---|---|---|
| 연결 방식 | 연결 지향 (3-way handshake) | 비연결 |
| 순서 보장 | O | X |
| 데이터 유실 | 재전송으로 복구 | 유실 그대로 |
| 속도 | 상대적으로 느림 | 빠름 |
| 오버헤드 | 높음 | 낮음 |
| 사용 예 | HTTP, SSH, 파일 전송 | DNS, 게임, 스트리밍 |

## 2. 왜 쓰나

### HTTP로 안 되는 것들이 있다

HTTP는 클라이언트가 요청하면 서버가 응답하는 구조다. 서버가 먼저 데이터를 보낼 수 없다. 채팅 앱에서 누군가 메시지를 보냈을 때 서버가 다른 클라이언트에게 즉시 전달하려면, HTTP만으로는 클라이언트가 계속 폴링하거나 Long Polling 같은 편법을 써야 한다.

소켓은 연결이 유지되는 동안 양방향으로 언제든 데이터를 보낼 수 있다. 서버 → 클라이언트 푸시가 자연스럽다.

### 낮은 레이턴시

HTTP는 각 요청마다 헤더를 붙이고, HTTPS는 TLS 핸드셰이크 비용도 있다. 소켓은 연결을 맺은 후에는 순수 데이터만 오간다. 헤더 오버헤드가 없어서 대량의 소규모 메시지를 빠르게 처리하는 데 유리하다.

### 지속 연결

HTTP/1.1의 Keep-Alive도 있지만 소켓은 기본이 지속 연결이다. 연결 수립 비용(TCP handshake)을 한 번만 지불하고 이후에는 재사용한다. 실시간으로 수천 개의 메시지를 주고받는 환경에서 이 차이가 누적된다.

### 프로토콜 자유도

HTTP는 정해진 형식이 있다. 소켓은 형식이 없다. 바이너리 프로토콜을 직접 정의하거나, JSON을 쓰거나, 길이 헤더 + 페이로드 방식을 구현하거나 — 완전히 자유다. 게임 서버처럼 자체 프로토콜이 필요한 경우에 유리하다.

## 3. 이럴 때 유리하고, 이럴 때 피해라

### 소켓이 유리한 상황

**실시간 양방향 통신**
서버가 클라이언트에게 먼저 데이터를 밀어야 할 때. 채팅, 알림, 실시간 대시보드가 여기 해당한다.

**고빈도 소규모 메시지**
초당 수백~수천 건의 작은 패킷을 주고받는 상황. 매 요청마다 HTTP 오버헤드를 감당하기 어렵다. IoT 센서 데이터 수집, 금융 틱 데이터, 게임 입력 이벤트 전송이 여기 해당한다.

**낮은 레이턴시가 생명인 경우**
온라인 게임의 캐릭터 위치 동기화, 자율주행 차량 간 통신처럼 수십 밀리초의 차이가 결과를 바꾸는 환경.

**스트리밍**
영상/음성 스트리밍에서 UDP 소켓은 패킷 하나가 유실되어도 재생을 멈추지 않고 다음 프레임을 받아 계속 재생한다. HTTP 기반 스트리밍(HLS, DASH)보다 레이턴시가 낮다.

**멀티플레이어 게임 서버**
수백 명의 플레이어가 동시에 같은 게임 세계 안에 있고, 각자의 상태가 다른 모든 플레이어에게 실시간으로 전파되어야 한다. HTTP 폴링으로는 불가능하다.

---

### 소켓을 피해야 하는 상황

**단순 요청-응답 API**
로그인, 상품 조회, 결제 처리 같은 일반적인 API는 HTTP REST가 맞다. 소켓으로 구현하면 인증, 라우팅, 캐싱, 로드밸런싱 같은 HTTP 생태계의 혜택을 포기하게 된다. 얻는 것보다 잃는 것이 많다.

**브라우저에서 서버 연결이 필요한 경우**
일반 TCP 소켓은 브라우저에서 직접 쓸 수 없다. 브라우저는 보안상 Raw TCP 소켓 접근을 막는다. 이때는 **WebSocket**을 써야 한다. WebSocket은 HTTP 업그레이드로 시작해서 소켓처럼 동작하는 프로토콜이다.

**팀이 인프라에 익숙하지 않은 경우**
소켓 서버는 연결 상태를 직접 관리해야 한다. 끊긴 연결 감지(heartbeat), 재연결 처리, 메시지 큐잉, 로드밸런서 세션 고정(sticky session) — HTTP 서버에서는 프레임워크가 해주는 것들을 직접 구현해야 한다. 준비가 안 된 팀이 소켓 서버를 운영하면 유지보수 부담이 크다.

**확장성이 중요한 경우 (잘못 쓰면)**
소켓은 연결을 유지하기 때문에 서버 한 대당 처리할 수 있는 동시 연결 수에 한계가 있다. C10K 문제(동시 1만 연결)가 대표적이다. 비동기 I/O(`epoll`, `kqueue`, `IOCP`)나 이벤트 루프 기반 프레임워크(Node.js, Netty, asyncio)를 쓰지 않으면 연결이 늘수록 서버가 빠르게 한계에 다다른다.

## 4. 특정 사례

### 사례 1 — 멀티플레이어 게임: UDP로 위치 동기화

FPS 게임에서 플레이어 위치 정보는 1초에 20~60번 전송된다. 이 패킷 하나가 유실되어도 다음 패킷이 오면 된다. 오히려 유실된 패킷을 재전송하면 이미 지난 위치 정보를 늦게 받는 것이므로 역효과다.

```
[클라이언트] ──UDP──→ [게임 서버]
              x,y,방향,타임스탬프
              (패킷 유실 허용)
```

TCP를 쓰면 유실된 패킷을 재전송하느라 레이턴시가 튀고 게임이 버벅댄다. 이래서 게임은 UDP를 선호한다. 단, 아이템 구매, 채팅처럼 신뢰성이 필요한 데이터는 TCP로 별도 채널을 열거나 UDP 위에 재전송 로직을 직접 구현한다(QUIC, ENet 등).

---

### 사례 2 — 실시간 채팅: TCP + WebSocket

브라우저 기반 채팅은 WebSocket이 표준이다. 서버에서 클라이언트로 메시지를 즉시 푸시할 수 있고, HTTP 폴링처럼 불필요한 요청이 없다.

```
[브라우저]  HTTP Upgrade →  [서버]
           ←────────────── 101 Switching Protocols
           ↕ WebSocket (양방향 지속 연결)
```

서버 측에서는 연결된 클라이언트를 Map으로 관리하고, 채팅방에 메시지가 오면 해당 방의 모든 연결에 브로드캐스트한다.

```python
# FastAPI + WebSocket (Python)
from fastapi import FastAPI, WebSocket
from typing import List

app = FastAPI()
connections: List[WebSocket] = []

@app.websocket("/ws")
async def websocket_endpoint(ws: WebSocket):
    await ws.accept()
    connections.append(ws)
    try:
        while True:
            msg = await ws.receive_text()
            for conn in connections:
                await conn.send_text(msg)
    except:
        connections.remove(ws)
```

---

### 사례 3 — IoT 센서 데이터 수집: TCP 지속 연결

온도/습도 센서 1000개가 5초마다 데이터를 서버로 보내는 상황. HTTP를 쓰면 5초마다 TCP handshake + HTTP 헤더 오버헤드가 1000번 반복된다. 소켓 지속 연결을 쓰면 handshake는 처음 한 번이고 이후에는 순수 데이터만 전송된다.

```
[센서 장치] ──TCP 지속 연결──→ [수집 서버]
           5초마다 데이터 전송
           (연결은 유지)
```

데이터 형식은 최소화한다. JSON보다 고정 길이 바이너리 패킷이 처리 속도가 빠르다.

```
[2 bytes: 장치ID] [4 bytes: 타임스탬프] [2 bytes: 온도*100] [2 bytes: 습도*100]
총 10 bytes
```

---

### 사례 4 — 금융 시세 데이터: Multicast UDP

주식 거래소는 시세 데이터를 수백 개의 클라이언트에게 동시에 뿌린다. 유니캐스트(1:1)로 1000개 클라이언트에게 각각 보내는 것보다, **멀티캐스트**(1:N)로 한 번만 보내고 라우터가 복제하는 방식이 효율적이다.

```
[거래소 서버] ──UDP Multicast──→ [라우터]
                                  ├──→ [증권사 A]
                                  ├──→ [증권사 B]
                                  └──→ [HFT 시스템]
```

패킷이 유실되면 해당 틱 데이터는 그냥 버린다. 다음 틱이 오면 그걸 쓴다. 신선도가 신뢰성보다 중요한 데이터이기 때문이다.

## 정리

소켓은 만능이 아니라 특정 문제를 해결하는 도구다.

| 필요한 것 | 선택 |
|---|---|
| 단순 API | HTTP REST |
| 브라우저 실시간 | WebSocket |
| 저레이턴시 게임 | UDP 소켓 |
| 신뢰성 있는 스트림 | TCP 소켓 |
| 1:N 브로드캐스트 | UDP Multicast |
| IoT 지속 연결 | TCP 소켓 |

HTTP가 충분한 곳에서 소켓을 쓰면 복잡도만 올라간다. 소켓이 필요한 곳에서 HTTP를 고집하면 폴링 지옥에 빠진다. 상황을 정확히 읽는 것이 먼저다.
