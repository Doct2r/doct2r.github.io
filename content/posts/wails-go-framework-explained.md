[일렉트론](/posts/electron-explained/)의 대안으로 [타우리](/posts/tauri-explained/)가 자주 언급되는데, Go로 같은 자리를 노리는 Wails도 있다. 흔히 "타우리의 Go판 아류" 정도로 소개되지만, 실제 시간순은 반대다 — Wails 1.0은 타우리가 세상에 처음 공개되기 반년 전에 이미 나와 있었다.

---

## 1. 탄생 — 레스틱 GUI를 만들려다

Wails는 리아 앤서니가 만들었다. 계기는 소박했다 — 백업 도구 레스틱(Restic)에 웹 화면을 붙이고 싶은데, 웹서버 없이 HTML·JS·CSS로 그걸 하고 싶었다는 것이다. 그러다 웹뷰 라이브러리를 발견했고, 여기에 Go 코드와 프론트엔드 코드를 동기화해주는 틀을 얹으면서 지금의 Wails가 됐다. 2017～2018년 사이 비공개로 개발되다가 2019년 5월 1일 깃허브에 공개됐고, 그해 12월 10일 1.0이 나왔다.

## 2. 정정 — Wails가 타우리보다 먼저다

타우리는 2019년 말 다니엘 톰슨이베토와 루카스 노게이라가 공동 창업해 2020년 초 알파 버전을 거쳐 2020년 6월 19일 처음 공개됐고, 정식 1.0은 2022년 6월에야 나왔다. 즉 Wails 1.0(2019년 12월)은 타우리가 세상에 처음 공개되기(2020년 6월)보다 약 반년 앞서 있었다. "타우리류의 프레임워크가 먼저 있고 Go 버전이 뒤늦게 따라 나왔다"는 흔한 인상은 실제 연대기와 맞지 않는다 — Go와 러스트가 거의 같은 시기에, 우연히도 Go 쪽이 살짝 먼저 같은 문제에 도달했던 것에 가깝다.

## 3. 핵심 구조 — OS 네이티브 웹뷰라는 같은 전략

Wails가 타우리와 진짜 공유하는 지점은 여기다 — 일렉트론처럼 크로미움을 통째로 번들링하는 대신, OS가 이미 갖고 있는 웹뷰를 그대로 빌려 쓴다. 윈도우에서는 크로미움 기반 WebView2를, macOS에서는 WKWebView를, 리눅스에서는 WebKitGTK를 CGo(또는 purego)를 통해 호출한다. 앱 용량과 메모리 사용량을 줄이는 원리 자체는 타우리와 동일하다.

## 4. Go↔JS 바인딩 — 리플렉션에서 정적 분석으로

Wails 2에서는 `Bind`에 넘긴 구조체를 런타임에 리플렉션으로 검사해, 그 구조체의 공개(대문자로 시작하는) 메서드에 대응하는 JS 래퍼 함수를 `wailsjs/go/` 아래 자동으로 생성한다. Wails 자체 문서는 이 방식에 "닭이 먼저냐 달걀이 먼저냐" 같은 문제가 있다고 스스로 인정한다 — 바인딩이 없으면 빌드가 안 되고, 빌드를 하지 않으면 바인딩을 생성할 수 없다는 순환 구조다. Wails 3에서는 이 메커니즘 자체를 리플렉션에서 정적 AST(추상 구문 트리) 분석기로 바꿨다 — 속도가 빨라졌고, 생성된 타입스크립트 코드에 원본 주석과 파라미터 이름까지 보존된다.

## 5. 자동 타입스크립트 생성 — 진짜 차별점

이 지점이 Wails와 타우리 코어의 실질적인 차이다. 타우리는 `#[tauri::command]` 매크로를 붙인 러스트 함수를 프론트엔드에서 `invoke()`로 호출하는 구조인데, 여기엔 타입스크립트 타입을 자동으로 만들어주는 기능이 코어에 없다 — 이걸 원하면 `tauri-specta`라는 커뮤니티 플러그인을 따로 붙여야 한다. 반면 Wails는 처음부터 이 자동 생성을 기본 기능으로 갖고 있다. 백엔드 함수 시그니처를 바꾸면 프론트엔드의 타입과 자동완성이 별도 설정 없이 그대로 따라온다는 뜻이다 — Wails가 실제로 타우리보다 앞서 있는 몇 안 되는 지점이다.

## 6. 보안 모델 — 이번엔 반대로 타우리가 앞선다

타우리 2.x는 기본적으로 아무 권한도 없는 상태에서 시작해, JSON으로 작성한 Capabilities 파일로 어떤 커맨드를 어떤 창에서 호출할 수 있는지 세밀하게 화이트리스트로 지정하는 구조다. Wails에는 이에 대응하는 세분화된 권한 체계가 없다 — `Bind`에 포함된 구조체의 공개 메서드는 기본적으로 프론트엔드의 `window.go` 아래 전부 노출된다. 여러 비교 자료가 타우리의 모델을 "더 유연하고 안전하다", Wails의 모델을 "더 단순하지만 세밀함이 떨어진다"고 평가하는데, 이건 오해가 아니라 실제로 문서화된 구조적 차이다.

## 7. 버전 히스토리

2022년 9월 22일 나온 Wails 2.0.0은 윈도우에서 최신 웹 표준을 쓸 수 있는 WebView2 채택, CGO 없이도 되는 윈도우 빌드, React·Vue·Svelte·Preact·Lit·바닐라JS 공식 템플릿, Vite 핫 리로드, 네이티브 메뉴·다이얼로그, 윈도우11의 미카·아크릴 배경 효과 지원까지 갖췄다. 지금 진행 중인 3버전은 아직 알파 단계(2026년 7월 기준 v3.0.0-alpha.114)인데, v2의 선언적 단일 윈도우 API 대신 절차적인 "서비스(Services)" 모델을 도입했고, v2에서는 우회책이 필요했던 멀티 윈도우 지원을 정식 기능으로 갖췄다.

## 8. 생태계 규모 — 작지만 실체는 있다

2026년 6월 기준 깃허브 스타 수는 일렉트론 약 12만 1,600개, 타우리 약 10만 7,600개, Wails 약 3만 4,700개다 — 셋 중 가장 작지만, "규모는 작아도 실재하는" 생태계라는 평가가 뒤따른다. 커뮤니티가 정리한 쇼케이스 목록에는 레디스 GUI인 Tiny RDM, 쿠버네티스 클라이언트 Klustr, 버프 스위트 스타일의 HTTP 인터셉션 도구 NetraX 같은 것들이 올라와 있다 — 다만 이런 사례들은 대체로 니치한 오픈소스 개발자 도구 쪽에 몰려 있고, 특정 기업이 실제로 채택했다는 독립적으로 확인 가능한 사례는 찾기 어렵다.

## 9. Go 대 러스트 — 바이너리와 메모리

바이너리 크기 면에서는 최소한으로 스트립한 러스트 바이너리가 약 300KB인 데 비해 같은 수준의 Go 바이너리는 약 2MB, 현실적인 CLI 규모에서는 Go가 10～15MB, 러스트+musl이 2～5MB 선이라는 수치가 자주 인용된다. 메모리도 Go가 가비지 컬렉터 오버헤드 때문에 러스트보다 2～5배 더 쓴다는 이야기가 흔한데, 이런 수치들의 출처는 대개 블로그·비교 사이트 수준이라 정확한 벤치마크라기보다는 방향성 정도로 참고하는 게 맞다.

## 정리

Wails는 타우리를 뒤쫓아 나온 아류가 아니라, 오히려 시간상으로는 앞서 같은 문제(웹 기술로 가벼운 데스크톱 앱을 만든다)에 도달한 쪽이다. 두 프레임워크가 OS 네이티브 웹뷰를 쓴다는 같은 전략을 택했지만, 그 위에서 벌어지는 트레이드오프는 서로 반대 방향이다 — Wails는 자동 타입스크립트 생성이라는 개발자 경험에서 앞서고, 타우리는 세분화된 권한 모델이라는 보안 설계에서 앞선다. 생태계 규모의 차이는 어느 쪽 기술이 더 나은가의 문제라기보다는, 러스트 진영이 더 일찍 더 큰 관심을 끌어모았다는 확산 속도의 차이에 가깝다.
