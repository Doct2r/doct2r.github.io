자바는 2004년부터 Maven으로 라이브러리를 중앙 저장소에서 받아 썼고, 루비는 2004년부터 RubyGems를, 파이썬은 2008년부터 pip를 썼다. 그런데 .NET 개발자들은 한동안 서드파티 라이브러리를 쓰려면 DLL 파일을 직접 다운로드해서 프로젝트에 참조로 걸어야 했다. 이 공백을 채운 게 NuGet이다.

---

## 1. NuGet이란

**NuGet**은 .NET 생태계의 표준 패키지 관리자다. 라이브러리를 하나의 패키지 파일(`.nupkg`)로 묶어 중앙 저장소(nuget.org)에 올려두고, 개발자는 명령 한 줄로 그 라이브러리와 그 의존성까지 한꺼번에 내려받아 프로젝트에 넣을 수 있다.

2010년 마이크로소프트가 처음 내놨을 때 이름은 **NuPack**이었다. 정식 출시를 앞두고 상표 문제로 다른 이름을 써야 했고, 그렇게 정해진 이름이 지금의 **NuGet**이다. 초기에는 마이크로소프트가 후원하는 오픈소스 재단인 **Outercurve Foundation** 아래서 개발됐고, 이후 2014년 마이크로소프트가 .NET 오픈소스 프로젝트들을 통합 관리하기 위해 세운 **.NET Foundation**으로 이관됐다([C#은 왜 나왔는가 — 자바 소송이 낳은 언어](/posts/csharp-origin/)에서 다룬 Roslyn 오픈소스화와 같은 흐름이다).

---

## 2. 패키지의 실체 — `.nupkg`

NuGet 패키지 파일(`.nupkg`)은 사실 ZIP 압축 파일이다. 안에는 실제 컴파일된 DLL과, 패키지 이름·버전·의존성 정보를 담은 XML 메타데이터 파일(`.nuspec`)이 들어 있다. 압축을 풀어보면 폴더 구조 안에 대상 프레임워크별로 DLL이 나뉘어 들어있는 걸 볼 수 있다 — 이게 NuGet의 가장 독특한 특징으로 이어진다.

---

## 3. 왜 "멀티 타겟"이 특이한가

.NET은 오랜 시간 동안 **.NET Framework**(윈도우 전용), **.NET Core**(크로스플랫폼), 그리고 이 둘을 통합한 **.NET 5 이상**으로 갈라져 있었다([C#은 왜 나왔는가 — 자바 소송이 낳은 언어](/posts/csharp-origin/) 참고). 같은 라이브러리라도 어떤 .NET 버전에서 쓰느냐에 따라 호환되는 컴파일 결과물이 달라야 한다.

NuGet은 이 문제를 패키지 포맷 차원에서 흡수했다. 하나의 `.nupkg` 안에 `net472`(.NET Framework 4.7.2용), `netstandard2.0`, `net8.0`처럼 **여러 타겟 프레임워크(TFM, Target Framework Moniker)별 DLL을 동시에** 담아둘 수 있다. 패키지를 설치하면 NuGet이 현재 프로젝트의 타겟에 맞는 DLL을 알아서 골라 참조한다. 다른 언어의 패키지 관리자에서는 보기 힘든, .NET 특유의 파편화된 역사가 만들어낸 구조다.

---

## 4. 쓰는 방식이 바뀌어온 과정

초기에는 프로젝트에 `packages.config`라는 별도 파일을 두고 패키지 목록을 관리했다. 2017년 .NET Core 도구체인이 자리 잡으면서, 프로젝트 파일(`.csproj`) 안에 패키지 참조를 직접 적는 **PackageReference** 방식이 기본이 됐다. 지금은 대부분 다음 중 하나로 패키지를 다룬다.

- `dotnet add package <이름>` — .NET CLI
- Visual Studio의 NuGet 패키지 관리자 UI, 혹은 `Install-Package <이름>` (패키지 관리자 콘솔)
- **Paket** — F# 커뮤니티에서 인기 있는 서드파티 대안 클라이언트로, 의존성 잠금(lock file) 관리에 더 엄격하다.

---

## 5. 생태계에서의 위치

NuGet은 마이크로소프트 자신의 패키지(ASP.NET Core, Entity Framework)뿐 아니라 서드파티 라이브러리 유통의 표준 창구가 됐다. 대표적인 예가 **Newtonsoft.Json(Json.NET)**이다. 제임스 뉴턴킹이 만든 이 JSON 라이브러리는 오랫동안 거의 모든 .NET 프로젝트의 사실상 표준이었다. .NET Core 3.0(2019년)부터 마이크로소프트가 `System.Text.Json`을 기본 내장하면서 선택지가 넓어졌지만, Newtonsoft.Json은 지금도 방대하게 쓰인다 — NuGet 없이는 이런 생태계 자체가 존재하기 어려웠을 것이다.

---

## 6. 다른 언어 패키지 관리자와 비교하면

npm(자바스크립트, 2010), Maven/Gradle(자바), pip(파이썬), Cargo(러스트, 2015)와 비교하면 중앙 레지스트리+메타데이터+의존성 자동 해석이라는 기본 골격은 동일하다. 다만 다른 생태계 대부분은 언어·런타임 버전이 하나로 통일돼 있어 패키지 하나가 여러 런타임 타겟을 동시에 담을 필요가 없다. NuGet이 멀티 타겟 구조를 갖게 된 건, 다른 패키지 관리자보다 늦게 나온 대신 .NET 자체가 겪은 플랫폼 분열의 역사를 그대로 흡수해야 했기 때문이다.

---

## 7. 결론

NuGet은 다른 언어들이 이미 갖고 있던 표준 패키지 관리 체계를 .NET 생태계가 뒤늦게 따라잡은 결과물이다. 다만 단순히 베낀 게 아니라, .NET Framework와 .NET Core로 갈라졌던 플랫폼 역사를 하나의 패키지 포맷 안에 여러 타겟 프레임워크를 담는 방식으로 흡수했다는 점이 고유한 특징이다. .NET이 오픈소스화되고 크로스플랫폼으로 전환되는 과정과 함께, NuGet도 Visual Studio 종속 도구에서 `dotnet` CLI 기반의 범용 인프라로 자리를 넓혀왔다.
