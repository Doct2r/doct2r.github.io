코드를 짜서 실행하면 결국 CPU가 처리한다. CPU는 0과 1로 된 기계어만 이해한다. 그렇다면 사람이 읽을 수 있는 Python이나 Java 코드는 어떻게 CPU에 닿는가? 그 과정에서 선택하는 방식이 AoT, 인터프리터, JIT이다. 각각 다른 시점에 다른 방식으로 코드를 변환한다.

---

## 1. 무엇인가?

### 컴파일 — 번역의 개념부터

코드가 실행되려면 어떤 형태로든 기계어로 변환되어야 한다. 이 변환을 **컴파일(Compile)**이라고 한다. 문제는 **언제** 변환하느냐다. 이 타이밍이 세 방식을 가른다.

```
[소스 코드] ──→ ??? ──→ [기계어] ──→ CPU 실행
              언제, 어떻게 변환하느냐
```

---

### AoT — Ahead-of-Time (실행 전 미리 컴파일)

실행하기 **전에** 소스 코드를 기계어로 완전히 변환한다. 개발자가 컴파일 버튼을 누르는 순간 완성된 실행 파일이 만들어진다.

```
개발 시점:
  [소스 코드] → [컴파일러] → [실행 파일(.exe/.bin)]

실행 시점:
  [실행 파일] → CPU 즉시 실행
```

**대표 언어/도구**: C, C++, Rust, Go, Swift (일부), WinUI3 Native AoT

```c
// C 코드
int add(int a, int b) { return a + b; }
```

```bash
$ gcc main.c -o main   # 이 시점에 기계어로 변환 완료
$ ./main               # CPU가 바로 실행. 변환 없음
```

---

### 인터프리터 — 줄 단위로 읽고 즉시 실행

소스 코드를 **실행 시점에 한 줄씩 읽으면서 즉시 처리**한다. 컴파일 단계가 없고, 인터프리터 프로그램이 코드를 해석해 실행한다.

```
실행 시점:
  [소스 코드]
      ↓ (한 줄 읽기)
  [인터프리터] → 해석 → 실행
      ↓ (다음 줄 읽기)
  [인터프리터] → 해석 → 실행
      ↓ ...
```

**대표 언어/환경**: 초기 Python (CPython), Ruby, PHP, BASIC, Shell 스크립트

```python
# Python 코드 — 저장하고 바로 실행 가능
x = 10
print(x * 2)  # 인터프리터가 이 줄에 도달하면 그때 해석
```

실제로 현대 CPython은 순수 줄 단위 실행이 아니라 바이트코드로 변환 후 실행하는데, 이는 후술한다.

---

### JIT — Just-in-Time (실행 중에 컴파일)

실행을 시작할 때는 인터프리터처럼 동작하다가, **자주 실행되는 코드(핫 코드)**를 발견하면 그 시점에 기계어로 컴파일해서 이후 실행 속도를 높인다.

```
실행 초기 (인터프리터 모드):
  [바이트코드] → 해석 → 실행 (느림)
      ↓ 프로파일러가 감시
  "이 함수가 1000번 호출됐다" → 핫 코드 감지
      ↓
  [JIT 컴파일러] → 기계어로 변환 → 캐시에 저장

이후 실행 (컴파일 모드):
  [캐시된 기계어] → CPU 즉시 실행 (빠름)
```

**대표 환경**: Java HotSpot JVM, JavaScript V8 엔진, .NET CLR, PyPy

---

## 2. 왜 나왔나?

세 방식은 각각 다른 문제를 풀기 위해 등장했다.

### AoT의 등장 — 컴퓨팅의 시작점

초기 컴퓨터(1950～60년대)는 기계어 또는 어셈블리어로 프로그래밍했다. 어셈블리어 역시 AoT 방식 — 어셈블러(컴파일러의 조상)가 어셈블리 코드를 기계어로 변환했다.

FORTRAN(1957), C(1972)가 등장하며 **고수준 언어를 기계어로 변환하는 컴파일러**가 정착됐다. AoT는 "코드를 미리 변환해 놓으면 실행이 빠르다"는 가장 직관적인 접근이다.

**AoT가 해결한 문제**: 사람이 읽기 쉬운 언어로 코드를 짜면서도, 실행은 최대한 빠르게.

### 인터프리터의 등장 — 이식성과 즉시 실행

AoT의 문제가 있다. 컴파일된 실행 파일은 **특정 CPU와 OS에 종속**된다. Windows x86용으로 컴파일하면 Mac ARM에서 바로 돌아가지 않는다.

1960～70년대 대학과 연구소에서 **BASIC, Lisp** 같은 언어가 인터프리터로 설계됐다. 인터프리터 하나만 각 플랫폼에 이식하면, 그 위에서 돌아가는 코드는 플랫폼을 가리지 않는다.

```
인터프리터 방식의 이식성:
  코드.py ──→ Python 인터프리터(Mac) ──→ 실행
  코드.py ──→ Python 인터프리터(Win) ──→ 실행
  코드.py ──→ Python 인터프리터(Linux) ──→ 실행
  (코드는 수정 없이 어디서든 동작)
```

또한 인터프리터는 **대화형(REPL) 실행**이 가능하다. 코드를 한 줄 입력하면 즉시 결과가 나온다. 빠른 실험과 스크립팅에 유리하다.

**인터프리터가 해결한 문제**: 이식성. 그리고 "짜자마자 바로 실행"하는 편의성.

### JIT의 등장 — 두 마리 토끼

Java(1995)가 "Write Once, Run Anywhere"를 내세웠다. JVM(Java Virtual Machine)이라는 인터프리터 위에서 돌아가는 언어였다. 이식성 문제는 해결됐다.

그런데 문제가 생겼다. JVM 위의 Java는 **네이티브 C/C++ 대비 10～50배 느렸다**. 인터프리터의 숙명적 느림이었다.

Sun Microsystems가 1990년대 후반 **HotSpot JIT 컴파일러**를 개발했다. 자주 실행되는 코드는 실시간으로 기계어로 변환해 캐시에 저장하는 방식이었다.

```
Java 코드 성능 진화:
  초기 JVM (순수 인터프리터): C 대비 10~50배 느림
  HotSpot JIT 도입 후: C 대비 1~3배 수준 (핫 코드 기준)
```

JavaScript에서 JIT의 혁명적 영향이 더 극적으로 나타났다. 2008년 구글이 Chrome에 **V8 JIT 엔진**을 탑재하면서 JavaScript 속도가 10～100배 빨라졌다. 이것이 Node.js와 현대 웹 앱 시대를 열었다.

**JIT가 해결한 문제**: 인터프리터의 이식성을 유지하면서 AoT에 가까운 실행 성능.

## 3. 장단점

### AoT

**장점**

- **실행 속도 최고**: 이미 기계어라 런타임 변환 없음
- **예측 가능한 성능**: 실행 중 갑자기 느려지는 구간이 없음
- **낮은 런타임 메모리**: JVM·인터프리터 프로세스가 필요 없음
- **콜드 스타트 빠름**: 첫 실행부터 최고 성능 (JIT는 웜업 필요)
- **보안**: 소스 코드를 배포하지 않아도 됨

**단점**

- **플랫폼 종속**: x86/ARM, Windows/Linux별로 따로 빌드
- **컴파일 시간**: 큰 프로젝트는 빌드에 수십 분 (Rust, C++)
- **동적 동작 어려움**: 실행 중에 새 코드를 로드하거나 타입을 바꾸기 어려움
- **배포 파일 크기**: 런타임을 포함하면 바이너리가 커질 수 있음 (Native AoT의 100MB 문제)

### 인터프리터

**장점**

- **이식성**: 인터프리터만 있으면 어디서든 동작
- **즉시 실행**: 컴파일 대기 없음. 코드 수정 → 즉시 결과
- **동적 유연성**: 실행 중 코드 평가(`eval`), 타입 변경, 플러그인 로드
- **REPL**: 대화형 실행으로 빠른 실험
- **디버깅 편의**: 소스 코드와 실행이 직접 대응

**단점**

- **느림**: 매번 해석 비용이 발생. CPU 집약 연산에서 AoT 대비 수십～수백 배 느릴 수 있음
- **메모리 오버헤드**: 인터프리터 프로세스 자체가 메모리를 점유
- **배포 시 소스 노출**: 순수 인터프리터 방식은 소스 코드 그대로 배포

### JIT

**장점**

- **이식성 유지**: 바이트코드는 플랫폼 독립적
- **실행 중 최적화**: 런타임 정보(실제 타입, 실제 실행 경로)를 보고 최적화 — AoT보다 나은 경우도 있음
- **적응형 성능**: 처음엔 느려도 시간이 지나면서 빨라짐
- **동적 기능 지원**: 인터프리터의 유연성 일부 보존

**단점**

- **웜업(Warm-up) 시간**: 핫 코드가 감지되고 컴파일될 때까지 초반 느림
- **메모리 소비**: JIT 컴파일된 기계어 캐시가 메모리를 차지
- **예측 불가 지연**: GC(가비지 컬렉터) pause, deoptimization이 갑자기 발생 가능
- **복잡한 구현**: JIT 엔진 자체가 매우 복잡한 소프트웨어

### 한눈에 비교

| | AoT | 인터프리터 | JIT |
|---|---|---|---|
| 실행 속도 | 최고 | 느림 | 중～고 (웜업 후) |
| 이식성 | 낮음 | 높음 | 높음 |
| 시작 속도 | 빠름 | 빠름 | 느림 (웜업) |
| 메모리 | 낮음 | 중간 | 높음 |
| 동적 기능 | 제한 | 최고 | 중간 |
| 배포 | 플랫폼별 빌드 | 소스 배포 | 바이트코드 배포 |

## 4. 한계점

### AoT의 한계 — "컴파일 시점에 모든 걸 알 수 없다"

AoT 컴파일러는 코드를 실행하기 전에 최적화한다. 실제로 어떤 데이터가 들어올지, 어떤 경로가 자주 실행될지 모른다.

```c
// AoT 컴파일러는 이 함수가 얼마나 자주 호출될지,
// x가 항상 양수인지 모른다
int process(int x) {
    if (x > 0) return x * 2;
    return -x;
}
```

JIT는 런타임에 "이 함수는 항상 양수 x로 호출된다"는 정보를 얻어 분기를 제거하는 최적화를 할 수 있다. AoT는 이런 적응형 최적화가 불가능하다.

또한 AoT는 **플랫폼 수만큼 빌드가 필요**하다. CI/CD에서 Windows/Linux/Mac/ARM/x86 조합을 모두 빌드하면 파이프라인이 복잡해진다. Rust 생태계가 이 문제를 잘 보여준다.

### 인터프리터의 한계 — "해석 비용은 피할 수 없다"

같은 연산을 1억 번 반복하는 루프에서 인터프리터는 매 반복마다 코드를 해석한다. 이 오버헤드는 구조적으로 제거할 수 없다.

```python
# 순수 인터프리터에서 이 루프는 10억 번 해석 비용 발생
total = 0
for i in range(1_000_000_000):
    total += i
```

Python이 수치 계산에서 C보다 100배 느린 근본 이유가 이것이다. NumPy가 필요한 이유도 — NumPy의 핵심 연산은 C로 작성된 AoT 컴파일 코드를 Python에서 호출하는 구조다.

### JIT의 한계 — "예측 불가능성과 복잡도"

JIT의 가장 큰 약점은 **성능이 비결정적**이라는 것이다.

- **Deoptimization**: JIT가 "이 변수는 항상 int다"라고 가정해 최적화했는데 갑자기 float이 들어오면 최적화를 취소(deopt)하고 느린 경로로 폴백한다. 이 순간 성능이 급락한다.
- **GC Pause**: Java, C#에서 가비지 컬렉터가 실행되는 동안 모든 스레드가 멈춘다. 실시간 시스템(게임, 금융 트레이딩)에서 치명적이다.
- **JIT 컴파일 오버헤드**: 핫 코드를 컴파일하는 동안 그 스레드의 성능이 일시 저하된다.
- **서버리스/람다 환경에서 치명적**: AWS Lambda 같은 환경은 함수가 짧게 실행되고 종료된다. JIT가 웜업되기 전에 실행이 끝나버린다 — JIT의 장점이 하나도 없다.

## 5. 이 세 가지 말고 다른 게 있나?

있다. 그것도 많다. 세 가지 분류는 설명의 편의지, 실제 세계는 훨씬 복잡하다.

### 바이트코드 컴파일 (Bytecode Compilation)

"순수 인터프리터"는 사실 현대에 거의 없다. Python(CPython)을 예로 들면:

```
Python 실행 과정:
  source.py → [컴파일러] → .pyc (바이트코드) → [VM이 인터프리트]
```

소스 코드를 **가상 머신(VM)이 이해하는 중간 언어(바이트코드)**로 먼저 변환한 후, VM이 그 바이트코드를 인터프리트한다. 순수 소스 줄 단위 실행보다 빠르지만 JIT보다는 느리다.

Java의 `.class` 파일, Python의 `.pyc` 파일이 이 바이트코드다.

### 트랜스파일러 (Transpiler / Source-to-Source Compiler)

소스 코드를 다른 언어의 소스 코드로 변환한다. 기계어가 아닌 다른 고수준 언어로 번역한다.

```
TypeScript → [tsc] → JavaScript
CoffeeScript → JavaScript
SCSS → CSS
Kotlin → JavaScript (Kotlin/JS)
```

웹 프론트엔드 개발에서 필수적이다. TypeScript를 브라우저가 직접 이해하지 못하므로 JavaScript로 변환해야 한다. Babel은 최신 JavaScript 문법을 구형 브라우저가 이해하는 오래된 문법으로 트랜스파일한다.

### PGO — Profile-Guided Optimization

AoT인데, 런타임 정보를 활용한다. 두 단계로 진행한다.

```
1단계: 프로파일링 빌드
  → 코드 실행하며 "어떤 함수가 자주 호출되나" 데이터 수집
  → 프로파일 데이터 저장

2단계: 최적화 빌드
  → 프로파일 데이터를 보고 컴파일
  → 자주 쓰이는 경로에 인라이닝, 분기 제거 적용
```

Rust, Clang, MSVC가 지원한다. JIT의 런타임 최적화를 AoT 방식으로 흉내 낸 것이다. 빌드 과정이 복잡하지만 실행 파일 성능이 일반 AoT보다 10～30% 향상된다.

### 바이너리 번역 (Binary Translation)

소스 코드가 아닌 **이미 컴파일된 기계어를 다른 아키텍처의 기계어로 변환**한다.

```
x86 기계어 → [바이너리 번역기] → ARM 기계어
(런타임에 실시간 또는 미리 변환)
```

- **QEMU**: x86 바이너리를 ARM 머신에서 실행. 가상화와 에뮬레이션에 사용
- **Apple Rosetta 2**: M1 Mac 출시 시 기존 x86 Mac 앱을 ARM에서 실행. 실행 전 미리 변환(AoT 방식)해 캐시에 저장하므로 QEMU보다 훨씬 빠름

Rosetta 2는 바이너리 번역인데 AoT 방식으로 구현된 흥미로운 사례다. 처음 실행 시 변환 후 캐시 → 이후 실행은 변환 없이 즉시 실행.

### WebAssembly (WASM)

2019년 W3C 표준. 새로운 **범용 컴파일 타깃**이다.

```
C/C++/Rust → [컴파일러] → .wasm
Python/JavaScript → [트랜스파일] → .wasm

.wasm → [브라우저/런타임] → JIT 컴파일 → 실행
```

WASM은 기계어도, JavaScript도 아닌 중간 형태다. 어떤 언어로 짜든 WASM으로 컴파일하면 브라우저에서 실행 가능하다. 브라우저는 WASM을 받아 JIT로 실행한다.

Figma, AutoCAD Web, Google Earth — 복잡한 앱이 브라우저에서 돌아가는 것이 WASM 덕분이다. 서버 측에서도 Wasmtime, Wasmer 같은 런타임이 WASM을 실행한다.

### 현대 런타임의 혼합 방식

현실에서 하나의 순수한 방식을 쓰는 경우는 드물다. 대부분 혼합이다.

```
V8 (Chrome, Node.js) 실행 파이프라인:
  JavaScript
    → [파서] → AST (추상 구문 트리)
    → [Ignition] → 바이트코드 (인터프리터 실행)
    → [Sparkplug] → 빠른 비최적화 기계어 (1단계 JIT)
    → [Maglev] → 중간 최적화 기계어 (2단계 JIT)
    → [TurboFan] → 완전 최적화 기계어 (3단계 JIT)
    → 핫니스 감소 시 → Deopt → 하위 단계로 폴백

JVM (HotSpot) 실행 파이프라인:
  .class 바이트코드
    → 인터프리터 실행
    → C1 컴파일러 (빠른 JIT, 가벼운 최적화)
    → C2 컴파일러 (느린 JIT, 공격적 최적화)
    → GraalVM Native Image (AoT 옵션)
```

Python의 **PyPy**는 CPython과 달리 JIT을 탑재했다. 같은 Python 코드를 PyPy로 실행하면 수 배 빠르다.

## 정리

```
방식           언제 변환         대표 사례
─────────────────────────────────────────────
AoT            실행 전 완전 변환  C, C++, Rust, Go
인터프리터      실행 중 줄 단위   BASIC, 초기 Python
바이트코드 VM   중간(VM 해석)     Python .pyc, Java 초기
JIT            실행 중 핫코드    Java HotSpot, V8, .NET
트랜스파일러   빌드 시 소스→소스  TypeScript, Babel, SCSS
PGO            AoT + 프로파일    Rust PGO, Clang PGO
바이너리 번역  런타임/AoT        Rosetta 2, QEMU
WASM           컴파일 타깃       Rust/C → 브라우저
```

"AoT냐 JIT냐"는 트레이드오프의 문제다. 절대적으로 옳은 선택은 없다. Rust로 짠 CLI 도구는 AoT가 맞고, 웹 브라우저에서 돌아가는 JavaScript는 JIT가 맞고, 빠른 데이터 처리 스크립트는 PyPy JIT가 맞다.

한 가지 확실한 것은, 현대 언어 런타임은 세 방식 중 하나를 고르는 게 아니라 여러 방식을 파이프라인처럼 조합해 각 단점을 보완한다는 점이다. V8이 인터프리터에서 3단계 JIT까지 자동으로 전환하는 것이 그 예다.
