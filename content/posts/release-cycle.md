"개발하고, 테스트하고, 배포한다." 이것이 틀린 말은 아니다. 하지만 이 세 단계가 어떤 순서로, 얼마나 자주, 누구에게, 어떤 절차를 거쳐 이루어지는지는 플랫폼마다, 앱의 성격마다, 사용자층마다 완전히 달라진다. 웹 서비스는 하루에 수십 번 배포할 수 있고, 항공기 소프트웨어는 릴리즈 하나에 수년이 걸린다. 같은 "개발 주기"라는 말이 완전히 다른 세계를 가리킨다.

---

## 기본 모델들

먼저 개발 방법론이 주기를 결정한다.

### 폭포수 (Waterfall)

```
요구사항 분석 → 설계 → 구현 → 테스트 → 배포 → 유지보수
```

각 단계가 완전히 끝난 후 다음으로 넘어간다. 중간에 돌아가기 어렵다. 1970년대 소프트웨어 공학의 기반이었고, 지금도 특정 영역에서 쓰인다. 요구사항이 초반에 확정되고 변경이 거의 없는 환경에서 유효하다. 방위산업, 항공, 대규모 인프라 프로젝트가 그 예다.

### 애자일 (Agile)

```
[계획 → 개발 → 테스트 → 리뷰] × 반복
  스프린트 1    스프린트 2    스프린트 3 ...
  (1~4주)      (1~4주)      (1~4주)
```

짧은 주기(스프린트)를 반복하며 점진적으로 기능을 추가한다. 요구사항이 변할 수 있고, 사용자 피드백을 빠르게 반영해야 할 때 쓴다. 2000년대 이후 대부분의 소프트웨어 개발이 이 계열로 이동했다.

### CI/CD

```
코드 푸시 → [자동 빌드 → 자동 테스트 → 자동 배포]
              ↑__________________________|
                     파이프라인 (몇 분)
```

지속적 통합(CI)과 지속적 배포(CD)의 조합. 코드를 올리면 자동으로 테스트하고 배포까지 이어진다. 파이프라인이 실패하면 알림이 온다. 사람이 승인 버튼을 누르는 단계를 최소화한다. 현대 웹 서비스의 표준이다.

## 플랫폼별 주기

### 웹 서비스 — "배포"와 "릴리즈"가 분리된다

웹은 이론상 코드를 바꾸는 순간 전 세계 사용자에게 즉시 반영된다. 이 유연함은 독특한 패턴을 만들어냈다.

**배포(Deploy) ≠ 릴리즈(Release)**

기능 플래그(Feature Flag)를 쓰면 코드는 배포하되, 사용자에게는 아직 보이지 않게 할 수 있다.

```
[코드 머지] → [자동 배포] → 사용자에게는 비활성화 상태
                              ↓ 준비되면
                           플래그 켜기 → 노출
```

이 덕분에 "금요일 배포 금지" 같은 규칙이 점점 의미 없어진다. 기능을 배포해두고 월요일 아침에 플래그만 켜면 된다.

**단계적 공개 (Staged Rollout)**

```
1% 사용자 → 이상 없으면 → 5% → 25% → 50% → 100%
```

전체 사용자에게 한 번에 배포하지 않는다. 문제가 생기면 100%에 도달하기 전에 롤백한다. 구글, 페이스북, 넷플릭스가 이 방식을 쓴다. 수억 명의 사용자가 있으면 1%도 수백만 명이기 때문에 충분한 데이터가 나온다.

**카나리 배포**

```
프로덕션 서버 100대 중 1대만 새 버전 실행
나머지 99대는 이전 버전
→ 1대에서 에러율 정상 → 점진적으로 확대
```

새 버전의 서버를 조금씩 늘려가는 방식이다. 이름은 광산의 카나리아(독가스 조기 경보용 새)에서 왔다.

**블루-그린 배포**

```
Blue(현재 프로덕션) ← 트래픽
Green(새 버전)      ← 준비 완료

전환: 트래픽을 Green으로 → Blue가 이제 대기

문제 생기면: 트래픽을 다시 Blue로 (즉각 롤백)
```

두 개의 동일한 환경을 유지하다가 트래픽을 전환한다. 롤백이 거의 즉각적이다.

### iOS — 심사가 주기를 지배한다

```
개발 → TestFlight 베타 → App Store 심사 제출
                              ↓
                     심사 (보통 1~3일, 최대 1주)
                              ↓
                    승인 → 단계적 출시 설정
                              ↓
                     1% → 2% → 5% → ... → 100%
```

Apple의 App Store 심사가 주기에 박힌 제약이다. 버그 하나를 고쳐도 심사를 다시 받아야 한다. 긴급 패치도 최소 하루를 기다려야 한다.

이 때문에 iOS 앱은 배포 전 품질 관리에 훨씬 공을 들인다. 출시 후 고치는 비용이 크기 때문이다.

**TestFlight**: Apple의 공식 베타 배포 플랫폼. 심사 없이 최대 10,000명의 테스터에게 배포 가능. 심사를 받으면 최대 90일 동안 배포된다. 출시 전 실제 기기에서 테스트하는 표준 방법이다.

**단계적 출시**: App Store Connect에서 1～100%를 설정할 수 있다. 문제가 생기면 출시를 일시 중지할 수 있다. 하지만 이미 업데이트를 받은 사용자를 되돌릴 수는 없다.

**iOS 버전 파편화**: Android보다 훨씬 적지만, 여전히 최소 지원 버전(Deployment Target)을 결정해야 한다. iOS 17 이상만 지원하면 편하지만, 아직 16을 쓰는 사용자를 놓친다.

### Android — 빠르지만 파편화가 숙제

```
개발 → 내부 테스트 → 비공개 테스트 → 공개 테스트
                                          ↓
                                   Google Play 심사 (보통 수 시간~수일)
                                          ↓
                                   단계적 출시 (%)
```

Google Play의 심사는 Apple보다 빠르다. 긴급 패치의 경우 수 시간 만에 통과되기도 한다.

**파편화 문제**: Android의 진짜 고난은 심사가 아니라 기기와 OS 버전의 다양성이다. Samsung Galaxy A 시리즈, Pixel, 샤오미, 오포가 같은 Android지만 다르게 동작할 수 있다. 최소 SDK 버전을 낮게 잡으면 더 많은 기기를 지원하지만 테스트해야 할 경우의 수가 폭발한다.

**Google Play Console의 기능**: 출시를 트랙(내부→알파→베타→프로덕션)으로 관리한다. 크래시 리포트, ANR(앱 응답 없음) 통계가 실시간으로 보인다. 특정 나라, 기기 모델에만 배포하는 것도 가능하다.

### 데스크톱 앱 (Windows / macOS) — 업데이트가 선택이다

```
개발 → 내부 테스트 → 베타 채널 → 안정 채널
```

웹과 모바일의 결정적 차이: **사용자가 업데이트를 거부할 수 있다.**

Slack, VS Code, Chrome처럼 백그라운드 자동 업데이트를 구현하면 대부분의 사용자가 최신 버전을 쓴다. 하지만 회사 IT 정책으로 자동 업데이트를 막아두는 경우도 많다. 기업용 소프트웨어에서는 "우리 회사 컴퓨터는 아직 버전 X를 써요" 상황이 흔하다.

**업데이트 채널 전략**

```
Nightly/Canary → Dev → Beta → Stable
(매일 빌드)    (주간)  (격주) (안정적)
```

Chrome, VS Code, Firefox가 이 구조를 쓴다. 얼리어답터는 Nightly를 쓰며 버그를 제보하고, 일반 사용자는 Stable을 쓴다. 같은 코드베이스에서 여러 속도로 배포가 이루어진다.

**Microsoft Store / Mac App Store**: 모바일 앱 스토어처럼 심사가 있다. 하지만 데스크톱 앱은 스토어 바깥으로 직접 배포하는 것이 여전히 일반적이다(인스톨러 직접 배포).

### 서버/백엔드 — 사용자가 모르게 바뀐다

서버 소프트웨어에는 "버전"이 사용자에게 노출되지 않는다. AWS의 EC2 서비스가 업데이트돼도 사용자는 모른다. 이 투명성이 가장 빠른 배포 주기를 가능하게 한다.

Netflix는 하루에 수백 번 배포한다고 알려져 있다. Amazon은 평균 11.7초마다 프로덕션 배포가 일어난다고 2011년에 발표했다. 이것이 가능한 이유는 마이크로서비스 구조(서비스를 독립적으로 배포 가능하게 분리), 강력한 자동화, 롤백 자동화 덕분이다.

## 앱 종류별 주기

### 게임 — 패치와 콘텐츠 주기가 따로 돈다

게임의 개발 주기는 두 가지 레이어가 있다.

```
레이어 1: 버그 패치
  문제 발생 → 핫픽스 → 가능한 빨리 (수일~수주)

레이어 2: 콘텐츠 업데이트
  시즌/확장팩 → 수개월~1년 주기로 대규모 공개
```

라이브 서비스 게임(포트나이트, 리그 오브 레전드 등)은 시즌제를 채택한다. 새 시즌이 시작될 때 대규모 업데이트가 배포된다. 그 사이에는 버그 패치와 소규모 추가가 이루어진다.

**콘솔 게임**: 플랫폼(PlayStation, Xbox)이 패치 승인 프로세스를 가지고 있다. 모바일 앱 심사와 비슷하다. 한국 게임이 해외 콘솔에 출시될 때 이 심사 기간이 계획에 들어가야 한다.

**싱글플레이어 게임**: 출시 후 큰 DLC 하나를 1～2년 뒤에 내는 것이 일반적이다. 라이브 서비스와 달리 지속적 업데이트 압박이 상대적으로 낮다.

### SaaS (서비스형 소프트웨어) — 웹의 속도, 기업의 신중함

Salesforce, Notion, Jira 같은 B2B SaaS는 특이한 위치에 있다.

기술적으로는 웹 서비스라 언제든 배포할 수 있다. 하지만 고객이 기업이다. 기업 고객은 갑작스러운 UI 변경을 싫어한다. 워크플로우가 바뀌면 직원 교육을 다시 해야 한다.

그래서 SaaS는 종종 이런 전략을 쓴다:

```
새 기능 → 옵트인(선택) 방식으로 먼저 공개
→ 일정 기간 후 기본 활성화
→ 이전 방식은 일정 기간 유지 후 제거
```

Notion이 새 에디터를 도입할 때, Figma가 UI를 개편할 때 이런 방식으로 점진적으로 전환한다. 갑자기 바꾸면 기업 고객에게 이탈 사유가 된다.

**릴리즈 노트 문화**: B2B SaaS에서 릴리즈 노트는 중요한 커뮤니케이션 수단이다. 고객이 "이번에 뭐가 바뀌었나"를 확인하고, 자신의 팀에게 알려야 하기 때문이다.

### 안전 최우선 소프트웨어 — 규제가 주기를 정한다

의료기기, 항공, 원자력, 자동차(자율주행) 소프트웨어는 완전히 다른 세계다.

**의료기기 소프트웨어 (FDA 규제, 미국)**

```
설계 → 검증(Verification) → 확인(Validation) → FDA 510(k) 또는 PMA 제출
                                                       ↓
                                              FDA 심사 (수개월~수년)
                                                       ↓
                                                   승인 → 출시
```

소프트웨어 변경이 생기면 이 과정을 다시 거쳐야 할 수 있다. 버그 하나를 고쳐도 규제 문서를 업데이트하고, 재검증이 필요하다. 이 때문에 의료기기 소프트웨어는 처음 설계부터 극도로 보수적이다.

**항공 소프트웨어 (DO-178C 표준)**

소프트웨어 레벨 A (항공기 추락으로 이어질 수 있는 기능) 기준으로, 코드 한 줄에 대해 여러 독립적인 검증이 필요하다. 개발 기간이 수년이고, 출시(인증) 후에는 수십 년 동안 쓰인다. 변경은 극히 제한적이다.

**"인증된 버전"의 개념**: 이런 영역에서는 소프트웨어 버전이 규제 인증서와 묶인다. 버전 2.3.1이 FDA 승인을 받았다면, 2.3.2로 업데이트하려면 다시 승인이 필요할 수 있다. 임의로 업데이트하는 것이 불법이 된다.

### 오픈소스 라이브러리 — 의존하는 사람이 주기를 본다

```
main 브랜치 (개발)
    ↓
rc (Release Candidate) 태그: v2.0.0-rc.1
    ↓
사용자 피드백, 버그 리포트
    ↓
안정화 후: v2.0.0 릴리즈
    ↓
유지보수: v2.0.1, v2.0.2 (버그 패치만)
```

오픈소스 라이브러리의 릴리즈는 해당 라이브러리를 의존성으로 쓰는 수천, 수만 개의 프로젝트에 영향을 준다. 그래서 **시맨틱 버저닝(Semantic Versioning)**을 쓴다.

```
MAJOR.MINOR.PATCH

MAJOR: 하위 호환성이 깨지는 변경 (1.x → 2.0)
MINOR: 하위 호환되는 새 기능 추가 (2.0 → 2.1)
PATCH: 버그 수정 (2.1.0 → 2.1.1)
```

`npm install react`를 하면 `^18.2.0`처럼 설치된다. `^`는 "18.x.x 범위 내 최신"을 의미한다. MAJOR가 바뀌는 업데이트는 자동으로 받지 않는다. 이것이 버전 관리의 약속이다.

**LTS (Long Term Support)**: Node.js, Python, Ubuntu 같은 플랫폼은 LTS 버전을 따로 지정한다. LTS 버전은 새 기능 추가 없이 보안 패치만 오랫동안 받는다. 기업 환경에서는 LTS만 쓰는 경우가 많다.

```
Node.js 예시:
  22 (Current)  → 최신 기능, 변경 잦음
  20 (LTS)      → 안정적, 2026년까지 보안 패치
  18 (LTS)      → 2025년 4월 지원 종료
```

## 테스트 단계의 실제 모습

"테스트"도 단순하지 않다.

```
단위 테스트 (Unit Test)
  함수 하나의 동작 검증, 개발 중 수시로 실행

통합 테스트 (Integration Test)
  여러 컴포넌트가 연결된 상태에서 검증

E2E 테스트 (End-to-End Test)
  실제 사용자처럼 전체 흐름 검증 (브라우저 자동화 등)

성능 테스트
  부하를 주었을 때 응답 시간, 처리량 측정

보안 테스트
  취약점 스캔, 침투 테스트

알파 테스트
  내부 팀이 미완성 버전을 실제로 사용

베타 테스트
  선별된 외부 사용자가 거의 완성된 버전을 사용
  (오픈 베타 / 클로즈드 베타)
```

어떤 테스트를 언제 하느냐가 주기의 속도를 결정한다. 전부 자동화하면 CI/CD 파이프라인으로 흡수된다. 일부는 사람이 직접 해야 한다(탐색적 테스트, 사용성 테스트).

---

## 주기를 결정하는 실제 변수들

개발 주기는 결국 이 변수들의 함수다.

**배포 비용**: 배포 자체가 얼마나 어렵고 위험한가. 웹은 거의 없고, 하드웨어 펌웨어는 극히 높다.

**롤백 가능성**: 잘못됐을 때 되돌릴 수 있는가. 웹 서비스는 몇 초, 모바일 앱은 불가능(이미 설치한 사용자), 의료기기는 불가능.

**사용자 허용 오차**: 사용자가 버그를 얼마나 용납하는가. 게임 얼리 액세스는 높고, 인터넷 뱅킹은 거의 없다.

**규제 환경**: 외부 기관의 승인이 필요한가. 필요하다면 그 기관이 주기를 결정한다.

**팀 크기와 구조**: 혼자 개발하면 프로세스가 단순하다. 100명이 동시에 작업하면 브랜치 전략, 코드 리뷰, 배포 승인 체계가 필요해진다.

**사용자 업데이트 통제권**: 사용자가 버전을 선택할 수 있는가. 없다면(웹, 서버) 개발팀이 모든 책임을 진다.

## 같은 팀이어도 주기가 다르다

하나의 회사 안에서도 서비스별로 주기가 다를 수 있다.

```
마케팅 웹사이트     → 하루에도 여러 번 배포 (거의 위험 없음)
모바일 앱           → 격주 ~ 월간 (스토어 심사 + 사용자 업데이트 주기)
결제 시스템         → 매우 신중하게, 변경 최소화 (장애 비용이 극도로 높음)
데이터 분석 파이프라인 → 비교적 자유롭게 (사용자 직접 노출 없음)
```

같은 회사의 같은 팀이라도 무엇을 다루느냐에 따라 배포에 접근하는 방식이 완전히 달라진다.

## 정리

```
속도로 줄 세우면:

빠름  웹 서비스 백엔드    → 하루 수십 번 가능
      웹 프론트엔드        → 거의 같음, 기능 플래그로 제어
      Android 앱          → 수일 (심사 빠름)
      iOS 앱              → 수일~1주 (심사 있음)
      데스크톱 앱         → 주간~월간 (자동 업데이트 채널 구조)
      게임 (라이브)       → 패치 수일, 콘텐츠 시즌 수개월
      오픈소스 라이브러리 → 안정성 우선, 수개월~반년
      기업용 소프트웨어   → 분기~반년 (도입 검증 기간 포함)
느림  안전 최우선 SW      → 수년 (규제 인증 포함)

핵심 변수:
  배포 비용 × 롤백 가능성 × 사용자 허용 오차 × 규제
```

"개발 → 테스트 → 릴리즈"는 방향은 맞다. 하지만 그 사이 간격이 1시간인지 5년인지, 테스트가 자동화 파이프라인인지 FDA 제출 서류인지, 릴리즈가 git push인지 인증 갱신인지는 전혀 다른 이야기다.
