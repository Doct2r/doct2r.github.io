[프론트엔드와 백엔드가 갈라지는 과정](/posts/frontend-backend-split-after/)에서 제이쿼리는 2006년 크로스브라우저 DOM 조작·Ajax 호출을 하나로 통일해 진입 장벽을 낮춘 다리 역할로 짧게 등장했다. 그런데 리액트·뷰·앵귤러가 웹 프론트엔드를 완전히 장악한 지금도, 제이쿼리는 전체 웹사이트의 68%에 남아 있다. 흥미로운 건 제이쿼리가 원래 풀던 문제 자체는 브라우저 표준이 이미 다 해결했다는 점이다 — 그런데도 왜 아직 안 지워졌을까.

---

## 1. 원래 풀던 문제

2006년 당시 IE6·IE7과 파이어폭스·사파리·오페라는 DOM 조작 방식도, 이벤트 바인딩 방식(`attachEvent` 대 `addEventListener`)도, XMLHttpRequest 동작도 제각각이었다. 제이쿼리의 `$()`는 이 차이를 전부 하나의 일관된 API 뒤로 숨겼다. 이게 제이쿼리가 폭발적으로 퍼진 진짜 이유였다.

## 2. 그 문제는 이미 표준이 해결했다

지금은 사정이 다르다. CSS 선택자로 DOM 요소를 바로 찾는 `querySelector`·`querySelectorAll`은 W3C 셀렉터 API로 2009년 12월 후보 권고안이 됐고 2013년 2월 21일 정식 권고안이 됐다 — IE8(2009)이 부분 지원을, IE9(2011)이 완전 지원을 시작하면서 사실상 모든 현역 브라우저가 갖추게 됐다. XMLHttpRequest의 번거로운 콜백 구조는 2015년 무렵 나온 네이티브 프라미스 기반 `fetch()` API로 대체됐고, 클래스 조작도 `classList` API가 2017년까지는 주요 브라우저 전체에서 크로스브라우저로 굳어졌다. "You Might Not Need jQuery"라는 프로젝트(허브스팟이 만들었고 11년째 유지되고 있다)는 이런 흔한 제이쿼리 패턴 하나하나를 순수 자바스크립트로 대체하는 법을 정리해뒀다 — 제이쿼리가 새 프로젝트에서는 더 이상 필요 없다는 정서를 코드로 증명한 셈이다.

## 3. 그런데도 왜 이렇게 많이 쓰이나

W3Techs가 2026년 7월 기준으로 집계한 수치는 놀랍다 — 전체 웹사이트의 **67.9%**가 제이쿼리를 쓰고 있고, JS 라이브러리를 쓰는 걸로 확인된 사이트로 좁혀도 그중 **87.2%**가 제이쿼리를 포함한다. 다만 이 수치를 읽을 때 조심할 게 있다 — 이건 "이 사이트의 주력 프레임워크가 제이쿼리다"라는 뜻이 아니라, 크롤링한 HTML 소스 안에 제이쿼리가 존재한다는 걸 탐지한 수치다. 리액트로 화면을 그리면서도 어딘가에 제이쿼리 플러그인 하나가 같이 로드돼 있는 사이트도 이 숫자에 함께 잡힌다.

## 4. 진짜 원인 — 워드프레스

이 숫자를 실질적으로 떠받치는 건 워드프레스다. 워드프레스 코어는 제이쿼리를 표준 내장 의존성 핸들(`wp_enqueue_script('jquery')`)로 번들해 제공하고, 다른 라이브러리와 충돌을 피하려고 `noConflict` 모드로 실행한다. 워드프레스 자체가 전체 웹사이트의 약 41～42%를 차지한다는 사실과 겹쳐보면, 이 하나의 CMS가 제이쿼리 통계 전체를 사실상 떠받치고 있다는 그림이 나온다. 드루팔은 조금 다른 길을 갔다 — OpenJS 재단이 jQuery UI의 수명 종료를 선언하면서, 드루팔은 코어에서 jQuery UI 자체를 제거하고 필요한 모듈은 외부 컨트리뷰트 패키지로 받게 했다. 다만 이건 jQuery UI 이야기지, 제이쿼리 코어 자체가 드루팔에서 빠졌다는 뜻은 아니다.

## 5. 제이쿼리 자신도 죽지 않았다

제이쿼리는 지금 OpenJS 재단의 생태계 지속가능성 프로그램 아래서 계속 관리되고 있다. 2026년 1월 17일에는 2016년 3.0 이후 거의 10년 만의 메이저 버전인 4.0.0이 나왔다 — 오래된 "매직" 동작과 지원 중단된 API를 정리하고, IE10 이하 지원을 끊었다(IE11 지원 제거는 5.0에서 예정돼 있다). 3.x 계열은 이제 치명적인 버그 수정만 받는다. 다만 형제 프로젝트들의 사정은 다르다 — jQuery Mobile은 2021년 10월 7일 완전히 폐기됐고, jQuery UI는 2021년부터 신규 기능 없이 보안·회귀 버그 수정만 받는 유지보수 모드에 들어갔다. 지금 실질적으로 계속 개발되는 건 제이쿼리 코어 하나뿐이다.

## 6. 레거시 관성

2008년부터 2018년 사이에 만들어진 수많은 기업 어드민 패널·사내 인트라넷은 데이트피커·폼 검증기 같은 제이쿼리 플러그인에 깊이 의존하고 있다. 브라우저가 하위 호환성을 계속 지켜주는 한, 이 코드는 굳이 뜯어고치지 않아도 멀쩡히 돌아간다. 다시 짜는 데 드는 비용과 리스크에 비해 "그냥 놔두면 계속 작동한다"는 쪽이 압도적으로 쉬운 선택이라, 이런 사이트들을 굳이 걷어낼 강제 요인 자체가 없다.

---

## 정리

제이쿼리가 풀던 기술적 문제는 이미 브라우저 표준 스스로가 해결했다. 그런데도 제이쿼리가 여전히 웹의 3분의 2에 남아 있는 건 기술적 필요 때문이 아니라, 워드프레스라는 거대한 설치 기반과 이미 잘 돌아가는 레거시 코드를 굳이 걷어낼 이유가 없는 관성 때문이다. 유행하는 대체재가 아무리 많이 나와도, 결국 오래 남는 건 가장 세련된 기술이 아니라 가장 널리 깔려 있는 기술이라는 걸 보여주는 사례다.
