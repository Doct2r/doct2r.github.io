for문도 없고, 변수에 값을 다시 대입할 수도 없고, "모나드"라는 정체불명의 개념이 튀어나오는 언어. 학생 때 호기심에 하스켈을 건드려본 사람 대부분이 "이상하고 느리다"는 인상만 남기고 떠난다. 그런데 이 이상한 언어가 은행 거래 시스템과 블록체인 한복판에서 30년 넘게 조용히 쓰이고 있다. 왜 이렇게 생겼고, 왜 아직도 거기 있는지 살펴본다.

---

## 1. 위원회가 만든 특이한 탄생 배경

대부분의 언어는 특정 회사나 개인의 필요에서 나온다(자바는 선, C#은 마이크로소프트, 루비는 마츠모토 개인). 하스켈은 다르다. **1987년 무렵 학계에는 이미 미란다(Miranda), Lazy ML 같은 순수 함수형 언어가 여러 개 난립**해 있었고, 연구자들은 서로 다른 언어로 논문을 쓰느라 소통이 안 되는 상황이었다.

이 문제를 풀기 위해 **사이먼 페이튼 존스, 폴 허드슨, 필립 와들러** 등이 참여한 위원회가 결성됐고, 여러 언어를 하나로 통합한 표준을 1990년 내놓았다. 이름은 조합논리학자 **하스켈 커리(Haskell Curry)**에서 따왔다. 즉 하스켈은 산업적 필요가 아니라 **학계의 합의로 태어난 드문 언어**다.

---

## 2. 왜 이렇게 이상하게 느껴지나

이상함의 정체는 크게 세 가지로 압축된다.

### 순수 함수형 — 부작용이 없다

하스켈 함수는 같은 입력에 항상 같은 출력을 낸다(참조 투명성). 변수에 값을 다시 대입하는 것 자체가 불가능하고, 화면에 뭔가를 출력하는 것조차 "부작용"이라 특별한 취급을 받는다. 대부분의 언어에서 당연한 것들이 하스켈에서는 전부 예외적인 절차를 거쳐야 한다.

### 지연 평가(Lazy Evaluation) — 가장 이질적인 특징

```
일반적인 언어: 코드가 실행되는 순서대로 값이 즉시 계산된다
하스켈       : 값이 실제로 필요해지는 순간까지 계산을 미룬다
```

이 덕분에 하스켈에서는 **무한히 긴 리스트**도 아무 문제 없이 정의할 수 있다 — 필요한 만큼만 계산하면 되기 때문이다. 하지만 반대로 "언제 계산되는지"를 예측하기 어려워지고, 이게 초보자에게 가장 큰 혼란을 준다.

### 모나드 — 악명 높은 그 개념

부작용이 없어야 하는데, 파일을 읽거나 예외를 처리하는 건 어떻게 할까? 이 문제를 풀기 위해 등장하는 게 **모나드(Monad)**다. 흔히 "부리토에 비유하면..." 식의 설명이 넘쳐나는데, 사실 이 현상 자체가 유명하다 — 하스켈 커뮤니티에는 **"모나드 튜토리얼의 오류(Monad Tutorial Fallacy)"**라는 말이 있을 정도다. 모나드를 스스로 이해한 사람은 예외 없이 "나만의 쉬운 설명"을 쓰고 싶어 하지만, 그 설명은 정작 이미 이해한 사람에게만 와닿고 처음 배우는 사람에게는 도움이 안 된다는 뜻이다.

## 3. "진짜 느렸다"는 경험, 사실은 이렇다

학생 때 하스켈이 느리게 느껴졌다면, 그건 하스켈이라는 언어 자체의 한계라기보다 **접했던 방식의 문제**일 가능성이 높다.

```
GHC(Glasgow Haskell Compiler)로 최적화 컴파일한 하스켈
  → 실제로는 C/C++에 근접하는 벤치마크가 나올 정도로 빠르다

GHCi(대화형 인터프리터)로 돌려본 하스켈
  → 컴파일 없이 한 줄씩 해석 실행 — 당연히 느리다
  → 학습 단계에서는 대부분 이 방식을 쓴다
```

여기에 지연 평가의 부작용도 한몫한다. 계산을 미루다 보니 **"썽크(thunk)"**라 불리는 미실행 계산들이 메모리에 계속 쌓이는 **스페이스 리크(Space Leak)** 현상이 초보자 코드에서 흔하게 발생한다. 특히 `foldl` 같은 기본 함수를 무심코 쓰면 이 문제가 바로 터진다 — 하스켈 안내서마다 이 함정을 별도로 경고할 정도로 유명한 초보자 함정이다. 즉 "이상하게 느리다"는 인상은 언어의 본질적 성능 문제가 아니라, **지연 평가라는 낯선 모델을 모른 채 짠 코드가 만든 결과**였을 가능성이 크다.

## 4. 그래서 어디서 쓰는가

| 사용처 | 용도 |
|---|---|
| **메타(구 페이스북)의 Sigma** | 스팸·악성 콘텐츠를 실시간으로 필터링하는 내부 시스템, 하스켈로 개발돼 대규모 트래픽을 처리 |
| **스탠다드차타드 은행** | 하스켈 기반 내부 금융 DSL(Mu)로 파생상품 등 금융 모델링 시스템 구축 |
| **Cardano 블록체인** | IOHK가 개발한 암호화폐 플랫폼 전체가 하스켈로 작성, 형식 검증 방법론 적용 |
| **Pandoc** | 마크다운·HTML·LaTeX·워드 문서를 서로 변환해주는 유명 도구 — 많은 사람이 매일 쓰면서도 하스켈로 짜였다는 걸 모른다 |
| **학계** | 프로그래밍 언어 이론·타입 시스템 연구의 사실상 표준 언어로 지금도 쓰인다 |

## 5. 왜 하필 금융·블록체인이 하스켈을 좋아하는가

이유는 하스켈의 "이상함"이 정확히 이 도메인에서는 장점으로 뒤집히기 때문이다.

```
금융·블록체인 코드의 특징 : 버그 하나의 비용이 극단적으로 크다(돈이 실제로 사라짐)

하스켈의 특징:
  - 강력한 타입 시스템 → 잘못된 상태 조합 자체가 컴파일 에러로 걸러짐
  - 순수 함수(부작용 없음) → 코드가 무엇을 하는지 수학적으로 검증하기 쉬움
  - 패턴 매칭의 완전성 검사 → 처리 안 한 경우의 수를 컴파일 시점에 경고
```

즉 "실행 전에 최대한 많은 실수를 잡아내야 하는" 도메인에서는, 하스켈이 개발 속도를 희생하는 대신 얻는 안전성이 충분히 남는 장사가 된다.

## 6. 왜 대중화는 안 됐나 — 루비와는 다른 궤적

[이전 글](/posts/ruby-decline/)에서 다룬 루비는 "떴다가 식은" 언어였다. 하스켈은 궤적 자체가 다르다 — **애초에 대중적으로 뜬 적이 없다.** 러닝커브가 워낙 가팔라서(순수 함수형+지연 평가+모나드를 동시에 익혀야 함) 대규모 채용이 어렵고, "빨리 뭔가를 만들어내는" 생산성보다 "정확성"을 우선하는 설계 철학 자체가 대부분의 산업 현장이 원하는 우선순위와 맞지 않았다.

대신 이 덕분에 하스켈은 **좁지만 깊은 틈새**를 30년 넘게 안정적으로 유지하고 있다. 유행을 타지 않았으니 유행이 지나가며 무너질 일도 없었다 — 처음부터 대중을 노리지 않은 언어이기 때문에 가능한 안정성이다.

## 정리

```
탄생 배경    → 1990년 학계 위원회가 난립하던 순수 함수형 언어들을 통합해 표준화, 산업적 필요가 아닌 학문적 합의의 산물
이상함의 정체 → 순수 함수형(부작용 없음), 지연 평가(계산을 미룸), 모나드(부작용을 다루는 우회로) — 이 세 개념의 조합
"느렸다"의 진실→ GHC 최적화 컴파일은 실제로 빠름. 학습 단계의 인터프리터 실행 + 지연평가로 인한 스페이스 리크가 진짜 원인
실사용처      → 메타 Sigma(스팸필터), 스탠다드차타드 은행(금융DSL), Cardano 블록체인, Pandoc(문서변환), 학계 연구
선호 이유     → 타입 시스템+순수성으로 버그를 컴파일 시점에 차단 — 실수 비용이 큰 금융·블록체인에 최적
대중화 실패   → 가파른 러닝커브와 정확성 우선 철학. 다만 애초에 유행한 적 없어 유행이 꺼지며 무너질 위험도 없는 안정적 틈새
```

하스켈이 이상하게 느껴지는 이유는 결함이 아니라 설계 목표가 완전히 다르기 때문이다. 빨리 만드는 것보다 확실하게 만드는 것을 택한 언어이고, 그 선택이 필요한 곳(돈이 걸린 시스템, 형식 검증이 필요한 코드)에서는 지금도 조용히, 그리고 확고하게 자리를 지키고 있다.
