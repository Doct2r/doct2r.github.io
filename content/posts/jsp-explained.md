[스프링과 스프링 부트](/posts/spring-vs-spring-boot/)의 탄생 배경을 다룬 적이 있는데, 그 이야기에는 한 조각이 빠져 있었다 — 스프링MVC가 오랫동안 기본 뷰 기술로 썼던 JSP다. JSP는 서블릿의 불편함을 가리려고 태어났는데, 정작 자기 안에 또 다른 불편함(스크립틀릿)을 낳았고, 그 불편함을 지우려는 시도들이 쌓이는 동안 스프링 생태계가 그 위에 올라탔다가 스프링 부트라는 새로운 배포 철학 하나 때문에 결별했다.

---

## 1. 탄생 — 서블릿의 불편함과 ASP라는 경쟁자

자바로 동적 웹페이지를 만드는 첫 방법은 서블릿이었다. 문제는 서블릿 안에서 HTML을 찍어내려면 `out.println("<html>...")` 식으로 자바 코드 안에 마크업 문자열을 통째로 밀어넣어야 했다는 점이다. 페이지 하나 고치려 해도 자바 코드를 다시 컴파일해야 하는 이 구조는 유지보수하기 끔찍했다.

썬 마이크로시스템즈는 1998년 JSP를 처음 발표했고, 라이언 케이블과 에두아르도 펠레그리요파스가 이끈 명세팀이 1999년 6월 JSP 1.0 정식 스펙을, 그해 12월 JSP 1.1을 내놨다. 이건 사실상 마이크로소프트가 1996년 내놓은 ASP(Active Server Pages)에 대한 자바 진영의 대응이었다 — HTML 안에 서버 코드를 끼워 넣는다는 ASP의 발상을 그대로 가져오되 언어만 자바로 바꾼 셈이다.

JSP의 실제 동작 방식을 알면 이게 "새로운 기술"이라기보다는 "서블릿 위에 얹은 문법 설탕"이라는 게 드러난다. JSP 파일은 처음 요청이 들어올 때 서블릿 컨테이너(예를 들어 톰캣의 JSP 엔진 재스퍼)가 `_jsp.java`라는 자바 서블릿 소스로 번역하고, 이걸 다시 `.class`로 컴파일해 실행한다. 즉 개발자가 보는 `.jsp` 파일과 실제로 돌아가는 건 결국 평범한 서블릿이다.

## 2. 문법의 변화 — 스크립틀릿에서 JSTL·EL로

초기 JSP는 `<% %>`(스크립틀릿), `<%! %>`(선언), `<%= %>`(표현식) 태그로 자바 코드를 HTML에 그대로 박아넣는 방식이었다. 이건 서블릿의 문제를 형태만 뒤집었을 뿐이다 — 이번엔 HTML 안에 자바 코드가 섞이며 유지보수가 힘들어졌다.

이 문제를 풀려고 2002년 JSTL(JSP Standard Tag Library, JSR 52)이 나왔다 — `<c:if>`·`<c:forEach>` 같은 태그로 제어 흐름을 표현해 스크립틀릿을 대체하는 게 목적이었다. 이듬해인 2003년 JSP 2.0(JSR 152)에서는 `${...}` 형태의 표현언어(EL)가 정식 도입됐고, 2005년 JSP 2.1(JSR 245)에서는 이 EL이 JSF(JavaServer Faces)의 표현언어와 통합돼 "통합 표현언어(Unified EL)"가 됐다. JSP 2.0 이후로는 스크립틀릿을 쓰지 말라는 게 사실상 업계 표준 권고가 됐다 — "JSP는 처리 단위가 아니라 뷰 단위여야 한다"는 원칙이 이때 자리 잡았다.

## 3. Model 1과 Model 2 — MVC를 JSP 위에 얹기

1998년 썬이 공개한 JSP 스펙 0.92 초안에는 JSP를 쓰는 두 가지 방식이 소개돼 있었다. "Model 1"은 JSP 페이지 자체가 화면 출력과 처리 로직을 동시에 떠맡고 페이지 간 이동은 하이퍼링크로 하는 방식이었다 — 초기 JSP 애플리케이션 대부분이 이런 형태였다. "Model 2"는 서블릿을 프론트 컨트롤러로 두어 요청을 먼저 가로채 처리하게 하고, 그 서블릿이 순수 화면 역할만 하는 JSP로 결과를 넘기는 방식이다. Model 2가 사실상 MVC 패턴을 JSP 생태계에 이식한 셈인데, 정작 썬의 자바 블루프린트 문서는 이걸 "정식 MVC로 부르는 건 과장"이라며 선을 그었다 — 이름과 실제 구현 사이에 거리가 있었다는 뜻이다.

## 4. 스트럿츠 — Model 2를 프레임워크로 만든 첫 시도

크레이그 맥클래너핸이 만든 아파치 스트럿츠는 2000년 5월 아파치 재카르타 프로젝트에 기부됐고, 2001년 6월 15일 1.0 정식판이 나왔다. `ActionServlet`이라는 프론트 컨트롤러가 요청을 가로채고, `struts-html` 같은 JSP 커스텀 태그로 화면을 그리는 구조로 Model 2 패턴을 실제 프레임워크로 만들어낸 첫 사례였다. 2000년대 초중반 내내 엔터프라이즈 자바 웹 개발의 사실상 표준으로 군림했다.

스트럿츠의 신뢰가 결정적으로 무너진 계기는 2017년이다. 3월 7일 아파치가 스트럿츠2의 치명적 원격 코드 실행 취약점 CVE-2017-5638(CVSS 10.0 만점)을 공개하고 같은 날 패치를 냈는데, 에퀴팩스는 이 패치를 적용하지 않았다. 사흘 뒤인 3월 10일 공격자들이 정확히 이 취약점으로 침투했고, 5월부터 7월 사이 1억 4,790만 명의 개인정보가 유출됐으며, 이 사건은 그해 9월 공개됐다. 패치가 나온 지 두 달 넘게 방치된 대가였다.

## 5. 스프링MVC와 JSP — 오래된 동거

스프링MVC(2004년 등장)는 오랫동안 JSP를 기본 뷰 기술로 사용했다. `InternalResourceViewResolver`가 컨트롤러가 반환한 논리적 뷰 이름을 `/WEB-INF/views/` 아래의 실제 JSP 파일로 매핑하는 방식이었고, 폼 바인딩 등을 위한 스프링 자체 태그 라이브러리(`spring:` 태그)도 갖추고 있었다. 스프링 생태계 안에서 JSP는 오랫동안 그냥 당연한 선택지였다.

## 6. 스프링 부트가 끊어낸 지점 — 실행 가능 JAR과 JSP의 근본적 불화

문제는 2014년 스프링 부트가 내세운 철학이었다 — `java -jar app.jar` 한 줄로 실행되는 내장 서버라는 단순함이 핵심 셀링 포인트였다. 그런데 JSP 컴파일은 톰캣의 재스퍼 엔진이 WAR가 풀린 실제 디렉터리 구조를 필요로 하는 방식으로 설계돼 있어서, 실행 가능 JAR 안에서는 제대로 동작하지 않는다 — 톰캣 내부에 하드코딩된 파일 패턴 때문이다. 스프링 부트 공식 레퍼런스 문서에는 아예 "JSP 제한사항"이라는 전용 챕터가 있고, 여기서 실행 가능 JAR로는 안 되고 WAR로만 배포해야 한다고 명시한다. 내장 서블릿 컨테이너로 Undertow를 쓰는 경우엔 JSP 자체를 지원하지 않는다는 제약도 있다. 즉 "그냥 jar 하나 실행하면 끝"이라는 스프링 부트의 핵심 가치 제안 자체가, JSP를 쓰는 순간 깨져버린다.

## 7. 타임리프의 등장 — 브라우저에서 그냥 열리는 템플릿

이 자리를 메운 게 타임리프다. 스페인 개발자 다니엘 페르난데스가 만들었고 2011년 7월 17일 1.0.0 정식판이 나왔다. 핵심 차별점은 "자연 템플릿(natural templating)"이다 — JSP 파일은 서버가 처리하기 전에는 브라우저에서 열어도 제대로 안 보이지만, 타임리프 템플릿은 `th:text` 같은 HTML 속성만 덧붙인 형태라 서버 없이 그냥 브라우저에서 열어도 유효한 HTML로 보인다. 스프링 부트가 `spring-boot-starter-thymeleaf`를 공식 추천 스타터로 채택하면서, 타임리프는 사실상 스프링 생태계의 새 기본값이 됐다.

## 8. 지금 JSP는 — 이름은 바뀌었지만 명맥은 남아 있다

2020년 12월 Jakarta EE 9에서 있었던 `javax`→`jakarta` 개명(다른 글에서 다룬 적 있다)으로 JavaServer Pages는 "Jakarta Pages(Jakarta Server Pages)"로 이름이 바뀌었고 패키지도 `jakarta.servlet.jsp`로 바뀌었다. 스펙 자체는 2025년 Jakarta EE 11까지 계속 관리되고 있다 — 완전히 죽은 기술은 아니라는 뜻이다. 다만 신규 프로젝트에서 JSP를 새로 선택하는 경우는 사실상 없고, 실행 가능 JAR로 배포되지 않는 오래된 엔터프라이즈 시스템을 유지보수하는 현장에만 남아 있는 기술이라는 게 지금의 현실적인 위상이다.

---

JSP의 궤적을 정리하면 이렇다 — 서블릿의 불편함을 가리려고 태어났지만 스크립틀릿이라는 자기 불편함을 낳았고, 그걸 지우려는 JSTL·EL·Model 2·스트럿츠가 쌓이는 동안 스프링 생태계는 그 위에 자연스럽게 올라탔다. 그리고 정작 JSP를 밀어낸 건 더 나은 문법이 아니라, "그냥 jar 하나 실행하면 된다"는 배포 방식의 단순함이었다.
