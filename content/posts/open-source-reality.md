오픈소스를 "공짜 소프트웨어"로 이해하는 사람이 많다. 틀리지는 않지만 절반만 맞다. 소스 코드가 공개되어 있다고 해서 그것을 만들고 유지하는 데 시간도 돈도 들지 않는 것은 아니다. 그리고 "아무도 소유하지 않는다"는 것도 정확하지 않다. 라이선스가 그 소유와 사용의 규칙을 정의한다. 오픈소스의 진짜 모습을 처음부터 짚어본다.

---

## 1. 오픈소스의 정의와 역사

### "자유"와 "무료"는 다른 말이다

오픈소스의 철학을 이해하려면 영어 단어 **free**의 두 가지 의미를 구분해야 한다.

- **free as in freedom**: 자유롭다. 코드를 보고, 수정하고, 배포할 권리
- **free as in free beer**: 무료다. 돈이 들지 않는다

오픈소스 운동의 출발점은 후자가 아니라 전자였다.

### 리처드 스톨먼과 GNU (1983)

1983년 MIT의 프로그래머 리처드 스톨먼(Richard Stallman)이 **GNU 프로젝트**를 선언했다. 당시 소프트웨어 산업은 소스 코드를 비밀로 잠그고 파는 방향으로 흘러가고 있었다. 스톨먼은 이것이 프로그래머 공동체의 협업 문화를 파괴한다고 보았다.

그가 주장한 **네 가지 자유**:
0. 어떤 목적으로든 프로그램을 실행할 자유
1. 프로그램이 어떻게 동작하는지 연구하고, 원하는 대로 수정할 자유 (소스 접근 필요)
2. 이웃을 돕기 위해 복사본을 배포할 자유
3. 수정된 버전을 배포해 커뮤니티에 기여할 자유

그리고 이 자유를 법적으로 보장하기 위해 **GPL(GNU General Public License)**을 만들었다.

### 리누스 토발즈와 Linux (1991)

1991년 핀란드 대학생 리누스 토발즈(Linus Torvalds)가 취미로 운영체제 커널을 만들고 공개했다. GNU의 도구들과 결합되어 **Linux**가 탄생했다. 지금 전 세계 서버의 대부분, Android 폰 전체, 슈퍼컴퓨터의 대부분이 Linux 위에서 돌아간다.

### OSI와 "오픈소스"라는 이름 (1998)

1998년 에릭 레이먼드(Eric Raymond)와 브루스 페렌스(Bruce Perens)가 **OSI(Open Source Initiative)**를 설립하고 "오픈소스(Open Source)"라는 용어를 제안했다. "자유 소프트웨어"라는 표현이 비즈니스 세계에서 거부감을 일으킨다고 판단했기 때문이다. 오픈소스는 기술적·실용적 측면을 강조하는 브랜딩이었다.

스톨먼은 이 분리에 반대했다. "오픈소스는 자유를 빠뜨린 개념"이라며 지금도 "자유 소프트웨어"라는 표현을 고집한다. 두 진영은 지향점이 약간 다르다.

## 2. 라이선스 — 오픈소스의 법적 뼈대

오픈소스는 "아무나 써도 된다"가 아니다. **라이선스**가 사용 조건을 정한다. 라이선스를 무시하면 저작권법 위반이다.

오픈소스 라이선스는 크게 **허용적(Permissive)**과 **카피레프트(Copyleft)** 두 계열로 나뉜다.

### 허용적 라이선스 — "써도 된다, 단 저작권 표시는 해라"

조건이 거의 없다. 상업적으로 써도 되고, 수정해도 되고, 소스를 공개하지 않아도 된다. 단, 원 저작권 고지를 유지해야 한다.

**MIT 라이선스**

가장 단순하고 가장 많이 쓰인다. 라이선스 전문이 한 페이지도 안 된다.

```
MIT License
Copyright (c) [연도] [저작자]

이 소프트웨어를 누구든지 무상으로 제한 없이 사용, 복사, 수정,
합병, 출판, 배포, 서브라이선스, 판매할 수 있으며...
단, 위 저작권 고지와 이 허가 고지를 모든 복사본에 포함해야 한다.
```

React, Rails, jQuery, .NET (일부) — 대부분의 유명 프레임워크가 MIT다. "써도 된다"는 의미에서 가장 자유롭다.

**Apache License 2.0**

MIT보다 조금 길지만 역시 허용적이다. 추가 조건: 특허권 관련 조항이 있다. 기여자가 자신의 특허를 사용자에게 무상으로 허가하며, 사용자가 기여자를 특허로 고소하면 라이선스가 즉시 취소된다.

Kubernetes, Android, TensorFlow가 Apache 2.0이다.

**BSD 라이선스 (2-Clause, 3-Clause)**

MIT와 유사하다. 3-Clause는 "이 소프트웨어를 홍보 자료에 사용할 때 저작자 이름을 쓰지 마라"는 조항이 추가된다. FreeBSD, OpenBSD가 BSD 라이선스다.

---

### 카피레프트 라이선스 — "써도 된다, 단 수정본도 공개해라"

카피레프트의 핵심 조건: 이 코드를 수정해서 배포하면, 수정된 코드도 **같은 라이선스로** 공개해야 한다. 오픈소스의 자유가 전파(감염)된다.

**GPL v2 / v3**

가장 강한 카피레프트. Linux 커널은 GPL v2다.

핵심 조건: GPL 코드를 포함한 소프트웨어를 배포하면, **전체 소프트웨어의 소스 코드**를 GPL로 공개해야 한다.

이것이 "GPL 감염"이라 불리는 이유다. 기업 입장에서는 GPL 코드를 자사 독점 소프트웨어에 통합하면 자사 코드 전체를 공개해야 하는 상황이 생길 수 있다. 그래서 많은 기업이 GPL 코드 사용을 피한다.

GPL v2와 v3의 차이: v3는 **Tivoization** 방지 조항이 추가됐다. Tivo는 Linux(GPL v2)를 쓰면서 하드웨어적으로 수정 설치를 막았다. v3는 이를 막기 위해 "하드웨어가 수정된 소프트웨어를 실행할 수 있어야 한다"는 조건을 추가했다. 리누스 토발즈는 이 조항을 싫어해서 Linux는 여전히 GPL v2다.

**LGPL (Lesser GPL)**

GPL의 완화 버전. 라이브러리 형태로 쓰일 때 적용. LGPL 라이브러리를 **동적 링크**해서 쓰는 것은 소스 공개 의무 없음. 수정한 라이브러리 자체는 공개해야 함.

GNU C Library (glibc)가 LGPL이다. 덕분에 모든 Linux 프로그램이 glibc를 쓸 수 있다.

**AGPL (Affero GPL)**

GPL의 강화 버전. "배포"의 정의를 네트워크 서비스까지 확장한다.

일반 GPL은 소프트웨어를 **배포(distribute)**할 때만 소스 공개 의무가 생긴다. 서버에서 돌리면서 사용자에게 서비스만 제공하면 소스를 공개하지 않아도 됐다. Google, Amazon 같은 클라우드 기업이 오픈소스를 서버에서 돌리며 소스를 공개하지 않는 "SaaS 허점"이 생겼다.

AGPL은 이를 막는다. 네트워크로 서비스를 제공해도 소스 공개 의무가 생긴다. MongoDB가 AGPL이었고, Nextcloud, Mastodon이 AGPL이다.

---

### 라이선스 계보 한눈에

```
허용적 (소스 공개 의무 없음)
  MIT ────────────── 가장 단순
  BSD 2/3-Clause ─── MIT와 유사, 홍보 제한 조항
  Apache 2.0 ─────── MIT + 특허 조항

카피레프트 (소스 공개 의무 있음)
  LGPL ─── 라이브러리 동적 링크는 예외
  GPL v2 ── 배포 시 전체 소스 공개
  GPL v3 ── GPL v2 + Tivoization 방지
  AGPL ─── GPL v3 + 네트워크 서비스도 포함
```

---

### 비공개 소스 기반 새 라이선스들

최근 기업들이 오픈소스의 이점(커뮤니티, 채택)을 누리면서 클라우드 기업의 무임승차를 막기 위한 **중간형 라이선스**를 만들었다. OSI 기준 오픈소스는 아니다.

**SSPL (Server Side Public License)**: MongoDB가 만들었다. AGPL보다 강하다. SSPL 소프트웨어를 서비스로 제공하는 **전체 서비스 스택**의 소스를 공개해야 한다.

**BSL (Business Source License)**: MariaDB가 만들었다. 특정 기간(보통 4년) 동안은 상업적 서비스로 사용 제한, 이후 자동으로 오픈소스 라이선스로 전환. Hashicorp(Terraform), Sentry가 BSL을 채택했다.

이 흐름은 "오픈소스의 배신"이라는 비판과 "지속가능성을 위한 현실"이라는 옹호 사이에서 논쟁 중이다.

## 3. 유지보수 현실 — 누가, 어떻게 하는가

### 오픈소스를 떠받치는 사람들

세계 인터넷 인프라의 상당 부분이 소수의 개인이 취미로 관리하는 오픈소스 위에 올라가 있다.

**OpenSSL**은 2014년 **Heartbleed** 취약점이 발견되기 전까지 전체 인터넷 HTTPS 트래픽의 절반 이상을 암호화하는 라이브러리였다. 그 유지보수를 담당하던 풀타임 개발자는 **단 1명**, 스티브 마켈이었다. 연간 예산은 2,000달러 수준이었다. Heartbleed는 이 현실을 세상에 드러냈다.

```
OpenSSL Heartbleed (2014):
  - 영향: 전 세계 HTTPS 사이트 약 17% (수십만 개)
  - 원인: 경계 검사 누락 (단순한 코드 오류)
  - 유지보수 인력: 사실상 1명
  - 연간 예산: 약 2,000달러
```

### 번아웃 문제

오픈소스 메인테이너(Maintainer)는 종종 자원봉사자다. 수백만 명이 사용하는 라이브러리를 무보수로 관리하며, 버그 리포트와 기능 요청이 쏟아진다.

**colors.js / faker.js 사건 (2022)**

npm에서 수백만 번 다운로드되는 라이브러리 `colors.js`와 `faker.js`의 메인테이너 Marak Squires가 의도적으로 라이브러리를 망가뜨렸다.

`colors.js` 새 버전은 실행하면 무한 루프로 쓰레기 텍스트를 출력했다. `faker.js`는 GitHub에서 삭제됐다. 메인테이너가 남긴 메시지:

> "대기업들은 내 무료 작업으로 수백만 달러를 벌지만, 나는 아무것도 받지 못한다."

수천 개의 프로젝트와 기업 서비스가 즉시 영향을 받았다. 사건은 오픈소스 지속가능성 문제를 다시 수면 위로 올렸다.

**Log4Shell 사태 (2021)**

Java 로깅 라이브러리 Log4j2에서 발견된 **Log4Shell**은 역대 최악의 취약점 중 하나로 평가됐다. CVSS 점수 10.0 만점.

```
Log4j2 사용 범위:
  Apple, Amazon, Twitter, Cloudflare, Steam,
  Minecraft, Tesla, LinkedIn, NASA...
  → 사실상 Java를 쓰는 서비스 전체
```

수십억 기기에 영향을 미쳤다. Apache Log4j 프로젝트 팀은 소규모 자원봉사자들이었고, 사태 수습을 위해 수일간 밤을 새워야 했다. 이들이 받은 보상은 없었다.

## 4. 돈의 문제 — 오픈소스는 정말 무료인가

### 비용이 드는 것들

오픈소스 소프트웨어 자체는 무료로 받을 수 있다. 하지만 그것을 쓰는 데는 비용이 든다.

- **인프라**: 코드를 CI로 테스트하는 서버, 문서 호스팅, CDN
- **개발자 시간**: 버그 수정, 기능 추가, 코드 리뷰
- **보안 감사**: 전문 보안 업체에 맡기는 취약점 검토
- **법무**: 라이선스 충돌, 특허 분쟁 대응

GitHub 자체도 돈이 드는 인프라다. 물론 오픈소스 프로젝트는 무료로 쓸 수 있지만, GitHub를 운영하는 Microsoft는 이것을 무한정 공짜로 제공하지 않는다. 기업 고객과 GitHub Actions 유료화로 수익을 낸다.

### 오픈소스의 수익화 모델

오픈소스를 만들면서 돈을 버는 방법은 여러 가지다.

**Dual Licensing (이중 라이선스)**

같은 소프트웨어를 두 가지 라이선스로 제공한다. 오픈소스 버전은 GPL(소스 공개 의무 있음), 상업용 버전은 별도 유료 라이선스.

```
MySQL 예시:
  GPL 버전  → 오픈소스 프로젝트에 무료
  상업 라이선스 → 소스 공개 없이 독점 제품에 쓰려면 Oracle에 돈 내야 함
```

Qt 프레임워크도 같은 방식이다. 이 모델은 오픈소스의 이점(커뮤니티, 채택)을 누리면서 상업적 수익을 낸다.

**오픈코어 (Open Core)**

핵심 기능은 오픈소스, 기업용 고급 기능은 유료.

```
GitLab CE (Community Edition) → 오픈소스, 무료
GitLab EE (Enterprise Edition) → 고급 기능 추가, 유료

Elasticsearch OSS → 오픈소스
Elastic Cloud → 관리형 서비스, 유료
```

**서비스로서의 오픈소스 (SaaS / Managed Service)**

코드는 공개하되 운영·관리 서비스로 돈을 받는다. "코드는 공개돼 있지만 직접 운영하는 것은 어렵다"는 현실을 이용한다.

```
MongoDB → 오픈소스
MongoDB Atlas → AWS/GCP/Azure에서 관리형으로 쓰면 MongoDB Inc.에 비용
```

AWS, Google, Azure가 오픈소스를 관리형 서비스로 팔아 막대한 수익을 올리는 것이 원 개발사와의 갈등 원인이기도 하다. MongoDB, Redis, Elasticsearch가 이에 반발해 라이선스를 바꿨다.

**지원·컨설팅**

Red Hat의 모델이다. Linux(오픈소스)를 기업 환경에서 안정적으로 운영하도록 지원·컨설팅·인증을 유료로 제공한다. IBM이 Red Hat을 340억 달러에 인수한 이유다.

**기부 / 스폰서십**

개인 개발자나 소규모 프로젝트에서 주로 쓰인다.

- **GitHub Sponsors**: 개발자에게 직접 후원
- **Open Collective**: 프로젝트 단위 투명한 자금 관리
- **Patreon**: 창작자 후원 플랫폼. 오픈소스 개발자도 사용

Vue.js의 에반 유(Evan You)는 Patreon으로 전업 오픈소스 개발자가 됐다. 현재 월 수만 달러를 스폰서십으로 받는다. 하지만 이 수준에 도달하는 개발자는 극소수다.

## 5. 기업과 오픈소스 — 이중적인 관계

### 기업의 오픈소스 참여

대형 기술 기업들은 오픈소스의 최대 기여자이기도 하다.

| 기업 | 대표 오픈소스 기여 |
|---|---|
| Google | Android, Kubernetes, TensorFlow, Chrome V8, Go, Dart |
| Meta | React, PyTorch, Llama |
| Microsoft | VS Code, TypeScript, .NET, WSL |
| Apple | WebKit, Swift, LLVM (Clang) |
| Amazon | OpenSearch (Elasticsearch 포크) |
| Linux Foundation | Linux 커널 (기업 기여자 다수) |

왜 대기업이 오픈소스에 기여하는가?

- **인재 채용**: 오픈소스 기여자 커뮤니티에서 우수한 개발자를 발굴
- **생태계 주도**: 자사가 만든 도구가 표준이 되면 그 위에 쌓이는 모든 것이 자사에 유리
- **비용 분산**: 혼자 개발하는 것보다 커뮤니티와 함께 개발하면 비용이 줄어든다
- **이미지**: 개발자 커뮤니티에서 호감을 얻는다

### 무임승차 문제

그러나 기업이 오픈소스를 쓰기만 하고 기여하지 않는 경우도 많다.

```
오픈소스 프로젝트의 전형적 기여 분포:
  핵심 유지보수: 소수의 개인 자원봉사자
  코드 기여: 기업 직원 일부 + 개인 기여자
  사용: 수백만 개 기업 (기여 없음)
```

**AWS와 Elasticsearch 갈등 (2021)**

AWS가 Elasticsearch를 포크해서 자체 관리형 서비스(Amazon Elasticsearch Service)를 운영했다. 원 개발사 Elastic에 라이선스 비용을 내지 않았다. Elastic은 이에 반발해 라이선스를 SSPL로 변경했다. AWS는 Elasticsearch를 별도로 포크해 **OpenSearch**라는 이름으로 유지하고 있다. 두 프로젝트가 분리된 채 경쟁 중이다.

## 6. 오픈소스의 지속가능성

### 현재 상황

2022년 Linux Foundation과 Harvard LLAB이 공동 발표한 보고서(Census II):

> "자유롭고 오픈소스인 소프트웨어(FOSS)의 핵심 라이브러리 대부분이 소수의 개인에 의존하며, 많은 경우 단 한 명의 메인테이너가 있다."

현대 소프트웨어 공급망은 거대한 오픈소스 라이브러리의 탑이다. npm 생태계에는 200만 개 이상의 패키지가 있다. 그 중 많은 것이 수십 년 된 코드를 혼자 관리하는 누군가에 의존한다.

```
현대 앱의 의존성 구조:
  내 앱
  └── 라이브러리 A (기업 개발)
      └── 라이브러리 B (오픈소스, 활발)
          └── 라이브러리 C (오픈소스, 관리자 1명)
              └── 라이브러리 D (오픈소스, 수년째 업데이트 없음)
```

라이브러리 D가 보안 취약점을 가지고 있어도, 아무도 고치지 않으면 그 위에 쌓인 모든 것이 위험해진다.

### 해결을 위한 시도들

**OpenSSF (Open Source Security Foundation)**

Heartbleed 이후 구글, 마이크로소프트, 메타 등이 설립한 재단. 오픈소스 보안 인프라에 자금을 지원한다.

**Sovereign Tech Fund (독일)**

독일 연방 정부가 운영하는 오픈소스 지원 기금. curl, Fortran 컴파일러, OpenBSD 등에 자금을 지원했다.

**GitHub Sponsors / Open Collective**

개인·기업이 오픈소스 개발자를 직접 후원하는 플랫폼. 자발적 참여에 의존해 불안정하다는 한계가 있다.

## 7. 오픈소스는 누구의 것인가

처음 질문으로 돌아간다.

**소스 코드 자체**: 저작자의 것이다. 오픈소스 라이선스는 "이 조건 하에 사용을 허락한다"는 것이지, 저작권을 포기하는 것이 아니다.

**GPL 코드의 파생물**: 라이선스가 공유를 강제한다. "누구의 소유도 아닌 공유지"가 되도록 설계됐다.

**기업 기여 코드**: 기업이 오픈소스에 기여한 코드는 해당 라이선스를 따른다. 기여 후 철회할 수 없다.

```
오픈소스 저작권의 실제 구조:
  Linux 커널 코드 한 줄 한 줄은 각각의 기여자가 저작권자
  → 코드는 GPL로 보호됨
  → 누군가 GPL을 위반하면 저작권자가 소송 가능
  → 저작권자가 수천 명이므로 "공동 소유" 형태
```

오픈소스를 "무주공산"으로 보는 것은 틀렸다. 각 코드 줄마다 저작권자가 있고, 라이선스가 그 사용 조건을 정한다. 그것을 무시하면 저작권 침해다.

## 정리

```
오픈소스 = 소스 공개 + 라이선스 조건

라이선스 계열:
  허용적 (MIT, Apache, BSD) → 자유롭게 쓰되 저작권 표시
  카피레프트 (GPL, AGPL)   → 수정·배포하면 소스도 공개
  중간형 (SSPL, BSL)       → 오픈소스 아닌 "소스 공개"

현실:
  ├ 유지보수: 대부분 소수의 자원봉사자
  ├ 비용: 인프라, 시간, 보안 감사 — 공짜 아님
  ├ 번아웃: colors.js, Log4j 사태가 보여준 현실
  └ 수익화: Dual License, 오픈코어, SaaS, 지원·컨설팅

소유권:
  "무주공산"이 아니다
  각 기여자가 저작권 보유, 라이선스가 사용 조건 정의
  라이선스 위반 = 저작권 침해
```

오픈소스 생태계는 엄청난 가치를 만들어낸다. 현대 인터넷의 대부분이 그 위에 서 있다. 그러나 그것을 떠받치는 사람들이 지속 가능하게 일할 수 있는 구조는 아직 제대로 갖춰지지 않았다. "공짜로 쓸 수 있다"와 "공짜로 만들어진다"는 완전히 다른 말이다.
