노드를 선으로 잇는 위지윅 프로그래밍은 몇 번이나 시도됐지만 범용 소프트웨어 개발의 주류가 되지 못했다. 그런데 절차에서 완전히 벗어나려 하면 다른 문제가 생긴다 — `A = B + C`를 쓴 다음 `A = 10`을 다시 쓰면, 이게 대입인지 모순된 연립방정식인지 헷갈린다. 이 두 관찰은 사실 같은 뿌리에서 나온다.

---

## 1. "전혀 주류가 못 됐다"는 전제는 절반만 맞다

정확히 짚자면, 비절차적·시각적 프로그래밍이 실패한 게 아니라 **특정 도메인에서는 이미 주류**다.

- **LabVIEW**(1986, 내셔널인스트루먼츠): 계측·제어 엔지니어링 분야에서 데이터플로우 노드 프로그래밍이 실제 표준이다.
- **Max/MSP와 Pure Data**(1980년대, 밀러 퍽): 전자음악·인터랙티브 미디어 분야에서 노드로 신호 흐름을 짜는 게 당연한 작업 방식이다.
- **언리얼 엔진 블루프린트**(2014): 게임 디자이너가 프로그래머 없이 게임 로직을 짜는 데 광범위하게 쓰인다.
- **Node-RED**(2013, IBM): IoT 기기 연동에서 니치를 확보했다.
- **엑셀 스프레드시트**: 전 세계에서 가장 널리 쓰이는 비절차적 프로그래밍 환경일 수 있다. 셀 참조와 수식은 절차형이 아니라 선언적 데이터플로우 모델이다.

반대로 **야후 파이프스**(2007~2015, 웹 데이터 처리 노드 도구)처럼 비슷한 시도가 실패해 서비스 종료로 끝난 경우도 있다. 시각적 프로그래밍이 실패하느냐 성공하느냐는 도메인에 달려 있다.

---

## 2. 왜 "범용 소프트웨어 개발"에서는 여전히 텍스트가 이기는가

- **압축도**: 텍스트는 한 줄에 훨씬 많은 정보를 담는다. 로직이 복잡해지면 노드 그래프는 선이 얽히는 "비주얼 스파게티"가 된다 — 실제로 블루프린트나 랩뷰 사용자들이 흔히 겪는 문제다.
- **검색·비교·버전관리와의 궁합**: 텍스트 코드는 grep, git diff처럼 수십 년간 쌓인 텍스트 도구 생태계를 그대로 쓸 수 있다. 노드 그래프는 특수 파일 포맷으로 저장돼 코드 리뷰와 병합이 훨씬 어렵다.
- **추상화 표현력**: 재귀나 고차함수 같은 고급 추상화는 텍스트로는 간결한 기호로 압축되지만, 노드로 그리면 오히려 더 번잡해진다.
- **속도**: 숙련자에게는 키보드 타이핑이 마우스로 노드를 배치하고 잇는 것보다 훨씬 빠르다.

---

## 3. `A = B + C` 다음 `A = 10`의 혼란은 왜 생기는가

이건 정확히 **명령형의 대입(assignment)**과 **선언형의 방정식(equation)**이 서로 다른 의미를 갖기 때문이다.

절차형 언어에서 `A = B + C`는 "이 순간 A에 B+C 값을 저장하라"는 명령이다. 그다음 `A = 10`은 A를 다시 10으로 덮어쓰라는 별개의 명령일 뿐이다. 시간에 따라 상태가 바뀌는 걸 그대로 반영한다.

반면 완전한 선언형·제약 기반 시스템에서 `A = B + C`는 "항상 성립해야 하는 관계"다. 이 경우 나중에 `A = 10`을 쓰면 이미 세운 제약과 모순되거나, 반대로 B와 C가 자동으로 재계산돼야 한다. 실제로 엑셀에서 셀에 수식을 넣은 뒤 그 셀에 값을 직접 입력하면, 엑셀은 모순을 해결하려 하지 않고 그냥 **수식을 지워버리고 값으로 덮어쓴다** — 이게 정확히 사용자가 우려한 혼란을 실제 소프트웨어가 마주하고 택한 해법이다.

1977년 존 배커스(포트란을 만든 그 사람)는 튜링상 수상 강연 "Can Programming Be Liberated from the von Neumann Style?"에서 이 문제의 근원을 정확히 짚었다. 그의 논지는, **폰 노이만 아키텍처 자체가 이미 "차례로 실행되는 명령들"이라는 시간적 순서를 전제**하고 있고, 대부분의 주류 언어는 이 하드웨어 모델을 그대로 노출하도록 설계됐다는 것이다. 인간의 사고가 자연스럽게 "먼저 이거, 그다음 이거"라는 시간적 순서로 작동하는 것과, 이 아키텍처가 정확히 맞아떨어진다.

---

## 4. 완전한 선언형은 이 혼란을 어떻게 없애는가

순수 함수형 언어는 재대입 자체를 금지하고 **불변성(immutability)**을 강제해 이 혼란을 원천적으로 없앤다. 한 번 `A = B + C`로 정의되면 A는 그 값에서 다시는 바뀌지 않는다([[haskell-explained]]에서 다룬 하스켈이 이 방식이다). 다만 이건 대가가 있다 — 반복문의 카운터 증가처럼 "값이 바뀌어야 하는" 상황을 표현하려면 재귀나 새 이름의 바인딩을 만드는 다른 방법을 배워야 한다. 혼란을 없애는 대신 학습 비용이 다른 곳으로 옮겨갈 뿐이다.

---

## 5. 결론

프로그래밍이 절차와 텍스트에서 "못 벗어나는" 게 아니다. 절차형과 선언형은 각자 다른 문제에 최적화된 도구이고, 우리는 이미 둘 다 널리 쓰고 있다 — SQL과 CSS는 선언형이고, 대부분의 애플리케이션 로직은 절차형이다. `A = B + C` 다음 `A = 10`이 헷갈리는 이유도 절차를 벗어나는 데 실패했다는 증거가 아니라, 애초에 절차형이 풀고 있던 문제(시간에 따라 바뀌는 상태)를 선언형이 다른 트레이드오프(불변성과 재귀)로 풀 때 나타나는 필연적인 대가다. 도메인이 흐름도에 가까울수록 시각적·선언적 방식이 이기고, 범용적이고 복잡한 로직일수록 텍스트·절차형이 이긴다 — 어느 한쪽이 열등해서가 아니라, 각자 원래 잘하는 자리가 다르기 때문이다.
