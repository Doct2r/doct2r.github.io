`if (x > 0) { ... }` 이 형태를 처음 본 언어가 무엇이었는지 기억하는 사람은 드물다. C 이후 나온 언어 대부분이 중괄호로 블록을 감싸고, 세미콜론으로 문장을 끝내고, `==`와 `&&`와 `++`를 그대로 쓴다. C는 언어 자체로도 여전히 쓰이지만([C언어, 컴파일러·커널·애플리케이션을 모두 아우르는 고대 언어](/posts/c-language-omnipresence/) 참고), 그보다 더 넓게 퍼진 건 이 문법 자체다. C의 문법을 물려받은 자식들을 계보로 정리한다.

---

## 1. 가장 직접적인 자식 — C++

1979년 벨 연구소의 비야네 스트로스트룹이 C에 클래스를 얹어 만들기 시작했다. 처음 이름부터 "C with Classes"였다. C++는 지금도 C의 문법을 거의 그대로 유지하면서 객체지향과 템플릿을 더한 가장 직계에 가까운 자식이다([C++, 누구의 것인가? 그리고 C의 확장판인가?](/posts/cpp-ownership-and-c/)에서 자세히 다뤘다).

---

## 2. 전혀 다른 방향으로 간 형제 — Objective-C

C++와 거의 같은 시기, 브래드 콕스와 톰 러브는 C에 완전히 다른 것을 얹었다. 1980년대 초 이들의 회사 스텝스톤(Stepstone)은 C 위에 **스몰토크식 동적 메시징 문법**을 결합한 **Objective-C**를 내놨다. C++가 정적인 v테이블 기반 객체 모델을 택한 반면, Objective-C는 런타임에 메서드를 동적으로 찾아 호출하는 방식을 택했다. 1988년 넥스트(NeXT, 스티브 잡스의 회사)가 이 언어를 라이선스해 NEXTSTEP의 기본 언어로 삼았고, 1996년 애플이 넥스트를 인수하면서 2014년 스위프트가 나오기 전까지 애플 생태계의 주력 언어 자리를 지켰다. 같은 C에서 갈라져 나왔지만 객체 모델의 철학이 정반대인 두 자식이다.

---

## 3. 문법은 물려받고 위험한 부분은 버린 자식 — Java

제임스 고슬링은 C++의 포인터, 다중 상속, 연산자 오버로딩 같은 복잡하고 위험한 기능을 걷어내면서도, C 스타일의 문법(`{}`, `;`, `if/for/while`)은 그대로 유지했다. C++ 개발자가 최소한의 학습으로 넘어올 수 있게 하려는 의도적 설계였다. 원래 셋톱박스용 언어로 시작한 이 언어가 어떻게 "인터넷 언어"로 재포장됐는지는 [새로운 언어의 셀링 포인트는 실제 탄생 이유와 같을까?](/posts/language-selling-point-vs-origin/)에서 다뤘다.

---

## 4. 마케팅 때문에 억지로 C 옷을 입은 자식 — JavaScript

자바스크립트의 문법이 C 계열을 닮은 건 브렌던 아이크의 원래 취향이 아니었다. 아이크는 스킴(Scheme)처럼 함수가 1급 시민인 언어를 만들고 싶어 했다. 하지만 1995년 넷스케이프 경영진은 당시 한창 뜨던 자바와 비슷해 보여야 한다고 요구했고, 그 결과 열흘 만에 나온 이 언어는 **문법은 C·자바를 닮았지만 내부 설계는 함수형 언어에 가까운** 독특한 혼종이 됐다. 겉모습과 속을 만든 사람의 의도가 이렇게 어긋난 사례는 드물다.

---

## 5. Perl과 뒤섞인 자식 — PHP

1995년 라스무스 러도프가 만든 PHP는 중괄호와 세미콜론 같은 C의 블록 문법을 그대로 가져오면서, 변수 앞에 `$`를 붙이는 관행은 Perl에서 빌려왔다. 두 언어의 문법이 뒤섞인 잡종이지만, 이 조합이 결국 웹 서버 스크립팅 언어로 크게 성공했다.

---

## 6. C의 공동 창시자가 직접 낳은 자식 — Go

Go의 설계자 세 명 중 한 명은 **켄 톰슨** — C를 만든 데니스 리치와 함께 유닉스를 만든 그 사람이다. 그래서 Go는 혈통상 C의 가장 적자에 가깝다고 볼 수도 있다. 다만 실제 문법은 세미콜론을 프로그래머가 직접 쓰지 않아도 되도록 컴파일러가 자동으로 줄바꿈을 해석해 삽입하는 등, C의 번거로운 부분을 상당히 다듬었다.

---

## 7. C++를 다시 설계해보자는 실험 — D

2001년 월터 브라이트가 만든 **D 언어**는 "C++를 처음부터 다시 설계하면 어떨까"라는 질문에서 나왔다. C/C++의 문법을 계승하면서 가비지 컬렉션, 모듈 시스템, 컴파일 타임 코드 실행 같은 현대적 기능을 얹었다. 큰 상업적 성공을 거두진 못했지만, C 계열 언어가 어디까지 현대화될 수 있는지를 보여주는 실험으로 지금도 자주 인용된다.

---

## 8. 문법은 물려받았지만 정신은 다른 곳에서 온 자식 — Rust

Rust는 중괄호와 세미콜론을 쓰니 겉으로는 C 계열처럼 보인다. 하지만 패턴 매칭, 대수적 데이터 타입(enum), 표현식 중심 문법 같은 핵심 설계는 C보다 OCaml 같은 ML 계열 함수형 언어에서 더 많이 가져왔다([Rust, C의 대안인가?](/posts/rust-c-alternative/) 참고). 겉모습만 C 가문의 자식이고, 실제로는 이웃집에서 자란 셈이다.

---

## 9. C 가문을 떠나기로 한 자식 — Swift

Objective-C의 후계자로 나온 스위프트는 오히려 C의 문법적 흔적을 지우는 방향으로 갔다. 세미콜론이 선택 사항이 됐고, `func`·`var`·`let` 같은 키워드는 C 계열보다 스크립트 언어에 더 가깝다. C 가문에서 태어났지만 성인이 되며 독립한 자식에 가깝다.

---

## 10. 결론

C의 진짜 유산은 언어 자체보다 **문법이라는 관습**이다. 객체 모델이 정반대인 C++와 Objective-C, 위험한 기능을 걷어낸 자바, 마케팅 때문에 억지로 옷을 입은 자바스크립트, 정신은 완전히 다른 곳에서 온 러스트까지 — 이 모든 언어가 서로 전혀 다른 이유로 태어났으면서도 `{}`와 `;`라는 같은 문법 유산을 공유한다. 새 언어를 설계하는 사람들이 지금도 습관적으로 중괄호를 선택하는 이유는, C가 반세기 전에 그 문법을 "기본값"으로 만들어놨기 때문이다.
