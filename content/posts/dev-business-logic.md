좋은 코드를 짜는 것과 그 코드로 먹고사는 것은 다른 문제다. 과거에는 선택지가 단순했다. 소프트웨어를 패키지로 팔거나, 기업에 납품하거나, 후원을 받거나. 지금은 수익 모델이 수십 가지로 늘었고, 개발 진입 장벽은 낮아졌고, 경쟁자는 전 세계 어디서든 나타난다. 선택지가 많아졌다는 것은 더 잘 알아야 한다는 뜻이다.

---

## 먼저, 사용자와 고객을 구분하라

개발자가 가장 자주 혼동하는 개념이다.

```
사용자 (User)   → 제품을 실제로 쓰는 사람
고객 (Customer) → 돈을 내는 사람
```

이 둘이 같은 경우도 있고, 완전히 다른 경우도 있다.

- **Google 검색**: 사용자는 검색하는 사람, 고객은 광고를 사는 기업
- **Slack**: 사용자는 직원들, 고객은 회사(비용을 결제하는 팀장·IT부서)
- **iOS 앱 (무료)**: 사용자는 앱을 쓰는 사람, 고객은 없음(광고주가 있다면 그쪽)
- **B2B SaaS**: 사용자는 실무자, 고객은 예산을 승인한 임원

이 구분이 왜 중요한가. **만족시켜야 할 대상이 달라지기 때문이다.** Slack의 사용자(직원)는 UI가 예쁘면 좋아하지만, 고객(IT부서)은 보안 감사 로그와 SSO 연동을 원한다. 어느 쪽을 먼저 만족시키느냐에 따라 제품의 방향이 달라진다.

## 수익 모델의 지도

수익 모델은 "어떤 가치를 누구에게 팔 것인가"를 정의한다.

### 일회성 판매 (Perpetual License)

```
사용자 → [돈] → 개발자
         ↑
       한 번
```

소프트웨어를 한 번 사면 영구 사용권을 얻는다. Photoshop이 CC 구독제로 전환하기 전, Sketch가 한때 쓰던 방식. 직관적이고 사용자가 좋아하지만, 개발자 입장에서 수익이 불규칙하다. 새 버전을 만들어야 추가 매출이 생기는 구조라 업그레이드를 유도해야 하는 압박이 있다.

**지금도 유효한 곳**: Affinity Photo/Designer(Adobe 대항마로 일회성 판매 전략 고수), 게임, 소규모 유틸리티 앱. "나는 구독 싫다"는 사용자층을 겨냥한 역차별화 전략이 되기도 한다.

### 구독 (Subscription / SaaS)

```
사용자 → [월 N원] → 개발자
         ↑ 매월 반복
```

현대 소프트웨어의 지배적 모델. 개발자 입장에서 예측 가능한 수입(MRR: Monthly Recurring Revenue)이 생긴다. 투자자도 좋아한다. 사용자가 쓰는 동안 계속 내기 때문에 지속적으로 가치를 제공해야 한다는 압박이 있다. 서비스를 멈추면 수익도 멈춘다.

**핵심 지표**: 이탈률(Churn Rate). 매달 5%가 해지하면 1년에 절반 이상이 사라진다. 이탈률을 낮추는 것이 신규 획득만큼 중요하다.

```
월 이탈률 5% 가정:
  월 1: 100명
  월 6: 74명
  월 12: 54명
  → 1년 만에 절반 이탈
```

### 프리미엄 (Freemium)

```
무료 사용자 → 가치 경험 → 일부가 → 유료 전환
   (대부분)                        (소수)
```

기본 기능은 무료, 고급 기능은 유료. Spotify, Notion, Slack의 기본 모델이다. 무료 사용자가 유료로 전환하는 비율(전환율)이 핵심이다. 업계 평균은 2～5%. 100명 모이면 2～5명이 돈을 낸다는 뜻이다.

**함정**: 무료 사용자도 서버 비용, 지원 비용을 발생시킨다. 전환율이 너무 낮으면 무료 사용자를 유지하는 비용이 수익을 초과한다. 무료 한도를 어디까지 줄 것인지가 핵심 설계 포인트다.

```
Notion의 선택:
  무료: 블록 수 제한 (초기엔 1000개)
  유료: 무제한 + 팀 기능
  → 개인 사용자는 오래 무료로 쓰지만, 팀이 되면 자연스럽게 전환
```

### 광고 (Advertising)

```
사용자 → [주의(Attention)] → 개발자 → [광고 지면] → 광고주
                                                        ↓ 돈
```

무료로 서비스를 제공하고, 사용자의 시간과 주의를 광고주에게 판다. Google, Meta, 대부분의 무료 앱이 이 구조다. 사용자 수가 매우 많아야 의미 있는 수익이 나온다. 개인 개발자가 선택하기에는 규모 달성이 어렵다. 또한 사용자의 이익과 광고주의 이익이 충돌하는 구조적 긴장이 있다.

### 인앱 결제 (In-App Purchase)

```
앱은 무료 → 내부에서 콘텐츠/기능/소모품 구매
```

게임에서 발전한 모델. 스킨, 캐릭터, 에너지 아이템. 확장팩. 사용자의 소수가 대부분의 매출을 만든다. 게임에서는 이 극소수를 "고래(Whale)"라고 부른다. 전체 수익의 80%를 5%의 사용자가 만들기도 한다.

**플랫폼 수수료**: App Store, Google Play는 인앱 결제에 15～30% 수수료를 가져간다. 100원 팔면 30원이 플랫폼으로 간다. 이 수수료를 피하기 위해 웹 결제로 유도하는 방법을 쓰기도 한다(다만 스토어 정책 위반 가능).

### 오픈 코어 (Open Core)

```
코어 기능: 오픈소스 (무료, 누구나)
엔터프라이즈 기능: 유료 (SSO, 감사 로그, 우선 지원 등)
```

GitLab, Elasticsearch, HashiCorp Terraform이 이 모델이다. 오픈소스로 개발자 신뢰와 도입을 얻고, 기업이 필요로 하는 기능을 유료로 판다. 개발자가 회사에 도입을 제안하면, 회사가 기업 플랜을 산다.

**긴장점**: 무료 기능을 얼마나 줄 것인가. 너무 많이 주면 기업도 유료로 안 간다. 너무 적게 주면 오픈소스의 매력이 없어진다.

### API / 플랫폼

```
개발자 → API 호출 → [사용량 × 단가] → 제공자
```

Stripe(결제), Twilio(SMS), SendGrid(이메일), OpenAI(AI)가 이 모델이다. 개발자를 고객으로 삼는다. 개발자가 서비스에 API를 연동하면, 그 서비스의 사용자가 많아질수록 API 호출도 늘어난다. 확장성이 좋다. 하지만 개발자를 설득해야 하는 특수한 마케팅 방식이 필요하다(개발자 관계, DevRel).

### 마켓플레이스 / 수수료

```
판매자 ←→ [플랫폼] ←→ 구매자
              ↓
         거래액의 N% 수수료
```

App Store, Steam, Etsy, 배달의민족이 이 구조다. 자신이 물건을 만들지 않고, 거래가 일어나는 장소 역할을 한다. 네트워크 효과가 강하다(판매자가 많을수록 구매자가 오고, 구매자가 많을수록 판매자가 온다). 초기 닭과 달걀 문제가 있다.

### 후원 / 도네이션

```
사용자 → [자발적 기여] → 개발자
```

GitHub Sponsors, Patreon, Ko-fi. 오픈소스 유지관리자나 개인 개발자가 주로 쓴다. 정기 후원과 일회성 후원. 충성도 높은 소수 사용자에게 의존한다. 예측 불가능하고 규모화가 어렵다. 하지만 개발자가 자기 페이스로 만들 수 있는 모델이기도 하다.

## 고객 획득 비용과 생애 가치

수익 모델을 정했다면, 다음은 숫자가 맞는지 확인해야 한다. 두 개의 핵심 지표다.

**CAC (Customer Acquisition Cost): 고객 한 명을 얻는 데 드는 비용**

```
CAC = 마케팅 비용 + 영업 비용
      ─────────────────────────
         신규 고객 수

예: 100만원 광고비로 10명 획득 → CAC = 10만원
```

**LTV (Customer Lifetime Value): 고객 한 명이 기간 동안 내는 총 금액**

```
LTV = 월 결제액 × 평균 이용 기간(월)

예: 월 1만원, 평균 18개월 이용 → LTV = 18만원
```

**생존 공식**:

```
LTV > CAC    → 지속 가능
LTV < CAC    → 고객을 얻을수록 손해 (서비스 종료 예정)
LTV = 3× CAC → 업계 건강한 기준
```

많은 스타트업이 여기서 망한다. 광고비를 써서 사용자를 모으는데, 그 사용자가 내는 돈보다 모으는 비용이 더 크다. 규모가 커질수록 손해가 커지는 구조다.

## 해자 (Moat) — 경쟁자가 따라올 수 없는 이유

"좋은 제품을 만들면 사람들이 알아서 온다"는 생각은 반만 맞다. 사람들이 왔어도, 경쟁자가 비슷한 것을 만들면 떠날 수 있다. **해자(Moat)**는 경쟁자가 쉽게 복사할 수 없는 우위다.

### 네트워크 효과

사용자가 많을수록 제품의 가치가 높아지는 구조.

```
카카오톡: 내 친구들이 다 카카오톡을 써야 나도 써야 한다
          → 경쟁자가 더 좋은 앱을 만들어도 갈아타기 어렵다

GitHub: 오픈소스 코드, 이슈, 기여 히스토리가 거기 있다
        → 다른 곳으로 옮기면 이 모든 것을 잃는다
```

네트워크 효과가 있는 제품은 한 번 임계점을 넘으면 매우 강한 해자가 생긴다.

### 전환 비용 (Switching Cost)

다른 서비스로 옮기는 것이 귀찮거나 비싼 구조.

```
Salesforce: 영업 데이터가 전부 여기 있음
            ERP 연동, 직원 교육, 커스텀 설정...
            → 옮기려면 수개월+수억 원

Notion: 노트, 데이터베이스가 여기에 있음
        → 내보내기가 되더라도 정리가 귀찮음
```

전환 비용이 높은 제품은 사용자가 불만족해도 쉽게 떠나지 않는다.

### 데이터 해자

사용하면 사용할수록 서비스가 사용자에게 맞춰지는 구조.

```
Spotify: 듣는 음악을 기록할수록 추천이 정확해짐
         → 10년치 청취 기록이 있는 사람은 떠나기 어려움

Gmail: 10년치 이메일 아카이브가 여기 있음
```

### 브랜드와 신뢰

특히 보안, 의료, 금융 분야에서 강하다. "우리 회사 데이터를 처음 들어본 서비스에 맡기겠는가." 브랜드는 오래 쌓이고 단기간에 복사할 수 없다.

### 규모의 경제

더 커질수록 단위 비용이 낮아지는 구조. AWS가 서버를 원가보다 싸게 팔 수 있는 이유는 규모가 압도적이기 때문이다. 개인 개발자가 이 해자를 쓰기는 어렵다.

## 개발자가 자주 놓치는 것들

### PMF — 제품-시장 적합성

**Product-Market Fit**: 만든 제품이 실제로 존재하는 시장의 수요에 맞는가.

```
PMF 없음: "이게 있으면 좋겠다고 생각해서 만들었는데 아무도 안 씀"
PMF 있음: "이거 없으면 안 되는데요?" 라는 말을 듣기 시작함
```

기술적으로 완벽한 제품이 망하는 이유의 1위는 PMF가 없어서다. 아무도 원하지 않는 문제를 해결하거나, 사람들이 이미 충분하다고 느끼는 방식으로 해결한 경우다.

**찾는 방법**: 만들기 전에 판다. 랜딩 페이지만 만들어 "이런 서비스가 있다면 얼마를 내겠습니까"를 물어본다. 사전 주문을 받는다. 코드 한 줄 없이 수요를 확인한다.

### "10배 법칙"

사람들은 현재 쓰는 것보다 **10배 좋아야** 바꾼다. 조금 더 좋은 것은 전환 비용을 극복하지 못한다.

```
전환 장벽:
  익숙함을 버리는 비용
  + 데이터 이전 비용
  + 학습 비용
  + 혹시 새 것이 별로면 다시 돌아오는 비용

이것을 넘으려면: 명백하게 훨씬 나아야 한다
```

경쟁자가 있는 시장에 진입할 때 "더 예쁘다", "더 싸다" 수준으로는 부족하다. 특정 사용자에게 특정 차원에서 압도적으로 나아야 한다.

### 가격이 메시지다

가격은 숫자가 아니라 신호다.

```
너무 싸다 → "별로겠지" (가치 신호 약화)
적당하다  → "써볼 만 하다"
비싸다    → "그만큼 가치가 있겠구나" (특정 시장에서)
```

기업용 소프트웨어에서 "무료로 드리겠습니다"는 때로 역효과가 난다. 기업 구매 담당자는 무료인 소프트웨어에 "이게 얼마나 오래 살아남을까", "지원은 누가 해주나"를 묻는다. 적정 가격을 받는 것이 신뢰의 신호가 된다.

**가격 앵커링**: 더 비싼 옵션을 먼저 보여주면 실제 가격이 싸 보인다.

```
Enterprise: ₩500,000/월
Pro:        ₩50,000/월   ← "Pro가 합리적이네"
Starter:    ₩10,000/월
```

Pro가 메인 타겟이지만, Enterprise가 있어야 Pro가 합리적으로 보인다.

### 사용자 수 ≠ 성공

100만 다운로드, 10만 DAU. 이 숫자가 빛나 보이지만, 돈이 되지 않으면 지속 불가능하다. 반대로, 1,000명의 사용자가 각각 월 100달러를 낸다면 월 10만 달러다. 규모가 수익을 결정하지 않는다. **수익화 방식이 수익을 결정한다.**

Kevin Kelly의 "1,000 True Fans" 이론: 당신의 작업에 진심인 팬 1,000명이 연간 100달러를 쓰면 연 1억 원(=100,000달러). 대중을 상대하지 않고 특정 집단을 깊게 파는 전략이 개인 개발자에게 더 현실적이다.

## 시장 선택 — 어디서 싸울 것인가

### B2C (소비자 대상)

- 판매 주기 짧음, 결정권자 = 사용자 본인
- 가격이 낮음 (월 수천 원～수만 원)
- 사용자 획득 비용이 낮아야 함 (바이럴, 광고)
- 이탈이 빠르고 쉬움
- 규모가 있어야 의미 있는 수익

### B2B (기업 대상)

- 판매 주기 김 (수개월～1년), 결정권자 ≠ 사용자
- 가격이 높음 (월 수십만～수천만 원)
- 영업, 데모, 계약이 필요
- 한 번 도입하면 쉽게 안 바꿈 (전환 비용)
- 소수 고객으로 큰 수익 가능

**개인 개발자에게는?** 초기에는 B2B small (소규모 기업, 10～100명 규모)이 현실적이다. 의사결정자가 한 명이고, 예산이 있고, 실제 업무 문제를 해결해주면 기꺼이 돈을 낸다.

### 버티컬 집중

특정 산업에 특화하는 전략. "모든 기업을 위한 CRM" 대신 "치과를 위한 CRM".

```
수평적 접근: 모든 산업 → 경쟁 치열, 마케팅 비용 큼
수직적 접근: 치과만 → 경쟁 적음, 타겟팅 정밀, 마케팅 효율적
```

수직 시장은 작아 보이지만, 그 안에서 필요한 기능, 용어, 규제를 깊이 이해하면 대형 플레이어가 진입하기 어렵다.

## AI 시대의 변화

2020년대 중반 이후 개발 진입 장벽이 크게 낮아졌다. AI 코딩 도구로 MVP(최소 기능 제품)를 만드는 데 걸리는 시간이 주 단위에서 일 단위로 줄었다.

이것이 의미하는 바:

```
진입 장벽 하락 → 경쟁자가 더 빠르게 등장
기술 우위의 반감기 단축 → 기능 복사가 더 빠름
```

이 환경에서 차별화의 무게 중심이 이동했다.

**기술에서 덜 중요해진 것**:
- "우리만 이 기능을 구현할 수 있다"는 기술적 희소성

**더 중요해진 것**:
- 특정 사용자 집단에 대한 깊은 이해
- 신뢰와 브랜드 (이 서비스가 내 데이터를 제대로 다룰 것인가)
- 커뮤니티 (사용자들이 서로 연결되는 구조)
- 실행 속도 (기능을 빨리 만드는 것보다 올바른 기능을 먼저 찾는 속도)
- 유통 (Distribution) — 좋은 제품을 만드는 것만큼 사람들에게 닿는 것이 중요

## 개발자가 스스로에게 물어봐야 할 질문들

제품을 만들기 전, 또는 만드는 중에 계속 물어봐야 하는 것들이다.

```
1. 누가 이것을 사는가?
   (사용자와 고객이 다른가?)

2. 왜 지금 쓰는 것을 버리고 이것으로 바꾸는가?
   (10배 법칙을 만족하는가?)

3. 고객 한 명을 얻는 데 얼마가 드는가? (CAC)
   그 고객이 기간 동안 얼마를 내는가? (LTV)
   LTV > 3 × CAC 인가?

4. 경쟁자가 똑같이 만들면 어떻게 되는가?
   (해자가 있는가? 있다면 무엇인가?)

5. 1,000명 모이면 얼마의 매출인가?
   그 금액으로 지속 가능한가?

6. 사용자가 "이거 없으면 안 된다"고 말하는가?
   아니면 "있으면 좋겠다" 수준인가?
```

"있으면 좋겠다(Nice to have)"와 "없으면 안 된다(Must have)"의 차이가 생존을 결정한다.

## 정리

```
수익 모델 선택지:
  일회성 판매   → 직관적, 수입 불규칙
  구독          → 예측 가능, 이탈 관리가 관건
  프리미엄      → 규모 필요, 무료한도 설계가 핵심
  광고          → 대규모 필요, 사용자↔광고주 긴장
  인앱 결제     → 고래 의존, 플랫폼 수수료 30%
  오픈코어      → 개발자 신뢰 + 기업 매출
  API/플랫폼    → 개발자가 고객, 사용량 기반

해자:
  네트워크 효과 > 전환 비용 > 데이터 > 브랜드 > 규모

핵심 공식:
  LTV > 3 × CAC
  PMF 먼저, 규모화 나중
  "없으면 안 된다"가 되어야 한다

AI 시대의 변화:
  기술 우위 → 단기화
  유통 + 신뢰 + 커뮤니티 → 장기 해자로 부상
```

기술을 잘 만드는 것은 필요조건이다. 하지만 그것만으로는 충분하지 않다. 누가 살 것인지, 왜 살 것인지, 경쟁자가 왜 못 따라오는지를 아는 것이 제품의 수명을 결정한다.
